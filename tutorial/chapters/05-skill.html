第 05 章 · 按需加载能力

让 Agent 按需加载工具子集: Skill

第 04 章让 agent 学会分身, 但所有 agent 看到的都是同一个 tool registry。 工具数到 30+ 之后, system prompt 里的工具描述就撑爆了, 模型也会被淹没 在不相关的工具里。这一章给 harness 加 "Skill" 机制: 按场景动态加载工具 子集, 同时 Skill 本身也是一种 "按需注入能力" 的元机制, 后续章节会复用。

在第 04 章基础上改了什么

这一章在 tool registry 之上加一层 "SkillLoader"。当 LLM 在 tool_call 里调用 load_skill 时, harness 把指定 skill 的工具集"激活" 进当前 loop 的工具列表 (而不是永久改 registry)。loop 结束后激活状态 丢弃, 不会污染父 agent 或后续 loop。 对应到代码, 改动集中在 4 个文件: src/skills.ts (新)、 src/tools/load_skill.ts (新)、src/agent.ts (改)、 src/index.ts (改接线, 注入 skills 配置)。

1. src/skills.ts: Skill 加载器 (新) 2. src/agent.ts: 维护当前激活的 skill 集合 3. src/tools/registry.ts: 暴露 getAll() 给 skill loader 4. src/system-prompt.ts: 工具描述前缀 (第 10 章 cache-friendly 的伏笔)

作者怎么想的: 这一章的思考链

想清楚现象
工具数到 30+ 之后, system prompt 里堆 30 段工具描述, 模型在第 5 轮 已经记不清 "我有哪些工具可用"。更糟的是, 任务不相关 (例如 "改 React 组件" 不需要 "deploy k8s" 工具) 的工具描述会误导模型 尝试用错工具。
想反例
最朴素的反例是"把工具描述塞进 system prompt, 让 LLM 自己挑"。 这有两个问题: 一是 system prompt 越长, token 成本越高 (无关任务 也要付账), 二是 LLM 在长 context 下选错工具的概率显著上升。
想接口和不变量
接口: interface SkillLoader { load(name): Tool[], list(): SkillMeta[] }。 不变量三条: (1) skill 激活状态是 loop 内的临时态, loop 结束丢弃, (2) 同一个 loop 内可激活多个 skill, 工具集取并集, (3) 工具描述 在 LLM 视角是稳定前缀 (skill 集合是稳定集合, 不是动态拼字符串)。
想怎么验证
fake LLM 第一轮调 load_skill("react"), 跑完后第二轮 LLM 收到的 system prompt 含 react skill 的工具描述; 第三轮 LLM 不再调 load_skill, 但 react 工具仍然可用 (loop 内激活状态保留)。

先观察: 两段故意有气味的实现

观察 1 · skill 状态写到 module-level 单例

// 教学简化版, 仓库真实实现见 GitHub 永久链接
let activeSkills: string[] = [];
export function loadSkill(name: string) { activeSkills.push(name); }

问:为什么不直接 module-level 存, 而要放到 loop 闭包?

答:module-level 会让父 agent 加载的 skill 出现在子智能体 工具列表里 (第 04 章的子智能体共享 tool registry, 但 skill 状态也 共享, 这就破坏了"按需隔离")。也违反第 04 章工厂模式的本意。

观察 2 · 工具描述按需拼接字符串

// 教学简化版
let systemPrompt = basePrompt + skillDescriptions.join("");

问:为什么不直接拼字符串?

答:字符串拼接会让 system prompt 每次调 LLM 都不同, 整个 prompt cache 失效, token 成本涨 3-5 倍。正确做法: skill 集合稳定 (例如 "react" 一旦加载就一直存在), 工具描述作为稳定 前缀, 不再因 LLM 调用而变化。这是第 10 章 cache-friendly 布局 的关键伏笔。

接口形状: 在写实现前钉死

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export interface SkillMeta {
  name: string;
  description: string;       // 给 LLM 看: 什么场景用
  tools: string[];            // 该 skill 包含的工具名
}

export interface SkillLoader {
  get(name: string): SkillMeta | undefined;
  list(): SkillMeta[];
  // 返回该 skill 包含的工具实例
  resolve(name: string): Tool[];
}

export interface ActiveSkillSet {
  // 激活 skill, 返回新增的工具 (供 agent 合并到当前工具列表)
  activate(name: string): Tool[];
  // 当前激活的 skill 名字列表
  names(): string[];
  // 当前激活的所有工具 (去重)
  tools(): Tool[];
}

loop 接入: skill 激活是 loop 内的临时态

关键设计: ActiveSkillSet 是 agent 实例闭包内的状态, 不是 module-level。loop 跑完一次 run() 之后, 状态保留 (LLM 在同一 run 内 多次 load_skill 是合理的); 但 agent 实例被销毁时, 状态自然消失。 子智能体拿不到父 agent 的 skill 状态 (第 04 章隔离)。 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export function createAgent(deps): Agent {
  const activeSkills = createActiveSkillSet();

  return {
    async run(query) {
      history.add({ role: "user", content: query });
      for (;;) {
        // 当前可用工具 = registry 基础 + 当前激活的 skill 工具
        const availableTools = [...registry.list(), ...activeSkills.tools()];
        const messages = history.getMessages();
        const assistant = await llm.chat({ messages, tools: availableTools });
        history.add(assistant);

        if (!assistant.tool_calls) return assistant.content;

        for (const call of assistant.tool_calls) {
          // load_skill 特殊处理: 不真执行, 而是激活 skill
          if (call.name === "load_skill") {
            const skillName = call.args["name"] as string;
            activeSkills.activate(skillName);
            history.add({ role: "tool", tool_call_id: call.id,
              content: `Loaded skill "${skillName}". Tools: ${
                skillLoader.get(skillName)?.tools.join(", ") ?? "(unknown)"
              }` });
            continue;
          }
          // 其它工具正常执行
          const tool = registry.get(call.name);
          /* ... */
        }
      }
    },
  };
}

注意: load_skill 不算 "真" 的 tool call, 它激活 skill 后 写一条 tool message 告诉 LLM "已加载 X, 可用工具: Y, Z"。下一轮 LLM 调用时, 这些工具就在 availableTools 里, LLM 看到 schema 后 可以直接调用。

对 prompt cache 的影响: 稳定 vs 动态

本章是第 10 章 cache-friendly 布局的关键伏笔。原则是:

  1. 稳定前缀: system prompt + 工具描述 (按 skill 集合)。 一旦 skill 集合在一次 run 内确定, 这部分就不变。
  2. 动态状态: history messages + reminder。 每轮都变, 但走普通 messages, 不进 system prompt。

错误做法是把 skill 工具描述拼到 system prompt 字符串里 (观察 2 的反例), 那会让 system prompt 每次都变, cache 失效。

反例梯度

新手错法 · A

常见错误:skill 激活状态写到 module-level 单例。

为什么错:第 04 章父子隔离被破坏, 子智能体看到父 agent 加载的 skill。

正确做法:activeSkills 在 createAgent() 闭包内, 工厂模式。

中级错法 · B

常见错误:工具描述拼到 system prompt 字符串。

为什么错:破坏第 10 章 cache-friendly 布局, 每次调 LLM 都要重传所有 skill 工具描述。

正确做法:工具描述作为 LLM.chat() 的 tools 字段, system prompt 保持稳定。

高级错法 · C

常见错误:load_skill 工具调成功但没写 tool message。

为什么错:第 02 章的 tool_call_id 配对约束, 不写 tool message 会让 messages 序列断裂。

正确做法:load_skill 激活后, 必写一条 role: "tool" 消息, content 描述已加载的工具列表。

边界错法 · D

常见错误:同一个 skill 加载多次, 工具集出现重复。

为什么错:availableTools 出现重复项, LLM 看到两个相同 name 的工具, 困惑。

正确做法:activeSkills.tools() 用 Map 去重, 同名工具取先注册的版本。

如何验证 (本章 Validation 卡片)

Validation · 第 05 章

load_skill 激活工具:fake LLM 第一轮调 load_skill("react"), 跑完后第二轮 LLM 收到的 tools 字段含 react skill 的工具 (例如 "use_react_component")。

tool message 必写:load_skill 调用后, history 末尾 出现 role: "tool" 消息, content 描述已加载的工具列表。

同 skill 多次加载去重:fake LLM 连续调 load_skill("react") 两次, availableTools 不出现重复项。

loop 结束不污染父:父 agent 调 load_skill 后 spawn 子智能体, 子智能体的 availableTools 不含 react skill 工具 (第 04 章父子隔离, 本章沿用)。

未知 skill 不崩:load_skill("不存在"), 写 error tool message, 继续 loop。

回望第 00–04 章: 哪些原则在本章兑现了

  • 工厂模式再次回报:activeSkills 在 createAgent() 闭包内, 父子隔离自然成立。
  • tool call 协议沿用:load_skill 是普通工具, 复用 Tool interface, 激活后写 tool message 满足协议。
  • 稳定前缀原则:工具描述作为 LLM.chat() 的 tools 字段, 不污染 system prompt 字符串, 为第 10 章做准备。
  • 不污染 history 的稳定结构:skill 状态本身不进 history.getMessages(), 只在 loop 闭包内。

前瞻张力: 留给后续章节

skill 加载后内容太长
第 06 章 compress 会处理"skill 工具返回内容太大" 的情况, 例如 read_skill 工具一次返回整个 README, 需要 P1 即时压缩。
skill 跨会话保留
第 09 章 memory 区分"会话内 skill" (本章的) 和"用户级偏好 skill" (例如"用户偏好 React 不用 Vue"), 后者需要持久化。
工具描述前缀稳定性
第 10 章 cache-friendly 布局会展开, skill 集合一旦在 run() 内确定就稳定, 工具描述作为稳定前缀。
skill 加载权限
第 07 章 permission 会扩展, load_skill 本身可能需要权限 (例如 "deploy" skill 是受限的)。

Prompt Card (本章任务)

Prompt Card · 第 05 章

目标:实现 Skill 加载机制, agent 可按需激活工具子集, 激活状态在 loop 闭包内临时保留。

场景:用户输入 "帮我做一个 React 组件", agent 调 load_skill("react"), 第二轮 LLM 看到 react 工具 (例如 use_react_component), 之后在该 run 内持续可用。

模块: src/skills.ts (新) 暴露 createSkillLoader() 和 createActiveSkillSet(); src/tools/load_skill.ts (新) 实现工具; src/agent.ts createAgent() 维护 activeSkills 闭包; src/index.ts 接线 skillLoader。

边界 (LLM 必须遵守的 checklist):

  • activeSkills 是 createAgent() 闭包内状态, 不是 module-level
  • load_skill 必写 role: "tool" 消息, content 描述已加载工具列表
  • 工具描述不拼到 system prompt 字符串 (走 LLM.chat() 的 tools 字段)
  • 同名工具去重 (用 Map 而非 Array.includes)
  • 子智能体拿不到父 agent 的 skill 状态 (第 04 章隔离沿用)

验证 (用 fake LLM + fake loader, 逐条落到 vitest):

  • load_skill("react") 后, 第二轮 LLM 收到的 tools 字段含 react 工具
  • load_skill 调用后, history 末尾出现 role: "tool" 消息
  • 连续两次 load_skill("react"), availableTools 不重复
  • 父 agent 调 load_skill 后 spawn 子智能体, 子智能体 tools 不含 react 工具
  • load_skill("不存在"), 写 error tool message, agent 不抛

本章练习

  1. 故意把 activeSkills 提到 module-level, 跑测试, 看"父子 skill 隔离" 是否抓到。
  2. 把工具描述改成 systemPrompt + skillDesc 拼接写法, 跑测试, 看是否破坏 system prompt 稳定性 (本节没显式测, 但可以为第 10 章留伏笔)。
  3. 在 load_skill 末尾忘记写 tool message, 跑测试, 看"tool message 必写" 是否抓到。

本章小结

本章给 harness 加了 "Skill" 机制, agent 可按需激活工具子集。 activeSkills 是 createAgent() 闭包内的临时态, 父子隔离, 工具描述 走 LLM.chat() 的 tools 字段而不是 system prompt 字符串拼接, 为 第 10 章 cache-friendly 布局做铺垫。下一章 (第 06 章) 我们处理 "context 撑爆" 的问题——Normalize / Block / Compress, 让 harness 在长任务下保持稳定。

下一章伏笔

第 05 章解决了"工具太多装不下" 的问题, 但 messages 本身仍然会无限 增长 (用户多轮对话 + 工具结果累积)。第 06 章会引入三层压缩: Normalize (合并相邻同角色消息) → Block (按消息块分组) → Compress (P1 即时压缩 + P2 全量压缩), 让 harness 在长任务下保持稳定的 context 长度。
