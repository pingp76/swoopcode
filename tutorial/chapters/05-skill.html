第 05 章 · 按需加载能力

Skill: 让 Agent 按需激活工具子集

第 04 章的 SubAgent 解决"长任务上下文" 问题, 但解决不了另一类 撑爆: 工具数到 30+ 之后, system prompt 里的工具描述就撑爆了, 模型也被淹没在不相关的工具里。 这一章给 harness 加"Skill" 机制: 按场景动态加载工具子集, 同时 Skill 本身也是一种"按需注入能力" 的 元机制, 后续章节会复用。 读完后, 你能讲清"工具描述是 stable prefix" 的伏笔, 并能用 fake LLM 验证"加载 skill 后工具可用, loop 结束不污染父"。

真实失败故事: 30 个工具的混乱

写代码之前, 先看一个真实场景。 跑你的 harness 已经积累到 30 个工具 (run_bash / run_read / run_write / run_edit / run_edit_exact / run_todo_* / run_task_* / run_memory_* / run_skill / run_subagent / run_async_* / run_schedule_* / run_output_* / run_web_* ...)。

用户让 agent "修改 React 组件"。 观察:

  1. system prompt 撑到 8000 token: 30 个工具的 description + JSON Schema 占了大半, 真正"用户场景" 的 system prompt 反而被 挤到边角。
  2. 模型被淹没: 30 个工具摆在面前, 模型选了 run_web_search 去搜 React 文档, 而不是用 run_read 看本地组件代码。 因为工具描述里 run_web_search 写得很显眼 (描述里说"搜任何信息"), run_read 写得很朴素 ("读文件")。
  3. 成本涨 3 倍: 8000 token 的 system prompt 每轮都重传, 即使不相关也要付账。 跑 50 轮对话, 仅 system prompt 就 400K token。
  4. 模型选错工具的频率上升: 10 个工具时模型选对率 95%, 30 个工具时降到 78%。 模型在长列表中迷失, 工具描述写得"有趣" 的更容易被选, 但有趣 ≠ 相关。

朴素想法 1: "压缩工具描述, 每个只写 1 句?" 模型看 1 句描述, 跟没看一样, 选错的频率更高。

朴素想法 2: "按 mode 切工具集?" 比如"代码 mode" 只给 run_*, "运维 mode" 只给 schedule_*。 但 mode 切换是用户的负担, 用户不知道当前 mode。

正确做法: Skill 机制。 平时只暴露 5-10 个核心工具, LLM 看到"我需要 React 工具" 时调 load_skill("react"), harness 临时把 React 相关工具加进当前 loop 的工具列表。 loop 结束后激活状态丢弃, 不污染父 agent 或后续 loop。

Skill 是什么: 不只是工具子集

用途: Skill 是一组工具 + 一段元信息的封装。 包含 3 个要素: skill 名 / description (给 LLM 看的"何时用") / tools 数组 (包含哪些工具)。 Skill 本身也是"按需注入能力" 的元机制 — 后面 hook / memory 章节会复用。

真实场景: harness 默认只暴露 5-10 个核心工具。 用户说"帮我做一个 React 组件", LLM 调 load_skill("react"), harness 把 use_react_component / react_props 等 5 个 工具加进当前 loop 的工具列表。 LLM 看到新工具, 调它们。 loop 结束后激活状态丢弃, 不污染父 agent。

设计思想: 经典按需加载模式 — 平时只暴露 必要的"地基工具", 场景化工具按需加载。 这跟 npm 的 import 一个道理: 全局 require 所有包会撑爆 内存, 按需 import 才省。

实现细节: 

interface SkillMeta {
  name: string;             // "react" / "kubernetes" / "code-review"
  description: string;       // 给 LLM 看: 什么场景用
  tools: string[];            // 该 skill 包含的工具名
  // 后续可以加 examples / whenToLoad / 依赖关系等
}

interface SkillLoader {
  get(name: string): SkillMeta | undefined;
  list(): SkillMeta[];
  // 返回该 skill 包含的工具实例
  resolve(name: string): Tool[];
}

interface ActiveSkillSet {
  // 激活 skill, 返回新增的工具 (供 agent 合并到当前工具列表)
  activate(name: string): Tool[];
  // 当前激活的 skill 名字列表
  names(): string[];
  // 当前激活的所有工具 (去重)
  tools(): Tool[];
}

GitHub · src/skills.ts SkillLoader / ActiveSkillSet 实现 (L1)

3 个接口, 3 个职责:

  1. SkillLoader: 从配置目录 (如 ~/.skills/) 加载所有 skill, 静态数据。 启动后不变。
  2. ActiveSkillSet: 当前 loop 内激活的 skill 集合。 闭包内状态, 父子 agent 各自一份。
  3. LLM 调 load_skill(name): 把 skill 加进 ActiveSkillSet, 工具列表更新。 这是普通的 tool call, 走 第 02 章的 tool_call 协议。

3 条不变量

不变量 1 · skill 激活状态是 loop 内的临时态, loop 结束丢弃
ActiveSkillSet 存在 agent 实例的闭包里, 不放 module-level。 父 agent 加载的 skill 不出现在子智能体工具列表 (第 04 章 父子隔离的延伸)。
不变量 2 · 同一个 loop 内可激活多个 skill, 工具集取并集
LLM 调 load_skill("react") 后再调 load_skill("typescript"), 两个 skill 的工具合并 进当前工具列表。 不需要"先 deactivate 再 activate", 简单加并集。
不变量 3 · 工具描述在 LLM 视角是稳定前缀
skill 集合一旦确定, 工具描述就稳定 (skill 是稳定集合, 不是动态拼字符串)。 不会因为 LLM 调用某个工具而触发工具描述变化, 不会破坏 prompt cache (第 10 章的伏笔)。

朴素反例: 状态写到 module-level 单例

朴素实现: skill 激活状态写 module-level。 

// ❌ 反例: module-level 单例
let activeSkills: string[] = [];
export function loadSkill(name: string) { activeSkills.push(name); }

GitHub · src/skills.ts 反例: module-level 单例 (L1)

立刻坏 3 件事:

  1. 父子污染: 父 agent 加载 "react" skill, spawn 子智能体, 子智能体工具列表里出现 react 工具。 第 04 章的父子隔离被破坏。
  2. 多 agent 互相干扰: 多个 agent 实例共享 activeSkills, 一个 agent 加载 skill 另一个 agent 也看到, 互不可见性被破坏。
  3. 无法测试隔离: 跑完一个测试 activeSkills 还有上次 加载的 skill, 下一个测试看到 "ghost skill", 测试 flaky。

解决方式: ActiveSkillSet 在 createAgent() 闭包内创建, 父子天然隔离。

朴素反例 2: 工具描述按需拼字符串

朴素实现 2: skill 激活时把工具描述拼到 system prompt 字符串。 

// ❌ 反例: 拼字符串
let systemPrompt = basePrompt + skillDescriptions.join("");

这条更危险 — 破坏 prompt cache。 解释:

  1. cache 命中率归零: system prompt 每轮都变, 即使只是 TODO 状态变化, 整个 system prompt 重新算 hash, cache 不命中。
  2. 成本涨 5-10 倍: 100 轮对话, 没用 cache 时 $1.50, 拼 上去后 $7.50-$15.00。
  3. cache-debug 无法追踪: 第 10 章的 stable prefix hash 算法假设 system prompt 稳定, 一旦拼字符串, hash 永远在变。

正确做法: skill 工具描述作为独立的 tool definition 数组 传给 LLM, 不拼到 system prompt。 LLM 通过 tool_calls 字段 调工具, 工具描述在 LLM 视角是稳定的 stable prefix。

loop 接入: skill 激活是 loop 内的临时态

用途: 把 SkillLoader 接入 agent.run, 让 LLM 调 load_skill 后能立刻看到新工具。 这是"按需加载" 在 agent 里的具体应用。

真实场景: LLM 跑一段对话, 中途调 load_skill("react"), 下一轮 LLM 看到 use_react_component 工具, 调它。 loop 结束后激活状态自动丢弃, 不会影响下一个 run。

设计思想: 经典按需加载 + 状态机 组合 — skill 加载 是异步的, 加载后下一轮 chat() 才看到新 tools, 同一个 run() 的 tools 数组稳定, 不破坏 prompt cache。 这跟第 10 章的 cache 布局 思想完全一致。

实现细节: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export function createAgent(deps: { llm: LLMClient; history: History; tools: ToolRegistry; skillLoader: SkillLoader; activeSkills: ActiveSkillSet }): Agent {
  return {
    async run(query) {
      deps.history.add({ role: "user", content: query });

      for (;;) {
        // 当前可用工具 = registry 基础 + 当前激活的 skill 工具
        const availableTools = [
          ...deps.tools.list(),
          ...deps.activeSkills.tools(),
        ];
        const messages = deps.history.getMessages();
        const assistant = await deps.llm.chat({ messages, tools: availableTools });
        deps.history.add({
          role: "assistant",
          content: assistant.content,
          tool_calls: assistant.toolCalls,
        });

        if (assistant.toolCalls.length === 0) return assistant.content ?? "";

        for (const call of assistant.toolCalls) {
          // load_skill 特殊处理: 不真执行, 而是激活 skill
          if (call.name === "load_skill") {
            const skillName = call.args["name"];
            const newTools = deps.activeSkills.activate(skillName);
            history.add({ role: "tool", tool_call_id: call.id,
              content: `Loaded skill "${skillName}". Tools: ${skillName in loader ? loader.get(skillName).tools.join(", ") : "(unknown)"}` });
            continue;
          }
          // 其他工具正常执行 (第 02 章的 loop)
          const result = await deps.tools.invoke(call.name, JSON.parse(call.function.arguments));
          history.add({ role: "tool", tool_call_id: call.id, content: result.content });
        }
      }
    },
  };
}

关键点: load_skill 是特殊 tool, 它激活 skill 后写一条 role: "tool" 消息告诉 LLM "已加载 X, 可用工具: Y, Z"。 下一轮 LLM 调用时, 这些工具就在 availableTools 里, LLM 看到 schema 后可以直接调。

对 prompt cache 的影响: stable vs 动态

本章是第 10 章 cache-friendly 布局的关键伏笔。 原则是:

  1. 稳定前缀: system prompt + 工具描述 (按 skill 集合)。 一旦 skill 集合在一次 run 内确定, 这部分就不变。
  2. 动态状态: history messages + reminder。 每轮都变, 但 走普通 messages, 不进 system prompt。

❌ / ✅: 错误做法是把 skill 工具描述拼到 system prompt 字符串里 (朴素反例 2), 那会让 system prompt 每次都变, cache 失效。 正确做法: 工具描述作为 LLM.chat() 的 tools 字段, system prompt 保持稳定。

LLM 视角: LLM 看到 system prompt (稳定) + tools (稳定, 因为 skill 集合不变) + messages (动态, 含 tool_call / tool_result / reminder)。 真正变的是 messages, 但 LLM provider 通常也缓存 messages 前缀 (Anthropic 支持 4 个 breakpoint, 详见第 10 章)。

ActiveSkillSet 闭包化

ActiveSkillSet 必须在 createAgent() 闭包内, 不写 module-level: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export function createActiveSkillSet(skillLoader: SkillLoader): ActiveSkillSet {
  const activated = new Map<string, SkillMeta>();  // name → skill
  const toolIndex = new Map<string, Tool>();        // toolName → Tool (去重)

  return {
    activate(name: string): Tool[] {
      const meta = skillLoader.get(name);
      if (!meta) return [];
      activated.set(name, meta);
      const resolved = skillLoader.resolve(name);
      const newly: Tool[] = [];
      for (const tool of resolved) {
        if (!toolIndex.has(tool.name)) {
          toolIndex.set(tool.name, tool);
          newly.push(tool);
        }
      }
      return newly;   // 返回新增的工具 (供 agent 调试用)
    },
    names(): string[] { return [...activated.keys()]; },
    tools(): Tool[] { return [...toolIndex.values()]; },
  };
}

GitHub · src/skills.ts createActiveSkillSet 完整实现 (L1)

关键: toolIndex 用 Map 而不是 Array, 天然去重。 同名工具 在两个 skill 出现时, 先注册的优先 (而不是报错或覆盖)。

❌ / ✅ 写法: 

// ❌ 激活时用 Array.includes 检查去重 (O(n))
function activate(name) {
  const resolved = skillLoader.resolve(name);
  for (const t of resolved) {
    if (!tools.find(x => x.name === t.name)) tools.push(t);
  }
}

// ✅ 用 Map 去重 (O(1))
function activate(name) {
  for (const t of resolved) {
    if (!toolIndex.has(t.name)) toolIndex.set(t.name, t);
  }
}

功能上等价, 但 10 个 skill 30 个工具时, O(n²) 的 includes 跑 1000 次循环, O(1) 的 Map 只跑 30 次。 测试上看不出, 但 LLM 调 100 个 skill 加载时, 性能差异明显。

loop 边界: 不变量 1 的实际意义

"skill 激活状态是 loop 内的临时态" 翻译成代码: 

// 教学简化版
export function createAgent(deps) {
  // 闭包内 activeSkills, 父子隔离天然成立
  const activeSkills = createActiveSkillSet(deps.skillLoader);

  return {
    async run(query) {
      // ... run 内可调 load_skill, 状态保留
      // run 跑完, activeSkills 仍存在 (下次 run() 复用)
      // agent 实例被销毁时, activeSkills 自然消失
    },
  };
}

注意: 不是"每次 run() 重置 activeSkills"。 同一个 agent 实例的多次 run() 共享 activeSkills — 这是合理的, 用户 在 run 1 加载了 "react" skill, run 2 想继续用, 不需要重新加载。 状态仅在 agent 实例被销毁时清空, 而 agent 实例通常在用户退出 REPL 时销毁。

load_skill 工具的特殊设计

load_skill 是特殊 tool, 它不在 ToolRegistry 里, 而是 agent.run 内部特殊处理。 原因:

  1. load_skill 不应该被普通 tool invoke 路径处理: 普通工具是 IO 操作 (读文件、跑命令), load_skill 是"修改 agent 自身状态"。 两条路径语义不同。
  2. load_skill 必须写 tool message: 跟其他工具一样, 写 一条 role: "tool" 消息告诉 LLM "skill 已加载"。 不写会让 messages 序列断裂 (第 02 章的教训)。
  3. load_skill 的 content 描述可用工具: 不是 "OK", 而是 "已加载 X, 可用工具: Y, Z"。 让 LLM 知道下一步能调 哪些工具。

❌ / ✅: 错误做法是 load_skill 激活后不写 tool message。 

// ❌ load_skill 激活了, 但没写 tool message
if (call.name === "load_skill") {
  activeSkills.activate(call.args.name);
  continue;   // 跳到下一个 tool call, 没 history.add(tool)
}
// 下一次 chat() 时, LLM 看到一个 assistant 调了 load_skill 但
// 没收到 tool result, 模型困惑

// ✅ load_skill 激活后必写 tool message
if (call.name === "load_skill") {
  activeSkills.activate(call.args.name);
  history.add({ role: "tool", tool_call_id: call.id, content: "Loaded skill X. Tools: Y, Z" });
  continue;
}

用途: 验证 skill 加载和去重逻辑的 fake 测试。 不依赖 真实 skill loader, 跑得快, 跑得稳。

设计思想: 经典fake LLM + fake loader 双 fake 套路 — 模拟 LLM 调 load_skill, 验证 activeSkills 状态正确, 验证下一轮 chat() 的 tools 字段含新工具。

test("load_skill 后, 工具列表含 skill 工具", async () => {
  const skillLoader = createSkillLoader({
    react: {
      name: "react",
      description: "React 组件工具集",
      tools: ["use_react_component"],
    },
  });
  skillLoader.register("use_react_component", { name: "use_react_component", description: "...", parameters: {} }, async (args) => ({ content: "OK" }));

  const activeSkills = createActiveSkillSet(skillLoader);
  const fakeLLM = createFakeLLM([
    // 第一次: LLM 调 load_skill
    { content: null, toolCalls: [{ id: "s1", function: { name: "load_skill", arguments: '{"name":"react"}' } }], finishReason: "tool_calls" },
    // 第二次: 调 use_react_component
    { content: null, toolCalls: [{ id: "t1", function: { name: "use_react_component", arguments: '{}' } }], finishReason: "tool_calls" },
    // 第三次: 回答
    { content: "Done.", toolCalls: [], finishReason: "stop" },
  ]);

  const agent = createAgent({ llm: fakeLLM, history, tools: emptyRegistry, skillLoader, activeSkills });
  await agent.run("Make a React component");

  // 验证: 第二次 chat() 时, tools 字段含 use_react_component
  const secondCallTools = fakeLLM.allCalls()[1].tools;
  expect(secondCallTools.map(t => t.function.name)).toContain("use_react_component");
});

test("同 skill 多次加载去重", async () => {
  const skillLoader = createSkillLoader({ react: { name: "react", tools: ["use_react_component"] } });
  const activeSkills = createActiveSkillSet(skillLoader);

  activeSkills.activate("react");
  activeSkills.activate("react");
  activeSkills.activate("react");

  expect(activeSkills.tools().length).toBe(1);

3 个常见误解

误解 1 · "Skill 应该进 system prompt 才有保证"
错。 Skill 工具描述走 LLM.chat() 的 tools 字段, 不进 system prompt。 跟 TODO 走 reminder 一样的原理: 功能上等价, 性能上 (prompt cache) 不同。
误解 2 · "Skill 之间互斥"
错。 Skill 是并集, 不是互斥。 LLM 调多个 skill, 工具列表合并。 "react" + "typescript" 两个 skill 共存。 只有同名工具会去重, 不是整个 skill 互斥。
误解 3 · "Skill 加载是 hot reload"
错。 skill 配置在 ~/.skills/ 目录, agent 启动时 skillLoader.scan() 加载一次。 加载后修改 skill 文件, 需要重启 agent 才生效。 "hot reload" 看似灵活, 实际引入 一致性问题 (LLM 看到 A, 你改完, 下一轮看到 B, 行为漂移)。

反例梯度

新手错法 · A

常见错误: skill 激活状态写到 module-level 单例。

为什么错: 第 04 章父子隔离被破坏, 子智能体看到父 agent 加载的 skill。

正确做法: activeSkills 在 createAgent() 闭包内, 工厂模式。

中级错法 · B

常见错误: 工具描述拼到 system prompt 字符串。

为什么错: 破坏第 10 章 cache-friendly 布局, 每次调 LLM 都要重传所有 skill 工具描述。

正确做法: 工具描述作为 LLM.chat() 的 tools 字段, system prompt 保持稳定。

高级错法 · C

常见错误: load_skill 工具调成功但没写 tool message。

为什么错: 第 02 章的 tool_call_id 配对约束, 不写 tool message 会让 messages 序列断裂。

正确做法: load_skill 激活后, 必写一条 role: "tool" 消息, content 描述已加载的工具列表。

边界错法 · D

常见错误: 同一个 skill 加载多次, 工具集出现重复。

为什么错: availableTools 出现重复项, LLM 看到两个相同 name 的工具, 困惑。

正确做法: activeSkills.tools() 用 Map 去重, 同名工具取先注册的版本。

回望: 哪些原则在本章兑现了

  • 工厂模式再次回报: activeSkills 在 createAgent() 闭包内, 父子隔离自然成立。
  • tool call 协议沿用: load_skill 是普通工具, 复用 Tool interface, 激活后写 tool message 满足协议。
  • 稳定前缀原则: 工具描述作为 LLM.chat() 的 tools 字段, 不污染 system prompt 字符串, 为第 10 章做准备。
  • 不污染 history 的稳定结构: skill 状态本身不进 history.getMessages(), 只在 loop 闭包内。

前瞻张力: 留给后续章节

skill 加载后内容太长
第 06 章 compress 会处理"skill 工具返回内容太长" 的情况, 例如 read_skill 工具一次返回整个 README, 需要 P1 即时压缩。
skill 跨会话保留
第 09 章 memory 区分"会话内 skill" (本章的) 和"用户级偏好 skill" (例如"用户偏好 React 不用 Vue"), 后者需要持久化。
工具描述前缀稳定性
第 10 章 cache-friendly 布局会展开, skill 集合一旦在 run() 内确定就稳定, 工具描述作为稳定前缀。
skill 加载权限
第 07 章 permission 会扩展, load_skill 本身可能需要权限 (例如 "deploy" skill 是受限的)。

Prompt Card (本章任务)

Prompt Card · 第 05 章

目标: 实现 Skill 加载机制, agent 可按需激活工具子集, 激活状态在 loop 闭包内临时保留。

场景: 用户输入 "帮我做一个 React 组件", agent 调 load_skill("react"), 第二轮 LLM 看到 react 工具 (例如 use_react_component), 之后在该 run 内持续可用。

模块: src/skills.ts (新) 暴露 createSkillLoader() 和 createActiveSkillSet(); src/tools/load_skill.ts (新) 实现工具; src/agent.ts createAgent() 维护 activeSkills 闭包; src/index.ts 接线 skillLoader。

边界 (LLM 必须遵守的 checklist):

  • activeSkills 是 createAgent() 闭包内状态, 不是 module-level
  • load_skill 必写 role: "tool" 消息, content 描述已加载工具列表
  • 工具描述不拼到 system prompt 字符串 (走 LLM.chat() 的 tools 字段)
  • 同名工具去重 (用 Map 而非 Array.includes)
  • 子智能体拿不到父 agent 的 skill 状态 (第 04 章隔离沿用)

验证 (用 fake LLM + fake loader, 逐条落到 vitest):

  • load_skill("react") 后, 第二轮 LLM 收到的 tools 字段含 react 工具
  • load_skill 调用后, history 末尾出现 role: "tool" 消息
  • 连续两次 load_skill("react"), availableTools 不重复
  • 父 agent 调 load_skill 后 spawn 子智能体, 子智能体 tools 不含 react 工具
  • load_skill("不存在"), 写 error tool message, agent 不抛

本章练习

  1. 故意把 activeSkills 提到 module-level, 跑测试, 看"父子 skill 隔离" 是否抓到。
  2. 把工具描述改成 systemPrompt + skillDesc 拼接写法, 跑测试, 看是否破坏 system prompt 稳定性 (本节没显式测, 但可以为第 10 章留伏笔)。
  3. 在 load_skill 末尾忘记写 tool message, 跑测试, 看"tool message 必写" 是否抓到。

本章小结

这一章给 harness 加了 "Skill" 机制, agent 可按需激活工具子集。 activeSkills 是 createAgent() 闭包内的临时态, 父子隔离, 工具描述 走 LLM.chat() 的 tools 字段而不是 system prompt 字符串拼接, 为 第 10 章 cache-friendly 布局做铺垫。 下一章 (第 06 章) 我们处理 "context 撑爆" 的问题——Normalize / Block / Compress, 让 harness 在长任务下保持稳定。
