第 05 章 · 按需加载能力

Skill: 让 Agent 按需激活工具子集

第 04 章的 SubAgent 解决"长任务上下文" 问题, 但解决不了另一类 撑爆: 工具数到 30+ 之后, system prompt 里的工具描述就撑爆了, 模型也被淹没在不相关的工具里。 这一章给 harness 加"Skill" 机制: 按场景动态加载工具子集, 同时 Skill 本身也是一种"按需注入能力" 的 元机制, 后续章节会复用。 读完后, 你能讲清"工具描述是 stable prefix" 的伏笔, 并能用 fake LLM 验证"加载 skill 后工具可用, loop 结束不污染父"。

先看一个具体的 skill 长什么样

在展开"为什么需要 skill" 之前, 先看一个真实可运行 的 skill 文件长什么样。 这样你脑子里先有一个"东西", 后面的 "为什么要这样做" 才好理解。

Skill 文件本质是一个 Markdown + frontmatter 的 文本文件, 存在项目根目录的 skills/ 目录下 (或者 用户的 ~/.skills/ 目录)。 文件名 = skill 名字。 

// 文件路径: skills/code-review/SKILL.md
// (或者 ~/.skills/code-review/SKILL.md)
---
name: code-review
description: |
  Code review 工具集。 当用户要求 "review this PR" / "看看这段代码
  有什么问题" / "帮我审一下这个文件" 时加载。 加载后 LLM 可以:
  1. 调 run_read_diff 看 PR 的 diff
  2. 调 run_lint_static 对单个文件跑静态检查
  3. 调 run_suggest_improvement 给改进建议
tools:
  - run_read_diff
  - run_lint_static
  - run_suggest_improvement
---

# Code Review Skill

加载这个 skill 后, 你获得了 3 个新工具:

- `run_read_diff(branch: string)`: 读指定 branch 的 diff
- `run_lint_static(file: string)`: 对单个文件跑静态检查
- `run_suggest_improvement(file: string)`: 给出改进建议

## 何时用

- 用户说 "review this PR" / "审一下这个 PR"
- 用户说 "这段代码有什么问题" / "帮我审一下"
- 用户说 "lint 一下" / "跑静态检查"

## 何时不用

- 用户只是读文件但没要求 review (用 run_read 即可)
- 用户想跑测试 (用 run_bash)

关键观察:

  1. 3 段组成: frontmatter 元数据 + 工具列表 + Markdown 说明。 工具列表是声明, Markdown 是给 LLM 读的自然语言。
  2. 文件即 skill: 一个文件就是一个 skill, 改文件 = 改 skill。 git 友好, 团队可以 PR review 改动。
  3. 声明式 vs 命令式: 工具列表是"声明" skill 含哪些工具, 不是"注册"。
  4. harness 启动时: 读所有 SKILL.md, 建立映射。
  5. LLM 调用时: 调 load_skill("code-review"), 按名字查找, 不需要 import / require。

外部参考: Claude Code Agent Skills 官方文档 — Anthropic 的 skill 概念和我们这里实现的是一致的: 一个目录 + SKILL.md frontmatter 描述, harness 启动时扫描, LLM 按需激活。 教学版的实现简化了一些边界 (如跨平台路径、版本号), 但核心思想 相同。 如果你想看生产级 skill 长什么样, 点上面这个链接。

真实失败故事: 30 个工具的混乱

了解了 skill 长什么样之后, 再看"为什么需要 skill"。

写代码之前, 先看一个真实场景。 假设你的 harness 已经积累到 30 个工具:

... (共 30 个)

30 个工具摆在 LLM 面前, 跑用户场景 "修改 React 组件"。 观察:

  1. system prompt 撑到 8000 token: 30 个工具的 description + JSON Schema 占了大半, 真正"用户场景" 的 system prompt 反而被 挤到边角。
  2. 模型被淹没: 30 个工具摆在面前, 模型选了 run_web_search 去搜 React 文档, 而不是用 run_read 看本地组件代码。 因为工具描述里 run_web_search 写得很显眼 (描述里说"搜任何信息"), run_read 写得很朴素 ("读文件")。
  3. 成本涨 3 倍: 8000 token 的 system prompt 每轮都重传, 即使不相关也要付账。 跑 50 轮对话, 仅 system prompt 就 400K token。
  4. 模型选错工具的频率上升: 10 个工具时模型选对率 95%, 30 个工具时降到 78%。 模型在长列表中迷失, 工具描述写得"有趣" 的更容易被选, 但有趣 ≠ 相关。

这就是 skill 要解决的问题。 思路很简单: 平时只暴露 5-10 个核心工具, 场景化工具按需加载。 用上面的例子 来说: 

// 默认情况 (skill 加载前): LLM 看到 6 个工具
- run_bash
- run_read
- run_write
- run_edit
- run_skill        // ← 关键: 入口工具, 让 LLM 主动加载更多
- run_subagent

// 用户说"修改 React 组件" 后, LLM 调 load_skill("react")
// 之后 LLM 看到 6 + 3 = 9 个工具:
- run_bash, run_read, run_write, run_edit, run_skill, run_subagent
- use_react_component   // ← react skill 提供的
- react_props           // ← react skill 提供的
- react_state           // ← react skill 提供的

// 用户说"apply 一下 k8s 配置" 后, LLM 调 load_skill("kubernetes")
// 之后 LLM 看到 6 + 3 = 9 个工具 (覆盖之前的 3 个):
- run_bash, run_read, run_write, run_edit, run_skill, run_subagent
- run_kubernetes_apply  // ← k8s skill 提供的
- run_kubernetes_get
- run_kubectl_logs

// 这样无论 harness 累计 30 个还是 100 个工具, LLM 一次只看到
// 5-10 个最相关的, 不被淹没

朴素想法 1: "压缩工具描述, 每个只写 1 句?" 模型看 1 句描述, 跟没看一样, 选错的频率更高。

朴素想法 2: "按 mode 切工具集?" 比如"代码 mode" 只给 run_*, "运维 mode" 只给 schedule_*。 但 mode 切换是用户的负担, 用户不知道当前 mode。

正确做法: Skill 机制。 平时只暴露 5-10 个核心 工具, LLM 看到"我需要 React 工具" 时调 load_skill("react"), harness 临时把 React 相关工具 加进当前 loop 的工具列表。 loop 结束后激活状态丢弃, 不污染 父 agent 或后续 loop。

Skill 是什么: 不只是工具子集

用途: Skill 是一组工具 + 一段元信息的封装。 包含 3 个要素: skill 名 / description (给 LLM 看的"何时用") / tools 数组 (包含哪些工具)。 Skill 本身也是"按需注入能力" 的 元机制 — 后面 hook / memory 章节会复用。

真实场景: harness 默认只暴露 5-10 个核心工具。 用户说"帮我做一个 React 组件", LLM 调 load_skill("react"), harness 把 use_react_component / react_props 等 3 个 工具加进当前 loop 的工具列表。 LLM 看到新工具, 调它们。 loop 结束后激活状态丢弃, 不污染父 agent。

设计思想: 经典按需加载模式 — 平时只暴露 必要的"地基工具", 场景化工具按需加载。 这跟 npm 的 import 一个道理: 全局 require 所有包会撑爆 内存, 按需 import 才省。

实现细节: 

// 简化版接口定义, 真实实现见 src/skills.ts
interface SkillMeta {
  name: string;             // "react" / "kubernetes" / "code-review"
  description: string;       // 给 LLM 看: 什么场景用
  tools: string[];            // 该 skill 包含的工具名, 例如 ["use_react_component", "react_props"]
  // 后续可以加 examples / whenToLoad / 依赖关系等
}

interface SkillLoader {
  // 启动时从配置目录 (如 ~/.skills/) 加载所有 skill
  scan(): SkillMeta[];
  // 按名字查一个 skill
  get(name: string): SkillMeta | undefined;
  // 把 skill 名字转成实际的 Tool 实例
  resolve(name: string): Tool[];
}

interface ActiveSkillSet {
  // 激活一个 skill, 把它的工具合并进当前集合
  activate(name: string): Tool[];
  // 当前激活了哪些 skill
  names(): string[];
  // 当前激活的所有工具 (去重后)
  tools(): Tool[];
}

GitHub · src/skills.ts SkillMeta 接口 (L41)

3 个接口, 3 个职责:

  1. SkillLoader: 从配置目录 (如 ~/.skills/) 加载所有 skill, 静态数据。 启动后不变。
  2. ActiveSkillSet: 当前 loop 内激活的 skill 集合。 闭包内状态, 父子 agent 各自一份。
  3. LLM 调 load_skill(name): 把 skill 加进 ActiveSkillSet, 工具列表更新。 这是普通的 tool call, 走 第 02 章的 tool_call 协议。

3 条不变量

不变量 1 · skill 激活状态是 agent 闭包内的临时态, agent 销毁时清空
ActiveSkillSet 存在 createAgent() 的闭包里, 不放 module-level (下面"反例" 一节会讲什么叫 module-level)。 父 agent 加载的 skill 不出现在子智能体工具列表 (第 04 章父子 隔离的延伸)。
不变量 2 · 同一个 agent 内可激活多个 skill, 工具集取并集
LLM 调 load_skill("react") 后再调 load_skill("typescript"), 两个 skill 的工具合并 进当前工具列表。 不需要"先 deactivate 再 activate", 简单加并集。
不变量 3 · 工具描述在 LLM 视角是稳定前缀
skill 集合一旦确定, 工具描述就稳定 (skill 是稳定集合, 不是动态拼字符串)。 不会因为 LLM 调用某个工具而触发工具描述变化, 不会破坏 prompt cache (第 10 章的伏笔)。 "对 prompt cache 的影响" 一节会展开讲。

朴素反例: 状态写到 module-level 单例

朴素实现: skill 激活状态写 module-level。

先解释什么叫 module-level。 Module-level 指的是 写在 .ts 文件最外层 (不在任何函数内) 的变量, 也叫 "模块顶层变量"。 比如: 

// === module-level 变量 ===
// 整个文件被 import 一次, 这两个变量就被创建一次。
// 整个 Node.js 进程里所有引用这个模块的代码, 共享这两个变量。
let counter = 0;                              // ← module-level
const CONFIG = { maxRounds: 10 };              // ← module-level
let activeSkills: string[] = [];              // ← module-level (反例!)

export function loadSkill(name: string) {
  activeSkills.push(name);                    // 改的是 module-level 变量
}

// === 反之: 函数内 (闭包内) 变量 ===
// 每次调用 createAgent() 都会创建一份新的, 多个 agent 不共享。
export function createAgent() {
  let activeSkills: string[] = [];            // ← 闭包内变量, 每次 create 新一份
  return {
    loadSkill(name: string) {
      activeSkills.push(name);                // 改的是这次 create 的那份
    },
  };
}

关键区别: module-level 变量 = 全进程共享 1 份, 闭包内变量 = 每次 create 新 1 份。

现在看反例, 把 activeSkills 写 module-level: 

// ❌ 反例: module-level 单例
let activeSkills: string[] = [];
export function loadSkill(name: string) { activeSkills.push(name); }

GitHub · src/skills.ts SkillMeta 接口定义 (L41)

立刻坏 3 件事:

  1. 父子污染: 父 agent 加载 "react" skill, spawn 子智能体, 子智能体工具列表里出现 react 工具。 第 04 章的父子隔离被破坏。
  2. 多 agent 互相干扰: 多个 agent 实例共享 activeSkills, 一个 agent 加载 skill 另一个 agent 也看到, 互不可见性被破坏。
  3. 无法测试隔离: 跑完一个测试 activeSkills 还有上次 加载的 skill, 下一个测试看到 "ghost skill", 测试 flaky。
图 1 · module-level 单例 vs 闭包内状态
❌ module-level 单例
整个 Node 进程共享 1 份 activeSkills。 父 agent 加载 "react" → 子 agent 也看到。 两个 agent 之间互相污染。
✅ 闭包内状态
createAgent() 每次返回新闭包, 各自 1 份 activeSkills。 父 agent 加载 "react" → 子 agent 看不到, 隔离天然成立。

解决方式: ActiveSkillSet 在 createAgent() 闭包内创建, 父子天然隔离。

朴素反例 2: 工具描述按需拼字符串

朴素实现 2: skill 激活时把工具描述拼到 system prompt 字符串。 

// ❌ 反例: 拼字符串
let systemPrompt = basePrompt + skillDescriptions.join("");

这条更危险 — 破坏 prompt cache。 解释:

  1. cache 命中率归零: system prompt 每轮都变, 即使只是 TODO 状态变化, 整个 system prompt 重新算 hash, cache 不命中。
  2. 成本涨 5-10 倍: 100 轮对话, 没用 cache 时 $1.50, 拼 上去后 $7.50-$15.00。
  3. cache-debug 无法追踪: 第 10 章的 stable prefix hash 算法假设 system prompt 稳定, 一旦拼字符串, hash 永远在变。

正确做法: skill 工具描述作为独立的 tool definition 数组 传给 LLM, 不拼到 system prompt。 LLM 通过 tool_calls 字段 调工具, 工具描述在 LLM 视角是稳定的 stable prefix。

loop 接入: skill 激活是 loop 内的临时态

用途: 把 SkillLoader 接入 agent.run, 让 LLM 调 load_skill 后能立刻看到新工具。 这是"按需加载" 在 agent 里的具体应用。

真实场景: LLM 跑一段对话, 中途调 load_skill("react"), 下一轮 LLM 看到 use_react_component 工具, 调它。 loop 结束后激活状态自动丢弃, 不会影响下一个 run。

设计思想: 经典按需加载 + 状态机 组合 — skill 加载 是异步的, 加载后下一轮 chat() 才看到新 tools, 同一个 run() 的 tools 数组稳定, 不破坏 prompt cache。 这跟第 10 章的 cache 布局 思想完全一致。

实现细节: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export function createAgent(deps: { llm: LLMClient; history: History; tools: ToolRegistry; skillLoader: SkillLoader; activeSkills: ActiveSkillSet }): Agent {
  return {
    async run(query) {
      deps.history.add({ role: "user", content: query });

      for (;;) {
        // 当前可用工具 = registry 基础 + 当前激活的 skill 工具
        const availableTools = [
          ...deps.tools.list(),
          ...deps.activeSkills.tools(),
        ];
        const messages = deps.history.getMessages();
        const assistant = await deps.llm.chat({ messages, tools: availableTools });
        deps.history.add({
          role: "assistant",
          content: assistant.content,
          tool_calls: assistant.toolCalls,
        });

        if (assistant.toolCalls.length === 0) return assistant.content ?? "";

        for (const call of assistant.toolCalls) {
          // load_skill 特殊处理: 不真执行, 而是激活 skill
          if (call.name === "load_skill") {
            const skillName = call.args["name"];
            const newTools = deps.activeSkills.activate(skillName);
            history.add({ role: "tool", tool_call_id: call.id,
              content: `Loaded skill "${skillName}". Tools: ${skillName in loader ? loader.get(skillName).tools.join(", ") : "(unknown)"}` });
            continue;
          }
          // 其他工具正常执行 (第 02 章的 loop)
          const result = await deps.tools.invoke(call.name, JSON.parse(call.function.arguments));
          history.add({ role: "tool", tool_call_id: call.id, content: result.content });
        }
      }
    },
  };
}

GitHub · src/agent.ts loop 接入 Skill (L1)

关键点: load_skill 是特殊 tool, 它激活 skill 后写一条 role: "tool" 消息告诉 LLM "已加载 X, 可用工具: Y, Z"。 下一轮 LLM 调用时, 这些工具就在 availableTools 里, LLM 看到 schema 后可以直接调。

对 prompt cache 的影响: 加 / 减工具时, 缓存怎么变

这一节直接看"加 / 减一个工具" 时, LLM 看到的 prompt 结构 怎么变。 变 = cache miss, 不变 = cache 命中。 一次 cache miss = 整段 prompt 重新计费, 很贵。

前提: LLM 收到的请求大致分两部分 — system + tools (放在请求顶部, 每次一样) 和 messages (对话历史, 每轮累加)。 顶部这两块就是 LLM 厂的 prompt cache 命中的对象。 所以我们只 看"加 / 减工具" 时, 这两块有没有变。

详细 cache 调试工具 (systemPromptHash / toolsHash / stable prefix 观察) 在第 10 章展开。 本章只建立"工具描述不进 system prompt" 的肌肉记忆。

先记住两件事:

  1. 稳定前缀: system prompt + 工具描述 (按 skill 集合)。 一旦 skill 集合在一次 run 内确定, 这部分就不变。
  2. 动态状态: history messages + reminder。 每轮都变, 但 走普通 messages, 不进 system prompt。

具体例子: 跑 1 个 50 轮对话的 React 组件任务

// 第 30 轮 (LLM 调 load_skill("typescript")): systemPrompt = "basePrompt + react 段 + typescript 段"

// - 第 30 轮: cache miss
// - 第 31-50 轮: 命中 (加了 typescript 后 20 轮都相同)

// 50 轮中 cache miss 3 次, 其余 47 次命中。 还行, 但不是最优。

// 50 轮对话, cache 命中情况:

为什么方案 B 更好

  2. 方案 A 的字符串拼接是隐式的: 开发者写 systemPrompt += skillDesc 时不知道这会破坏 cache, 第 10 章加 cache debug 时才"哦原来 30 个工具时 cache 全 miss 是这个原因"。 方案 B 把工具描述和 system prompt 在代码层面 物理隔离, 拼字符串 = 编译期就明显的反模式。
  3. 方案 B 的 tools 是 LLM 视角的稳定集合: 同一 agent 同一 run 内, "激活了哪些 skill" 是不变的, 工具描述在 LLM 视角 是"固定长度的工具列表", 不是"动态拼的字符串"。 LLM 推理 时看工具描述的 token 块也稳定, cache 命中率高。

ActiveSkillSet 闭包化

ActiveSkillSet 必须在 createAgent() 闭包内, 不写 module-level: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export function createActiveSkillSet(skillLoader: SkillLoader): ActiveSkillSet {
  const activated = new Map<string, SkillMeta>();  // name → skill
  const toolIndex = new Map<string, Tool>();        // toolName → Tool (去重)

  return {
    activate(name: string): Tool[] {
      const meta = skillLoader.get(name);
      if (!meta) return [];
      activated.set(name, meta);
      const resolved = skillLoader.resolve(name);
      const newly: Tool[] = [];
      for (const tool of resolved) {
        if (!toolIndex.has(tool.name)) {
          toolIndex.set(tool.name, tool);
          newly.push(tool);
        }
      }
      return newly;   // 返回新增的工具 (供 agent 调试用)
    },
    names(): string[] { return [...activated.keys()]; },
    tools(): Tool[] { return [...toolIndex.values()]; },
  };
}

GitHub · src/skills.ts createSkillManager 完整实现 (L262)

关键: toolIndex 用 Map 而不是 Array, 天然去重。 同名工具 在两个 skill 出现时, 先注册的优先 (而不是报错或覆盖)。

❌ / ✅ 写法: 

// ❌ 激活时用 Array.includes 检查去重 (O(n))
function activate(name) {
  const resolved = skillLoader.resolve(name);
  for (const t of resolved) {
    if (!tools.find(x => x.name === t.name)) tools.push(t);
  }
}

// ✅ 用 Map 去重 (O(1))
function activate(name) {
  for (const t of resolved) {
    if (!toolIndex.has(t.name)) toolIndex.set(t.name, t);
  }
}

功能上等价, 但 10 个 skill 30 个工具时, O(n²) 的 includes 跑 1000 次循环, O(1) 的 Map 只跑 30 次。 测试上看不出, 但 LLM 调 100 个 skill 加载时, 性能差异明显。

skill 激活状态: 存在哪, 谁改, 啥时候清

上一节说 activeSkills 在 createAgent() 闭包内。 但闭包 太抽象, 我们用 3 个具体问题讲清"skill 激活状态" 到底是个什么 东西:

  1. 存在哪? 一个 Map, key 是 skill 名字, value 是该 skill 的工具列表。 物理位置: createAgent() 闭包内的内存。
  2. 谁改它? 3 个地方: load_skill tool 调用时加, deactivate_skill tool 调用时减, REPL 关闭时整个 清空。
  3. 啥时候清空? 只有 1 个时机清空: REPL 关闭 / agent 实例销毁。 其他时机 (run 结束 / turn 结束 / 任务完成) 都不清。

这 3 条回答了"skill 激活状态" 是什么。 下面用具体场景验证:

图 3 · skill 激活状态的具体行为
场景 A · 同一个 run 内的多轮对话
轮 1: LLM 调 load_skill("react") → activeSkills = {react}
轮 2: LLM 看到 react 工具, 调 use_react_component → activeSkills 仍然 {react}
轮 3: LLM 调 react_props → activeSkills 仍然 {react}

→ 多轮共享同一份 activeSkills, 因为都在同一个 run 内。
场景 B · 同一个 agent 实例的多个 run
run 1: LLM 调 load_skill("react") → activeSkills = {react}
run 1 结束: 不清空, activeSkills 仍然是 {react}
run 2: 用户输 "再帮我做个 Vue 组件", LLM 调 load_skill("vue")
    → activeSkills = {react, vue}

→ 跨 run 共享, 累积式。 这是反直觉点: 不清空。
场景 C · REPL 关闭 / 实例销毁
关闭 REPL → agent 实例销毁 → 闭包释放 → activeSkills 被
    GC 回收, 数据消失。
下次开 REPL → 新建 agent 实例 → 新空 Map。

→ 这才是"清空" 唯一发生的时机。

为什么不在 run 结束清空?

因为用户在 run 1 加载了 "react" skill, run 2 很可能继续做 React 相关的事。 如果 run 1 结束就清, run 2 一开始 LLM 又看不到 react 工具, 又得调一次 load_skill, 多一轮对话。 体验差。 所以 run 结束不清, 跨 run 累积, 直到 REPL 关闭才整体清空。

和"父子隔离" 怎么配合?

父 agent 加载的 skill, 子 agent 看不到。 原因是 父子各是独立的 createAgent() 调用, 各自闭包内 activeSkills。 父 agent 的 activeSkills 在父闭包, 子 agent 闭包内是新建的空 Map。 所以不会污染。

图 4 · 父子 agent 的 skill 状态完全独立
父 agent 闭包内 activeSkills
{react, typescript} (父 LLM 自己加载的)
父销毁前一直保留。
子 agent 闭包内 activeSkills (run_subagent 时创建)
{} (新建, 空 Map)
子 agent 看不到父的 skill, 父子隔离天然成立。

load_skill 工具的特殊设计

load_skill 是特殊 tool, 它不在 ToolRegistry 里, 而是 agent.run 内部特殊处理。 原因:

  1. load_skill 不应该被普通 tool invoke 路径处理: 普通工具是 IO 操作 (读文件、跑命令), load_skill 是"修改 agent 自身状态"。 两条路径语义不同。
  2. load_skill 必须写 tool message: 跟其他工具一样, 写 一条 role: "tool" 消息告诉 LLM "skill 已加载"。 不写会让 messages 序列断裂 (第 02 章的教训)。
  3. load_skill 的 content 描述可用工具: 不是 "OK", 而是 "已加载 X, 可用工具: Y, Z"。 让 LLM 知道下一步能调 哪些工具。

❌ / ✅: 错误做法是 load_skill 激活后不写 tool message。 

// ❌ load_skill 激活了, 但没写 tool message
if (call.name === "load_skill") {
  activeSkills.activate(call.args.name);
  continue;   // 跳到下一个 tool call, 没 history.add(tool)
}
// 下一次 chat() 时, LLM 看到一个 assistant 调了 load_skill 但
// 没收到 tool result, 模型困惑

// ✅ load_skill 激活后必写 tool message
if (call.name === "load_skill") {
  activeSkills.activate(call.args.name);
  history.add({ role: "tool", tool_call_id: call.id, content: "Loaded skill X. Tools: Y, Z" });
  continue;
}

3 个常见误解

误解 1 · "Skill 应该进 system prompt 才有保证"
错。 Skill 工具描述走 LLM.chat() 的 tools 字段, 不进 system prompt。 跟 TODO 走 reminder 一样的原理: 功能上等价, 性能上 (prompt cache) 不同。 详细对比看 本章"对 prompt cache 的影响" 一节。
误解 2 · "Skill 之间互斥"
错。 Skill 是并集, 不是互斥。 LLM 调多个 skill, 工具列表合并。 "react" + "typescript" 两个 skill 共存。 只有同名工具会去重, 不是整个 skill 互斥。
误解 3 · "Skill 加载是 hot reload"
错。 skill 配置在 ~/.skills/ 目录, agent 启动时 skillLoader.scan() 加载一次。 加载后修改 skill 文件, 需要重启 agent 才生效。 "hot reload" 看似灵活, 实际引入 一致性问题 (LLM 看到 A, 你改完, 下一轮看到 B, 行为漂移)。

反例梯度

新手错法 · A

常见错误: skill 激活状态写到 module-level 单例。

为什么错: 第 04 章父子隔离被破坏, 子智能体看到父 agent 加载的 skill。

正确做法: activeSkills 在 createAgent() 闭包内, 工厂模式。

中级错法 · B

常见错误: 工具描述拼到 system prompt 字符串。

为什么错: 破坏第 10 章 cache-friendly 布局, 每次调 LLM 都要重传所有 skill 工具描述。

正确做法: 工具描述作为 LLM.chat() 的 tools 字段, system prompt 保持稳定。

高级错法 · C

常见错误: load_skill 工具调成功但没写 tool message。

为什么错: 第 02 章的 tool_call_id 配对约束, 不写 tool message 会让 messages 序列断裂。

正确做法: load_skill 激活后, 必写一条 role: "tool" 消息, content 描述已加载的工具列表。

边界错法 · D

常见错误: 同一个 skill 加载多次, 工具集出现重复。

为什么错: availableTools 出现重复项, LLM 看到两个相同 name 的工具, 困惑。

正确做法: activeSkills.tools() 用 Map 去重, 同名工具取先注册的版本。
