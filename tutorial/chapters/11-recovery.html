第 11 章 · 跑生产时的异常恢复

Recovery: 7 类错误 × 4 种恢复动作的纯函数决策

前 10 章的 harness 在 happy path 上很顺, 但生产里 LLM 一定会 报错: 网络抖 / 限流 / 余额耗尽 / 上下文撑爆 / 输出被截断 / 认证失效 / 协议错误。 每种错误对应不同的恢复动作 (backoff / compact / continue / fail), 决策错一次就是几小时的挂机等待 或几万的 token 浪费。 这一章加 src/recovery.ts 模块 — 7 类错误统一分类 (classifyLLMError) + 4 种恢复动作决策 (decideRecovery, 纯函数) + 3 段提示文案 (recovery notice / failure message) + 1 个 sleep 工具。 读完后, 你能讲清"为什么 context_length 用 compact 而不是 backoff" 和"为什么 decideRecovery 不在内部修改 state", 并能用纯函数测试覆盖 4 种动作的边界。

真实失败故事: LLM 限流 1 小时, harness 傻等

写代码之前, 先看一个真实痛点。 用户的 team 上线 harness 跑 生产, 周一早上 9 点全公司 30 个用户同时跑, OpenAI 限流 429。

  1. 症状: 30 个 user 同时被 429 限流, harness 傻傻 同步等待 60 秒 (硬编码 backoff), 一个一个 retry, 跑 1 小时 才把队列清完。
  2. 更深的问题: 限流 5 次后, harness 应该 fail 让 用户知道"现在过载了, 稍后再试", 而不是继续 retry。 傻等 让 30 个用户全部卡死, 没人能 ctrl-c。
  3. 类似的惨案: context_length 错误 (413) 应该 compact 上下文, 但团队写了通用 backoff, 每次都带同样长的 上下文重试, 反复 413, 浪费 5 次重试 budget 才 fail。
  4. 真问题: 错误分类 + 恢复决策写进 if/else, 散在 agent.ts 各处, 改一次要重读 200 行。 应该集中到 recovery 模块, 纯函数决策, agent 只调 decideRecovery(...)。

朴素想法 1: "所有错误都 retry 一下?" 错。 401 认证错误 retry 100 次还是 401; quota 耗尽 retry 100 次还是耗尽; context_length retry 100 次还是太长。 不同错误需要不同动作, 不是"重试次数" 的差异。

朴素想法 2: "重试 budget 全局共享?" 错。 1 个 user 的 rate_limit 不应该消耗下一个 user 的 budget。 重试计数必须 per-request, 每次 agent.run(query) 重建 RecoveryState。

正确做法: 加 src/recovery.ts — classifyLLMError 把任何 LLM 错误归并到 7 种类型, decideRecovery 纯函数决策 4 种动作, formatFailureMessage 给用户 可读的中文提示。 这是 Reference 章节 "模式 13 · Error→Action 错误转动作" + "模式 19 · Idempotent 幂等" 的具体应用。

7 类错误: 不可恢复的先识别

用途: 实际 LLM 调用失败的原因可能几十种 (HTTP 401/403/413/429/500/502/503/504, SDK 异常, 网络 timeout, DNS 失败, 协议错误, 余额耗尽...), 但 agent 主循环不应该认识所有。 应该先归并成少数几类, 每类对应一种恢复策略。

真实场景: user 看到 "Error: Request failed with status code 429", harness 应该知道这是 rate_limit, 走 backoff; 看到 "Error: Invalid API key", harness 知道这是 credential, 直接 fail 不 retry。 不同错误对应不同动作, 决策不能错。

设计思想: 收敛 + 顺序 — 把 几十种 provider 方言收敛成 7 种领域类型; 分类顺序很重要 (credential / quota 这种"不可恢复" 必须先识别, 不然浪费 重试 budget)。 看 src/recovery.ts 第 22 行的 LLMErrorKind union: 

export type LLMErrorKind =
  | "network"
  | "rate_limit"
  | "credential"
  | "quota"
  | "context_length"
  | "output_interrupted"
  | "unknown";
图 1 · 7 类错误的分类顺序与对应动作
1. credential (401/403) · 不可恢复
API key 错 / 权限不够。 fail, 提示用户检查 LLM_PROVIDER / API key / baseURL。 永远不 retry。
2. quota (429 + quota/billing) · 不可恢复
账户余额耗尽。 fail, 提示"token 额度或账户余额不足"。 等不回来, 充值才行。
3. rate_limit (其他 429) · 临时
限流, 临时性。 backoff 3 秒 × 5 次。 5 次后 fail, 提示"LLM 服务暂时不可用"。
4. context_length (413) · 上下文过长
上下文撑爆模型窗口。 compact 压缩历史, 1 次。 1 次后仍 fail。
5. network (5xx/timeout) · 网络抖动
服务器临时挂或网络抖。 backoff 3 秒 × 5 次。 5 次后 fail。
6. output_interrupted · 输出被截断
流式输出中途断开。 continue (从断点续), 2 次。 2 次后 fail。
7. unknown · 协议错误 / 代码 bug
无法识别。 fail, 输出原始 error message。 不盲目重试, 防止 bug 放大。

实现细节: 分类顺序刻意设计 — 先识别 credential (401/403 + 关键字), 再识别 quota (429 + quota/billing 关键字), 再识别 rate_limit (其他 429), 再识别 context_length, 再识别 network, 最后 unknown。 顺序错了, credential 会被误判为 rate_limit, 浪费 5 次 retry budget。

classifyLLMError: 不依赖 OpenAI SDK 类型

用途: 分类函数必须不依赖具体 LLM SDK 的类型, 只通过宽松结构 (error.status / error.code / error.message) 读取。 这样换 provider (OpenAI / Anthropic / 自部署 / Kimi) 都不用 改分类函数。

真实场景: team 原本用 OpenAI SDK, 后来加 Anthropic SDK, 又加自部署 vLLM。 三个 SDK 的 error 类型不同 (OpenAI 是 APIError, Anthropic 是 AnthropicError, vLLM 直接抛 Error)。 如果 classifyLLMError 用 instanceof OpenAI.APIError, 换 provider 就崩。 应该用 duck typing, 读 error.status 这种宽松字段。

设计思想: 鸭子类型 + 启发式 — 不强求特定 SDK 类型, 只读常见字段; 不是完美真理, 启发式分类 已经覆盖 99% 场景。 看 src/recovery.ts 第 109 行的 classifyLLMError: 

export function classifyLLMError(error: unknown): LLMErrorKind {
  const status = extractNumber(error, "status");
  const message = extractString(error, "message").toLowerCase();

  // 1. credential: 401/403 或认证关键字
  if (status === 401 || status === 403 ||
      message.includes("api key") || message.includes("unauthorized") ||
      message.includes("forbidden") || message.includes("credential")) {
    return "credential";
  }
  // 2. quota: 429 + 额度关键字 (rate_limit 之前先识别)
  if (status === 429 && (message.includes("quota") || ...)) {
    return "quota";
  }
  // 3. rate_limit: 其他 429
  if (status === 429) return "rate_limit";
  // 4. context_length: 413 或上下文长度关键字
  if (status === 413 || message.includes("context length") || ...) {
    return "context_length";
  }
  // 5. network: 5xx 或网络关键字
  if (status >= 500 && status <= 504 ||
      message.includes("timeout") || message.includes("econnreset") || ...) {
    return "network";
  }
  return "unknown";
}

实现细节: 用 extractNumber / extractString 内部辅助函数 (第 326/334 行) 安全 提取字段, 不依赖 SDK 形状。 message 全部 .toLowerCase() 做大小写不敏感匹配, 覆盖 "API Key" / "api key" / "Api Key" 各种 写法。

4 种恢复动作: continue / compact / backoff / fail

用途: 7 类错误 → 4 种动作, 一一映射, 决策不能 含糊。 continue = 输出续; compact = 压缩重试; backoff = 等 重试; fail = 停止。 4 种动作覆盖所有可恢复 + 不可恢复场景。

真实场景:

  • LLM 流式输出跑到 8k token 时网络抖断了, 错误是 output_interrupted, 动作 = continue (从 8k 断点续, 不从头来)
  • LLM 返回 413, 动作 = compact (压缩历史后重试, 同等长度仍太长才 fail)
  • LLM 返回 401, 动作 = fail (config 错误, 永远不重试)

设计思想: 动作类型枚举 — 看 src/recovery.ts 第 39 行的 RecoveryAction union: 

export type RecoveryAction = "continue" | "compact" | "backoff" | "fail";
图 2 · 4 种恢复动作的语义对比
continue · 输出续
针对 output_interrupted, 追加 continuation reminder 后再调 LLM, 期望从断点继续。 不修改历史, 节省 token。
compact · 压缩重试
针对 context_length, 强制压缩历史 (第 06 章 P0/P1/P2) 再调 LLM。 1 次 budget, 仍过长就 fail。
backoff · 等重试
针对 network / rate_limit, sleep retryDelayMs (默认 3 秒) 后重试同一次 LLM 调用。 最多 5 次。
fail · 停止
针对 credential / quota / unknown, 或 budget 用完。 停止 run(), 返回用户可读中文提示。

实现细节: 4 种动作的协议 — continue / compact 在 agent.ts 调 LLM 之前, 由调用方负责执行 动作; backoff 在 LLM 调用之间, 调用 sleep(); fail 在最外层 try/catch 里 throw 或 return, 由 repl.ts 渲染中文提示。

decideRecovery 纯函数: 不在内部修改 state

用途: 决策函数必须纯 — 同样 输入永远同样输出, 不修改 state, 不调外部 API, 不读时间。 这才能单元测试覆盖。

真实场景: 测试想验证"network + retry 5/5 → fail", 但如果 decideRecovery 内部 state.apiRetryCount++, 测一次后 state 变了, 第二次跑同样的输入, 期望 fail 但实际 backoff, 测试 flaky。 纯函数没这个问题, 同样的输入两次 都返回同样的动作。

设计思想: 决策 vs 执行分离 — decideRecovery 只算"应该做什么", 调方负责 "做"和"state++"。 看 src/recovery.ts 第 198 行的 decideRecovery: 

export function decideRecovery(
  kind: LLMErrorKind,
  state: RecoveryState,
  config: RecoveryConfig = DEFAULT_RECOVERY_CONFIG,
): RecoveryAction {
  switch (kind) {
    case "network":
      return state.apiRetryCount < config.maxApiRetries ? "backoff" : "fail";
    case "rate_limit":
      return state.apiRetryCount < config.maxApiRetries ? "backoff" : "fail";
    case "context_length":
      return state.compactRetryCount < config.maxCompactRetries
        ? "compact" : "fail";
    case "output_interrupted":
      return state.continueRetryCount < config.maxContinueRetries
        ? "continue" : "fail";
    case "credential":
    case "quota":
    case "unknown":
      return "fail";
  }
}

实现细节: 故意不在内部 ++ — 注释里明确说"decideRecovery 是纯函数, 故意不在内部递增 state, 这样调用方能清楚控制什么时候消耗一次重试预算, 测试 也更容易覆盖。 常见坑: 在决策函数里顺手修改计数, 导致日志 显示次数和真实次数错位。"

默认参数 config = DEFAULT_RECOVERY_CONFIG 让调用方 不传 config 也能用 (用默认值), 但测试时显式传 small config (maxApiRetries: 0) 验证"立即 fail" 边界。

RecoveryState 作用域: per-request, 不全局

用途: 重试 budget 必须每次 agent.run() 重建, 不持久化, 不全局共享。 1 个 user 的 rate_limit 不应该 消耗下一个 user 的 budget。

真实场景: 30 个 user 同时用 harness, user A 触发 rate_limit 重试 5 次后 fail, 但 user B 的 budget 也耗尽了 (因为全局共享), user B 第一次 LLM 调用就 fail。 这就是 "重试 budget 全局" 的惨案。

设计思想: 作用域清晰 — RecoveryState 的注释里明确写"单次 agent.run() 的恢复计数器, 不应该跨 user turn 复用"。 看 src/recovery.ts 第 86 行的 createRecoveryState: 

export function createRecoveryState(): RecoveryState {
  // RecoveryState 是"单次 agent.run() 的恢复计数器", 不应该跨 user turn 复用。
  // 如果把它做成全局状态, 一个用户请求触发的 rate limit 可能导致下一个请求直接 fail。
  return { apiRetryCount: 0, compactRetryCount: 0, continueRetryCount: 0 };
}

实现细节: 工厂函数而非全局变量, 强制"每次 新建"。 agent.run(query) 第一行 const recovery = createRecoveryState(), 整个 run() 期间所有 retry 共享这一个 instance, run() 返回时 销毁。 这是 Reference 章节 "模式 1 · 工厂 + 闭包" 的应用 — 状态在闭包内, 外部不能 "误改全局"。

DEFAULT_RECOVERY_CONFIG: 5/1/2/3 秒的经验值

用途: 给 4 个数字合理的默认值, 用户不传也能用。 教学项目优先直观而非通用, 把 经验值写死, 不开放配置中心。

真实场景: 默认 maxApiRetries=5, retryDelayMs=3000。 user 跑 5 次 retry × 3 秒 = 15 秒后 fail, 给用户"现在过载" 提示, 不傻等 1 小时。

设计思想: 经验值 + 教学优先 — 看 src/recovery.ts 第 75 行的 DEFAULT_RECOVERY_CONFIG: 

export const DEFAULT_RECOVERY_CONFIG: RecoveryConfig = {
  maxApiRetries: 5,
  maxCompactRetries: 1,
  maxContinueRetries: 2,
  retryDelayMs: 3000,
};

实现细节: 4 个数字的理由:

  • maxApiRetries=5: 网络抖 5 次仍 fail 几乎都是真 挂了 (不是临时问题), 5 次 ≈ 15 秒 (含 3 秒 sleep × 5), 用户 不会觉得太慢。
  • maxCompactRetries=1: 压缩 1 次仍过长 = 历史太 长, 继续压缩会丢关键信息, 1 次够。
  • maxContinueRetries=2: 输出截断 2 次仍截断 = 模型 输出超长 (本身有问题), 2 次够。
  • retryDelayMs=3000: 3 秒是经验值, 太短仍触发 rate_limit, 太长用户体验差。 想要 jitter (随机化) 留 P2 阶段。

3 段提示文案: notice / failure / reason

用途: 用户在 REPL 看到的中文提示必须人话, 不能"Error: 429"。 3 段不同用途的提示:

图 3 · 3 段提示文案的边界
formatRecoveryNotice · 恢复中
"LLM 调用失败, 正在重试 3/5, 3 秒后继续..." — 走 logger.warn, 用户看到"系统在等, 别急"。
formatFailureMessage · 最终失败
"LLM token 额度或账户余额不足, 请稍后或补充额度后再试。" — 走 repl.ts 渲染, 给用户具体行动指引。
error.message · 原始 SDK 错误
"Request failed with status code 429" — 附加在 failure message 括号里, 给开发者看, 用户看不懂。

实现细节: 看 src/recovery.ts 第 242 行的 formatRecoveryNotice + 第 276 行的 formatFailureMessage。 failure message 故意分 7 类, 每类一句中文, 不复用通用模板, 因为 credential 和 quota 的"用户应该做什么" 完全不同。

sleep: 单独导出, 测试可 mock

用途: backoff 动作需要 sleep, 但单测不应该 真等 3 秒。 把 sleep 单独导出, 测试用 vi.useFakeTimers() mock, 跑 1000 个 retry 测试也不会真等。

真实场景: 测试"network 重试 5 次, 第 6 次 succeed", 不 mock sleep 就要等 5×3=15 秒, vitest 跑 100 个 recovery 测试要 25 分钟。 mock 之后瞬时跑完。

设计思想: 副作用提出来 — sleep 是副作用, 应该可替换, 不内联在 backoff 流程里。 看 src/recovery.ts 第 318 行的 sleep: 

export function sleep(ms: number): Promise<void> {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

实现细节: agent.ts 调 if (action === "backoff") await sleep(retryDelayMs), 简单, 可读, 可测试。 这是 Reference 章节 "模式 17 · Test Doubles 测试替身" 的应用 — 副作用函数单独导出, 测试时换实现。

主循环集成: 4 个恢复点

用途: recovery 模块在 agent.ts 集成有 4 个点:

  1. LLM 调用前: try/catch 包住 llm.chat(), catch 后调 classifyLLMError → decideRecovery → 执行动作。
  2. backoff 执行: 调 sleep(retryDelayMs), state.apiRetryCount++, continue 循环。
  3. compact 执行: 调 compressor.compress(history, level: "aggressive"), state.compactRetryCount++, continue 循环。
  4. continue 执行: append 一条 user message "请从断点 继续", state.continueRetryCount++, continue 循环。

设计思想: 4 个动作 4 个分支 — 每个动作的执行逻辑不同, 用 switch 处理, 不混在 if/else 链里。 公共部分 (state 递增 + 日志) 提到 switch 之前。

实现细节: agent.ts 第 3 步 (call LLM) 的伪代码: while (true) { try { return await llm.chat(messages, tools); } catch (err) { const kind = classifyLLMError(err); const action = decideRecovery(kind, recovery, config); logger.warn(formatRecoveryNotice(action, kind, recovery, config)); switch (action) { case "backoff": await sleep(config.retryDelayMs); recovery.apiRetryCount++; continue; case "compact": ...; case "continue": ...; case "fail": throw new Error(formatFailureMessage(kind, err)); } } }。

fake test: 纯函数覆盖 4 种动作

用途: recovery 模块的测试不需要 LLM, 纯函数测试, 输入错误 + state, 期望动作。 这是教学版 最大优势 — 业务逻辑和副作用彻底分离, 测业务逻辑零成本。

真实场景: 用户写测试覆盖 4 种动作的 12 个 边界 (4 动作 × 3 状态), 加上 classifyLLMError 的 7 类错误 分类测试, 总共 20 个单测覆盖 recovery 模块, 5 分钟跑完。

设计思想: 纯函数 = 单元测试黄金 — decideRecovery / classifyLLMError 都是纯函数, 同样输入永远同样 输出, 测试不需要 setup/teardown。 看测试示例: 

test("network + 0/5 -> backoff", () => {
  expect(decideRecovery("network", { apiRetryCount: 0, compactRetryCount: 0, continueRetryCount: 0 }))
    .toBe("backoff");
});

test("network + 5/5 -> fail", () => {
  expect(decideRecovery("network", { apiRetryCount: 5, compactRetryCount: 0, continueRetryCount: 0 }))
    .toBe("fail");
});

test("context_length + 0/1 -> compact", () => {
  expect(decideRecovery("context_length", { apiRetryCount: 0, compactRetryCount: 0, continueRetryCount: 0 }))
    .toBe("compact");
});

test("credential 任何 state 都 fail", () => {
  expect(decideRecovery("credential", { apiRetryCount: 0, compactRetryCount: 0, continueRetryCount: 0 }))
    .toBe("fail");
  expect(decideRecovery("credential", { apiRetryCount: 5, compactRetryCount: 0, continueRetryCount: 0 }))
    .toBe("fail");
});

test("classifyLLMError: 401 -> credential, 429 + quota -> quota, 其他 429 -> rate_limit", () => {
  expect(classifyLLMError({ status: 401, message: "" })).toBe("credential");
  expect(classifyLLMError({ status: 403, message: "forbidden" })).toBe("credential");
  expect(classifyLLMError({ status: 429, message: "insufficient_quota" })).toBe("quota");
  expect(classifyLLMError({ status: 429, message: "rate limit exceeded" })).toBe("rate_limit");
  expect(classifyLLMError({ status: 413, message: "context length exceeded" })).toBe("context_length");
  expect(classifyLLMError({ status: 500, message: "" })).toBe("network");
  expect(classifyLLMError(new Error("weird thing"))).toBe("unknown");
});

test("decideRecovery 是纯函数: 同样输入多次调用结果相同", () => {
  const state = { apiRetryCount: 3, compactRetryCount: 0, continueRetryCount: 0 };
  const r1 = decideRecovery("network", state);
  const r2 = decideRecovery("network", state);
  expect(r1).toBe(r2);
  expect(state.apiRetryCount).toBe(3);   // 内部没修改 state
});

实现细节: 6 个测试覆盖 4 动作 + 7 类分类 + 纯函数性质。 不需要 mock, 不需要 fake timer (sleep 单独 测试, 走 vi.useFakeTimers), 全程 <1ms 跑完。

常见误解: 4 种动作 ≠ 重试次数

误解 1: "所有错误都 retry N 次?" 错。 4 种 动作类型不同, 不是次数不同。 context_length retry 5 次 = 浪费 5 次 budget 在不可能成功的请求上, 应该 compact 1 次仍 fail 就 fail。

误解 2: "quota 等一下就好?" 错。 quota = 余额 耗尽, 等 1 小时还是耗尽, 必须 fail 提示用户充值。 rate_limit ≠ quota, 两者都可能是 429, 但 message 含 quota/billing 关键字 是 quota, 否则是 rate_limit。

误解 3: "decideRecovery 应该自己 +1?" 错。 决策和执行分离 — decide 只算"应该做什么", 调用方负责"做" 和 "state++"。 内置 ++ 会让日志和真实次数错位, 测试也难。

误解 4: "formatFailureMessage 是装饰, 可选?" 错。 用户看不懂 "Error: 429", 必须有 中文 failure message 告诉用户"现在过载, 等下再试" 或 "余额不足, 去充值"。 这不是 nice-to-have, 是核心功能。

反例梯度: 4 个常见错误

反例 1 · 所有错误都 backoff 5 次

用户写 if (error) { for (let i = 0; i < 5; i++) { await sleep(3000); return await llm.chat(...); } }。 错。 context_length 错误 5 次重试, 每次都带同样长的上下文, 全部 413 失败, 浪费 15 秒 + 5 次 token 浪费。 正确: 先 classify → decide, context_length 走 compact 1 次, 仍 fail 才放弃。

反例 2 · 错误分类顺序错

先判断 status === 429 返回 rate_limit, 再 判断 message 含 quota。 错。 顺序反了, quota (429 + "insufficient_quota") 会被先识别为 rate_limit, 走 backoff 5 次, 5 次后 fail 提示"LLM 服务暂时不可用", 但真实是 "余额不足"。 正确顺序: credential → quota → rate_limit。

反例 3 · RecoveryState 全局

把 let apiRetryCount = 0 放在模块顶层, 所有 agent.run 共享。 错。 user A 触发 rate_limit 重试 5 次, 把 全局 budget 用完, user B 第一次 LLM 调用就 fail。 正确: createRecoveryState() 在 agent.run 第一行调用, run() 返回时销毁。

反例 4 · 输出中文 "未知错误"

对 unknown 错误, 给用户显示 "未知错误, 请稍后重试"。 错。 unknown 错误可能是协议错误 / SDK bug, 用户重试 100 次还是 fail。 正确: 输出 "LLM 调用出现未知错误: ${errMsg}", 把原始 error.message 暴露给开发者, 同时引导用户 "请提 issue"。

Validation: 4 条不变量检验

  1. 分类顺序正确: 同样的 error (status=429 + message="insufficient_quota") 永远返回 quota, 不会先识别为 rate_limit。 验证: 单测覆盖 4 个互斥 case (401/429+quota/429 普通/413), 看返回值。
  2. 纯函数性质: 同样输入 (kind + state) 多次调用 decideRecovery, 返回值永远相同, state 内部不被修改。 验证: 单测, 调用 2 次, state 字段对比。
  3. budget 上限: 4 种动作的 budget 用完后, 必须 返回 fail, 不能继续 retry。 验证: 单测覆盖 (5,0,0) → fail, (0,1,0) → fail, (0,0,2) → fail。
  4. credential/quota/unknown 立即 fail: 这 3 类 任何 state 都返回 fail, 不消耗 budget。 验证: 单测 (0,0,0) 和 (5,1,2) 都 expect fail。

回望: 哪些原则在本章兑现了

  • 错误转动作: 7 类错误 → 4 种动作, 一一映射, 决策 纯函数, 不混执行。
  • 决策 vs 执行分离: decideRecovery 只算"应该 做什么", 调方负责"做"和"state++"。
  • 作用域清晰: RecoveryState 每次 agent.run() 重建, 不全局共享。
  • 人话提示: 中文 failure message 告诉用户具体 行动 (充值 / 改 config), 不输出 "Error: 429"。
  • 副作用可替换: sleep 单独导出, 测试用 fake timer mock。

前瞻张力: 留给后续章节

Tool throw 错误
本章只覆盖 LLM 调用错误。 tool 内部 throw (如 fs.readFile ENOENT) 走另一套恢复 — write "Tool error" tool message 不 fail, 让 LLM 自己决定重试或换工具。 错误分类不能把 tool error 和 LLM error 混在一起。
用户主动取消 (ctrl-c)
user 按 ctrl-c 取消当前 run(), 应该 graceful shutdown — 保存 history + 释放 lock, 不应该 mid-write 文件导致 half state。 这是 OS 信号的捕获, 不是 recovery 范畴, 但和 fail 动作的"立即停止" 边界要切清。
Subagent 错误传播
子 agent 抛 credential 错误, 父 agent 应该 fail 还是继续? 当前策略: 父 agent 把子 agent 错误包成 tool message "Subagent failed: ${reason}", 让 LLM 自己决定重试或换子 agent 任务。 这是 Reference 章节 "模式 14 · Concurrent Limit" 的延伸。

Prompt Card (本章任务)

Prompt Card · 第 11 章

目标: 给 harness 加 recovery 模块, 7 类错误 分类 + 4 种恢复动作 + 3 段提示文案, agent 主循环按动作 走分支。

场景: 30 个 user 同时用 harness, OpenAI 限 流 429, 每个 user 走 backoff 5 次 × 3 秒 = 15 秒, 5 次后 fail 提示"LLM 服务暂时不可用", user ctrl-c 不卡死。

模块: src/recovery.ts (新) 暴露 classifyLLMError(error) / decideRecovery(kind, state, config) / formatRecoveryNotice(...) / formatFailureMessage(...) / sleep(ms) / createRecoveryState(); src/agent.ts (改) 第 3 步 try/catch, 按动作 走分支。

边界 (LLM 必须遵守的 checklist):

  • 7 类错误固定枚举, 分类顺序 credential → quota → rate_limit → context_length → network → unknown
  • 4 种动作 continue / compact / backoff / fail, 一一映射, 不可新增
  • decideRecovery 纯函数, 不在内部修改 state, 调方负责 ++
  • RecoveryState 每次 agent.run() 重建, 不全局共享
  • credential / quota / unknown 任何 state 都 fail, 不消耗 budget
  • failure message 必须是中文, 告诉用户具体行动, 不输出 "Error: 429"
  • sleep 单独导出, 测试用 fake timer mock, 不内联在 backoff 流程

验证 (用纯函数 + vitest, 逐条断言):

  • classifyLLMError 覆盖 7 类输入, 输出对应 kind
  • decideRecovery 覆盖 4 动作 × 3 state 边界 (0/budget/budget+1)
  • credential / quota / unknown 任何 state 都 fail
  • decideRecovery 是纯函数: 同输入 2 次返回相同, state 不变
  • sleep 单独测试, 用 vi.useFakeTimers 验证 1000 次不真等

本章练习

  1. 故意把所有错误都 backoff 5 次 (不分类), 跑 rate_limit 场景 测试, 看"统一重试" 是否抓到 (5 次后 fail 但浪费 5 次 budget 在不可能成功的请求上)。
  2. 故意把分类顺序写错 (先 429, 再 quota), 跑 classifyLLMError({status: 429, message: "insufficient_quota"}) 测试, 看"quota 应该不被识别为 rate_limit" 是否抓到。
  3. 把 RecoveryState 提到模块顶层全局变量, 跑两个 agent.run 串联 测试, 看"全局共享 budget" 是否抓到 (第二个 run 第一次 LLM 调用就 fail, 因为 budget 被第一个用完)。
  4. 在 decideRecovery 内部 state.apiRetryCount++, 跑"纯函数" 单测, 看"纯函数性质" 是否抓到 (state 被修改, 同输入 2 次返回不同)。

本章小结

Recovery 是给 LLM 调用的错误转动作, 7 类 错误 × 4 种动作的纯函数决策。 核心是 5 个设计:

  • 7 类错误: network / rate_limit / credential / quota / context_length / output_interrupted / unknown, 分 类顺序固定 (credential → quota → rate_limit → ...)。
  • 4 种动作: continue / compact / backoff / fail, 一一映射, 不可新增。
  • 纯函数决策: decideRecovery 不修改 state, 调方 负责 ++, 单元测试覆盖 12 个边界。
  • per-request state: RecoveryState 每次 agent.run() 重建, 不全局共享, 30 个 user 互不干扰。
  • 人话提示: 中文 failure message 告诉用户具体 行动, 不输出 "Error: 429"。
