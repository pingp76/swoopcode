第 11 章 · 出错时继续保持结构

LLM 出错时不要崩：Recovery

真实模型调用会失败：限流、网络抖动、上下文超限、格式异常。本章把错误分类和恢复动作纳入 agent loop。

你已经知道什么，还不知道什么

你已经有压缩器和稳定请求布局。现在要承认一个现实：即使 harness 写得正确，provider 仍然会失败。

读这一章时请把注意力放在 agent loop 的接入点：LLM 调用失败后，根据错误类别选择 backoff、compact、continue 或 fail。代码片段只服务于理解架构边界，不要求你逐行背实现。

本章怎么学

先把自己放进一个真实 query：没有 Recovery 时，当前 agent 会在哪里卡住。

再故意设计一个朴素方案，观察它如何破坏 loop、状态或安全边界。这个失败比“正确答案”更重要，因为它能帮你写出更好的重建 prompt。

最后才看实现：先看图和数据流，再看少量接口、源码入口和验证方法。读完后，你应该能用自己的话描述 Recovery 的职责，而不是只记住文件名。

本章场景

Agent 正在帮用户修测试，突然 LLM API 返回 rate limit。或者 messages 太长，provider 返回 context length error。直接抛异常会让用户丢失当前进度。

Recovery 层要先分类错误，再选择恢复动作：限流 backoff，超上下文 compact，流式中断可能 continue，不可恢复错误明确失败。

先试一个朴素方案

朴素方案是 try/catch 后统一 retry 三次。如果三次还失败，就把错误抛给用户。

这个朴素方案的问题在于：它把 Recovery 当成普通实现细节，而不是 agent harness 的一条稳定边界。 

for (let i = 0; i < 3; i++) {
  try { return await llm.chat(messages); }
  catch (error) { await sleep(1000); }
}
throw error;

朴素方案为什么不够

错误类别不同
rate limit 适合 backoff，context length 需要 compact，参数错误 retry 没意义。
预算会泄漏
retry 状态如果跨 turn 复用，会让下一次用户请求莫名失败。
审计缺失
恢复动作不写入事件流，后续无法解释 agent 为什么压缩或等待。

回到 Agent Loop

Recovery 接入 loop 的位置是：LLM 调用失败后，根据错误类别选择 backoff、compact、continue 或 fail。理解这个位置，比记住具体函数名更重要。

Recovery 属于单次 agent.run 的控制流。它读取错误、当前预算和压缩能力，决定下一步动作，但不应永久污染下一轮用户请求。

从原理落到代码

到这里我们已经知道 Recovery 为什么存在、插在 loop 的哪里，以及朴素方案会怎样失败。下面才进入实现层：只抓住能固定架构的接口、状态边界和接线路径。

读源码时不要从文件顶部一路啃到底。更有效的顺序是：先找入口函数，再找它读写的状态，最后看测试如何证明这些状态变化是真的发生了。

一次真实运行 walkthrough

  1. Agent 构建 messages 并调用 LLM。
  2. LLM 抛出 context length error。
  3. Recovery classifier 判断为可通过 compact 恢复。
  4. Agent 调用 compressor 生成更短 messages。
  5. 同一次 agent.run 内重试 LLM。
  6. 重试成功后继续正常 tool/answer loop。
  7. 如果超过恢复预算，返回明确失败并保留可诊断信息。

关键接口和伪码

下面的接口和伪码只保留本章最关键的形状。真正写代码时，可以让 coding agent 参考源码地图中的文件补齐细节。 

type RecoveryAction =
  | { kind: "backoff"; delayMs: number }
  | { kind: "compact" }
  | { kind: "continue" }
  | { kind: "fail"; reason: string };

function decideRecovery(error: unknown, state: RecoveryState): RecoveryAction {
  if (isRateLimit(error)) return { kind: "backoff", delayMs: state.nextDelay };
  if (isContextLength(error)) return { kind: "compact" };
  return { kind: "fail", reason: formatError(error) };
}

设计直觉

Recovery 让 agent 失败得有结构。不是所有错误都能修，但所有错误都应该被分类、记录，并给用户一个可理解结果。

把 provider 差异收敛在 recovery/adapter 层，可以避免 agent loop 到处散落 if provider。

架构拆分

classifier
把 provider 错误转成领域类别。
policy
决定 backoff、compact、continue、fail。
agent loop
执行恢复动作，并控制本次运行预算。

状态与边界

run-local
retry 次数、backoff 预算、compact 尝试次数。
History
不应写入半截失败 assistant 消息。
Transcript
可以记录恢复事件，便于审计。

Prompt Card

Prompt Card

目标：实现 LLM 错误分类和恢复策略，支持 backoff、compact、continue、fail。

场景：LLM 限流、上下文超限或流式中断时，agent 尝试恢复而不是直接崩溃。

模块：recovery.ts、llm.ts、agent.ts、compressor.ts、transcript.ts。

接线：Agent 捕获 LLM 错误，调用 recovery decision，再执行对应动作。

边界：不要无限 retry；恢复预算只属于当前 agent.run；不可恢复错误要明确失败。

例外和 corner case：rate limit、network、context length、invalid request、stream interrupted。

验证：mock 各类错误，断言恢复动作、重试次数和最终状态。

源码入口：src/recovery.ts、src/agent.ts、src/llm.ts。

文档同步：说明错误分类比简单 retry 更重要。

容易踩坑

Design Trap

常见错误：所有错误都 retry。

为什么错：参数错误和权限错误 retry 只会浪费时间，context length 需要先压缩。

正确做法：先分类，再选择恢复动作，并设置预算。

怎么验证：mock invalid request 时应 fail，不应 retry 三次。

  • retry 预算跨 turn 复用。
  • 压缩失败后继续追加 messages。
  • 把 provider 原始错误泄漏成用户难懂的长堆栈。

如何验证

Validation

手工 query：人为设置很小上下文预算触发 compact，再观察是否继续执行。

确定性测试：mock LLM 依次抛 rate limit、成功，验证 backoff 后重试。

集成观察：Transcript 中能看到 recovery 事件。

失败信号：同一个错误无限循环或吞掉失败。

  • src/recovery.test.ts 覆盖决策。
  • src/agent.test.ts 覆盖集成。
  • 用 fake compressor 测试 context length 恢复路径。

如果实现失败，先查哪里

  • 检查错误是否被正确分类。
  • 检查恢复预算是否在本次 run 内初始化。
  • 检查 compact 后 messages 是否真的变短。
  • 检查失败原因是否给用户可理解信息。
