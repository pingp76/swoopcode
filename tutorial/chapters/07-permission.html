第 07 章 · 给工具画边界

Permission: 工具调用前要不要问人

第 06 章的压缩解决了"context 撑爆" 问题, 但 harness 现在能调 run_write / run_bash 改文件, 没人把关的话, LLM 可能误删 / 误改重要文件。 这一章给 harness 加 Permission 系统: 三种模式 (plan / auto / default), 黑白名单, 路径边界, ask 降级。 读完后, 你能讲清"plan 模式" 和 "default 模式" 的区别, 并能用 fake terminal 验证"ask 时被拒, tool result 写 Permission denied" 的协议。

真实失败故事: 跑 5 分钟测试卡死主循环

写代码之前, 先看一个真实场景。 跑一段真实场景: 用户用 harness 跑一个 "全项目 e2e 测试", 命令是 npm run test:e2e, 跑 5 分钟。

  1. 误删: LLM 试图"清理项目里的临时文件", 跑 rm -rf /tmp/some-keep-this, 重要数据没了。
  2. 误改: LLM 试图"fix 一个 typo", 但它用 run_edit 模糊匹配, 把同名 pattern 的 5 个文件全改了。 用户只想要 1 个。
  3. 网络外泄: LLM 试图"看 React 文档", 调 curl https://... 把本地文件内容发到外网。
  4. 危险命令: LLM 试图"重启服务", 跑 sudo systemctl restart nginx, 整个环境挂了。

朴素想法 1: "LLM 应该聪明到不犯这些错?" 错。 LLM 是"下一个 token 预测器", 它在压力下 (上下文满、跑长任务) 会选错工具。 把安全责任完全推给 LLM, 就像让司机"靠记忆" 开车 不系安全带。

朴素想法 2: "禁止 LLM 调危险工具?" 那 LLM 啥都干不了。 需要人工把关。

正确做法: Permission 层在 LLM 和工具之间加一道 人工把关。 危险操作前先问用户, 用户拒绝就跳过, 不假装成功。

三种模式: plan / auto / default

用途: Permission 系统提供 3 种模式, 用户的"信任程度" 不同。 默认 default, 用户启动时通过 /mode plan|auto|default 切换。

真实场景: 用户首次使用 harness, 不知道 harness 行为, 选 plan 模式"看着它做"; 跑 CI / 批处理时选 auto 信任 harness; 日常使用选 default 平衡。

设计思想: 经典策略模式 — 同一操作在不同 模式下有不同决策 (allow / ask / deny), 用查表代替 if/else 链。 这是 Reference 章节的"模式 8 · Strategy 策略模式" 的具体应用。

模式 行为 典型场景
plan 所有写操作都要用户确认, 只读操作不确认 用户首次使用, 不知道 harness 行为, 想"看着它做"
auto 所有操作直接执行, 不问 用户信任 harness, 跑批量任务 (CI / 自动化)
default 按"危险度" 决定: 危险操作问, 安全操作直接做 日常使用, 大部分操作自动, 危险操作确认

3 个模式的边界:

  1. plan 模式: 防御性, 适合新用户。 但用户会被"问得太多" 烦。 跑 10 轮对话可能问 8 次。
  2. auto 模式: 信任模式, 跑 CI / 批处理必备。 但用户 失去"撤销" 机会 — 误删了没办法。
  3. default 模式: 平衡模式, 大部分情况自动, 危险 操作问。 适合日常使用。

默认是 default, 用户启动时通过 /mode auto / /mode plan / /mode default 切换。 mode 状态 存在 sessionEventBuffer (第 08 章), 跨 turn 保留。

ask 流程: 怎么问用户

Permission 检查命中"需要问" 时, 走 ask 流程:

  1. harness 检测到危险操作 (run_write / run_bash 危险命令)
  2. 调 terminal.askUser(prompt) 弹出确认
  3. 用户在 REPL 看到 Allow run_write to /etc/hosts? (y/n)
  4. 用户输入 y 或 n
  5. harness 根据回答继续 (y: 执行, n: 跳过)

关键: ask 必须同步等用户回答。 REPL 不能继续读 下一行, 直到用户回答当前 ask。 这跟普通 readline 不同 — readline 是一问一答, ask 是一问一等 + 必须答。

朴素反例: 权限检查写在工具里

// ❌ 反例: 权限写在工具里
async function runWrite(args) {
  if (args.path.startsWith("/etc")) {
    if (!confirm("Allow run_write to /etc/hosts? (y/n)")) {
      return { content: "Permission denied", error: true };
    }
  }
  await fs.writeFile(args.path, args.content);
  return { content: "OK" };
}

// 同样逻辑要在 run_bash / run_edit / run_delete / 等等各写一份

5 件事立刻坏掉:

  1. 重复代码: 同样的"问 / 拒绝" 逻辑在 5 个工具里各写一份, 维护噩梦。
  2. 模式切换写不出: plan / auto / default 怎么切? 每个工具都要知道当前 mode, mode 状态散在各处。
  3. 测试难写: 测试时每个工具都要 mock 自己的 confirm。 不能用一个 fake terminal 统一替换。
  4. 规则不一致: run_write 问 /etc 路径, run_bash 问 哪些命令, run_edit 问哪些 pattern — 各自一套规则, 容易有漏洞。
  5. 工具作者必须懂权限: 写 run_web_fetch 的工程师 要决定"问不问", 工具作者增加心智负担。

解决方式: Permission 层在 ToolRegistry.invoke 之前拦截, 工具 不知道也不关心权限。

PermissionManager: 集中权限决策

把权限检查抽出来, 集中在一个 manager: 

interface PermissionManager {
  check(input: {
    toolName: string;
    args: Record<string, unknown>;
    mode: "plan" | "auto" | "default";
  }): PermissionDecision;
  setMode(mode: "plan" | "auto" | "default"): void;
  addBlacklist(pattern: string): void;
  addWhitelist(pattern: string): void;
}

type PermissionDecision =
  | { kind: "allow" }
  | { kind: "deny"; reason: string }              // 硬黑名单 (危险命令), 不问
  | { kind: "ask"; prompt: string };              // 需要用户确认

GitHub · src/permission.ts PermissionManager 实现 (L1)

3 种 decision 的边界:

  1. allow: 直接执行, 不问用户。 工具白名单 / 安全操作。
  2. deny: 硬黑名单, 拒绝, 不问。 比如 rm -rf /、 sudo、/etc 写。 即使 auto 模式也拒绝。
  3. ask: 调 terminal.askUser, 问用户。 plan 模式全 ask, default 模式危险操作 ask, auto 模式不 ask。

❌ / ✅: 错误做法是 deny 和 ask 混为一谈。 

// ❌ deny 和 ask 都是 ask, 让用户决定
async function check(toolName, args, mode) {
  if (isDangerous(toolName, args)) {
    const allowed = await ask(`Allow ${toolName}?`);
    return allowed ? { kind: "allow" } : { kind: "deny" };
  }
  return { kind: "allow" };
}
// 问题: rm -rf / 也让用户决定, 用户可能误答 y

// ✅ deny 是硬规则, 不让用户选择
async function check(toolName, args, mode) {
  if (isBlacklisted(toolName, args)) return { kind: "deny", reason: "hard blacklist" };
  if (isDangerous(toolName, args) && mode !== "auto") {
    return { kind: "ask", prompt: `Allow ${toolName}?` };
  }
  return { kind: "allow" };
}

黑名单: 危险命令的硬规则

黑名单是 hard rule, 永远拒绝, 不让用户选择。 常见黑名单: 

const HARDCODED_BLACKLIST = [
  /rm\s+-rf\s+\//,                    // rm -rf /
  /sudo/,                              // sudo 命令
  /mkfs/,                             // 格式化
  /dd\s+if=/,                         // dd 写盘
  /:\(\)\s*\{.*\s*:\s*'\1;.*\}/,  // fork bomb
  /chmod\s+-R\s+777\s+\//,             // chmod 777 根
];

GitHub · src/command-safety.ts HARDCODED_BLACKLIST (L1)

这些规则来自 command-safety.ts, 跨工具共享 (run_bash 和 ExecutionPolicy 都用)。 集中在一处, 避免两处实现漂移。

黑名单的判断: 先用正则匹配命令字符串, 匹配成功就 deny。 这种 简单匹配能挡掉 90% 的常见误操作。 复杂 shell 语法解析不可靠时 宁可拒绝。

3 种模式的决策表

工具 + 操作 plan default auto
run_read (任何路径) allow allow allow
run_bash (安全命令如 ls / cat / echo) ask allow allow
run_bash (危险命令如 rm / sudo) deny deny deny
run_write (workspace 内) ask ask allow
run_write (workspace 外) deny deny deny
run_edit (workspace 内) ask ask allow
run_web_fetch (任意 URL) ask ask allow
黑名单 (任何模式) deny deny deny

关键点: 黑名单是硬规则, 跨模式都拒绝。 workspace 外写也是硬规则, 不让用户"破例" — workspace 是项目边界, 突破就破坏隔离。

路径边界: workspace 内的 hard rule

用途: 写操作必须限制在 workspace 内, 这是 hard rule, 跟 模式无关。 防止 LLM 误写到 /etc / 用户的 home / 任何项目外。

真实场景: LLM 调 run_write("/etc/hosts", ...) 修改系统 hosts, 任何模式都拒绝。 workspace 是项目边界, 突破就 破坏隔离。

设计思想: 路径边界用 path.resolve() 解析相对路径和 .., 然后判断 resolved 是否在 workspaceRoot 下。 不在就 deny。

实现细节: 

// 教学简化版
function isPathInWorkspace(path: string, workspaceRoot: string): boolean {
  const resolved = path.resolve(workspaceRoot, path);
  return resolved.startsWith(path.resolve(workspaceRoot) + path.sep);
}

// 决策
if (toolName === "run_write" && !isPathInWorkspace(args.path, workspaceRoot)) {
  return { kind: "deny", reason: "path outside workspace" };
}

Terminal.askUser 同步等待

上一节提到, ask 决策要调 terminal.askUser(prompt) 弹出 确认. 这一节专门讲 askUser 的同步等待语义 — 它跟普通 readline.question() 不一样, 必须理解清楚 才能写对代码.

先讲 "同步" 是什么意思

"同步" 在这里指调用方必须等返回值, 才能继续往下走. 翻译成代码: askUser() 返回一个 Promise<boolean>, await 它就卡住当前这行代码, 直到用户回答 ("y" 或 "n") 才继续. 在用户回答之前, agent loop 的整个下一步 都被冻住, LLM 不能调下一个工具, agent 也不返回.

举个具体例子: 普通 readline.question() vs askUser()

图 1 · 两种问法的执行时序对比
普通 readline.question() — 一问一答, 不卡
第 1 行: const name = await rl.question("你叫什么? ")
    ↓ 用户输入 "Alice", 按回车, 拿到 "Alice"
第 2 行: console.log("hi", name)
第 3 行: const age = await rl.question("多大了? ")
    ↓ 用户输入 "30"
第 4 行: console.log("age:", age)

特点: 每次问, 等回答, 再问下一个. 但不是"必须答" — 用户可以 ctrl-c 退出, 抛异常.
askUser() — 一问一等, 必须答, 卡整个 agent loop
第 1 行: const ok = await terminal.askUser("允许 rm 吗? (y/n) ")
    ↓ 整个 REPL 提示暂停, 等用户输入
    ↓ 用户输入 "y" 或 "n", agent loop 整条暂停线
第 2 行: if (ok) { tool.invoke(...) } else { /* 写拒绝 */ }

特点: 必须答 (y 或 n) 才能继续. 整个 agent loop 在等用户, LLM 不能调下一个工具, 整个 harness 暂时不 推进. 用户回答前, 时间冻住.

"同步等" 的 3 个具体含义

  1. 代码卡在 await: await terminal.askUser(...) 这一行执行后, 整个 JavaScript event loop 在等 resolve, 后续代码不跑. 类似 await fetch(...) 卡在网络 IO 上, 只是这里是卡在"用户敲键盘".
  2. agent loop 整个不推进: harness 跑着, 突然调 askUser, 等 30 秒用户不答, harness 整个冻 30 秒, 不调 LLM, 不调工具, 不输出任何东西. 用户能看到 REPL 提示"等待用户输入".
  3. 必须答才能继续: readline.question() 抛 ctrl-c 异常 也能跑下去, askUser() 必须拿到 y 或 n (或类似) 才能 resolve. 不能 "超时默认拒绝" 或 "超时默认允许" — 都是反模式 (后面 "deny 假装成功" 一节展开).

为什么必须这么设计?

因为 ask 的语义是"用户决定这个工具能不能跑", 跟 普通问"你叫什么名字" 是两件事:

  • 普通 question: 收集信息, 用户不答也能继续 (用 default 或异常), 不影响程序正确性.
  • ask: 安全决策, 用户不答 = 不知道允不允许, 不能 跑这个工具. 必须等用户拍板. 让 agent 假装"用户默认允许" 就回到了 Permission 缺失的老路.

实现细节: 

// 教学简化版
interface Terminal {
  question(prompt: string): Promise<string>;        // 普通 readline: 收集信息, 可中断
  askUser(prompt: string): Promise<boolean>;       // 安全决策: 必须答, 卡整个 loop
  println(text: string): void;
  close(): void;
}

GitHub · src/terminal.ts Terminal 接口 (L1)

注意类型差异: question() 返回 Promise<string> (用户答的字符串), askUser() 返回 Promise<boolean> (y/n 转成 true/false). 两个语义不同, 不能混用.

真实 readline 实现的 askUser: 

async function askUser(prompt: string): Promise<boolean> {
  // 暂停 readline 主循环, 否则用户在主 readline 输的内容会被吞
  this.rl.pause();
  process.stdout.write(`${prompt} (y/n) `);
  return new Promise((resolve) => {
    const onLine = (line: string) => {
      this.rl.removeListener("line", onLine);
      this.rl.resume();
      resolve(line.trim().toLowerCase() === "y");
    };
    this.rl.on("line", onLine);
  });
}

关键细节:

  1. rl.pause() 暂停主 readline 循环: REPL 已经在跑一个 readline (主输入循环), 如果不暂停, 用户在 ask 期间敲的 内容会被主 readline 吞掉, askUser 永远等不到回答. rl.pause() 让主 readline 停止接收输入, 这次只听 askUser 自己的 listener.
  2. return new Promise((resolve) => ...) 一直挂着: askUser 函数返回 Promise 后, await 那一行卡住, 整个 agent loop 等待 resolve. resolve 之前, agent loop 一步都不推进.
  3. 用户输入 y/n 后 resolve: onLine listener 拿到用户 敲的一行, parse 成 boolean, resolve() 让 await 返回, 代码继续. 同时 rl.removeListener + rl.resume() 恢复主 readline 循环.

伪代码的时序: 

// 1. agent 跑, 准备调 run_bash("rm -rf /tmp/old")
// 2. permission 检查: mode=default, command 不在白名单 → 决策 = ask
// 3. 调 await terminal.askUser("Allow rm -rf /tmp/old? (y/n) ")
// 4. **卡住** — 整个 agent loop 暂停
// 5. 用户看到 REPL 提示, 敲 "y" 按回车 (或 "n")
// 6. resolve(true/false), agent loop 继续
// 7. if (ok) { tool.invoke(...) } else { /* 写 Permission denied tool message */ }
// 8. 继续下一轮 LLM 调用

这跟第 02 章 "tool_call 协议" 的关系: askUser 卡住期间, LLM 看不到 tool result, 因为 tool result 还没写. 等用户答完, tool result 才写进 history, 下一轮 LLM 调用才看到. 整个过程 LLM 视角是"它调了工具, 工具在跑, 工具跑完了, 返回了结果", 它 不知道中间 user 介入过. 这跟 LLM 视角的"工具正常返回" 完全等价 — askUser 是在 harness 内部做的安全检查, LLM 不感知.

拒绝时怎么写 tool result

拒绝时, tool message 应该明确写"Permission denied", 不能假装成功: 

// 教学简化版
if (decision.kind === "deny") {
  history.add({
    role: "tool",
    tool_call_id: call.id,
    content: `Permission denied: ${decision.reason}. 请重新决定或换其他方案。`,
    // 不标 error: true, 因为这是预期行为, 不是工具错误
  });
  continue;  // 跳过 invoke
}
if (decision.kind === "ask" && !(await terminal.askUser(decision.prompt))) {
  history.add({
    role: "tool",
    tool_call_id: call.id,
    content: "Permission denied by user. 请重新决定或换其他方案。",
  });
  continue;  // 用户拒绝, 跳过 invoke
}

❌ / ✅: 不要拒绝时 throw。

ask 降级: 用户不在时怎么办

ask 流程的边界: 用户不在 REPL 旁边怎么办?

  1. CI / 批处理: 跑 auto 模式, 不 ask。 permission 系统 检测到 plan/auto 模式时不调 askUser。
  2. 用户不在但 harness 在跑: ask 会卡住。 permission 系统给 ask 设 timeout, 默认超时 (比如 60 秒), 超时按 deny 处理。
  3. 异步执行 (第 13 章): Async Run 默认 auto 模式, 不 ask。

超时降级很重要, 否则 harness 卡在 ask 等用户, 跑 1 小时没动静, 用户以为死了。

3 个常见误解

误解 1 · "Permission 应该写在工具里"
错。 写进工具里 = 重复代码 + 难测试 + 规则不一致。 Permission 必须在 ToolRegistry.invoke 之前拦截, 工具不感知权限。
误解 2 · "黑名单和 ask 是同一回事"
错。 黑名单是 hard rule, 不让用户选择。 ask 是 user choice, 用户可以 y 或 n。 rm -rf / 绝不能 ask, 必须 deny。
误解 3 · "auto 模式 = 不检查权限"
错。 auto 模式 = 不 ask, 但黑名单仍然 deny。 rm -rf / 在 auto 模式也拒绝。 auto 只是"不打扰用户", 不是"放飞"。

反例梯度

新手错法 · A

常见错误: 权限检查写进每个工具里。

为什么错: 重复代码, 模式切换写不出, 工具作者增加 心智负担, 规则不一致。

正确做法: PermissionManager 在 ToolRegistry.invoke 之前拦截, 工具不感知权限。

中级错法 · B

常见错误: 拒绝时 throw, 让 loop 崩。

为什么错: 拒绝是业务行为, 不是 harness 错误, throw 会让 整个 loop 崩, messages 序列错乱。

正确做法: 拒绝时写 "Permission denied" tool message, LLM 看到后自己调整。

高级错法 · C

常见错误: rm -rf / 让用户决定 (ask 而非 deny)。

为什么错: 黑名单是 hard rule, 用户可能误答 y, 误删 整个系统。 跨模式都拒绝。

正确做法: 黑名单和 ask 分开, 黑名单永远 deny, 不让 用户选择。

边界错法 · D

常见错误: ask 没设 timeout, harness 卡住等用户。

为什么错: 用户不在 REPL 旁边, ask 会永远卡住, 跑 1 小时 没动静, 看起来像死循环。

正确做法: ask 设 60 秒 timeout, 超时按 deny 处理, 避免挂死。

如何验证 (本章 Validation 卡片)

Validation · 第 07 章

plan 模式全 ask: plan 模式下, run_write 触发 terminal.askUser, 用户回答决定 allow / deny。

黑名单跨模式 deny: auto 模式下, run_bash("rm -rf /") 仍返回 deny, 不调 askUser。

回望: 哪些原则在本章兑现了

  • 关注点分离: PermissionManager 独立于 ToolRegistry, 工具 不感知权限。
  • 工厂模式: PermissionManager 在 Composition Root 创建, 注入 agent。
  • 硬规则与软规则分开: 黑名单 = hard, ask = user choice, 不混淆。
  • 工具调用协议: 拒绝时写 tool message, 复用第 02 章的协议, LLM 能正常推理。

本章小结

Permission 是给工具的"安全网"。 核心是 3 个设计:

  • 3 种模式: plan 全 ask, default 危险 ask, auto 不 ask。 黑名单跨模式都 deny。
  • PermissionManager 集中: 工具不感知权限, 重复代码 消失, 测试简单。
  • 拒绝写 tool message: 不 throw, LLM 收到 "Permission denied" 后自己调整。

    • 下一章 (第 08 章) 扩展 Permission 的思路 — Hook 系统, 让用户 在 loop 周围挂自定义逻辑 (PreToolUse / PostToolUse), 不只是 ask / deny。
