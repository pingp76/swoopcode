第 07 章 · 执行动作前的安全闸门

给工具画边界：Permission

工具让 agent 真正能改变世界，Permission 则决定它什么时候可以动手。本章把权限检查放在工具执行前，让 plan/default/auto 三种模式有清楚边界。

你已经知道什么，还不知道什么

你已经知道工具调用如何进入 loop。现在要补上执行前的安全闸门：并不是模型提出 run_write，harness 就应该立刻写文件。

PermissionManager 不是简单的 yes/no。它要结合模式、工具名、命令安全、路径边界、用户确认和子智能体继承策略。

本章怎么学

先从本章场景和朴素方案进入问题，不急着记函数名。

再用 loop 位置和架构图建立心智模型：新增模块到底改变了哪条数据流或状态流。

最后再看接口、源码地图和验证方法，把概念压缩成能交给 coding agent 的 Prompt Card。

本章场景

用户让 agent “清理项目里的临时文件”。模型可能提出 rm -rf，也可能写出项目外路径。如果 harness 不检查，教学项目也可能把真实工作区搞坏。

正确流程是：工具调用进入执行层前，PermissionManager 先判断 allow / ask / deny。只有 allow 才执行；ask 需要用户确认；deny 会作为可恢复错误写回 LLM。

先试一个朴素方案

朴素方案是相信模型：“既然用户让 agent 工作，模型应该知道不要做危险事。”或者提供一个 auto 模式直接允许所有工具。

这不是 agent harness，而是裸奔的脚本执行器。模型会误解、幻觉、复制用户危险命令，也可能被不可信文件内容诱导。 

if (toolCall.name.startsWith("run_")) {
  return registry.execute(toolCall.name, toolCall.arguments);
}

朴素方案为什么不够

模式语义消失
plan 模式应该只计划不写入，default 模式应该敏感操作询问，auto 也不应绕过硬性安全黑名单。
路径逃逸
只看工具名不看参数，会允许 write ../outside.txt 这类越界写入。
子智能体权限扩大
child agent 做只读分析时，不应该继承父 agent 的写权限。

回到 Agent Loop

Permission 插在 “工具执行前”。Agent 已经收到 tool call，但还没调用 handler。权限层根据工具、参数、当前模式和路径策略返回 allow / ask / deny。

deny 不是进程异常，而是 tool_result 错误，让模型知道这个动作不能做，并尝试更安全的方案。

架构图解

从原理落到代码

到这里已经有了场景、失败原因和架构图。下面才进入实现层：只抓住接口形状、状态边界和源码入口。

读源码时先找入口函数，再找它读写的状态，最后看测试如何证明这些状态变化是真的发生了。

一次真实运行 walkthrough

  1. LLM 返回一个写文件 tool call。
  2. Agent 根据工具名和参数构造 permission request。
  3. PermissionManager 检查当前模式：plan/default/auto。
  4. 路径类工具先解析到绝对路径，再确认是否仍在 projectRoot 内。
  5. 危险命令命中硬黑名单时直接 deny。
  6. default 模式下敏感写操作返回 ask，Terminal 向用户确认。
  7. 用户拒绝时，Agent 写入错误 tool_result；用户允许时才执行工具。

关键接口和伪码

权限结果最好显式建模，而不是用 throw 表示所有情况。ask/deny/allow 是业务决策，真正的系统异常才应该抛出。 

type PermissionDecision =
  | { action: "allow" }
  | { action: "ask"; reason: string }
  | { action: "deny"; reason: string };

interface PermissionManager {
  checkToolUse(request: {
    toolName: string;
    args: unknown;
    projectRoot: string;
    mode: "plan" | "default" | "auto";
  }): Promise<PermissionDecision>;
}

设计直觉

Permission 的价值是把用户意图、模式选择和工具风险放在 harness 层判断，而不是期待模型自觉。

即使 auto 模式也不意味着无限制执行。安全黑名单、路径边界和资源限制属于硬边界，不应该被模式轻易绕过。

架构拆分

PermissionManager
根据工具请求做 allow/ask/deny 决策。
Terminal
承载 default 模式下的用户确认交互。
Tool Handler
仍要做参数和路径校验，不能假设权限层已经覆盖全部风险。

状态与边界

进入 LLM
deny/ask 被拒绝后的原因可以作为 tool_result 错误进入 messages。
不进入 LLM
用户确认的内部 UI 状态、绝对路径细节和安全策略实现。
子智能体
权限应显式继承或收窄，不应隐式复制父状态。

场景：agent 要写文件或运行命令时，harness 必须先判断是否允许、询问或拒绝。

容易踩坑

Design Trap

常见错误：默认允许所有 run_* 工具。

为什么错：工具名只说明动作类别，不说明参数是否安全。

正确做法：权限层检查工具名、参数、模式和项目边界，工具层再做业务校验。

怎么验证：尝试写 projectRoot 外路径时，无论模式如何都不应成功。

  • 把 ask 写成 LLM 自问自答，而不是询问真实用户。
  • 只用字符串 startsWith 检查路径，符号链接和 .. 可能绕过。
  • Hook 放在权限前后不清晰，导致观察到的事件语义混乱。

如何验证

Validation

手工 query：切换 plan/default/auto 模式，分别尝试写文件和危险命令。

确定性测试：测试 PermissionManager 的 allow/ask/deny 返回。

集成观察：Agent 收到 deny 后应继续 loop，而不是进程崩溃。

失败信号：用户拒绝后工具仍执行，或 deny 没有写回 LLM 可见结果。

  • src/permission.test.ts 覆盖权限模式。
  • src/tools/files.test.ts 覆盖路径边界。
  • src/agent.test.ts 覆盖权限拒绝如何进入 tool_result。

如果实现失败，先查哪里

  • 检查 tool call 是否在执行 handler 前经过 PermissionManager。
  • 检查模式来源是否和 CLI / runtime 状态一致。
  • 检查路径 normalize 后是否仍在 projectRoot。
  • 检查 deny reason 是否足够清楚，模型能据此换方案。
