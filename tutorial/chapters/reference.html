6 段 Prompt Card 模板

这是第 00 章确立的模板, 这里只列反例对照。完整示例见第 01 章 Prompt Card。

反例对照 · 差 → 改 → 好

差的卡片 (5 段, LLM 会偷懒):

目标: 实现 agent loop
模块: history.ts, llm.ts, agent.ts, index.ts
验证: 能跑通
边界: 注意架构
场景: 用户输入 query, agent 调用 LLM 返回文本

问题:"能跑通" 不是断言, "注意架构" 不可验证, 没说 messages 怎么拼。

改 (5 段, 有信息量但 LLM 仍会猜):

目标: 实现最小 agent loop, 多轮对话保留上下文
模块: createHistory, createLlm, createAgent, createRepl, index.ts
验证: 连续两次 run 后第二次 LLM 收到 messages 包含第一轮
边界: agent.ts 不读环境变量, 不直接 new LLM client
场景: 用户先说"我喜欢简洁", 再问"我喜欢什么风格"

改进:有 messages 顺序断言, 有"绝对不能" 的边界。

好 (6 段, LLM 没空间偷懒):

目标: 实现最小 agent loop, 多轮上下文由 History 提供
场景: 用户依次输入 "我喜欢简洁" 与 "我喜欢什么风格",
      agent 第二次回复应包含"简洁"
模块:
  - src/history.ts: createHistory(), 内部 messages: Message[]
  - src/llm.ts: createLlm(config), 暴露 chat(messages)
  - src/agent.ts: createAgent(deps), 暴露 run(query)
  - src/repl.ts: createRepl(deps), 暴露 start()
  - src/index.ts: 创建 history/llm/terminal, 传给 agent 和 repl
接线: index.ts 内只做 new + 传参, 不出现 if 分支
边界 (LLM 必须遵守):
  - agent.ts 内不出现 process.env
  - agent.ts 内不出现 new LLMClient
  - history.getMessages() 返回浅拷贝
  - 空 query 不写入 history
验证:
  - fake LLM 返回 "收到" 时 agent.run("x") === "收到"
  - 连续两次 run, fake LLM 第二次收到的 messages.length === 3
  - 第二次收到的 messages[0].role === "user"
  - run("") 不增加 history 长度

关键差异: "边界" 是可枚举 checklist, "验证" 每条 都能落到 vitest 一行断言, "接线" 写明实例是不是同一份。

vibe coding 4 轮拆卡

6 段卡片是总图, 喂给 LLM 要分 4 轮:

  1. 第 1 轮 · 接口: 只贴 "目标 + 场景 + 模块", 让 LLM 给出 interface 草案 (不写实现, 只钉形状)。
  2. 第 2 轮 · 接线: 贴 "模块 + 接线", 让 LLM 给出 index.ts 接线, 工厂是 stub。
  3. 第 3 轮 · 边界: 贴 "边界 checklist", 让 LLM 按 checklist 写每个工厂实现。
  4. 第 4 轮 · 验证: 贴 "验证断言清单", 让 LLM 写测试, 优先 fake LLM + messages 顺序断言。

4 轮独立可验证, 任一轮不通过单独回退。

章节 Prompt Pack 总目录

每章一句话总结, 方便写"重建这个项目" 的总 prompt 时查阅。

00 · 元方法
用 LLM 写 LLM agent 的元方法: 4 轮拆卡 + review + 防自欺。
01 · 最小 loop
History + LLMClient + Agent + REPL + Composition Root 五件套。
02 · 工具调用
tool registry + tool_call 协议 + agent 主循环加 tool 分支。
03 · TODO
TODO 状态机 + reminder 注入 + 不污染 system prompt。
04 · SubAgent
父子 history/todo 隔离 + 防递归 + maxRounds。
05 · Skill
activeSkills 闭包化 + 按需激活 + 工具去重。
06 · Compress
三层压缩管道 (Normalize / Block / Compress) + 不写回 history。
07 · Permission
同步拦截 + deny 必写 tool message + 子智能体 ask 降级。
08 · Hook
三个事件 (SessionStart / PreToolUse / PostToolUse) + exitCode 0/1/2。
09 · Memory
user / project 双 scope + key 命名规范 + SessionStart 注入。
10 · Cache
稳定前缀 / 动态状态 / 自然增量 三段布局。
11 · Recovery
6 种失败分类 + 不同策略 + retry 上限 + 不跨 run 共享。
12 · Task
结构化字段 + 状态机严格 + renderActive 过滤 + id 由 harness 生成。
13 · Async Run
后台派发 + drain 通知 + 冲突检测 + 复用 P1 压缩。
14 · Schedule
cron 调度 + 复用 async run + 单独 source 标签 + timezone 解析。
15 · Hardening
原子写 + 日志轮转 + output 引用计数 + dry-run + 时间统一。

张力地图: 章节之间的矛盾点

harness 设计不是"哪一章节对的", 而是"在哪些约束下做哪种选择"。下面 是章节之间反复出现的张力, 回头查这里。

压缩 vs cache (06 vs 10)
压缩让 messages 短, 但破坏 system prompt 字符串的 cache 前缀稳定性。P2 总结的 LLM 调用本身吃 cache 配额, 需要 stable context 隔离。
并行 vs 串行工具 (02 vs 13)
同一个工具并行跑 (例如两个 run_bash) 容易写同一文件, async run + 冲突检测是折中方案。
memory vs task (09 vs 12)
memory 是"用户级偏好" (跨项目), task 是"项目级 plan" (单项目)。用户换项目后项目级 fact 隐藏, user 级保留。
reminder 频率 (03 vs 13 vs 14)
TODO reminder / async run notification / schedule notification 都走 reminder 标签, 各自 source 区分。频率高时 reminder 累积, 需要各自内部去重。
权限共享 vs 隔离 (07 vs 04)
父子 agent 共享 permission 策略 (子不能绕过父), 但 history / todo 隔离。共享与隔离的边界是 harness 设计核心。
eval 实时性 vs 重放 (eval vs 11)
eval 跑真实 LLM 能反映现状, 但每次跑结果可能不同; 重放 deterministic LLM stub 稳定但可能与真实行为漂移。两者结合用 (deterministic 主 + live 副)。

下一步

现在你已经走完整套教程。建议的下一步:

  1. 挑一章 (建议从 02 章开始, 不依赖 00/01 的元方法), 用本章 6 段模板自己写一份 Prompt Card。
  2. 用第 01 章的 fake LLM Cookbook 写测试, 验证你写的卡片能让 LLM 实现通过 Validation。
  3. 挑一章做 vibe coding 4 轮拆卡练习, 每轮跑通 Validation 才进下一轮。
  4. 读 eval 专题章, 学习如何把"卡片的实现" 自动化回归 (eval 反馈 prompt 优化)。
  5. 读 model-policy 专题章, 学习不同大模型的差异 (cache / tool call / 错误码)。
