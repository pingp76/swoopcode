4 条核心原则

整个项目所有模式都围绕 4 条核心原则展开。 理解这 4 条, 后面的 22 个模式只是它们的"具体应用"。 这 4 条不是凭空选的 — 它们各自 对应一类反复出现的"系统级"问题, 任何一条违反都会让 harness 在 某个维度上崩。

原则 1 · Composition Root 唯一组装

解决什么问题 : 业务模块互相 import, 改一个全挂; 难测试, 换部署环境要改 10 处。

原理 : 整个项目 只有 1 个文件 import 一切、 new 一切: src/index.ts 的 main() 。 业务模块 之间互相不 import, 只接收注入。 业务模块对"用什么 LLM / 哪个 history" 一无所知, 全部由 Composition Root 决定。

带来的好处 : 换测试 = 换 main() 一处; 换部署 = 换 main() 一处; 业务模块保持纯净, 永远不依赖具体实现。 业务模块之间的依赖关系图"塌缩成一个点", 维护复杂度从 O(n²) 降到 O(n)。

原则 2 · 工厂模式, 状态在闭包内

解决什么问题 : 单例导致父子 agent 状态污染; class 的 this 绑定问题让 callback 出 bug; 测试难隔离。

原理 : 几乎所有"有状态" 的对象 (agent / history / todoManager / skillLoader / memoryManager / taskManager) 都不是单例, 也不是 class, 而是 工厂函数返回的闭包对象 。 状态 藏在闭包里, 实例之间天然隔离。 闭包没有 this 绑定问题 (callback / 解构时不会丢上下文)。

带来的好处 : 父子 agent 隔离、并发测试隔离、多 session 隔离"自动" 成立, 不用业务模块写一行隔离代码。 仓库 0 个 class, 100% 工厂 + 闭包, 模式高度一致。

原则 3 · 稳定前缀优先, 状态走 reminder

解决什么问题 : LLM provider 的 prompt cache 是按 "前缀匹配" 工作的 (Anthropic 1/10, OpenAI 1/2), 任何"小修改" 都会让整个 cache 失效; system prompt 里塞状态会让 cache 命中率 归零, 成本涨 5-10 倍。

原理 : 把 prompt layout 严格分成两部分 — 稳定前缀 (system prompt + tools, 写后不改, 进 cache) + 动态 tail (history + reminder, 每轮变, 不进 cache 但算增量价格)。 状态 (TODO / memory / skill hint) 走 user reminder, 永远不进 system prompt。

带来的好处 : 跨 session / 跨用户 / 跨项目一致的行为规则 + 个性化的状态注入, 两者互不污染, cache 命中率最大化。

原则 4 · narrow interface + adapter 翻译

解决什么问题 : 业务代码直接调外部 SDK (OpenAI / Anthropic / 文件系统 / 终端) 时, 业务被 provider 特定细节污染; 换 provider = 重写业务; 错误码不一致 (Anthropic 429 / OpenAI 429 / Google RESOURCE_EXHAUSTED)。

原理 : 所有外部依赖 (LLM / 终端 / 工具 / 文件系统) 都 走 窄接口 (1-3 个方法), 实现藏在 adapter 里。 Adapter 内部处理 network / retry / cache / provider 错误码翻译, 业务模块 只看内部统一类型 (e.g. LLMErrorKind 7 类)。

带来的好处 : 业务模块 0 改动换底层; 测试用 fake 替换 真实实现; provider-specific 错误码统一翻译成内部错误类型, recovery 模块不依赖具体 provider。

模式 1 · 工厂函数 + 闭包 (Factory + Closure)

出现频率 : 整个项目最常用的模式, 出现在几乎所有 manager / store / loader 上。

解决什么问题 : 单例导致父子 agent 共享状态 (子 agent 调 load_skill 影响父); class 的 this 绑定问题 (把方法传 setTimeout / 事件回调, this 丢失); 测试无法隔离 (一个测试残留状态影响下一个测试)。

原理 : 函数式编程 + 闭包 — 工厂函数返回的对象捕获 闭包内的局部变量, 这些变量对外不可见, 只能通过返回的方法访问。 每次调 createXxx() 都得到全新闭包, 状态天然隔离。 闭包没有 this , 传 callback 永远不会丢上下文。

为什么不直接用 class

朴素想法: "用 class 写, 状态放 private 字段, 不是更清晰?" 错。 class 在 TypeScript 里有两个问题:

  1. this 绑定问题 : 把方法当 callback 传 (e.g. setTimeout ), this 丢失, 访问 this.items 变 undefined 。 闭包没有这个问题 — 闭包捕获的是变量, 不是 this 。
  2. 测试隔离 : 测一个方法要 new TodoManager() , 测完销毁。 闭包工厂一行 createTodoManager() 即可, 自动隔离。
  3. 父子隔离自动 : 父子 agent 各自调 createTodoManager() , 闭包天然隔离。 class 要手动传实例, 容易写错。

这就是为什么仓库 100% 用工厂 + 闭包, 一个 class 都没有。

常见陷阱

陷阱 · 闭包状态被意外共享
在工厂函数外定义 const, 然后在工厂内捕获 — 这会"按模块共享"而不是 "按实例共享"。 状态必须 在工厂函数体内 声明。
陷阱 · 闭包引用了大对象导致内存泄漏
闭包捕获的对象无法被 GC。 Async Run / Schedule 完成后必须 finishRun() 显式清理闭包内的 map, 否则长期跑内存爆炸。

模式 2 · 窄接口 + Adapter 翻译 (Narrow Interface + Adapter)

出现频率 : 外部依赖 (LLM / 终端 / 工具 / Provider / LLM Provider Error) 全部用这模式。

解决什么问题 : 业务代码直接调 openai.chat.completions.create() 时, 业务被 provider 特定字段名污染; Anthropic / OpenAI / Google 三个 provider 字段格式完全不同; 换 provider = 重写业务; provider 错误码各不相同 (Anthropic 429 / OpenAI 429 / Google RESOURCE_EXHAUSTED) 业务需要逐个识别。

原理 : 经典 适配器模式 + 接口隔离原则 。 定义一个跟外部细节 无关 的窄接口 (只含业务真正需要 的 1-3 个方法), 业务模块只 import 这个接口类型, 不 import 任何 外部 SDK。 Adapter 实现这个接口, 内部把所有 provider 特定逻辑 (字段名 / 错误码 / 重试 / 缓存) 翻译成内部统一类型。

带来的好处 : 业务模块 0 改动换 provider; 测试用 fake 替换; provider-specific 错误码统一翻译; 重试 / 缓存 / streaming 这些"中间件" 关注点藏 adapter 内部。

// 教学简化版
// 1. 定义窄接口
interface LLMClient {
  chat(args: { messages; tools? }): Promise<LLMResponse>;
}

interface LLMResponse {
  content: string | null;
  toolCalls: ChatCompletionMessageToolCall[];
  finishReason: string | null;
}

// 2. Adapter 实现接口, 内部处理 provider 特定逻辑
export function createOpenAILLMClient(config: ResolvedLLMConfig): LLMClient {
  const openai = new OpenAI({ apiKey: config.apiKey, baseURL: config.baseURL });
  return {
    async chat({ messages, tools }) {
      const completion = await openai.chat.completions.create({
        model: config.model,
        messages,
        tools,    // OpenAI 格式转换在 adapter 内部
      });
      // OpenAI 特定: choice[0].message.tool_calls
      const message = completion.choices[0].message;
      return {
        content: message.content,
        toolCalls: message.tool_calls ?? [],
        finishReason: completion.choices[0].finish_reason,
      };
    },
  };
}

GitHub · createOpenAILLMClient (LLMClient 适配器) (L1)

关键点:

  1. 接口只暴露 1 个方法 : chat() , 不暴露 temperature / max_tokens 等 provider 特有参数。 这些参数由 adapter 内部处理, 业务模块不感知。
  2. Adapter 翻译 provider 特定字段 : OpenAI 的 choices[0].message 翻译成 LLMResponse 。 换 Claude 时, Claude adapter 自己处理 content[0].text 。
  3. 业务模块只看接口 : agent.ts import LLMClient 类型, 不 import OpenAI。 换 Claude = 换 adapter, 业务代码 0 改动。

3 个边界设计

写 adapter 时, 必须遵守 3 条:

  1. 接口只暴露业务关心的 : 业务不需要 "messages 在 Anthropic 哪个字段", 只需要 "LLMResponse.content / toolCalls"。 其他字段不外泄。
  2. 错误统一翻译 : Provider 错误码 (Anthropic 429 / OpenAI 429 / Google RESOURCE_EXHAUSTED) 全部翻译成 LLMErrorKind.rate_limit 。 Recovery 模块 (第 11 章) 不看 provider。
  3. Adapter 自己处理 network / retry / cache : 这些 "中间件" 关注点藏 adapter 内部, 业务模块不感知。

常见陷阱

陷阱 · 接口设计过大
LLMClient 暴露 setTemperature() / setMaxTokens() 等 5 个 setter, 业务模块就要关心这 5 个。 正确做法: 接口只有 chat() 一个方法, 这些 setter 藏在 adapter 内部或 通过 RuntimePolicy 注入。
陷阱 · 业务模块 import 了 adapter
import { createOpenAILLMClient } from "./llm.js" — 这破坏了窄接口的隔离。 业务应该 import 类型 LLMClient , 让 Composition Root 决定用哪个 adapter 实现。

模式 3 · 依赖注入 (Dependency Injection)

出现频率 : 几乎所有 manager 的构造函数 / 工厂函数。

解决什么问题 : 业务代码自己 new PermissionManager() / new Logger() 时, 业务被具体实现绑死; 改 logger 格式要改 业务代码; 测试时无法替换 logger 收集输出 (因为 logger 在业务内部 new 出来的, 外面引用不到)。

原理 : 控制反转 (Inversion of Control) — 业务模块不再控制依赖的创建, 而是 接收 别人创建的 依赖 (通过构造函数 / 工厂参数)。 依赖的"组装" 集中在 Composition Root, 业务模块保持纯净。

带来的好处 : 测试可以注入 fake 实现 (e.g. ScriptedLLMClient 替代真实 LLM); 改 logger / permission / tool 不改业务代码; 业务模块 0 配置即可运行 (因为依赖是注入的, 不需要 自己读 env / config)。

常见陷阱

陷阱 · 默认值放在构造函数内, 不可注入

陷阱 · 注入"上帝对象" 一个 deps 包揽 10 个东西
业务模块的 deps 膨胀到 10+ 个, 难以阅读。 正确: 拆成几个相关 集合 (e.g. { llm, history, messages } 一起, { logger, metrics } 一起)。

模式 3.5 · 0 循环依赖: 桥接模式 (Bridge Pattern)

出现频率 : 仓库里几乎每个跨模块集成点都用这个模式 (memory + hook, skill + registry, async run + subagent, todo reminder + agent 主循环 ...)。 这是整个项目架构设计的 核心原则, 模式 3 (依赖注入) 和模式 4 (Composition Root) 都是为了实现这个原则。

解决什么问题 : 两个模块互相需要对方的能力 (memory 需要把内容发到 LLM, LLM 需要在启动时拿到 memory 段), 但两个模块不能互相 import (会栈溢出, 也违反分层)。 怎么"互相调用但不互相 import"?

朴素想法 1 · 直接互相 import : memory.ts 顶部写 import { agent } from "./agent.js", agent.ts 顶部写 import { memoryManager } from "./memory.js"。 一启动两个模块互相 找, 栈溢出。 即便不溢出, 也违反关注点分离 — memory 模块该管文件读写, 不该知道 "agent 长什么样", 反之亦然。

朴素想法 2 · 一个模块直接调另一个 : agent.ts 启动时调 memoryManager.buildPromptSection() 拼 system prompt。 表面行, 但违反关注点分离 + 难以扩展。 agent 主循环不该关心"memory 是什么" / "怎么拼"。 哪天加 Skill 注入 规范, 是不是又改 agent.ts? 加 Schedule 提醒是不是又改? agent.ts 会变成"巨型 switch", 谁都不敢动。

正确做法 · 桥接文件 : 写一个第三文件, 这个文件同时 import 两个 模块, 作为"桥"调用两边, 但两边互相不 import。

具体例子: memory + hook 的桥接 

// src/hooks/memory-inject.ts — 桥接文件, 唯一同时 import 两边的
import type { HookHandler } from "../hooks.js";
import type { MemoryManager } from "../memory.js";

// 返回 handler 函数, 闭包持有 memoryManager
export function createMemoryInjectHandler(
  memoryManager: MemoryManager,
): HookHandler {
  return (event) => {
    const section = memoryManager.buildPromptSection();
    if (!section) return { exitCode: 0 };
    return { exitCode: 2, message: section };
  };
}

Composition Root (index.ts) 把 3 个模块拼起来:
  const memory = createMemoryManager(...)
  const hookRunner = createHookRunner({
    SessionStart: [createMemoryInjectHandler(memory)]  ← 桥接文件被调用
  }, logger)
  const agent = createAgent({ hookRunner })              ← agent 不知道 memory

4 条铁律

  1. 业务模块之间不直接 import: memory.ts 不 import hooks.ts (反之亦然), agent.ts 不 import memory.ts。 任何两个业务模块 都通过第 3 个文件桥接。
  2. 桥接文件命名清晰: hooks/memory-inject.ts 的文件名就说明"这是 hook 系统的 memory 注入桥"。 一眼能看出 这是个桥, 不是业务逻辑。
  3. 桥接文件只做翻译, 不做业务: 它把 memory.buildPromptSection() 的字符串返回包成 {exitCode: 2, message}, 没有任何业务逻辑。 业务 都在被桥接的模块里。
  4. Composition Root 负责把桥"插上去": 业务模块 + 桥接文件都是"独立单元", Composition Root (index.ts) 负责 把它们拼起来 — 哪个 handler 注册到哪个事件, 哪个 manager 注入到哪个 handler。 业务模块和桥接文件都不知道自己会被怎么 拼。

这个原则在整个仓库的 5 个体现

集成场景 桥接文件 做什么
memory → system prompt hooks/memory-inject.ts 调 buildPromptSection(), 包成 {exitCode: 2, message}
todo reminder → user message tools/run-todo-*.ts 调 todoManager.list(), 在 tool result 里格式化
async run → subagent async-runs.ts 内部 launchSubagentRunner 调 createAgentFn(...) 派生 child agent, 注入只读工具集
schedule → async run schedules.ts 内部 onAsyncRunFinish 调 asyncRunManager.start(...) 启动后台执行
cache debug → agent agent.ts 内部 inspect 调用 调 cacheTracker.inspect({messages, tools}) 输出日志

违反这条原则会怎样

  • 栈溢出: memory.ts import agent.ts, agent.ts import memory.ts → 启动时两个 模块互相找, 无限递归。
  • 巨型 switch: agent.ts 直接读 memory, 又直接读 skill, 又直接读 schedule, 几百行 import + if-else, 没人敢改。
  • 不可测: memory 测 memory 时被迫 mock agent, 互相 耦合, 测试写不下去。

为什么模式 3 (依赖注入) 和模式 4 (Composition Root) 都要服务这个原则

  • 模式 3 (依赖注入): 业务模块不直接 new 别的模块, 而是通过工厂参数接收。 这样模块 之间没有 import 关系, 都能独立测试。
  • 模式 4 (Composition Root): 所有"new" / "createXxx" 调用集中在 index.ts。 业务模块只看 到"我已经创建好的依赖", 不知道谁创建的, 也没有 import 别的模块。

两个模式一起实现了"0 循环 import" + "0 业务模块互相知道" 的目标。 桥接模式是这两个模式在跨模块集成点的具体应用 — 当两个业务模块要互通时, 写第 3 个文件当桥, 桥里 import 两边 然后做翻译。

模式 4 · Composition Root (唯一组装点)

出现频率 : 项目只有 1 个 Composition Root: src/index.ts 的 main() 。

解决什么问题 : 业务模块互相 import, 改一个全挂; 难 部署到不同环境 (CLI / GUI / CI / 测试); 测试时无法替换真实 实现 (因为业务自己 new 了)。

原理 : 经典 依赖反转 的极端形式 — 整个 项目的"对象图构造" 集中到 1 个函数, 业务模块之间不互相 new 对方。 Composition Root 是唯一 知道所有实现细节 的地方。 切换环境 = 切换 Composition Root, 业务模块 0 改动。

带来的好处 : 换测试 = 换 main() 一处; 换部署 = 换 main() 一处; 业务模块保持纯净, 永远不依赖具体 实现。 业务模块之间的依赖关系图"塌缩成一个点", 维护复杂度 从 O(n²) 降到 O(n)。

模式 5 · 注册表模式 (Registry Pattern)

出现频率 : ToolRegistry / SkillLoader / HookRunner / MemoryStore / TaskStore / ScheduleStore / OutputStore。

解决什么问题 : 工具/技能/hook 数量动态增长 (从 5 个 涨到 30+); 调用方要"按名字查找" + "按类别枚举"; 翻译 (工具 def → LLM ChatCompletionTool) 应该跟调用解耦。

原理 : 经典 注册表模式 — 把"按名字 查找对象" 抽成一个数据结构 (通常是 Map<string, T> ), 外界通过 register(name, obj) 和 get(name) 交互。 注册和查询分离, 翻译 (LLM 格式转换) 藏 registry 内部。

带来的好处 : 调用方不 import 具体工具类; 动态加载 (从配置文件扫) 不需要改业务代码; 翻译 (tool def → LLM 格式) 在 registry 集中, 不会散在业务模块各处; 未知名字统一返回 error, 不抛 throw。

模式长什么样

// 教学简化版
interface ToolRegistry {
  register(name: string, def: ToolDefinition, executor: ToolExecutor): void;
  get(name: string): ToolDefinition | undefined;
  list(): ToolDefinition[];
  getToolDefinitions(): ChatCompletionTool[];   // 转成 LLM 格式
  invoke(name: string, args: unknown): Promise<ToolResult>;
}

export function createToolRegistry(): ToolRegistry {
  const tools = new Map<string, { def: ToolDefinition; executor: ToolExecutor }>();

  return {
    register(name, def, executor) {
      tools.set(name, { def, executor });
    },
    get(name) { return tools.get(name)?.def; },
    list() { return [...tools.values()].map(t => t.def); },
    getToolDefinitions() {
      return [...tools.values()].map(t => ({
        type: "function",
        function: {
          name: t.def.name,
          description: t.def.description,
          parameters: t.def.parameters,
        },
      }));
    },
    async invoke(name, args) {
      const entry = tools.get(name);
      if (!entry) return { content: `Tool not found: ${name}`, error: true };
      return entry.executor(args, { /* tool context */ });
    },
  };
}

GitHub · createToolRegistry 注册表 (L1)

关键点:

  1. 注册 / 查询 / 调用 分离 : 注册时存 def + executor, 查询时返回 def, 调用时 invoke executor。 三件事各管各的。
  2. getToolDefinitions() 翻译 : 内部存 ToolDefinition, 对外返回 LLM 期望的 ChatCompletionTool 格式。 Adapter 翻译 在 registry 内部, 业务模块不感知。
  3. 未知工具返回 error : 调不存在的工具不抛 throw, 返回 { content: "Tool not found", error: true } 。 LLM 收到后能继续推理。

3 个变种

变种 用途 示例文件

动态 loader 从配置目录扫描加载 (skill / memory / task) src/skills.ts / src/memory.ts / src/tasks.ts

常见陷阱

陷阱 · registry 自身有顺序依赖
注册顺序决定调用顺序, 业务依赖了"特定顺序"。 正确: registry 只管 存储, 顺序由调用方决定, 或者按"插入顺序" 显式记录。
陷阱 · registry 翻译函数返回不一致格式
getToolDefinitions() 第一次返回 {type: "function"} , 第二次返回 {type: "tool"} — 业务崩溃。 正确: 翻译 函数应该是 纯函数 , 同样输入永远同样输出。

模式 6 · Builder / Pipeline 模式

出现频率 : agent.prepareMessages() 的消息处理管道 / Compressor 的 P0/P1/P2 / 各种"输入 → 多步处理 → 输出" 的场景。

解决什么问题 : 消息处理 / 压缩 / 恢复是多步操作, 每步 顺序不能换, 散在 if/else 里难调试; 单步测试难 (要准备完整数据才能 测中间一步)。

原理 : 经典 管道模式 (Unix pipe 思想) — 把多步操作建模成"每步是纯函数, 输入 = 上一步输出, 顺序固定"。 每步 不可变 (不修改原数组, 返回新数组), 可以单独 测, 可以单独换实现。

带来的好处 : 步骤可单测 (想测 P0 衰减? 直接调 compressor.decay(blocks) , 不用先准备完整 messages); 步骤顺序错位立刻能看出 (调换顺序会出 bug, 测试会挂); 步骤是 纯函数, 无副作用, 调试容易。

模式长什么样

// 教学简化版, agent 主循环的 prepareMessages 管道
function prepareMessages(input: {
  history: History;
  currentLoopIndex: number;
  todoManager: TodoManager;
  memoryManager: MemoryManager;
  hooks: HookRunner;
}): ChatCompletionMessageParam[] {
  let blocks = getEntries(input.history);             // 1. 拿 entries
  blocks = annotate(blocks, input.currentLoopIndex);   // 2. 标 metadata
  blocks = normalize(blocks);                          // 3. 合并相邻同角色
  blocks = group(blocks);                              // 4. 分组成 text/tool_use/summary
  blocks = input.compressor.decay(blocks);             // 5. P0 衰减
  blocks = input.compressor.compact(blocks);           // 6. P2 总结
  blocks = input.hooks.run("LLMCall", blocks);         // 7. hook 注入
  return flatten(blocks);                              // 8. 还原成 messages
}

GitHub · prepareMessages Pipeline (8 步处理) (L1)

关键点:

  1. 每步是纯函数 : 输入 blocks, 输出 blocks, 不修改原数组。 顺序错了换行即可, 不会乱。
  2. 步骤固定, 不重排 : getEntries → annotate → normalize → group → decay → compact → hooks → flatten 。 任何一步乱序, 要么 metadata 丢, 要么压缩破坏消息块。
  3. 可单步测试 : 想测 P0 衰减? 直接调 compressor.decay(blocks) , 不用先准备完整 messages。

模式 7 · State Machine (状态机)

出现频率 : Task 状态 (6 个) / Async Run 状态 (6 个) / Schedule trigger 状态 / Recovery 状态。

解决什么问题 : 任务/Run 有 5+ 个状态, 状态间转换规则 复杂 (e.g. 失败能重试, 完成不能再回退); 不显式枚举, 业务代码会 写出"任意状态切换" 的混乱代码; 审计时不知道"任务到底处在 什么状态"。

原理 : 经典 有限状态机 (FSM) — 用 state enum + transition matrix 显式枚举所有合法 状态和合法转换。 任何状态切换必须经过"状态机校验函数", 不允许 业务代码直接 set status。

带来的好处 : 状态转换错误立刻被状态机捕获 (e.g. in_progress → pending 抛错); 状态机校验集中, 业务模块不能绕过; 审计能信任"任务当前 status" 是合法的。

模式长什么样

// 教学简化版, Task 状态机
type TaskStatus = "pending" | "in_progress" | "completed" | "failed" | "cancelled" | "deleted";

const ALLOWED_TRANSITIONS: Record<TaskStatus, TaskStatus[]> = {
  pending:     ["in_progress", "cancelled"],
  in_progress: ["completed", "failed", "cancelled"],
  completed:   [],     // 终态
  failed:      ["pending"],  // 失败可以重试
  cancelled:   [],     // 终态
  deleted:     [],     // 终态
};

function updateTaskStatus(task: Task, newStatus: TaskStatus): void {
  const allowed = ALLOWED_TRANSITIONS[task.status];
  if (!allowed.includes(newStatus)) {
    throw new Error(`Cannot transition from ${task.status} to ${newStatus}`);
  }
  // 额外校验: pending → in_progress 必须先满足依赖
  if (newStatus === "in_progress" && !dependenciesCompleted(task)) {
    throw new Error(`Task ${task.id} is blocked by ${task.blockedBy.join(", ")}`);
  }
  task.status = newStatus;
  task.updatedAt = Date.now();
  if (newStatus === "completed") task.completedAt = Date.now();
}

关键点:

  1. 转换矩阵显式枚举 : 哪个状态可以转哪个状态, 一张表查 一清二楚。 不允许任意状态切换 (那等于没状态机)。
  2. 业务校验放在 transition 函数 : 依赖检查 / 权限检查 / 时间窗检查, 都放在 updateStatus() 内部, 不让业务模块 自己检查后乱调 setStatus。
  3. 终态不可转出 : completed / cancelled / deleted 是终态, 不允许转出。 避免"撤销归档" / "恢复已删除" 这类破坏审计的操作。

模式 8 · Strategy (策略模式)

出现频率 : RuntimePolicy / FoundationModelProfile / Permission 3 模式 / 3 种 trigger / 3 种 compression mode。

解决什么问题 : 同一功能有多种实现 (3 种 permission mode, 3 种 schedule trigger, 3 种 compression mode); 业务代码用 if/else 链判断"用户选了哪个 mode" 时, 加新 mode 改 10 处; 模式选择规则不集中, 难审计。

原理 : 经典 策略模式 — 把"模式选择" 抽成 静态查表 (mode × 操作 → 决策), 业务代码 只调 checkPermission(mode, op) 拿结果, 不写 if/else。 新增 mode = 加一行, 不改 check 函数。

带来的好处 : 业务代码 if/else 链消失; 新 mode 改动 局部化 (只改表); 模式规则集中, 审计和测试都简单; 黑名单独立 于 mode, 跨策略硬规则不污染 mode 决策。

模式长什么样

// 教学简化版, Permission 3 模式
type PermissionMode = "plan" | "auto" | "default";

const PERMISSION_DECISION_TABLE: Record<PermissionMode, Record<string, "allow" | "ask" | "deny">> = {
  plan: {
    "run_read": "allow",
    "run_write": "ask",     // 全部 ask
    "run_bash": "ask",
    "rm -rf /": "deny",     // 黑名单永远 deny
  },
  auto: {
    "run_read": "allow",
    "run_write": "allow",    // auto 不 ask
    "run_bash": "allow",
    "rm -rf /": "deny",     // 黑名单仍然 deny
  },
  default: {
    "run_read": "allow",
    "run_write": "ask",     // 危险 ask
    "run_bash": "allow",     // 安全 allow
    "rm -rf /": "deny",
  },
};

function checkPermission(mode: PermissionMode, toolName: string, args: any): "allow" | "ask" | "deny" {
  // 先查黑名单
  if (isBlacklisted(toolName, args)) return "deny";
  // 再查模式表
  return PERMISSION_DECISION_TABLE[mode][toolName] ?? "ask";
}

模式 9 · Observer (观察者模式) - 改造版

出现频率 : TranscriptStore 事件流 / CacheDebugger hash 追踪 / Eval trace。

解决什么问题 : 调试时需要"回放整个 session"; 审计 需要"LLM 实际看到什么"; eval 需要"case 跑时发生了什么"。 没事件流, 这些都做不到 — 只能看 LLM 最终回复, 信息全丢。

原理 : 经典 观察者模式 的简化版 — 系统 关键操作时 (用户消息 / 助手消息 / 工具调用 / 错误恢复) append 一条事件到 事件流 。 事件流是 append-only 的 (不能改/删), 每个事件有 单调递增序号 , 跟 history sequence 对齐, 调试时按序号回放。 简化版没"订阅者", 只有 "存下来, 后续读"。

带来的好处 : 调试时按事件序号回放, 整个 session 一 清二楚; transcript 跟 LLM 看到的 messages 分离, transcript 保留所有内部 metadata (不被 P0/P1/P2 压缩); 审计能信任事件 流 (append-only, 不可篡改)。

模式 10 · Atomic Write (原子写)

出现频率 : 1 个核心工具 atomicWriteJsonSync() , 被所有持久化模块复用。

解决什么问题 : 进程突然被杀 (Ctrl+C / OOM / 断电) 时, 如果直接 fs.writeFileSync() 写到一半就留半截 JSON, 下次启动 JSON.parse 抛 SyntaxError, 整个 Task / Schedule / Memory 系统不可用。

原理 : POSIX 文件系统保证 rename() 是 原子的 — 要么旧文件保留, 要么新文件替换, 不会 有"两个都损坏" 的中间状态。 利用这个性质: 先写临时文件 ( tmpPath ), 写完 fsync 强制刷盘, 然后 rename(tmpPath, finalPath) 原子替换。

带来的好处 : 断电 / 进程被杀不会留半截 JSON; 所有 持久化模块用同一函数, 不会"两处实现漂移"; 写完直接返回 (同步), 不需要异步复杂度。

模式长什么样

// 教学简化版
import * as fs from "fs";
import * as path from "path";

export function atomicWriteJsonSync(filePath: string, data: unknown): void {
  // 1. 写临时文件 (含 pid + timestamp 避免并发写冲突)
  const tmpPath = `${filePath}.tmp.${process.pid}.${Date.now()}`;
  const json = JSON.stringify(data, null, 2);

  const fd = fs.openSync(tmpPath, "w");
  try {
    fs.writeSync(fd, json);
    fs.fsyncSync(fd);   // 强制刷盘, 不留 OS 缓存
  } finally {
    fs.closeSync(fd);
  }

  // 2. 原子 rename 替换 (POSIX rename 是原子的)
  fs.renameSync(tmpPath, filePath);
}

GitHub · atomicWriteJsonSync 原子写工具 (L1)

关键点:

  1. tmp + rename : 写到 tmp, rename 替换。 rename 在 POSIX 系统是原子的, 要么全成功要么全失败, 不会有"两个都损坏"。
  2. fsync 强制刷盘 : 不调的话数据在 OS 缓存, 断电丢。 fsync 告诉 OS "现在就写到磁盘"。
  3. tmp 路径含 pid + timestamp : 避免并发写覆盖 (多个进程同时写, 临时文件名不冲突)。

所有持久化模块都用这个工具

模块 用原子写

ScheduleStore schedule.json / occurrence-<id>.json
MemoryStore 每个 memory 一个 JSON
OutputStore 每个 output 一个文本 / JSON

所有持久化模块都 import src/atomic-write.ts 的同一函数, 不自己实现, 避免两处实现漂移。

陷阱 · 忘记 fsync
只写 tmp + rename, 不调 fsync — 数据在 OS 缓存, 断电就丢, 原子写等于不原子。 正确: 每次写完必须 fsyncSync 。
陷阱 · 多个进程写同一文件
进程 A 写 tmp.foo.pidA, 进程 B 写 tmp.foo.pidB, rename 时 互相覆盖。 正确: tmp 路径 必须含 pid + timestamp , 避免冲突。

模式 11 · Reminder 模式 (稳定前缀外的动态状态)

出现频率 : TODO reminder / Memory reminder / Skill hint / Async Run notification / Task Group progress。

解决什么问题 : 状态 (TODO / memory / skill) 需要 LLM 看到, 但拼到 system prompt 会破坏 prompt cache (每轮都变, 缓存 失效); 走 messages 又会被 LLM 误以为是用户真实输入。

原理 : 利用 LLM 对 <system-reminder> 标签的 结构化识别能力 — LLM 知道这是"系统注入 的状态", 不是用户输入。 把状态包在 reminder 里, 走 user message 末尾, 既不进 stable prefix (不破坏 cache), 又能被 LLM 正确识别 (不会被当成 user 输入)。

带来的好处 : 状态动态变化不破坏 cache (走 messages 增量 价格); LLM 不会混淆 reminder 和用户输入 (统一标签); 空 reminder 不注入, 不污染 (空字符串直接跳过)。

模式 12 · Cache-friendly Layout (缓存友好布局)

出现频率 : 整个项目最重要的隐式架构。 每个写 system prompt / tools / messages 的地方都隐式遵守。

解决什么问题 : LLM provider (Anthropic 1/10, OpenAI 1/2) 的 prompt cache 按"前缀匹配" 工作, 任何"小修改" 让整个 cache 失效; system prompt 里塞状态 = cache 命中率归零, 成本涨 5-10 倍; 用户想"便宜" 跑 harness, cache 是必选项。

原理 : 把 prompt layout 严格分成 稳定前缀 (system prompt + tools, 写后不改) + 动态 tail (history + reminder, 每轮变)。 稳定前缀进 cache 享受折扣, 动态 tail 不进 cache 但只算增量价格。 用 cache-debug 模块 算本地 stable hash, 验证"我们没破坏稳定前缀"。

带来的好处 : cache 命中率最大化, 成本降 5-10 倍; 业务模块不需要关心 cache, 只要遵守"system prompt + tools 稳定" 这条规则; 调试时 cache-debug 报"stableChanged=true" 立刻知道哪次操作破坏了 cache。

模式 13 · 错误分类 + 恢复动作 (Error → Action)

出现频率 : Recovery (7 类错误 → 4 个动作) / Permission (黑名单 deny / ask / allow) / ToolResult 错误 (business error vs throw)。

解决什么问题 : 错误种类多 (network / rate_limit / credential / context_length / output_interrupted / unknown), 全部 retry 浪费 (credential 重试 100 次也通不过), 部分 retry 也不行 (context_length 重试还是超限); 不同错误要不同处理; 业务代码"看到错误就 retry" 简单但低效, 看到错误就 throw 让 loop 崩也很糟。

原理 : 经典 错误分类 思想 — 按错误的 性质 (不是按消息文本) 分类, 每类对应一个 恢复 动作 。 Adapter 翻译 provider 特定错误码 → 内部统一 LLMErrorKind , 恢复逻辑只看内部类型。 配合 状态 上限 防止无限循环。

带来的好处 : 业务代码"catch → 分类 → 选动作" 路径 清晰; retry 不再"一刀切"; 上限保护防止永远卡住; provider 错误码不污染业务。

模式长什么样

// 教学简化版, Recovery 7 类错误 → 4 个动作
type LLMErrorKind = "network" | "rate_limit" | "credential" | "quota" | "context_length" | "output_interrupted" | "unknown";

type RecoveryAction =
  | { kind: "backoff"; delayMs: number }
  | { kind: "compact" }
  | { kind: "continue" }
  | { kind: "fail"; reason: string };

const ERROR_TO_ACTION: Record<LLMErrorKind, () => RecoveryAction> = {
  network: () => ({ kind: "backoff", delayMs: 1000 * 2 ** attemptCount.backoff }),
  rate_limit: () => ({ kind: "backoff", delayMs: 1000 * 2 ** attemptCount.rate }),
  context_length: () => ({ kind: "compact" }),
  output_interrupted: () => ({ kind: "continue" }),
  credential: () => ({ kind: "fail", reason: "API key 错误" }),
  quota: () => ({ kind: "fail", reason: "配额用尽" }),
  unknown: () => ({ kind: "fail", reason: "未知错误" }),
};

// Adapter 把 provider 错误翻译成 LLMErrorKind
function translateAnthropicError(err: any): { kind: LLMErrorKind; message: string } {
  if (err.status === 429) return { kind: "rate_limit", message: err.message };
  if (err.status === 401) return { kind: "credential", message: err.message };
  if (err.status === 400 && err.message.includes("context_length")) {
    return { kind: "context_length", message: err.message };
  }
  return { kind: "unknown", message: err.message };
}

模式 14 · Concurrent Limit (并发限制)

出现频率 : Async Run (max 3) / LLM retry (max 5) / Schedule overlap (allow/skip) / Task 依赖图 (BFS 限深度)。

解决什么问题 : 资源是有限的 (本地 CPU / 内存 / 文件 句柄 / provider 配额), 无限并发 = 资源耗尽 = harness 整体挂; 启动第 N+1 个 run 应当 立刻 拒绝, 让调用方 (LLM) 看到错误后自己决定 (等 / 取消 / 换工具), 而不是默默 排队把系统拖垮。

原理 : 经典 信号量 的简化版 — 在 manager 内部维护一个计数器, 启动时检查 + 递增, 完成后递减。 配对操作确保不会泄漏计数。 上限是经验值 (本地资源默认 3, CI 可 调到 5)。

带来的好处 : 资源受保护 (不会耗尽); 调用方立刻知道 拒绝原因 (而不是默默排队); 配对操作保证计数不泄漏 (启动 + 完成后一对, 缺一就错)。

模式 15 · Identity Check (id 与目录名同步)

出现频率 : TaskStore / ScheduleStore / MemoryStore / OutputStore 都有"目录名 = 数据 id" 的身份校验。

解决什么问题 : 用户可能手动改目录名 (重命名 / 移动) 或 改 JSON 文件内的 id 字段; 改完后目录名和 id 字段不一致, 读取 时按目录名查到但 id 是另一个, 数据错乱; 不校验, 后续操作 (按 id 删除 / 更新) 找不到对象, 静默失败。

原理 : 经典 防御性编程 的"双重保险" — 目录名和内容 id 字段必须一致, 不一致就是数据被外部破坏过。 读取 时校验, 不匹配就 skip + warn, 不自动修复 (自动修可能误改用户 真实意图)。

带来的好处 : 检测到外部破坏时优雅降级 (skip + warn), 不让一个坏 group 影响其他 group; 用户看到 warn 知道"我的数据 被改了"; 不自动修, 避免"猜错用户意图" 的灾难。

模式 16 · Test Factory (测试工厂)

出现频率 : 仓库里所有 test 文件都复用同一套 fake factory, 例如 FakeLLMClient、 ScriptedLLMClient、ScriptedTerminal、Fake ToolRegistry。

解决什么问题 : 测试不能依赖真实 LLM — CI 通常无法调用 OpenAI API; 测试需要精确控制 LLM 返回, 才能断言 messages 顺序等业务行为, 而不是"赌 LLM 这次刚好给正确答案"; 直接 mock 真实 SDK 又太重, 例如 openai SDK 有 10+ 个方法, mock 起来容易漏掉关键调用。

带来的好处 : 业务代码零改动即可注入 fake 进行测试; 断言可以精确到"LLM 看到了怎样的 messages 顺序"; 不依赖网络和 API key; 运行毫秒级, 不用等待真实 LLM 响应。

模式 17 · Stable Identity (id 不变)

出现频率 : 仓库所有"长生命周期" 数据 (Task / Memory / Schedule / Async Run / Skill) 都有 stable id。

解决什么问题 : 跨 run / 跨 session 引用"之前创建的 那个 task" 时, 如果 id 随内容变化, 引用就断了; 长生命周期数据 需要"id 永远不变" 才能被外部持续引用 (e.g. "task_abc12345 blockedBy task_def67890" 永远要有效); UUID 难读 / 易打错, LLM 在 tool_call 里引用也容易错。

原理 : 经典 内容寻址 (Content Addressing) 的简化版 — id = hash(原始创建内容), 内容改了 id 不变, 改名 / 改 status / 改 owner 都不变。 用短 hash (8 字符) 代替 UUID, LLM 易读易引用。 加前缀分类 ( task_ / tg_ ), 一眼看出类别。

带来的好处 : 跨 run 引用永远有效; LLM 在 tool_call 里易读易打; 分类前缀防止混淆; 不需要集中 ID 生成器 (内容 哈希本身就是 id)。

模式 18 · Transcript-First (transcript 优先)

出现频率 : 调试 / eval / audit 都靠 transcript。 几乎所有关键操作都 append transcript event。

解决什么问题 : 调试时需要"回放整个 session"; 审计 需要"LLM 实际看到什么 + 实际做了什么"; history 被 P0/P1/P2 压缩后会丢信息, transcript 必须保留全部; "我上一轮到底为什么 调了这个工具" 这种问题, 不靠 transcript 答不上。

原理 : 经典 审计日志 (Audit Log) 思想 — 所有关键操作 (用户消息 / 助手消息 / 工具调用 / 错误恢复) 都 append 一条到 事件流 , 不修改 / 不删除。 事件流跟 history 分离, history 是 LLM 看到的, transcript 是 调试/审计看到的。 transcript 保留所有内部 metadata, 不被压缩 影响。

带来的好处 : 调试时按事件序号回放, 整个 session 一清二楚; transcript 不被 P0/P1/P2 压缩, 长期保留完整记录; 审计能信任 (append-only, 不可篡改); 跟 LLM 看到的 messages 分离, 不会 污染 LLM 视角。

模式长什么样

// 教学简化版
class Agent {
  async run(query: string) {
    this.transcript.append({ kind: "user_message", content: query });
    for (;;) {
      const response = await this.llm.chat({ ... });
      this.transcript.append({ kind: "assistant_message", content: response.content, tool_calls: response.toolCalls });
      if (response.toolCalls.length === 0) break;
      for (const call of response.toolCalls) {
        const result = await this.tools.invoke(...);
        this.transcript.append({ kind: "tool_result", toolName: call.name, result: result.content });
      }
    }
    return response.content;
  }
}

出现频率 : Eval Runner 的核心抽象 — 不依赖真实 组件就能跑 case。

模式 21 · No Catch Throw (业务错误不 throw)

出现频率 : 工具执行 / 权限拒绝 / 子 agent 失败 / judge 输出 解析失败, 全部用 error: true 或专属 status 返回, 不 throw。

解决什么问题 : throw 让整个 loop 崩, messages 序列断裂 (assistant 调了 N 个工具, 但 N 个 tool message 都没写, LLM 看到 错位); LLM 看不到任何错误信息, 不能继续推理; 用户看到"500 错误" 一头雾水, 不知道"为什么失败、怎么修"。

原理 : 业务错误是"预期的失败", 不是"代码 bug"。 业务 错误应该作为"正常返回" 出现在 LLM 视野里 (通过 tool message), 让 LLM 继续推理 ("啊, 文件不存在, 我换路径"); 只有真正的 harness bug (配置错误 / 内存溢出 / 内部 invariant 违反) 才用 throw 上抛 让 recovery 处理。

带来的好处 : 业务错误让 LLM 看到, 继续推理; loop 不崩, messages 序列完整; 用户看到清晰的错误信息; recovery 只处理 真正的 harness 错误, 不被业务错误打扰。

// 2. 权限拒绝
if (!permissionManager.check(...).allow) {
  history.add({ role: "tool", tool_call_id: call.id,
    content: "Permission denied by user." });  // 不 throw
  continue;   // 跳过 invoke
}

// 3. 子 agent 失败
async function runSubagent(args) {
  try {
    return { content: await childAgent.run(args.task) };
  } catch (e) {
    return { content: `子任务失败: ${e.message}`, error: true };  // 不 throw
  }
}

// 4. judge 输出解析失败
function parseJudgeOutput(raw: string): EvalJudgeResult {
  try { return JSON.parse(raw); } catch {}
  // 4 层降级都失败, 返回 judge_failed fallback
  return { enabled: true, passed: false, score: 0,
    summary: "judge_failed", problems: [...], ... };
}

GitHub · PermissionManager 业务错误不 throw (deny 写 tool message) (L1)

关键点:

  1. 业务错误用 error: true 表达 : LLM 收到后能继续推理 ("啊, 文件不存在, 我换路径")。 不 throw 让 loop 崩。
  2. throw 只用于真正的 harness 错误 : 配置错误 / 内存 溢出 / 内部 invariant 违反, 这种 throw 是合理的。 业务错误绝不能 throw。
  3. fallback 不让系统挂 : judge 解析失败 fallback, async run 超时 fallback, 权限拒绝 fallback, 所有失败都有 "降级路径" 而不是"死路"。

常见陷阱

陷阱 · 把所有 throw 都 catch 住
try/catch 包一切 throw 当 fallback — 把"代码 bug" 和"业务 错误" 混为一谈, harness 错误也被吞掉, 排查难。 正确: 业务 错误显式返回 error: true, harness 错误让它 throw 上抛。
陷阱 · fallback 信息太简
fallback 只返回 "Error" — LLM 看到"Error" 不知道下一步 怎么办。 正确: 返回具体错误 + 建议下一步 (e.g. "Permission denied by user. 请重新决定或换其他方案。")。

模式 22 · Idempotent Operations (幂等操作)

出现频率 : load_skill 多次激活 / Async Run 重试 / Schedule 重新触发 / Memory create (允许重复 id, 覆盖更新)。

解决什么问题 : 网络抖动导致 LLM 重试时, 同一个 load_skill 被调 2 次, 应该不重复激活; 用户误操作"按 2 次 同一按钮" 应该不创建 2 个 run; Schedule tick 重新触发时, 不应 该创建重复 occurrence; Memory 用相同 id 创建时, 应该是"更新" 而非"创建"。

原理 : 经典 幂等性 (Idempotency) — 操作 重复 N 次的效果 = 操作 1 次的效果。 用"按 id 去重" 实现: 重复 id 时静默接受 (返回已有对象) 而非报错, 或者用 Set 存储已激活 名字 (重复激活时啥都不做)。

带来的好处 : 重试 / 误操作不会产生副作用; transcript 里出现 2 次同操作, 但 effect 只有 1 次 (审计友好); 业务代码 不用"if 第一次" 这种状态判断, 始终假设"幂等执行"。

GitHub · createAsyncRunManager 幂等操作 (重复 runId 拒绝) (L1)

关键点:

  1. 重复操作不产生副作用 : 调两次 load_skill("react") 和调一次, 效果一样。 不写 error, 不写 warning, 安静地幂等。
  2. 幂等 ≠ 拒绝 : 不要 "重复 id 报错", 应该 "重复 id 静默 接受"。 报错会让 LLM 困惑, 静默接受让它下次继续。
  3. 幂等是 audit-friendly : transcript 里出现 2 次 load_skill("react"), 但实际只激活 1 次。 审计时不会误判"调了 2 次" — 看 effect 而非 call。

常见陷阱

陷阱 · 把幂等做成"重复报错"
"runId 已存在" 当 error 返回 — 实际是同一个 run, 不该报错。 正确: 重复 runId 时静默返回已有对象, 或者拒绝但用 error: true 而不是 throw。
陷阱 · 幂等判断写在业务层
业务层每次调之前都 "if (already exists)" — 重复逻辑散落各 处。 正确: 幂等判断写在 manager/store 内部, 业务层无脑调 即可。
