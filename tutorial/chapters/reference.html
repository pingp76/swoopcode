先把自己放进一个真实 query：没有 Reference Pack 时，当前 agent 会在哪里卡住。

再故意设计一个朴素方案，观察它如何破坏 loop、状态或安全边界。这个失败比“正确答案”更重要，因为它能帮你写出更好的重建 prompt。

本章场景

你准备让另一个 coding agent 从零重建类似项目。此时不该把 20 多章原文全复制过去，而应该提炼：目标、模块边界、接线顺序、corner case、验证命令。

Reference 的任务就是把这些材料归档成 Prompt Pack 和验证手册。

先试一个朴素方案

朴素方案是把所有章节 Prompt Card 直接拼成一个超长 prompt，一次性让 coding agent 实现全部功能。

这个朴素方案的问题在于：它把 Reference Pack 当成普通实现细节，而不是 agent harness 的一条稳定边界。 

prompt = chapter00 + chapter01 + ... + chapter15 + topicA + topicB;

朴素方案为什么不够

任务过大
一次性实现全部功能会打破递进设计，也让 review 和测试失去边界。
验收模糊
超长 prompt 往往没有分阶段验收，失败时不知道该回滚哪里。
教学节奏丢失
本项目的价值在于逐步长出 agent，而不是一口气堆完功能。

回到 Agent Loop

Reference Pack 接入 loop 的位置是：读完整套教程后，把分散的概念整理成可复用 prompt 和验收清单。理解这个位置，比记住具体函数名更重要。

Reference 不接入 runtime loop，它服务学习者的构建 loop：读一章、写 prompt、实现、验证、复盘，再进入下一章。

从原理落到代码

到这里我们已经知道 Reference Pack 为什么存在、插在 loop 的哪里，以及朴素方案会怎样失败。下面才进入实现层：只抓住能固定架构的接口、状态边界和接线路径。

读源码时不要从文件顶部一路啃到底。更有效的顺序是：先找入口函数，再找它读写的状态，最后看测试如何证明这些状态变化是真的发生了。

一次真实运行 walkthrough

  1. 选择一个章节，例如工具调用。
  2. 提取 Prompt Card 中的目标、模块、接线和边界。
  3. 让 coding agent 只实现这一章，不提前做后续章节。
  4. 运行该章对应的确定性测试和手工 query。
  5. 阅读源码地图，确认实现边界没有漂移。
  6. 更新 summary 或教程，再进入下一章。

设计直觉

Reference 要帮助学生形成架构意识：不是背函数名，而是知道每个能力插在 loop 的哪里，状态属于哪里，怎么验证。

Prompt Card

Prompt Card

目标：把整套教程整理成可分阶段执行的 Prompt Pack 和验证手册。

场景：学生读完教程后，用自己的话指导 coding agent 逐章重建项目。

模块：教程章节、prompt pack、术语表、验证清单。

接线：每张卡片都指向源码入口、测试入口和文档同步位置。

边界：不要一次性让 agent 实现全部章节；不要把 prompt 写成使用手册。

例外和 corner case：章节依赖、跨模块边界、live 测试 opt-in、文档和实现漂移。

验证：抽取任意章节 Prompt Card，看是否能指导另一个 agent 生成合理改动计划。

源码入口：doc/web-tutorial-plan.md、tutorial/chapters/*、src/eval/README.md。

文档同步：Reference 本身就是教程收束页。

容易踩坑

Design Trap

常见错误：把教程当 API 手册读。

为什么错：本教程的目标是训练架构 prompt 能力，而不是记住工具命令列表。

正确做法：每章都追问：新增能力插在 loop 哪里，状态在哪里，如何验证。

怎么验证：不看源码细节，也能用自己的话写出本章实现 prompt。

  • 一次性实现所有章节。
  • 只复制 Prompt Card 不理解边界。
  • 把 live smoke 当默认 CI 门禁。

如何验证

Validation

手工 query：选一章，口述场景、设计、边界、验证，再看正文补漏。

确定性测试：检查每章都有 Prompt Card 和验证项。

集成观察：站点导航、页内目录、上一章/下一章都可用。

失败信号：学生只能复述功能列表，无法说明 loop 接入点。

  • 每章至少有场景、loop 位置、源码地图、Prompt Card、验证。
  • Prompt Card 面向实现 agent 能力，不面向写教程。
  • 验证层级区分 deterministic/live/judge。

如果实现失败，先查哪里

  • 找不到概念时先查术语表。
  • 实现漂移时查 summary.md。
  • 测试失败时先看 trace。
  • 章节导航问题查 content.js。
