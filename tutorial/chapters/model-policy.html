专题 A · 模型差异

不同大模型不是只换模型名

Claude / GPT / Gemini 在 tool call 协议、cache 边界、错误码、输出 截断规则上都不一样。本专题讲 harness 应当如何吸收这些差异, 不让模型 切换变成"改十几处代码" 的体力活。

在第 15 章基础上改了什么

这一章不修改 harness 主结构, 而是在 src/llm-adapter.ts 下 收口所有模型差异。LLM Provider 维度拆 4 个 adapter (Anthropic / OpenAI / Google / 本地兼容), 每个 adapter 实现 LLMClient interface, 把模型差异 (tool call 协议 / cache 边界 / 错误码 / 截断 规则) 收口在 adapter 内部。Composition Root 通过 Config 字段选 adapter, harness 其他部分完全不感知。

1. src/llm-adapter.ts: 模型差异适配 2. src/llm-providers.ts: provider 解析 3. src/foundation-models.ts: 基座模型画像

作者怎么想的: 这一章的思考链

想清楚现象
harness 默认跑 Claude 3.5 Sonnet, 用户想换 GPT-4o, 改了一行 model 名, 结果 tool call 字段名不一样、cache 边界不一样、错误 码不一样, harness 立刻崩。现象是"模型切换像换引擎, 不是换轮胎"。
想反例
最朴素的反例是"在主代码里 if (provider === "anthropic") {...} else if (provider === "openai") {...}"。这有两个问题: 一是 主代码被 provider 分支污染, 难以维护; 二是新增 provider 时要 改十几处 if, 容易漏。
想接口和不变量
接口: interface LLMClient { chat(messages): Promise<AssistantMessage> }。 不变量三条: (1) 暴露给 harness 的是统一 AssistantMessage 形态 (role / content / tool_calls), 内部差异完全不外泄, (2) cache 边界由 adapter 内部实现, harness 不知道细节, (3) 错误码统一收敛到 6 种 LLMErrorKind (第 11 章), adapter 负责把 provider 特定错误翻译过来。
想怎么验证
同一 scenario 跑 4 个 adapter, 断言 trace 里 tool_calls 字段 完全一致 (虽然 provider 内部表示不同), LLMErrorKind 分类一致 (虽然 provider 错误码不同)。

先观察: 两段故意有气味的实现

观察 1 · 在主代码里 if provider

// 教学简化版, 仓库真实实现见 GitHub 永久链接
// 错误: provider 分支散落在主代码
async function run(query: string) {
  history.add({ role: "user", content: query });
  let assistant;
  if (config.provider === "anthropic") {
    assistant = await callAnthropic(messages);  // Anthropic 特有格式
  } else if (config.provider === "openai") {
    assistant = await callOpenAI(messages);  // OpenAI 特有格式
  } else if (config.provider === "google") {
    assistant = await callGoogle(messages);  // Google 特有格式
  }
  history.add(assistant);
  /* ... */
}

问:为什么不直接 if provider?

答:三件事同时坏掉 —— 污染: agent.ts 不该知道 provider 存在, 职责被越界; 维护: 新增 provider 要改十几处 if, 容易漏; 测试: 主代码被 provider 分支污染, 单元测试要 mock 多个 provider。

观察 2 · 错误码不翻译直接抛

// 教学简化版
catch (err) {
  // 错误: provider 错误码直接抛
  throw err;  // harness 看到的是 Anthropic 特定错误码
}

问:为什么不直接抛 provider 错误?

答:三件事同时坏掉 —— 跨模型: 切到 OpenAI 后, 第 11 章的 Recovery 模块不知道是 rate_limit 还是 rate_limit_exceeded; 抽象: harness 不该知道 provider 内部错误码, 这是 adapter 职责; 演化: provider 升级 SDK 改了错误码, harness 全挂。

4 个 adapter 收口的差异

不同 provider 在以下 4 个维度上有差异, adapter 负责收口:

1. tool call 协议

Anthropic
tool_use 字段, content 是数组 (text + tool_use 块)。
OpenAI
tool_calls 字段, content 字符串 + tool_calls 数组并列。
Google
functionCall 字段, parts 数组, 命名 functionCall。
本地兼容 (Ollama / vLLM)
通常模仿 OpenAI, 但 tool name 可能需要规范化 (例如不允许 ".")。

adapter 内部把 4 种表示都翻译成统一的 AssistantMessage.tool_calls。

2. cache 边界

Anthropic
prompt cache 4 个 breakpoint (system / messages / tools / 末尾), 自动 5 分钟 TTL, 命中按 1/10 计费。
OpenAI
自动 cache, 无显式 breakpoint, 命中按 1/2 计费 (o1 系列)。
Google
implicit cache, TTL 短, 命中按 1/4 计费。
本地兼容
无 cache, 所有 token 全价。

第 10 章的"稳定前缀 / 动态状态 / 自然增量" 布局在 Anthropic 效果 最好 (cache hit rate 最高), 在 OpenAI 中等, 在本地模型无效但也没坏处。 adapter 在 metadata 里返回 cache hit 状态, harness 记录到 cache-debug (第 10 章)。

3. 错误码

Anthropic
HTTP 429 (rate_limit) / 529 (overloaded) / 400 (context_overflow) / 500 (api_error)。
OpenAI
HTTP 429 / 503 / 400 / 500, 错误码字符串 "rate_limit_error" / "context_length_exceeded"。
Google
HTTP 429 / 503 / 400 / 500, 错误码字符串 "RESOURCE_EXHAUSTED" / "INVALID_ARGUMENT"。

adapter 翻译成 6 种 LLMErrorKind (第 11 章), Recovery 模块只认 LLMErrorKind, 不认 provider 错误码。

4. 输出截断规则

Anthropic
stop_reason === "max_tokens", content 截断在最后一个完整 token。
OpenAI
finish_reason === "length", content 截断在最后一个完整 token。
Google
finishReason === "MAX_TOKENS", 行为类似。

adapter 统一翻译成 finishReason === "length" + truncated === true, Recovery 模块 (第 11 章) 一致处理。

基座模型画像

同样是 "claude-3-5-sonnet", 也有 4 个版本 (claude-3-5-sonnet-20240620, claude-3-5-sonnet-20241022, ...), 不同版本 cache 行为 / 工具协议 / 错误码都略有差异。本章用 foundation-models.ts 维护一份 "基座模型画像", 描述每个具体版本的差异, harness 启动时根据 model 名选画像, 应用到 adapter。

反例梯度

新手错法 · A

常见错误:在 agent.ts 主代码里 if provider === "anthropic"。

为什么错:主代码被 provider 污染, 维护噩梦, 测试难写。

正确做法:4 个 adapter 实现 LLMClient interface, agent.ts 完全不感知 provider。

中级错法 · B

常见错误:provider 错误码不翻译直接抛。

为什么错:Recovery 模块看不懂 provider 错误码, 跨模型立刻挂。

正确做法:adapter 把 provider 错误码翻译成 6 种 LLMErrorKind, 统一收敛。

高级错法 · C

常见错误:hardcode "claude-3-5-sonnet" 的 cache 行为, 切到 GPT-4o 后不工作。

为什么错:cache 行为是 provider 特性, 不是 agent 假设。不同 provider 表现不同, 切模型后假设失效。

正确做法:从 modelProfile.supportsCache / cacheBreakpoints 读, 不 hardcode。

边界错法 · D

常见错误:本地模型跑 tool call 失败, 切到 Claude 之后才发现是 tool name 不合法 ("fs.read" 有点)。

为什么错:tool name 在不同 provider 规则不同 (Anthropic 允许 ".", OpenAI 不允许), harness 不该 hardcode 假设。

正确做法:tool registry 暴露 getNormalizedName() 给 adapter, adapter 在注册时做规范化。

如何验证 (本章 Validation 卡片)

Validation · model-policy 专题

4 个 adapter 暴露统一 LLMClient:每个 adapter 实现 chat(messages): Promise<AssistantMessage>, 内部 provider 表示完全翻译成统一形态。

provider 错误码翻译成 LLMErrorKind:fake provider 抛 429 错误, adapter 抛 LLMErrorKind === "rate_limit"。

切模型 = 改环境变量, 不改代码:设 ANTHROPIC_API_KEY 跑通, 切到 OPENAI_API_KEY 同样跑通, agent.ts 不变。

tool name 规范化:tool name 含 "." 时, OpenAI adapter 内部替换为 "_" (OpenAI 限制), 但 trace 里仍然 显示原始 name。

modelProfile 决定 cache 行为:fake profile supportsCache=false, harness 不调用 cache-debug tracker; supportsCache=true, harness 调用并记录 hit/miss。

回望第 00–15 章: 哪些原则在本章兑现了

  • Composition Root 唯一感知 provider:其他模块只看到 LLMClient interface。
  • 不变量在 adapter 内部兑现:cache 边界、错误码、截断规则, 全部 adapter 收口。
  • 配置即适配:切模型 = 改 env, harness 主体不变。
  • 事实与视图分离:modelProfile 是"事实" (provider 描述), adapter 是"视图" (实际调用)。

前瞻张力: 留给后续章节

多 provider 同时跑
未来 harness 允许同一个 run() 内调多个 provider (例如用 GPT-4 选 tool, 用 Claude 总结), 4 个 adapter 要能并存。
model profile 动态加载
新 model 版本发布时, foundation-models.ts 要更新, 跑一次 eval 验证 cache / tool / 错误码行为。
local model 协议差异
Ollama / vLLM / LM Studio 各自的 tool call 协议略有差异, adapter 抽象要能容下。
cost-aware adapter selection
不同 provider 价位不同, harness 应当能在"质量 vs 成本" 之间动态选 adapter (例如用户配置"省钱模式")。

本章小结

本专题给 harness 加了 LLM Adapter 层, 用 4 个 adapter 收口 provider 在 tool call 协议 / cache 边界 / 错误码 / 截断规则 上的差异。harness 其他部分完全不感知 provider, 切模型 = 改 环境变量。modelProfile 决定每个具体模型版本的行为, 不 hardcode provider 假设。这是 harness 跨模型可移植性的关键。
