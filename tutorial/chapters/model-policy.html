专题 A · LLM 选型策略层

Model Policy: 任务类型 → Model 推荐的策略层

第 15 章提过 LLM provider 概念 (OpenAI / Anthropic / 自部署 vLLM) + 5 个常见 model, 但 harness 怎么知道"这个任务该用 哪个 model"? 用户让 LLM 跑"5 天迁移" 该用 GPT-4o (质量) 还是 Qwen2.5-7B (成本)? 跑"5 分钟 git status" 该用 Claude Haiku (快) 还是 Claude Opus (贵)? 选错就是几倍账单浪费或 质量崩盘。 这一专题加 Model Policy 策略层: 5 种任务分类 (long_running / short_interactive / bulk_summarize / complex_reasoning / experimental) → 5 种 推荐 model, 配 --model-policy auto|fast|quality|budget 4 种策略切换, 用户在质量 / 速度 / 成本间选边。 v2 更新: Anthropic Messages 协议已经实现 (src/llm-adapter.ts 新增 createAnthropicMessagesAdapter), 5 个 model 画像里 GLM-5.2 / Claude Sonnet 4.5 / Claude Opus 4.5 等 3 个声明 preferred: "anthropic-messages", 走 Anthropic 协议; GPT-4o / Qwen 等走 OpenAI Chat Completions 协议。 同一份 history 同时支持 2 套协议, 协议差异全部收敛在 LLM 边界, agent.ts / 压缩 / 权限 / 工具执行不知道协议 存在。 读完后, 你能讲清"为什么不能 1 个 model 跑所有任务" + "auto 策略怎么工作" + "OpenAI 和 Anthropic 协议怎么共存" + "怎么扩展第 6 个 model"。

真实失败故事: 5 分钟任务跑了 2 小时, 账单 ×10

写代码之前, 先看一个真实痛点。 用户的 team 跑 1 个月 harness, 发现账单异常 — 明明只是跑"git status" 这种 5 秒任务, 实际 账单 ×10。

  1. 症状 1: 大材小用: 用户跑简单的"扫一下 src/ 找 TODO" 任务, harness 默认用 GPT-4o (贵的 model), 5 分钟任务花 $0.50, 实际 Qwen2.5-7B 跑同样任务 30 秒花 $0.005。 100 倍 浪费。
  2. 症状 2: 小材大用: 用户跑"5 天迁移 500 文件" 用 Qwen2.5-7B (便宜的 model), 跑 1 周, 质量崩盘, 70% 转换 错误, 团队 leader 花了 2 周手动修正。 实际该用 GPT-4o 跑 1 天 质量好。
  3. 症状 3: 切换靠人: team 想"小任务用便宜, 大任务 用贵", 每次手动改 LLM_MODEL=..., 容易忘, 出现 "昨天我手动改了, 今天又跑贵 model" 的混乱。
  4. 真问题: 缺少Model Policy 策略层 — harness 知道任务类型 (短 / 长 / 简单 / 复杂), 应该自动选 合适 model, 不该让用户每次手动配。 同时给用户 4 种 策略 (auto / fast / quality / budget) 切 偏好, 满足"我今天要省钱" / "我今天要质量" 的临时切换。

朴素想法 1: "用最贵的 model 跑所有任务?" 错。 大材小用, 账单 ×100, 简单任务浪费钱。 该用便宜的快 model。

朴素想法 2: "用最便宜的 model 跑所有任务?" 错。 小材大用, 质量崩盘, 团队返工, 实际更贵。 该用贵的慢 model 做关键任务。

正确做法: 加 Model Policy 策略层 — harness 内部按任务类型分类 (5 种), 每种推荐 一个 model, 用户在 4 种策略 (auto / fast / quality / budget) 切偏好, 临时改变单次任务的 model。 这是 Reference 章节 "模式 8 · Strategy 策略模式" + "模式 4 · 依赖注入" 的联合应用。

先梳理语言: 8 个核心概念是什么

上面看到痛点和朴素方案, 这一节先给词表。 Model Policy 这一专题用了 8 个核心概念, 后面 14 个小节会 反复出现, 第一次看容易混 (provider / protocol / adapter / model profile / task type / strategy / fallback chain / compatible provider ...)。 v2 加了 Anthropic 协议后概念 数从 5 个涨到 8 个, 这一节先梳理一遍, 后面 读各小节脑子里有锚点。

8 个概念分 3 组:

1. provider 层 (3 个, 决定连哪个 endpoint)

  • LLMProviderId: provider 的枚举类型, v2 加了 anthropic_compatible, 现有 6 个: openai_compatible / anthropic_compatible / minimax_cn / kimi_platform_cn / kimi_code_cn / zhipuai_cn。 每个 providerId 对应 1 个 endpoint 配置 (apiKey + baseURL + 模型名)。
  • LLMProviderProtocol: provider 的原生传输 协议, 2 种: openai-chat-completions (OpenAI 兼容) / anthropic-messages (Anthropic Messages)。 决定 LLM 调哪个 adapter: OpenAI 协议走 createOpenAIChatCompletionsAdapter, Anthropic 协议 走 createAnthropicMessagesAdapter。 这 2 个 adapter 接口对称, 暴露同样的 buildRequest / parseNonStreamingResponse / parseStreamingEvent / finishStreaming 4 个方法。
  • FoundationModelProfile: 1 个 model 的完整 画像, 包含 id / displayName / provider / 协议偏好 (preferred + fallbacks + implemented) / 上下文窗口 / 思考 / 推理 / 工具 / 缓存 / 模态 / 优化提示 / 已知怪癖。 1 个 model = 1 个 profile, 存 src/foundation-models.ts 的 modelProfiles 数组, 启动时加载。

2. 任务 / 策略层 (3 个, 决定用哪个 model)

  • TaskType: 5 种任务特征枚举: long_running (1 小时+) / short_interactive (1-10 分钟) / bulk_summarize (大批量) / complex_reasoning (10-60 分钟高难度) / experimental (新功能试错)。 LLM 调 agent.run(taskType: TaskType) 显式声明, harness 不启发式猜。
  • Strategy: 4 种用户偏好枚举: auto (按任务类型查表) / fast (无脑 Haiku) / quality (无脑 Opus/o1) / budget (无脑 Qwen/Llama 本地)。 CLI 用 --model-policy auto|fast|quality|budget 切换, 也可以 REPL 里 /m fast 临时改。
  • FallbackChain: model 不可用时 (rate limit / 服务挂) 的降级链, 静态枚举: {gpt-4o: [claude-sonnet-4-5, claude-haiku-4-5], claude-opus-4-5: [gpt-4o, claude-sonnet-4-5], ...}。 链 可以跨协议 (OpenAI 失败 → 试 Anthropic 协议跑 Claude), 因为 fallback 试的是 adapter.call(), 不绑协议。

3. 协议细节 (2 个, v2 新加)

  • compatibleProviders: 1 个 model 画像可以 被多种 provider 复用。 典型场景: GLM-5.2 原生 provider 是 minimax_cn 走 OpenAI 协议, 但 用户可以显式配 LLM_PROVIDER=anthropic_compatible 让 GLM-5.2 走 Anthropic 协议 (因为 minimax 起了 Anthropic 兼容 网关)。 字段存在 FoundationModelProfile.compatibleProviders, resolveFoundationModelProfile 校验时用 profileSupportsProvider 检查 primary + compatible 列表。
  • AnthropicMessagesAdapter: v2 新加的协议 适配器, 把内部 OpenAI-shaped history 翻译成 Anthropic Messages 协议 (system 是顶层字段, 不在 messages 里; tool use 块独立 type, 不用 function_call; usage 字段 cache_read / cache_creation 分开; max_tokens 固定, 不用 max_completion_tokens 兼容逻辑)。 4 个方法: buildRequest / parseNonStreamingResponse / parseStreamingEvent / finishStreaming, 跟 OpenAI adapter 对称。

具体例子: "team 想从 OpenAI 切到 Anthropic 跑 GLM-5.2" 的完整流程

  1. 看 model 画像: 看 src/foundation-models.ts 里 GLM-5.2 画像, 发现 provider: "minimax_cn" + compatibleProviders: ["anthropic_compatible"] + protocol.preferred: "anthropic-messages" + protocol.implemented: ["anthropic-messages", "openai-chat-completions"]。 意味着用户可以选 anthropic 协议跑 GLM-5.2, 也保留了 OpenAI fallback。
  2. 用户改环境变量: export LLM_PROVIDER=anthropic_compatible + export LLM_BASE_URL=https://minimax-anthropic-gateway.com + export LLM_MODEL=GLM-5.2 + export ANTHROPIC_API_KEY=sk-...。
  3. resolveLLMProviderConfig 解析: 看到 provider=anthropic_compatible, 返回 ResolvedLLMConfig, protocol="anthropic-messages"。
  4. resolveFoundationModelProfile 校验: 用 profileSupportsProvider(profile, "anthropic_compatible") 检查 GLM-5.2 允许 anthropic_compatible (在 compatibleProviders 列表里), 校验通过, 拿到完整 profile。
  5. RuntimePolicy 选定 adapter: 看 profile.protocol.preferred = "anthropic-messages", 创 createAnthropicMessagesAdapter(policy)。
  6. LLM 调时: agent.run() 调 llmClient.chat(...), 看到 policy.protocol.selected === "anthropic-messages", 走 chatAnthropicMessages 路径, 用 anthropicClient.messages.create(...) 发请求。
  7. Anthropic 协议适配: adapter 把内部 OpenAI-shaped history 翻译成 {system, messages, tools, max_tokens}, tool use 块独立 type: "tool_use", 不用 function_call。
  8. 响应回流: Anthropic 返回 content blocks (text / tool_use), adapter 解析回 LLMResponse, agent 继续。 全程 agent.ts / 压缩 / 权限不知道协议存在。

关键观察: 用户只改 3 个环境变量 (provider + baseURL + apiKey), 不改任何业务代码, 就能从 OpenAI 协议切到 Anthropic 协议。 这就是为什么协议差异 "全部收敛在 LLM 边界" — 上层永远只看 OpenAI-shaped history。

看完这 8 个概念和具体例子, 后面 14 个小节 (5 种任务类型 / 4 种策略 / 默认映射表 / auto 策略 / 3 个 override / execution profile / 优雅降级 / cost 估算 / 主循环集成 / fake test / 常见误解 / 反例梯度 / Validation) 都在讲这 8 个概念的某个 细节。 看到陌生名字, 翻回这节找锚点。

v2 新增: Anthropic Messages 协议怎么共存

场景: v1 之前 harness 只支持 OpenAI Chat Completions 协议, 想接 Anthropic Claude 只能让用户跑 "Anthropic 兼容 OpenAI 的网关" (afoo / one-api), 多一层 转发。 v2 直接在 src/llm-adapter.ts 实现 Anthropic Messages 协议, 用官方 @anthropic-ai/sdk 客户端, 不再需要第三方网关。

朴素想法 1: "在每个调 LLM 的地方加 if provider === 'anthropic' ? 错。 agent.ts / 压缩 / 权限 / 工具执行 几十个 调 LLM 的地方, 每个都加 if = 协议逻辑污染业务, 换 provider 改一片。 正确做法是协议差异全部收敛在 LLM 边界 — agent.ts 只看 OpenAI-shaped history, 翻译交给 adapter。

朴素想法 2: "把内部 history 改成 Anthropic 格式? 错。 改 1 处意味着改所有用 history 的地方 (agent / compressor / memory / tools / permission ...), 范围更大, 风险更高。 应该 保留内部 OpenAI-shaped history, 翻译只发生在 "调 LLM 那一刻"。

正确做法: 加 1 个 AnthropicMessagesAdapter, 跟 OpenAIChatCompletionsAdapter 接口对称, 暴露 同样的 4 个方法 (buildRequest / parseNonStreamingResponse / parseStreamingEvent / finishStreaming), LLM 客户端 根据 policy.protocol.selected 选一个。 看 src/llm-adapter.ts 第 414-455 行 Anthropic adapter: 

export function createAnthropicMessagesAdapter(
  policy: RuntimePolicy,
): AnthropicMessagesAdapter {
  return {
    buildRequest(input) {
      const converted = convertMessagesToAnthropic(input.messages);
      const request: AnthropicPreparedLLMRequest = {
        model: policy.model,
        messages: converted.messages,
        stream: policy.request.prefersStreaming,
        maxOutputTokens: policy.request.maxOutputTokens,
      };
      if (converted.system !== undefined) {
        request.system = converted.system;  // 关键差异: system 是顶层字段
      }
      if (input.tools !== undefined && input.tools.length > 0) {
        request.tools = input.tools.map(convertToolToAnthropic);
      }
      return request;
    },
    // ... 4 个方法对称, parseResponse / parseEvent / finishStream
    // 区别只是协议细节
  };
}

2 套协议的核心差异 (翻译时处理的 5 个点):

图 1 · OpenAI Chat Completions vs Anthropic Messages 关键差异
OpenAI Chat Completions
  • system 在 messages 里: messages: [{role: "system", content: "..."}]
  • tool_call 在 assistant message 里: assistant.tool_calls: [{id, type, function: {name, arguments}}]
  • tool 响应是独立 role: {role: "tool", tool_call_id, content}
  • max_tokens 字段名: max_completion_tokens (新) / max_tokens (旧)
  • usage 字段: prompt_tokens / completion_tokens
Anthropic Messages
  • system 是顶层字段: 不在 messages 数组, 单独 system: "..."
  • tool_use 是 content block: content: [{type: "tool_use", id, name, input}]
  • tool 响应是 user 消息的 content block: {role: "user", content: [{type: "tool_result", tool_use_id, content}]}
  • max_tokens 字段固定: max_tokens, 不需要兼容
  • usage 字段: input_tokens / output_tokens + 缓存字段 cache_read_input_tokens / cache_creation_input_tokens

真实场景: user 把 model 从 gpt-4o 切到 claude-sonnet-4-5

  1. 改配置: export LLM_PROVIDER=anthropic_compatible + export LLM_MODEL=claude-sonnet-4-5 + export ANTHROPIC_API_KEY=sk-ant-...。 3 个环境变量, 不改任何 TypeScript 代码。
  2. resolveLLMProviderConfig 解析: 拿到 ResolvedLLMConfig { provider: "anthropic_compatible", protocol: "anthropic-messages", apiKey, baseURL, model: "claude-sonnet-4-5" }。
  3. resolveFoundationModelProfile 校验: 拿到 Claude Sonnet 4.5 画像, protocol.preferred = "anthropic-messages" + protocol.implemented = ["anthropic-messages", "openai-chat-completions"]。
  4. 创 adapter: LLM 客户端看到 policy.protocol.selected === "anthropic-messages", 创 createAnthropicMessagesAdapter(policy), 调 anthropicClient.messages.create(...) 走 Anthropic SDK。
  5. history 翻译: convertMessagesToAnthropic 把内部 OpenAI-shaped history 翻译: system 消息从 messages 数组移到顶层 system 字段; assistant 的 tool_calls 拆成 content blocks (text + tool_use); tool 消息聚成 user 消息 的 tool_result blocks; 整个流发生在 LLM 边界, agent.ts 完全 不知道。
  6. 响应回流: Anthropic 返回 content blocks, adapter 解析回 LLMResponse (含 content / tool_calls / usage / cache 字段), agent 继续下一轮。 usage 字段把 cache_read_input_tokens 映射到内部 cache_hit_tokens, cache debug 模块正常工作。

关键不变量 (4 条):

  • 内部 history 永远是 OpenAI-shaped: agent / 压缩 / 权限 / 工具 / memory 只接触 1 套消息格式, 协议差异不 泄漏到上层。
  • adapter 接口对称: 2 个 adapter 都暴露 buildRequest / parseNonStreamingResponse / parseStreamingEvent / finishStreaming 4 个方法, LLM 客户端能用同一套代码路径调 2 个 adapter。
  • fallback 跨协议: FallbackChain 不绑协议, 试 adapter.call() 不绑具体 adapter, OpenAI 失败可以 fallback 到 Anthropic 协议跑 Claude。
  • cache debug 字段映射: OpenAI 的 prompt_tokens_details.cached_tokens ↔ Anthropic 的 cache_read_input_tokens, adapter 翻译时统一映射到 内部 cache_hit_tokens, 上层 cache debug 模块只看 1 个字段名。

v1 → v2 不变的事: 5 种任务类型 / 4 种策略 / 默认映射表 / auto 策略 / execution profile / 优雅降级 / cost 估算 / 主循环集成 / fake test 都不变, 这些是业务层 概念。 v2 只在 LLM 边界 (provider + adapter + protocol 字段) 加新东西, 业务层 0 改动。

5 种任务类型: 决定 model 推荐

用途: 不是所有任务都用同一种 model, 任务本身 有 5 种典型特征: 跑多长 / 多频繁 / 多复杂 / 多容错 / 多便宜。 按特征选 model 才是合理。

真实场景: 用户让 LLM 跑"5 天迁移 500 文件" 是 long_running (跑 1 周, 不能崩, 质量优先); "5 分钟 git status" 是 short_interactive (快进快出, 速度优先); "1 小时扫 100 个 PR 摘要" 是 bulk_summarize (大批量, 成本优先); "半天设计 复杂算法" 是 complex_reasoning (高难度, 质量 优先); "试试新模型的 beta 功能" 是 experimental (新东西, 容错优先)。

设计思想: 特征分类 + 静态枚举 — 5 种是固定枚举, 不允许"我自己加第 6 种" (保持简单); 每种 配推荐 model, 但允许策略 (auto/fast/quality/budget) 覆盖。 看 任务类型定义:

图 1 · 5 种任务类型的边界
long_running · 长时间任务
跑 1 小时以上, 不能崩, 质量优先。 推荐: GPT-4o / Claude Sonnet (高质量 + 稳定)。 例子: 5 天迁移, 全项目扫描。
short_interactive · 短对话
跑 1-10 分钟, 快进快出, 速度优先。 推荐: Claude Haiku / GPT-4o-mini (快 + 便宜)。 例子: git status, ls, 单文件改。
bulk_summarize · 批量摘要
跑 1 小时, 但任务重复, 成本优先。 推荐: Qwen2.5-7B / Llama-3.1-8B (本地便宜)。 例子: 100 个 PR 摘要, 1000 个文件总结。
complex_reasoning · 复杂推理
跑 10-60 分钟, 难度高, 质量优先。 推荐: Claude Opus / o1 (最强推理)。 例子: 算法设计, 架构决策, 复杂 bug 修复。
experimental · 实验性任务
用户主动选"试试新 model", 容错优先。 推荐: 最新 beta model (新功能多但可能崩)。 例子: "试用 GPT-5 预览版"。

实现细节: 任务类型在 LLM 调 agent.run() 时 声明 (传 taskType 参数), 不靠 harness 启发式猜测。 让用户 / 上层调度器 (Schedule / Async Run) 显式声明, 避免 "harness 猜错任务类型选错 model" 的尴尬。

4 种策略: 用户切偏好

用途: 5 种任务类型 → 1 个推荐 model, 但用户 经常想"今天我就要省钱" / "今天我就要质量"。 提供 4 种 策略 切偏好, 临时改变 model 选择。

真实场景: team 平时用 auto (自动按任务类型选); 临时跑"周末实验" 切 fast (全用便宜 model 跑快); 跑"重要 demo" 切 quality (全用最贵 model); 预算紧的月切 budget (全用最便宜 model, 质量可能差)。

设计思想: 4 种策略 + 覆盖机制 — Reference 章节 "模式 8 · Strategy 策略模式" 的应用。 4 种 策略固定枚举, 不可新增第 5 种; 每种策略决定"对每种任务类型 选什么 model"。 看策略定义:

图 2 · 4 种策略对 5 种任务类型的覆盖
auto · 自动 (默认)
按任务类型用推荐 model。 long_running → GPT-4o; short_interactive → Haiku; bulk_summarize → Qwen; complex_reasoning → Opus; experimental → 用户的 LLM_MODEL 环境变量。
fast · 全快
所有任务都用最快最便宜的 model。 全用 Claude Haiku / GPT-4o-mini。 速度快, 成本低, 质量可能差。
quality · 全质
所有任务都用最贵最强的 model。 全用 Claude Opus / o1。 质量最高, 速度慢, 成本高。
budget · 全省
所有任务都用最便宜的本地 model。 全用 Qwen2.5-7B / Llama-3.1-8B (本地 vLLM)。 成本最低, 质量可能崩。

实现细节: 策略用 --model-policy auto|fast|quality|budget CLI 切换, 也可以 /model-policy show 在 REPL 看当前 策略 + 5 种任务类型对应的 model。 策略选择后, 派生一个 modelResolver(taskType) → modelName 函数, agent 调 LLM 前查 resolver 选 model。

default 任务类型 → model 映射表

用途: 5 种任务类型 → 5 个推荐 model 是默认 表, 写在 src/execution-policy.ts 的常量。 表是可改的 (新 model 出, 改表), 不是 hard code 在 if/else 里。

真实场景: 2026 年 GPT-4o 是 long_running 推荐, 2027 年 GPT-5 出来后, 改 default 表 long_running: "gpt-5", 所有 harness 升级后自动用 GPT-5, 不改 agent.ts。

设计思想: 查表代替 if/else — Reference 章节 "模式 8 · Strategy 策略模式" 的延伸, "查表" 是策略模式的具体实现。 看 execution-policy.ts 里的 default 表: 

// 5 种任务类型 → 推荐 model (default 表)
const DEFAULT_MODEL_FOR_TASK: Record<TaskType, ModelName> = {
  long_running: "gpt-4o",
  short_interactive: "claude-haiku-4-5",
  bulk_summarize: "qwen2.5-7b",
  complex_reasoning: "claude-opus-4-5",
  experimental: process.env.LLM_MODEL ?? "gpt-4o",
};

实现细节: 表是 readonly, agent.ts 通过 createModelResolver({policy, table}) 拿 resolver, resolver 内部按 (policy, taskType) 查表。 表改 = 改常量, 不 改 resolver。 这是 Reference 章节 "模式 5 · Composition Root 组合根" 的应用 — 表在 Composition Root 注入, 运行时不变。

auto 策略: 怎么"自动"

用途: auto 策略是默认, 任务类型 → model 的映射完全靠 default 表, 不靠运行时判断。 "自动" 的 含义是"用户不用选 model, harness 按任务类型自动选"。

真实场景: team 跑 10 个不同任务 (5 分钟 git / 1 小时迁移 / 30 分钟设计 / ...), 不传 taskType 让 harness 猜 (这会错), 让上层 (Async Run / Schedule) 显式传。 显式声明, 不启发式猜测。

设计思想: 声明式, 不启发式 — harness 不启发式判断"这个任务是 long 还是 short", 启发式容易猜错; 任务类型由调用方显式 声明 (Async Run 默认 short_interactive, Schedule 默认 bulk_summarize, 用户手动 run 默认 short_interactive)。

实现细节: agent.run({query, taskType}) 接收 taskType, 内部调 modelResolver(taskType) 选 model。 默认 taskType = "short_interactive" (用户最常见的 "短对话" 场景)。 想覆盖, 显式传 taskType。

fast / quality / budget: 3 个 override

用途: 3 个策略 (fast / quality / budget) 是 全 override, 不管任务类型, 全部任务用同一种 model (最快 / 最强 / 最便宜)。 适合"用户明确知道今天要什么" 的场景。

真实场景: team 周五下午, 老板说"今天剩下的 任务都用最快 model, 质量无所谓, 跑快就行", --model-policy fast 一下, 所有后续任务用 Claude Haiku。 周一早上切回 auto。

设计思想: 3 个常量 model, 不查表 — fast / quality / budget 各自对应 1 个固定 model (最快 / 最强 / 最便宜), 不查表, 直接 hard-code 关联。 

const STRATEGY_TO_MODEL: Record<FastQualityBudget, ModelName> = {
  fast: "claude-haiku-4-5",
  quality: "claude-opus-4-5",
  budget: "qwen2.5-7b",
};

function createModelResolver({policy}: {policy: ModelPolicy}): ModelResolver {
  if (policy === "auto") {
    return (taskType) => DEFAULT_MODEL_FOR_TASK[taskType];
  }
  if (policy === "fast" || policy === "quality" || policy === "budget") {
    return () => STRATEGY_TO_MODEL[policy];
  }
  throw new Error(`Unknown policy: ${policy}`);
}

实现细节: 4 种策略共 3 个常量 model (auto 走 default 表 5 个, fast/quality/budget 各 1 个), 总共 5 + 3 = 8 个 model name。 改策略 = 改 3 个常量, 改 default 表 = 改 5 个常量。 简单, 教学友好。

execution profile: readonly / ci / workspace_write

用途: 任务类型决定model, 执行 profile 决定权限。 两者正交, 不耦合。 profile 3 种 (readonly / ci / workspace_write) 沿用第 14 章 Schedule 的定义。

真实场景: long_running 任务 + readonly profile = "5 天跑扫描 (只读, 用 GPT-4o)"; long_running + workspace_write = "5 天跑迁移 (读写, 用 GPT-4o)"; 两者 model 一样, 权限不同。

设计思想: model × profile 两维 — Reference 章节 "模式 5 · Composition Root 组合根" 的延伸。 modelResolver 选 model, ExecutionPolicy 校验 profile 权限, 两者 不互相 import。

实现细节: profile 透传到 Async Run (第 13 章) 的 permissionProfile 字段, 由 executionPolicy 的 validateCommand / validateResources 校验。 model 和 profile 是独立维度, 组合出 5 × 3 = 15 种 任务场景。

graceful degradation: model 不可用怎么办

用途: 推荐的 model 不可用 (rate limit / 服务挂 / 配额用完) 时, harness 不能直接 fail, 应该优雅降级 到次优 model。 这条 fallback 链是 Model Policy 的隐藏价值。

真实场景: team 跑 bulk_summarize, 推荐 Qwen2.5-7B (本地 vLLM), vLLM 服务挂, fallback 到 GPT-4o-mini (云端 OpenAI), 速度慢但能用; 再 fallback 到 Claude Haiku; 再不行 fail 提示用户 "所有 model 都不可用"。

设计思想: Fallback 链 + 优雅降级 — Reference 章节 "模式 13 · Error→Action 错误转动作" 的应用。 不可用 = fail 之前先试 fallback。 fallback 顺序写死 (推荐 → 次优 → 次次优 → fail), 不动态计算。 

const FALLBACK_CHAIN: Record<ModelName, ModelName[]> = {
  "gpt-4o": ["claude-sonnet-4-5", "claude-haiku-4-5"],
  "claude-opus-4-5": ["gpt-4o", "claude-sonnet-4-5"],
  "claude-haiku-4-5": ["gpt-4o-mini", "qwen2.5-7b"],
  "qwen2.5-7b": ["claude-haiku-4-5", "gpt-4o-mini"],
  // ...
};

async function callWithFallback(model: ModelName, args): Promise<Response> {
  const chain = [model, ...(FALLBACK_CHAIN[model] ?? [])];
  for (const m of chain) {
    try {
      return await llmClient.call(m, args);
    } catch (err) {
      if (isRetryableError(err)) continue;  // 试下一个
      throw err;  // 不可恢复错误立即 throw
    }
  }
  throw new Error("All models in fallback chain failed");
}

实现细节: 5 个推荐 model 各自配 fallback 链, 链长 2-3, 教学版够用。 链必须是有限的 (不能 无限循环, 也不能跳到自身), 用 TypeScript 类型 + 单元测试保证。

v2 增强: fallback 跨协议。 v1 之前 fallback 链 都绑同一个 provider, GPT-4o 失败只能试 GPT-4o-mini (同 OpenAI 协议), 选 model 受协议限制。 v2 加了 Anthropic 协议后, fallback 链可以跨协议: GPT-4o (OpenAI 协议) 失败 → 试 Claude Sonnet (Anthropic 协议) → 试 Claude Haiku (Anthropic 协议)。 看 v2 增强后的 fallback 链: 

const FALLBACK_CHAIN: Record<ModelName, ModelName[]> = {
  // v1: 同协议 fallback
  "gpt-4o": ["claude-sonnet-4-5", "claude-haiku-4-5"],  // OpenAI → Anthropic
  "claude-opus-4-5": ["gpt-4o", "claude-sonnet-4-5"],   // Anthropic → OpenAI
  "qwen2.5-7b": ["claude-haiku-4-5", "gpt-4o-mini"],     // OpenAI → Anthropic → OpenAI
  // ...
};

// 跨协议 fallback 关键: FallbackChain 试的是 model name,
// 不是 adapter。 adapter 是 model profile 解析时根据 protocol
// 字段自动选的, fallback 链不需要知道协议存在。
// 换句话说: 协议细节在 model 画像层消化, fallback 链只看
// model name → 拿 model profile → 选 adapter → 调 LLM。

为什么能跨协议: 关键是 callWithFallback 试的是 model (model name), 不是 adapter。 每个 model name 在 modelProfiles 数组里有 1 个画像, 画像含 protocol.preferred 字段, 选哪个 adapter 是画像解析时决定的, fallback 循环不知道也不 关心协议存在。 协议细节在 LLM 边界内部消化, 业务层只看到 "试下一个 model name"。

cost 估算: 让用户知道"这次大概多少钱"

用途: 用户跑任务前, harness 估算"这次大概花 多少", 避免跑完发现"我跑了 $5"。 估算基于 (model 价目表 + token 估算), 不精准但"够参考"。

真实场景: user 跑 long_running 任务, 启动前 harness 输出 "Estimated cost: $0.30-$0.50 (GPT-4o, ~50k input + 5k output tokens)"。 user 知道"大概 5 毛", 决定跑; 如果输出 "Estimated cost: $5-$10 (Claude Opus, 复杂推理)", user 可能 改成 GPT-4o 跑。

设计思想: 价目表 + token 估算 — Reference 章节 "模式 19 · Idempotent 幂等" 的延伸 (估算可重 复, 不靠 LLM 自己估)。 看 cost 估算实现: 

// 价目表: 每 1M token 的美元价
const MODEL_PRICING: Record<ModelName, { input: number; output: number }> = {
  "gpt-4o": { input: 2.5, output: 10.0 },
  "claude-opus-4-5": { input: 15.0, output: 75.0 },
  "claude-haiku-4-5": { input: 0.8, output: 4.0 },
  "qwen2.5-7b": { input: 0.0, output: 0.0 },  // 本地免费
  // ...
};

function estimateCost(model: ModelName, inputTokens: number, outputTokens: number): { min: number; max: number } {
  const p = MODEL_PRICING[model];
  return {
    min: (inputTokens / 1_000_000) * p.input + (outputTokens / 1_000_000) * p.output,
    max: ((inputTokens * 1.2) / 1_000_000) * p.input + ((outputTokens * 1.5) / 1_000_000) * p.output,
  };
}

实现细节: 输出 min / max 两个值, min = 精确估算, max = 实际可能 1.2-1.5 倍 (LLM 输出不一定可控)。 启动前 estimateCost() 算一次, 输出 "Estimated cost: $0.30-$0.50"。 不 精准, 但够参考。 这是 Reference 章节 "模式 19 · Idempotent" 的应用 — 估算可重复, 同一任务多次跑估算一致。

主循环集成: 在哪 3 个时机选 model

用途: Model Policy 集成到 agent 主循环有 3 个 时机:

  1. agent.run() 启动前: 接收 taskType, 调 modelResolver(taskType) 选 model, 调 estimateCost(...) 算预算, 输出 "Using model X, estimated cost $Y"。
  2. 每次 LLM 调用前: 拿 model 调 llmClient.call(), 失败时走 fallback 链 (v2 跨协议: OpenAI 失败 → 试 Anthropic)。
  3. v2 新加: REPL /m 命令切换 model: 用户跑着跑着 想换 model (比如发现当前 task 比预想复杂, 切到 quality), 在 REPL 敲 /m claude-opus-4-5, runtime policy 即时更新, 下一轮 LLM 调用就用新 model。 关键: 切换不重 启 agent.run(), history 保留, 继续干活。 协议也随 model 自动切 (claude-opus-4-5 走 Anthropic, gpt-4o 走 OpenAI)。

设计思想: 显式选择 + 启动一次, 但支持 中途切 — agent.run() 启动时选 model (时机 1), 整个 run 期间默认不变, 但允许中途切 (时机 3)。 切换 model = 改 RuntimePolicy, 不重启 agent.run(), history 保留。 这是 Reference 章节 "模式 1 · 工厂 + 闭包" 的延伸 — model 选完在闭包内, 但闭包能接收外部 update。

实现细节: 集成代码在 src/agent.ts 第 0 步 (setup) — 接收 taskType → 调 modelResolver → 拿 model → 调 estimateCost → 注入到 run() 闭包。 之后所有 LLM 调用都 用这个 model, 失败走 fallback。 时机 3 的切换通过 runtimePolicy() 函数 (不是值) 拿, 每次 LLM 调用 重新 runtimePolicy(), 拿到的是最新 policy, 支持 /m 即时切换。

fake test: 用 fake model 测 fallback 链

用途: Model Policy 测试需要fake LLM client — 不真调 OpenAI / Anthropic, 用 stub 返回"rate_limit" / "ok" / "fail", 验证 fallback 链走通。

真实场景: 用户写测试 "推荐 GPT-4o + 链 [Claude Sonnet, Claude Haiku]; fake client 第一次 GPT-4o 返回 rate_limit, 第二次 Claude Sonnet 返回 ok, 验证: 最终 model = Claude Sonnet, fallback 链用了 2 步, 不再 试 Claude Haiku"。 

test("fallback 链: 推荐 model rate_limit -> 次优 model 成功", async () => {
  const fakeLlm = createFakeLlmClient({
    "gpt-4o": () => ({ error: "rate_limit" }),
    "claude-sonnet-4-5": () => ({ content: "ok" }),

常见误解: 1 个 model 不能跑所有

误解 1: "1 个最强 model 跑所有?" 错。 大材 小用, 简单任务浪费钱 (100 倍)。 应该 cheap model 跑简单, 强 model 跑复杂。

误解 2: "auto 策略 = 启发式判断?" 错。 auto 是 "用户不用选 model, 按任务类型查表选", 不是 harness 启发式猜 任务类型。 任务类型由调用方显式声明。

误解 3: "fallback 是无脑降级?" 错。 fallback 是有序的 (推荐 → 次优 → 次次优 → fail), 不 是随机选。 链写死在常量里, 改链 = 改常量。

误解 4: "model 一旦选定, 整个 run 不变?" 对。 整个 agent.run() 期间用同 model, 切换 = 重新 run()。 中途切换 会破坏 prompt cache (第 10 章) + 上下文不一致。

反例梯度: 4 个常见错误

反例 1 · 1 个 model 跑所有

team 觉得"用最好的最省心", 全部任务用 Claude Opus。 错。 跑 5 分钟 git status 用 Opus 5 分钟 × $0.015/分钟 = $0.075; 用 Haiku 5 分钟 × $0.0008/分钟 = $0.004。 100 倍 浪费。 正确: 5 种任务类型 → 5 个 model, 按类型选。

反例 2 · auto 启发式猜

让 harness 启发式判断"这个 query 长 = long_running, 短 = short_interactive"。 错。 启发式猜错率高 (1 句话也可能 是复杂问题)。 正确: 任务类型由调用方显式声明, harness 只查表选 model, 不猜。

反例 3 · 没有 fallback 链

推荐 model 不可用 (rate limit) 就 throw fail。 错。 1 个 model 不可用 = 整个 harness 死, 体验差。 正确: 配 fallback 链 (推荐 → 次优 → 次次优), 不可用就试下一个, 全失败才 fail。

反例 4 · 中途切换 model

让 agent.run() 中途根据 token 消耗切换 model (发现花 太多就换便宜)。 错。 切换 model 破坏 prompt cache (model B 看不到 model A 的 cache 块), 也破坏上下文一致性 (model A 看到的 conversation model B 看到的不一样)。 正确: 整个 run 用同 model, 切换 = 重新 run()。

Validation: 8 条不变量检验 (4 v1 + 4 v2)

  1. default 表完整: 5 种任务类型都有推荐 model, 缺 一个 TypeScript 编译失败 (Record 强制完整)。 验证: 单测覆盖 5 个 taskType 都能查表。

  7. v2: fallback 跨协议: callWithFallback 试 model name 不试 adapter, OpenAI 失败 → 试 Anthropic 协议 跑 Claude。 验证: 单测覆盖 "GPT-4o rate_limit + Claude Sonnet ok → 链用 2 步, 最终返回 ok"。

回望: 哪些原则在本章兑现了

  • 策略模式: 4 种策略决定 model 选, 5 种任务类型 决定 default 表, 两者正交。
  • 声明式 vs 启发式: 任务类型由调用方显式声明, harness 查表, 不启发式猜。
  • 有限 fallback 链: 链写死, 不动态算, 不无限循环。
  • 幂等估算: cost 估算可重复, 同一任务多次跑 估算一致。
  • 显式切换: 整个 run 用同 model, 切换 = 重新 run。

本章小结

Model Policy 是给 LLM 选型的策略层, 5 种任务 类型 × 4 种策略 × 3 种 execution profile = 60 种任务场景, 每种有合适的 model + 权限 + 成本。 核心是 5 个设计:

  • 5 种任务类型: long_running / short_interactive / bulk_summarize / complex_reasoning / experimental, 固定 枚举。
  • 4 种策略: auto / fast / quality / budget, 固定 枚举, 切用户偏好。
  • 声明式 taskType: 调用方显式声明, harness 查表 选 model, 不启发式猜。
  • 有限 fallback 链: 推荐 → 次优 → 次次优 → fail, 写死, 不动态算。
  • 幂等 cost 估算: 价目表 × token 估算, 启动时输出 min/max, 实际可偏差 1.5 倍。
