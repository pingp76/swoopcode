专题 B · 测试不确定系统

如何测试一个不确定的 Coding Agent

Coding Agent 调真实 LLM, 但测试不能一开始就依赖真实 LLM。本专题讲 harness 是怎么把"测试不确定系统" 拆成 4 层梯度的: 用 scripted LLM 让行为可重放, 用 trace 把中间过程结构化, 用 live regression 在固定路径 上做现实校准, 用 LLM judge 处理开放式语义质量。MCP 和 Agent Team 也属于同一套方法论, 但当前仅作为 harness 原型保留。

为什么 Coding Agent 难测

跑一个 coding agent 的时候, 你面对的"被测对象"其实有 3 层不确定性。 朴素做法是直接连真实 LLM 跑断言, 这 3 层不确定性会同时放大。 下面先用 3 个真实失败故事说明"朴素做法哪里坏", 再讲我们的方法。

失败故事 1 · 修了 prompt, 测试全挂

某次更新里我们改了一行 system prompt 措辞, 想让模型"更简洁地总结"。 第二天 CI 红了 200 多个 case, 但代码逻辑根本没动。

根因: 那些 case 在断言"最终回复的完整文本"。 LLM 稍微改一下措辞, 文本差异就触发 golden snapshot 失败。 我们把这种失败叫做假阳性: 不是代码 bug, 是测试方法不对。 修复方式是改 case, 不是改代码。

这类问题的核心: 不要断言模型"说了什么", 要断言模型"做了什么"。 "说了什么" 由 LLM 决定, 你控制不了; "做了什么" (调了哪个工具、读了什么文件) 由 harness 决定, 你能测。

失败故事 2 · 改了 tool schema, 真实 LLM 行为变了

某次更新里我们给 run_bash 工具加了一个新参数 timeoutMs。 跑真实 LLM 时, 模型开始尝试用它, 但因为某些模型对 schema 变化敏感, 选择工具的策略整体漂移 —— 本来用 run_read 的场景, 现在用 run_bash + cat。

根因: 测试完全依赖真实 LLM 行为, 任何上游变动都会放大成 case 失败。 这类问题更难定位: 你不知道是"模型行为变了" 还是"代码逻辑错了"。 修复方式是把模型行为和代码逻辑分开测。

失败故事 3 · 修了"代码 review 风格" 的 agent, 没法用代码判断

我们做了一个 review agent, 输入一段代码, 它输出"哪里可以改进"。 没有正确答案, 没有 golden snapshot, 没法用 expect(output).toBe(...)。

根因: 有些任务本身就是开放式的, 没有任何代码判断 "这个 review 好不好"。硬要写断言, 只能写非常宽松的正则, 等于没测。 修复方式是让另一个 LLM 来评这个 LLM (judge), 但要严格 控制"评的依据" 和"评的输出格式"。

3 类不确定性 → 3 种测试方式

上面 3 个故事对应 3 类不确定性, 每一类需要不同的测试方式:

Runtime 不确定性
消息顺序对不对? tool_call 和 tool_result 配没配对? 权限检查有没有漏? Context 超限时压缩行为是否正确? 一次更新会不会改坏这些? 测试方式: 用 scripted LLM (预设 response 序列), 让模型行为完全可预测, 测的就是 harness 本身。
模型不确定性
同一个 query, 同一个模型, 不同时间可能返回略不同的回答。 不同模型行为差异更大 (Kimi 强 coding, GLM 强长上下文, Qwen 强 agent)。 测试方式: 用真实 LLM 跑固定 case 集, 用结构性 hard 断言 (工具被调、文件存在) 判断"能力不退化", 不判断 "措辞变没变"。偶尔开启 judge 评开放式质量。
任务本身的不确定性
"改 README 的标题" 是结构化的, 改完用 diff 断言就行; "代码 review 这段是否合理" 是开放式的, 只能让另一个 LLM 评。 测试方式: 用 judge LLM 读 trace 摘要 (不是完整日志), 给出结构化评分, judge prompt 不进 agent system prompt。

核心方法: 把 3 类不确定性分开, 每一类用最合适的方式测。 混在一起测, 任何一类波动都会污染其他两类的判断。

整体架构: Driver 边界中立

4 层梯度都需要一个中立 runner 来执行 case。 朴素做法是在 runner 里直接 import { createAgent } from "../agent.js", 这条路走 3 步就会撞墙。下面先讲"撞墙过程", 再讲"Driver 边界怎么解"。

朴素做法的 3 个撞墙

  1. 主代码改了, 测试全挂 (假阳性): runner 绑死当前项目 内部类型, 任何重构都会让 case 失败, 但失败原因可能是 "case 写了私有事件细节", 不是代码 bug。
  2. 换被测对象写不出来: 假设想测一个 CLI 黑盒工具 (比如 包装另一个 agent), 因为 runner 写死了 createAgent() 调用, 没法换别的实现。
  3. tool / permission / transcript 私有事件泄漏: case 写着 "transcript 事件的 historySequence == 5" 这种断言, 测的是实现细节, 改 main loop 就全挂。

这 3 个撞墙都被同一个抽象层解决: CodingAgentDriver 接口。 Runner 只看这个接口, 不看 driver 内部。

CodingAgentDriver 接口

interface CodingAgentDriver {
  startCase(context: AgentCaseContext): Promise<void>;
  send(input: AgentInput): Promise<AgentTurnResult>;
  readEvents?(): Promise<AgentRuntimeEvent[]>;
  close(): Promise<void>;
}

interface AgentCaseContext {
  caseId: string;
  workspaceRoot: string;        // 临时目录, driver 内部用
  metadata?: Record<string, unknown>;
}

interface AgentInput {
  stepId: string;              // 多 step case 用
  query: string;
}

interface AgentTurnResult {
  stepId: string;
  finalOutput: string;
  exitCode?: number;
  events?: AgentRuntimeEvent[];  // tool_call / llm_call 等
}

async function runEvalCase(case: EvalCase): Promise<EvalRunResult> {

  const workspace = await createEvalWorkspace(case.workspace);

  }

  await driver.close();
  return buildResult(case, stepTraces, workspace);
}

断言分两类

断言是"模型做了什么" 的代码表达。我们把所有断言拆成两类, 解决"测实现细节 vs 测行为" 的张力。

Portable 断言: 跨 driver 可用

Portable 断言只看 finalOutput、workspace 文件、step result, 不依赖 driver 内部事件。换 driver 时, 这些断言不需要改。 case 主体应该尽量用 portable 断言。

断言 检查什么 例
finalOutputContains 最终回复包含文本 { kind: "finalOutputContains", text: "approved" }
finalOutputMatches 最终回复匹配正则 { kind: "finalOutputMatches", pattern: "(denied|permission|cannot)" }
allStepsCompleted 所有 step 跑完 { kind: "allStepsCompleted" }
fileExists workspace 文件存在 { kind: "fileExists", path: "README.md" }
fileNotExists workspace 文件不存在 (越界 / 拒绝后验证) { kind: "fileNotExists", path: "blocked.txt" }
fileContains workspace 文件包含文本 { kind: "fileContains", path: "README.md", text: "Usage" }
noWritesOutsideWorkspace 没有任何工具写到 workspace 外 { kind: "noWritesOutsideWorkspace" }
exitCodeIs CLI driver 退出码 { kind: "exitCodeIs", code: 0 }

Instrumented 断言: 需要 driver 发射 runtime events

Instrumented 断言依赖 driver 内部观察, CLI 黑盒 driver 不一定能用。 in-process driver 通过 wrapToolRegistryForTrace 和 scripted-terminal 自动发射这些事件, 所以 in-process case 可以放心使用。

断言 检查什么 例
toolCalled 工具被调过 (可设 minCount) { kind: "toolCalled", toolName: "run_read" }
toolNotCalled 工具没被调 { kind: "toolNotCalled", toolName: "run_write" }
toolCalledOneOf 一组工具中至少一个被调 { kind: "toolCalledOneOf", toolNames: ["run_memory_list", "run_memory_read"] }
toolCallCount 工具调用次数等于指定值 { kind: "toolCallCount", toolName: "run_bash", count: 1 }
toolArgsContain 工具参数包含文本 { kind: "toolArgsContain", toolName: "run_read", text: "package.json" }
toolResultContains 工具结果包含文本 (验证工具返回, 而不是最终回复) { kind: "toolResultContains", toolName: "run_task_group_read", text: "Live plan" }
stepToolCalled 指定 step 中工具被调 { kind: "stepToolCalled", stepId: "step-2", toolName: "run_write" }
stepToolNotCalled 指定 step 中工具没被调 (验证"先观察后写") { kind: "stepToolNotCalled", stepId: "step-1", toolName: "run_write" }
noToolErrors 没有工具返回 error: true { kind: "noToolErrors" }
allToolsSucceeded 所有 tool_result 不带 error { kind: "allToolsSucceeded" }
permissionPromptShown 权限确认弹窗出现过 { kind: "permissionPromptShown" }
transcriptEventTypes transcript 事件类型序列匹配 { kind: "transcriptEventTypes", expected: ["user_message", "tool_result", ...] }

复杂 case 还需要 4 个补充断言

跑全系统 live case 时, 上面的断言不够用, 我们又加了 4 个:

  • fileNotExists —— 验证越界路径 / 权限拒绝后文件没生成
  • toolCalledOneOf —— 模型可能用多个等价工具达成目标, 不想写死
  • toolResultContains —— 看工具结果本身是否含目标内容 (而不是只看最终回复)
  • stepToolCalled / stepToolNotCalled —— 验证"第一步只读, 第二步才写"

断言选择经验法则

  1. case 的"主断言" 全部用 portable
  2. 需要验证"模型走了某条路径" 时加 instrumented (例如验证"权限拒绝后没继续写")
  3. 开放式质量用 judge, 不要硬塞到 hard 断言

Scripted LLM: 确定性测试的支柱

Scripted LLM 是 deterministic 层的核心。设计极简: 每次 chat() 消耗一个预设 response, response 用完抛错。 实现大概 60 行, 但解决了"模型行为不可预测" 的根本问题。

为什么这种"看起来太朴素"的设计是对的

朴素的设计有两个好处:

  1. 强迫 case 作者把模型行为显式写出来: 不能依赖模型"自然" 做对。 跑测试如果失败, 一眼能看到"实际行为" 和"case 期望" 的差异, 不是"真实 LLM 升级后行为变了"。
  2. 把模型不确定性关在 case 里: 同样的 case 跑 1000 次, 行为都一样。 CI 跑过 = 代码逻辑正确, CI 挂了 = 代码逻辑错。模型波动不会假阳性。

工具调用的 response 必须成对出现

工具调用场景下, scripted response 必须成对: 第一次 finishReason: "tool_calls" + toolCalls 数组, 第二次 finishReason: "stop" + content。 

scriptedResponses: [
  // 第 1 轮: 模型决定调 run_bash
  {
    content: null,
    toolCalls: [{ id: "call_1", name: "run_bash", args: { command: "cat package.json" } }],
    finishReason: "tool_calls",
  },
  // 第 2 轮: 模型看到工具结果, 给最终回复
  { content: "package.json says test = npm test", toolCalls: [], finishReason: "stop" },
]

3 个常见 scripted bug

写 case 时容易踩 3 个坑, 几乎每个新 case 作者都会遇到一次:

Bug 1 · response 配对错
第一次写了 finishReason: "stop" 但同时写了 toolCalls, 或者反过来。Driver 会用工具数判定"还有下一轮", 配对错会导致 case 跑到一半 response 用完, 抛 Eval case <id> has no scripted LLM response for call <n>。
Bug 2 · toolCall id 漂移
scripted 里写 id: "call_1", 但真实 OpenAI 协议里 tool_call_id 拼写不一致。Driver 在 tool_result 配对时 用的是 toolCall.id, 拼错会导致 tool message 找不到 对应 assistant 消息, 整个 conversation 序列错乱。
Bug 3 · tool args 类型错
scripted 里 args 期望是 object, 但 case 作者 写成 JSON 字符串。Driver 会自动 JSON.stringify, 字符串再 stringify 就成了 "{\"command\": \"ls\"}" (外层多引号), 模型看到时 JSON 解析失败。

解决方式: 第一版 case 都跑一遍 dry run, 确认 response 数量正确、 id 拼写一致、args 是 object。后期可以加 scripted LLM 单元测试自动检查。

临时 workspace + 临时 agentHome

第一原则: case 跑完不能污染任何东西。 这一条不做到, 后面所有判断都是空中楼阁。

Workspace: 每个 case 独立临时目录

async function createEvalWorkspace(plan?: EvalWorkspacePlan): Promise<EvalWorkspace> {
  const root = await fs.mkdtemp(path.join(os.tmpdir(), "swoopcode-eval-"));
  // 写入 initialFiles
  if (plan?.initialFiles) {
    for (const [relPath, content] of Object.entries(plan.initialFiles)) {
      const abs = safeResolve(root, relPath);   // 拒绝 .. 和绝对路径
      await fs.mkdir(path.dirname(abs), { recursive: true });
      await fs.writeFile(abs, content, "utf8");
    }
  }
  return {
    root,
    readFile: (p) => fs.readFile(safeResolve(root, p), "utf8"),
    exists: (p) => fs.access(safeResolve(root, p)).then(() => true).catch(() => false),
    cleanup: async () => { /* case 跑完默认删 */ },
  };
}

GitHub · createEvalWorkspace 临时目录 (L50)

4 条安全防线:

  1. mkdtemp 生成唯一目录: 多个 case 并行跑不会冲突。
  2. initialFiles 拒绝绝对路径和 ..: case 写 { "/etc/passwd": "..." } 会被 safeResolve 拒绝, 防止 case 作者不小心写到 tmp 外面。
  3. readFile / exists 也走 safeResolve: 断言 fileContains("/etc/passwd", "root") 一样被拒。
  4. cleanup 默认删目录, keepOnFailure: true 时保留: 失败的 case 把 workspaceRoot 写进 trace, 方便人工排查。

AgentHome: full-tools 必须隔离

tools.kind = "full" 装载的是完整工具系统, 包括 run_memory_create、run_task_group_create、 run_schedule_create、run_skill 等。 这些工具默认写到 ~/.swoopcode 下的 memoryDir / tasksDir / schedulesDir / skillsDir。

如果不隔离: 跑一个 test 创建了 release-keyword = LIVE-MEM-42 的 memory, 会污染用户真实数据; 第二次跑还会读到上次的污染, 看起来 case 跑通了, 其实是脏数据。

为什么不用 in-memory store?

有人会问: "为什么不直接用内存对象模拟 store, 跑得更快?" 答案是持久化路径是 agent 的一部分。Memory / Task / Schedule 都有原子写、索引重建、跨进程恢复这些逻辑, 内存模拟会跳过这些真实路径, case 永远测不到"原子写失败" "索引文件损坏" 这类边界 bug。 临时目录 + 真实模块, 才是端到端测试该有的样子。

Judge: 开放式语义的补充

Hard 断言判断事实: 工具是否调用、文件是否含 sentinel。 Judge 判断质量: TODO 是否反映真实执行步骤、汇总是否忠实于 成员结果、Agent 是否没有夸大成功。

没有 judge, 这类问题只能用宽松正则, 等于没测。 但 judge LLM 本身也有不确定性, 必须严格控制。

关键顺序

  1. 先跑 hard 断言, 失败就 case failed, 不再跑 judge
  2. Hard 通过后, judge 用另一个 LLM 读 trace 摘要, 输出 passed / score / summary / strengths / problems / evidence / needsHumanReview
  3. Judge 失败也导致 case failed, 但报告里要明确标 (judge failed) 防止混淆

Judge prompt 与 agent 隔离

judge prompt 不进 agent system prompt。 它是 eval runner 的单独 LLM 调用, 与被测 agent 隔离。 Judge 模型可以和被测 agent 不同 (用 JUDGE_MODEL 覆盖), 常见选择是用更轻量的模型做 judge 降本。

4 层 JSON 解析降级

Judge LLM 可能返回 markdown code block、额外文本、无效 JSON。 解析器采用 4 层降级:

  try { return JSON.parse(raw); }

  const codeBlock = raw.match(/```json\s*([\s\S]+?)\s*```/);

    try { return JSON.parse(codeBlock[1]); } catch {}

  // 第 3 层: 括号深度计数器 + 字符串引号跟踪
  const balanced = extractBalancedJson(raw);
  if (balanced) {
    try { return JSON.parse(balanced); } catch {}
  }

  // 第 4 层: 降级
  return {
    enabled: true,
    passed: false,
    score: 0,
    summary: "judge_failed",
    strengths: [],
    problems: ["judge output was not parseable JSON"],

  };
}

如何写好一个 case

讲完 4 层梯度 + 3 种 driver + 12 个断言类型, 最后讲"如何写好一个 case"。 这是个心法问题, 不是 API 问题。

3 条心法

心法 1 · 先写失败用例, 再写 driver
写 case 的正确顺序: 先写 EvalCase (期望行为), 再用 fake driver 跑通断言, 最后才换 in-process driver 接入。 这样写出来的 case 不依赖具体 driver 内部, 也不会"为了 driver 写 case"。
心法 2 · 用 sentinel 而非完整文本
真实 LLM 措辞不可预测。如果 case 断言"模型说了什么", 措辞稍微变就挂。改用 sentinel: 写"输出必须含 LIVE-WRITE-001", 写"文件必须含 status: ready"。 任务核心要求 + 精确短字符串, 比完整文本 golden snapshot 稳定得多。
心法 3 · live case 要写"能力退化检测", 不写"完美行为"
Live regression 不是"证明 agent 完美", 是"证明核心能力不退化"。 期望是"v0.1 跑通的 6 个 case, v0.2 还跑通"。如果 v0.2 多调了 几次工具但最终结果一样, 不算退化; 如果少调了关键工具导致 文件没改对, 算退化。

case 评审清单

写完一个 case, 跑一遍下面 5 条检查:

  1. 断言是否都是 portable? grep 'kind:', 看看有没有 toolCalled / transcriptEventTypes 这种 instrumented 断言当主断言的。
  2. case 是否能跑通 fake driver? 把 tools.kind 改成 fake, 跑一次。 如果挂, 说明 case 依赖 driver 内部。
  3. 是否有 maxCalls / maxRounds? live case 必须有, 否则可能无限循环。
  4. workspace 边界是否合理? initialFiles 不能有 .. 和绝对路径。
  5. 断言失败时 trace 是否能定位? 故意改错一处 (比如把 sentinel 改成 LIVE-WRITE-002), 跑一次, 看 trace 能不能立刻告诉失败原因。

Eval 和 Agent 主循环如何解耦

一个常被新同学问的问题: "为什么 agent.ts 不需要知道自己被测? 主循环 不写'如果是 test 就...' 的分支吗?"

立刻坏 3 件事:

  1. 主代码和测试逻辑绑死, 改任何一边另一边都挂
  2. 测试测的是"加 if 后的代码", 不是"真实代码"
  3. production 行为和 test 行为可能漂移, case 跑通但 prod 挂

反例梯度

新手错法 · A

常见错误: Eval Core 直接 import { createAgent } from "../agent.js"。

为什么错: runner 绑死当前项目内部类型, 换 driver 写不出; case 写"transcript 事件 sequence = 5" 这种私有细节, 主代码改了全挂。

正确做法: Eval Core 只认识 CodingAgentDriver 接口, 当前项目实现藏在 src/eval/drivers/learn-claude-code/ 下。

中级错法 · B

常见错误: 跑 live case 不设 maxCalls / maxRounds。

为什么错: LLM 卡住或循环时, case 跑几分钟不退出, 阻塞 CI。

正确做法: 每个 live case 都设 maxCalls / maxRounds / Vitest timeout (通常 30-60s)。

高级错法 · C

常见错误: Live case 写进用户真实 ~/.swoopcode。

为什么错: memory / task / schedule 污染用户数据, 第二次跑 还会读到上次结果, 假阳性。

正确做法: tools.kind = "full" 必须配 agentHome: "temp"; case 结束后清空临时目录。

边界错法 · D

常见错误: Judge 失败时让 case 通过, 或者让 judge 覆盖 hard 失败。

为什么错: judge LLM 也有不确定性, 让 judge 覆盖硬规则等于 把模型不确定性放回主路径, 整个测试失去意义。

正确做法: 顺序固定: hard 先跑, hard 失败 case 直接 failed; judge 单独评分, judge 失败 case 也标 failed (报告里加 (judge failed) 区分)。

Prompt Card (本章任务)

Prompt Card · 专题 B

目标: 给 harness 加一套测试不确定系统的方法论, 让 runtime 不确定性和模型不确定性分开测, 切测试范围 = 换 driver / 换 LLM plan, 不改 case 主体。

场景: 项目主循环大改后, 跑 npm run test:eval 看 scripted suite 是否挂; 发布前跑 EVAL_LIVE_REGRESSION=1 + EVAL_JUDGE=1 看 core tools 真实 LLM 行为是否退化; 复杂工具系统变更后跑 EVAL_LIVE_FULL=1 看 TODO/Memory/Skill/SubAgent 是否正常。

必须实现的能力 (功能导向, 不限定代码结构):

  • 中立 driver 边界
    • eval runner 只认识 CodingAgentDriver 接口 (startCase / send / readEvents / close)
    • runner 不直接 import harness 主代码
    • 换被测对象 = 换 driver 实现, runner 不动
  • 4 层梯度
    • deterministic (scripted LLM, 默认 CI 跑)
    • replay (fixture 文件, 默认 CI 跑)
    • live smoke (真实 LLM, 显式 opt-in)
    • live regression (真实 LLM + hard 断言 + 可选 judge, 显式 opt-in)
  • 断言分两类
    • portable: fileExists / fileContains / finalOutputContains / noWritesOutsideWorkspace (跨 driver 可用)
    • instrumented: toolCalled / toolCallCount / permissionPromptShown (依赖 runtime events)
    • case 主体尽量用 portable, instrumented 作为补充调试
  • case 结构
    • 每个 case 有独立临时 workspace, 跑完默认清理
    • full-tools case 强制用临时 agentHome, 不污染用户真实数据
    • steps 数组复用同一 driver 实例, 支持多 turn 共享 context
    • 步骤级断言可挂 stepId, 验证"第一步只读, 第二步才写"
  • trace 是事实来源
    • case 失败时 trace 必须能告诉"哪一步、哪个工具、什么参数、什么结果"
    • trace 不写仓库目录, 由 EVAL_TRACE_DIR 显式开启
    • runtime events 标准化: tool_call / tool_result / llm_call / llm_response / permission_prompt / log / raw / driver_error
  • judge 边界
    • hard 断言先跑, hard 失败 case 直接 failed, 不再跑 judge
    • judge 用单独的 LLM 调用, prompt 不进 agent system prompt
    • judge 必须输出 JSON, 解析失败标 judge_failed 不影响 hard result
    • judge 模型可与被测 agent 不同 (用 JUDGE_MODEL 覆盖)
  • live safety
    • 所有 live suite 默认 skip, 显式 env 开启
    • 每个 live case 设 maxCalls / maxRounds / Vitest timeout
    • live case 不用完整回复文本做 golden snapshot
    • live case 只用临时 workspace + 临时 agentHome

验证 (用 fake driver + scripted LLM, 逐条断言):

  • eval runner 只看 driver 接口, core 模块不 import harness 主代码
  • deterministic case 用 scripted LLM, 跑通 hard 断言
  • 同一 case 多 step 复用 driver 实例, 共享 history
  • trace JSON 含 runtime events + assertion evidence, 失败时能定位
  • judge mock 返回非 JSON, case 标 failed 但不抛异常
  • live case 默认 skip, 设 EVAL_LIVE_REGRESSION=1 才跑

本章练习

  1. 故意在 runner.ts 写 import { createAgent } from "../agent.js", 跑测试, 看"Eval Core 不知道 driver 内部"是否抓到 (切 driver 后挂)。
  2. 写一个 live case 不设 maxCalls, 用一个会循环调工具的 scripted response 模拟, 跑测试, 看"Live case 默认 skip / maxCalls" 是否抓到 (超时挂)。
  3. 让 tools.kind = "full" 配 agentHome 指向真实路径, 跑测试, 看"临时 agentHome 隔离" 是否抓到 (写入真实数据)。

本章小结

测试 Coding Agent 的核心是把 3 类不确定性分开。

  • Runtime 正确性 用 deterministic scripted LLM 守住, 默认 CI 跑, 不依赖 API key。
  • 工具系统能力 用 live regression 跑真实 LLM + hard 断言, 发布前 / 大改后触发, 默认 skip。
  • 开放语义质量 用 judge 补充 hard 跑不到的部分, judge 失败 也标 failed 但不影响 hard result。

4 层梯度 + 临时 workspace + 临时 agentHome + 中立 driver 边界 + 标准化 trace + 解析鲁棒的 judge = 一套"跑得通"和"跑得稳"都能验的方法论。

MCP 和 Agent Team 也属于同一套方法论, 但当前仅作为 harness 原型 保留, 等真实 runtime 落地后再恢复 opt-in 运行。

3 个常见 prompt card 错误

学生 vibe eval case 时最容易踩的 3 个坑:

错误 1 · 边界用空话, 不可验证
"代码要清晰" / "跑得通" / "边界要严" — 这种 prompt 让 LLM 写 出来"看起来对" 但实际跑不过测试。 每条边界必须能转成一条 expect()。
错误 2 · 断言依赖措辞 (golden snapshot)
写 "finalOutputContains("看起来很对")" 这种断言, 真实 LLM 升级后措辞变化就挂。 改用 sentinel 短字符串: finalOutputContains("LIVE-WRITE-001")。
错误 3 · 写完 case 不跑 dry run
LLM 生成 case 后直接进 live suite, 跑超 maxCalls / fixture 文件读不到 / 工作区边界违规, 排查 1 小时。 流程: scripted LLM dry run 一次 (5 秒) → fake driver 跑通 (3 秒) → 上 live (1 分钟)。

把这节当 checklist 用

写完 prompt card 后, 用这 5 个反问自检:

  1. 目标段: "跑通后, 我能说'这个能力不退化' 吗?" 不行就重写。
  2. 场景段: 真人读完, 知道"用户说什么 + 期望什么" 吗? 不知道就 加具体例子。
  3. 工具步骤: 钉死 LLM plan + 工具集 + 步骤数, 别留模糊。
  4. 边界段: 每条边界都能写一条 expect() 吗? 不能就 删或重写。
  5. 验证段: 故意改 1 个 sentinel, 立刻挂吗? 不挂就断言写错了。

5 条都过了, 你的 prompt card 就可以喂给大模型 vibe 出来一个 "跑得通、跑得稳" 的 eval case。
