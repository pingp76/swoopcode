第 09 章 · 跨会话记忆

让 Harness 记得用户: Memory

前面八章让 harness 在单次会话内能聊天、调工具、跑子任务、压缩、拦权限、 留 hook。但用户偏好 (例如"我喜欢简洁解释") 仍然在每次新会话都丢。 这一章给 harness 加 Memory 模块, 区分"项目级短期事实" 和"用户级长期 事实", 把后者持久化, 让 harness 在新会话里也能想起。

在第 08 章基础上改了什么

这一章在 Composition Root 加 memory 持久化层。memory 数据落盘到 ~/.claude/memory/<scope>/<key>.md 这样的 markdown 文件, scope 区分 user (用户级) 和 project (项目级)。 agent 主循环在 SessionStart 时把 memory 注入到 system prompt, 会话过程中 LLM 可通过 save_memory 工具更新 memory。memory 与 history 严格分离, 不进 messages 序列。 对应到代码, 改动集中在 3 个文件: src/memory.ts (新)、 src/tools/save_memory.ts (新)、src/agent.ts (改 SessionStart 注入 memory)。

1. src/memory.ts: Memory 持久化 (新) 2. src/agent.ts: SessionStart 注入 memory 到 system prompt 3. src/project-context.ts: 项目级上下文 (新)

作者怎么想的: 这一章的思考链

想清楚现象
用户在新会话里说"我喜欢简洁解释", agent 完全不知道用户在上一会话 说过这件事。现象是"harness 短期记忆 OK, 长期记忆缺失"。
想反例
最朴素的反例是"把 memory 写在 history 里"。这有两个问题: 一是 history 每次新会话都从空开始, memory 仍然丢; 二是 history 是"对话 上下文", memory 是"长期事实", 两者职责不同。
想接口和不变量
接口: interface MemoryStore { get(scope, key), set(scope, key, value), list(scope) }。 不变量三条: (1) memory 数据不进 history.getMessages(), (2) memory 落盘是 append-only (更新是写新文件, 不直接改原文件, 第 15 章 atomic-write 兜底), (3) 区分 user 级与 project 级, user 级跨项目共享, project 级仅在当前项目可见。
想怎么验证
fake memoryStore 暴露 spy, 跑完一次会话, spy.set 被调用写入"用户 喜欢简洁"; 第二次会话启动, SessionStart 注入的 system prompt 含 "用户偏好: 简洁解释", LLM 第二轮能引用这个事实。

先观察: 两段故意有气味的实现

观察 1 · memory 写在 history 里

// 教学简化版, 仓库真实实现见 GitHub 永久链接
// 错误: 试图用 history 保存 memory
history.add({ role: "system", content: "User preference: 简洁解释" });
history.persist();  // 错误: history 不该有 persist 方法

问:为什么不复用 history 的持久化能力?

答:三件事同时坏掉 —— 职责: history 是"对话上下文", memory 是"长期事实", 两者职责不同; 性能: history 每次 LLM 调用都 重新发送, memory 写进去意味着每次都重传无关事实; 协议: memory 改了 history 的语义, 后续 compress / replay 都会混乱。

观察 2 · 不区分 scope, 所有 memory 写一个文件

// 教学简化版
const memory = await fs.readFile("~/.claude/memory", "utf8");
memory += `\n${newFact}`;
await fs.writeFile("~/.claude/memory", memory);

问:为什么不分 scope?

答:用户换项目之后, 项目级 fact (例如"本项目用 React") 跟着 user 走, 污染新项目。区分 user / project scope 之后, 项目级 fact 在新项目自动隐藏, 不会被 LLM 误用。

接口形状: 在写实现前钉死

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export type MemoryScope = "user" | "project";

export interface MemoryEntry {
  scope: MemoryScope;
  key: string;       // 例如 "user.preference.explanation_style"
  value: string;     // markdown 内容
  updatedAt: number; // epoch ms
}

export interface MemoryStore {
  get(scope: MemoryScope, key: string): Promise<MemoryEntry | null>;
  set(scope: MemoryScope, key: string, value: string): Promise<void>;
  list(scope: MemoryScope): Promise<MemoryEntry[]>;
  // 渲染为 system prompt 片段
  renderForPrompt(scope: MemoryScope): Promise<string>;
}

scope 划分规则

user 级
跨项目共享的事实, 例如 "user.preference.explanation_style = 简洁", "user.preferred_language = zh-CN"。这些 facts 在任何项目任何会话 都生效。
project 级
仅在当前项目可见的事实, 例如 "project.tech_stack = React + TypeScript", "project.naming = camelCase"。这些 facts 在新项目自动隐藏, 不会污染 其他项目。
key 命名规范
<scope-prefix>.<category>.<name>, 例如 user.preference.explanation_style。key 稳定, 不允许 LLM 自由发挥改名。

loop 接入: SessionStart 注入到 system prompt

// 教学简化版, 仓库真实实现见 GitHub 永久链接
// 第一次 run() 时, SessionStart 阶段
if (!sessionStarted) {
  sessionStarted = true;

  // 1. 加载 memory 渲染为 system prompt 片段
  const userMemory = await memoryStore.renderForPrompt("user");
  const projectMemory = await memoryStore.renderForPrompt("project");

  // 2. 拼接到 system prompt (用稳定前缀, 第 10 章 cache-friendly 伏笔)
  const basePrompt = history.getSystemPrompt() ?? "";
  history.setSystemPrompt(
    `${basePrompt}\n\n# User Memory\n${userMemory}\n\n# Project Memory\n${projectMemory}`
  );

  // 3. SessionStart Hook (第 08 章)
  const hookResult = await safeRunHook(/* ... */);
  if (hookResult.exitCode === 1) return hookResult.message;
}

// save_memory 工具
async function saveMemory(args) {
  await memoryStore.set(args.scope, args.key, args.value);
  return { toolCallId: call.id, content: `Saved ${args.scope}.${args.key}` };
}

关键设计: memory 拼到 system prompt 的固定位置 ("# User Memory" / "# Project Memory"), 即使 memory 内容变化, 这一行 prefix 稳定, 不会破坏第 10 章 cache-friendly 布局。memory 内容本身在 LLM 视角 是动态的, 但 prefix 锚点不变。

落盘: append-only + atomic write

memory 落盘必须用 atomic write (第 15 章会展开): 写新文件到 ~/.claude/memory/<scope>/<key>.md.tmp, fsync 之后 rename 到正式路径。直接 fs.writeFile 在断电时会留半截文件, 下次启动解析失败。

set() 内部追加 metadata header (updatedAt, scope), 这样 list() 时 不需要读文件内容就能展示元信息。文件本身是 markdown 格式, 人类可读, 也方便用户用编辑器手动修正。

反例梯度

新手错法 · A

常见错误:memory 写进 history。

为什么错:每次 LLM 调用都重传, token 浪费; 职责混淆; compress / replay 都会混乱。

正确做法:memory 独立落盘, SessionStart 注入 system prompt, 不写 history。

中级错法 · B

常见错误:不区分 scope, 所有 memory 写一个文件。

为什么错:用户换项目后项目级 fact 跟着走, 污染新项目。

正确做法:按 scope 写到不同目录, 渲染 system prompt 时按当前项目过滤。

高级错法 · C

常见错误:LLM 写 memory 时 key 自由发挥, 每次都新建一个 key。

为什么错:"user.preference.style" 和 "user.preference.explanation_style" 实际是同一件事, 写两份互相覆盖。

正确做法:key 命名规范, save_memory 工具在 schema 里硬约束 key 格式, harness 拒绝不合规的 key。

边界错法 · D

常见错误:memory 写入不原子, 断电后留半截文件。

为什么错:下次启动解析失败, user 长期偏好丢失。

正确做法:第 15 章的 atomic-write 模块兜底, write tmp + fsync + rename。

回望第 00–08 章: 哪些原则在本章兑现了

  • 事实与视图分离:memory 独立于 history, 是"长期事实", history 是"对话上下文"。这条原则在第 06 章 compress 隐含, 本章显式兑现。
  • 稳定前缀原则:memory 拼到 system prompt 的固定位置, prefix 锚点不变, 为第 10 章 cache-friendly 布局做准备。
  • 职责单一:memory 模块只管"读 / 写 memory 文件", 不参与 messages 构建 (那是 prepareMessages 的事)。
  • 错误降级原则:memory 读失败 (文件不存在) 降级为"空 memory", 不让主 loop 崩。

本章小结

本章给 harness 加了 Memory 模块, 区分 user 级与 project 级长期事实。 memory 独立于 history, 通过 SessionStart 注入 system prompt 的固定 位置 (prefix 稳定, 内容动态), 既给 LLM 长期事实, 又不破坏 cache 友好布局。下一章 (第 10 章) 我们会展开 prompt cache 友好布局的 完整设计: 哪些进稳定前缀, 哪些走 reminder, 哪些靠 history。
