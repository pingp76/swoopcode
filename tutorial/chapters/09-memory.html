第 09 章 · 跨会话的长期记忆

Memory: 4 类 tag 持久化团队规范与用户偏好

第 08 章的 Hook 给了"主循环周围挂逻辑" 的扩展点, 但 SessionStart handler 想读的"团队规范 / 用户偏好" 从哪来? 用户的 home 目录里 躺着一个 MEMORY.md, 团队 git 仓库里有一个 .claude/memory/ 目录, 但 LLM 不知道哪些"应该记住"。 这一章 加 Memory 系统: 4 类 tag (user / feedback / project / reference) × Markdown frontmatter 存储 × 自动生成索引 × Jaccard 相似度去重, 启动时通过 SessionStart handler 注入 system prompt。 读完后, 你能讲清"4 类 tag 的语义" (用户偏好 vs 团队规范 vs 项目 上下文 vs 参考资料), 并能用 fake filesystem 验证"name 合法 + 索引 重建 + 相似度去重" 3 条不变量。

真实失败故事: 用户说"我之前说过" harness 完全不知道

写代码之前, 先看一个真实痛点。 用户的 team 用 harness 跑 3 个月后, 反复问 LLM 同样的偏好:

  1. 用户偏好: 用户每次都说"中文回答 + TypeScript 不要 any", harness 每次都按默认回答。
  2. 团队规范: team 反复要求"PR 标题用 conventional commit 格式", harness 每次都按普通 commit 写。
  3. 项目上下文: 用户每次说"这个项目用 pnpm 不用 npm", harness 每次都跑 npm install。
  4. 参考资料: 用户每次说"读 README.md 第 3 节", harness 每次都搜错文件。

朴素想法 1: "把这些写进 system prompt?" 错。 system prompt 是 稳定前缀, 频繁改会破坏 prompt cache (第 10 章), 改一次成本 ×10。 system prompt 应该冷启动时一次写, 不 经常改。

朴素想法 2: "让 LLM 自己读 MEMORY.md?" 浪费 token, 而且 LLM 不知道 "应该读哪里", 容易读错。 应该 harness 主动按 type 筛选, 只把"高价值低频变" 的条目注入 system prompt。

正确做法: 加 src/memory.ts 模块 — 每条 memory 是独立 Markdown 文件 (frontmatter + body), 4 类 tag 区分, harness 在 SessionStart 启动时扫一次目录, 把所有条目按 type 排序后注入 system prompt, 之后 100 轮对话 cache 命中。 这是 Reference 章节 "模式 11 · Cache-friendly 缓存友好" 的具体应用。

先看一个例子: memory 注入后, system prompt 长什么样

在讲 4 类 tag / frontmatter 格式 / 工厂闭包之前, 先看一个 完整的注入示例。 学生脑子里先有"东西", 后面的 "怎么造这个东西" 才好懂。

假设用户在 home 目录的 ~/.memory/ 下存了 3 条 memory: 

// 文件 1: ~/.memory/chinese-answer.md
---
name: chinese-answer
description: 用户偏好中文回答
type: user
createdAt: 2026-06-12T10:00:00.000Z
updatedAt: 2026-06-12T10:00:00.000Z
---

用户偏好中文回答。 所有解释、注释、错误信息都用中文。
代码标识符 (变量名 / 函数名) 保留英文, 不要翻译。

// 文件 2: ~/.memory/no-any.md
---
name: no-any
description: TypeScript 不用 any
type: user
createdAt: 2026-06-12T10:05:00.000Z
updatedAt: 2026-06-12T10:05:00.000Z
---

TypeScript 代码不用 any。 优先用 unknown + 类型守卫, 或者
具体类型 (interface / type alias)。 实在无法确定类型时
才用 any, 并加注释说明。

// 文件 3: ~/.memory/commit-format.md
---
name: commit-format
description: PR 标题用 conventional commit
type: feedback
createdAt: 2026-06-12T10:10:00.000Z
updatedAt: 2026-06-12T10:10:00.000Z
---

PR 标题用 conventional commit 格式: type(scope): subject。
type 用 feat / fix / docs / refactor / test / chore。
scope 用模块名 (agent / tools / hooks ...)。 subject 中文
即可, 50 字以内。

这 3 条 memory 在 harness 启动时被 buildPromptSection() 拼成一段文字, 注入到 system prompt 的最前面: 

// system prompt 注入后的样子 (前 20 行)
[Memory Index]
- [user] chinese-answer: 用户偏好中文回答
- [user] no-any: TypeScript 不用 any
- [feedback] commit-format: PR 标题用 conventional commit

# Memory Details

[user] chinese-answer
用户偏好中文回答。 所有解释、注释、错误信息都用中文。
代码标识符 (变量名 / 函数名) 保留英文, 不要翻译。

[user] no-any
TypeScript 代码不用 any。 优先用 unknown + 类型守卫, 或者
具体类型 (interface / type alias)。 实在无法确定类型时
才用 any, 并加注释说明。

[feedback] commit-format
PR 标题用 conventional commit 格式: type(scope): subject。
type 用 feat / fix / docs / refactor / test / chore。
scope 用模块名 (agent / tools / hooks ...)。 subject 中文
即可, 50 字以内。

LLM 收到这段 system prompt 后, 行为立刻变了:

  • 用户问"帮我看下 src/agent.ts 的 bug", LLM 用中文回答, 不是英文。
  • 用户让 LLM "写个 helper", LLM 写的代码不出现 any, 优先用 unknown + 类型守卫。
  • 用户让 LLM "提个 commit", LLM 写 fix(agent): 修复 P0 衰减的 tool_call 配对错误, 不是 update code。

4 个关键观察:

  1. memory 段在 system prompt 最前面: 优先于 harness 默认 规则。 LLM 先看到"用户的偏好", 再看到"工具列表" / "通用规则"。
  2. 每条 memory 都标了 type: [user] / [feedback] 等前缀让 LLM 知道"这条是用户偏好还是 团队规范", 不同来源权重不同 (用户 > 团队 > 项目, 后面 "4 类 tag" 一节展开)。
  3. description 是一句话摘要: 让 LLM 不读 body 也能 大概知道这条 memory 是什么, 节省 token。 body 在 LLM 需要 细节时才读。
  4. 整个段在一次 run 内不变: 3 条 memory 的内容稳定, system prompt 100 轮对话都是同一段, cache 命中率高 (第 10 章 cache-friendly 展开)。 改了某条 memory 才需要重发 system prompt。

看到这里, 你应该能回答 3 个问题:

  • memory 是什么: 一段 Markdown + frontmatter, 存 ~/.memory/ 目录, 名字 = name 字段。
  • memory 怎么进 prompt: 启动时 buildPromptSection() 拼到 system prompt 最前面。
  • memory 为什么有用: 用户的偏好 / 团队规范 / 项目 上下文能跨 session 保留, LLM 不再"我之前说过" 失忆。

后面的章节展开"4 类 tag 怎么定" / "frontmatter 怎么写" / "工厂 + 闭包怎么实现"。 有了这个具体例子, 看后面的抽象概念就不抽象了。

4 类 tag: user / feedback / project / reference

用途: 不同来源 / 不同变更频率 / 不同优先级的 memory 条目应该分开存, 启动时按 type 决定注入 顺序。 4 类 tag 是固定枚举, 不允许自定义第 5 类 — 多了维护成本 上升, 少了语义冲突。

真实场景:

  • 用户把"中文回答" 存为 type: user
  • team 把"PR 用 conventional commit" 存为 type: feedback
  • 项目把"用 pnpm" 存为 type: project
  • 用户把"读 README.md 第 3 节" 存为 type: reference

4 类各管一摊, 不会把"个人偏好"和"团队规范"混在一起。

设计思想: 固定枚举 + 字段类型 — 4 类不是任意字符串, 是编译期可见的 union, 写错会 TypeScript 编译 失败。 看 src/memory.ts 第 30 行的 MemoryType union: 

export type MemoryType = "user" | "feedback" | "project" | "reference";
图 1 · 4 类 memory 的语义边界
user · 用户偏好
"中文回答" "不要用 any" "注释用中文" — 来自单个用户, 跨项目, 几乎不变。
feedback · 团队反馈
"PR 用 conventional commit" "测试要覆盖边界" — 来自团队, 跨项目但周期更新。
project · 项目上下文
"这个项目用 pnpm" "src/ 在 packages/core 下" — 来自项目, 项目内稳定, 跨项目无意义。
reference · 参考资料
"读 README.md 第 3 节" "API 文档在 https://..." — 高频查, 路径稳定但内容可能改。

实现细节: 4 类 tag 的语义正交, 不允许 出现"既是 user 又是 feedback" 的情况 — 真出现, 说明该拆成两条 memory。 启动注入顺序按 type 排序 (user → feedback → project → reference), 保证 system prompt 里的相对位置稳定, cache 命中。

文件格式: frontmatter + body 的 Markdown

用途: 每条 memory 是独立 .md 文件, 方便 git diff / review / 合并冲突解决。 文件名 = ${name}.md, 文件 内 frontmatter name 字段必须和文件名一致 (双重校验)。

真实场景: team 改了一条 feedback memory, 提 PR, reviewer 看 git diff 看到 frontmatter 改了 description + body 改了 一段, 接受合并。 如果用 JSON, 整个文件一行, review 看到的是 "改了一个字符但不知道改哪"。

设计思想: 人类可读 + 工具可解析 — 复用 skills.ts 的 parseFrontmatter() (第 05 章), 不引入 YAML 依赖。 看 src/memory.ts 第 151 行的 serializeMemory: 

export function serializeMemory(entry: MemoryEntry): string {
  const lines = [
    "---",
    `name: ${entry.meta.name}`,
    `description: ${entry.meta.description}`,
    `type: ${entry.meta.type}`,
    `createdAt: ${entry.meta.createdAt}`,
    `updatedAt: ${entry.meta.updatedAt}`,
    "---",
    entry.body,
  ];
  return lines.join("\n");
}

实现细节: frontmatter 5 个必填字段 (name / description / type / createdAt / updatedAt) — 缺一个就 warn 日志 + 跳过该文件, 不抛错。 这条"宽容失败" 是 Reference 章节 "模式 17 · Test Doubles 测试替身" 的反向应用 — 解析器要容忍 坏文件, 不因为一个坏文件就让 agent 启动失败。

name 字段有双重校验: 文件名 ${name}.md ↔ frontmatter name 必须一致。 不一致就 warn + 跳过。 这条"对称校验" 是 AGENTS.md "Validation Rules" 的体现 — name 是 identity 字段, 文件名和 frontmatter 必须由一个模块明确验证。

createMemoryManager 工厂 + 闭包缓存

用途: MemoryManager 接口在 Composition Root 创建, 通过闭包缓存 scan 过的 entries 列表, read() / list() 走内存不走磁盘, 性能稳定。

真实场景: SessionStart 启动时调一次 scan() 扫目录 (10 个文件, 大约 5ms), 之后 100 轮对话每次都调 buildPromptSection() 注入 system prompt, 这 100 次 调都走内存缓存, 不再读磁盘。 内存里的 entries 在 create() / delete() 后自动重建索引。

设计思想: 工厂 + 闭包 — 这是 Reference 章节 "模式 1 · 工厂 + 闭包" 的标准应用: 内部状态 (cachedEntries + memoryDir + logger) 全在闭包内, 外部只能通过 MemoryManager 接口访问。 看 src/memory.ts 第 331 行的 createMemoryManager 工厂: 

export function createMemoryManager(options: {
  memoryDir: string;
  logger: Logger;
}): MemoryManager {
  const { memoryDir, logger } = options;
  let cachedEntries: MemoryEntry[] = [];  // 闭包内状态

  function scan(): MemoryEntry[] { /* 读目录 + 解析 */ }
  function read(name: string): MemoryEntry | null { /* 查缓存或读单文件 */ }
  function create(input: CreateMemoryInput): MemoryEntry { /* 写文件 + 重建索引 */ }
  // ...

  return { scan, read, create, list, findSimilar, delete, buildPromptSection, rebuildIndex, getMemoryDir };
}

实现细节: scan() 每次都清空 cachedEntries = [] 然后重读, 保证结果反映磁盘最新 状态; read(name) 优先查缓存, 缓存没有再读单文件 (允许 "scan 之后外部改文件" 的边界情况)。 闭包隔离让多个 MemoryManager 实例 (父子 subagent 各一个) 互不污染。

name 合法性 + type 合法性 + name == 文件名 三重校验

用途: 写入和读取都要校验, 不合法的 name (路径穿越 / 隐藏文件 / 特殊字符) 一律拒绝。 这是 Reference 章节 "模式 15 · Identity Check 身份校验" + AGENTS.md "Validation Rules: 字段 一致性" 的具体应用。

真实场景:

  • LLM 调 memory_create({name: "../../etc/passwd", ...}) 想穿越目录, name 合法性校验拒绝
  • LLM 调 memory_create({name: "foo", ...}) 但磁盘上文件叫 bar.md, name == 文件名校验拒绝

设计思想: 3 重校验:

  1. name 合法字符: 只允许 [a-z0-9_-]+, 看 src/memory.ts 第 127 行的 isValidName。 拒绝大写 / 空格 / 点 / 斜杠, 防止路径穿越和 case-insensitive 文件系统的歧义。
  2. type 合法枚举: 只允许 4 类 tag, 看 src/memory.ts 第 134 行的 isValidType。 拒绝任意字符串, 防止"我自己加一类" 破坏 4 类语义。
  3. name == 文件名: frontmatter name 必须 等于文件名 (去掉 .md), 看 src/memory.ts 第 394 行的对称校验。 防止"改了 frontmatter 没改名" 或"改了名没改 frontmatter" 的 身份错位。

实现细节: 3 重校验失败都只 warn 日志 + 跳过, 不抛错。 一个坏文件不能阻止整个 agent 启动 — 用户可能手改了 frontmatter, harness 应该宽容。 这是 Reference 章节 "模式 13 · Error→Action 错误转动作" 的应用 — 错误 = warn + 跳过, 不是 throw。

MEMORY.md 自动索引: 不手写

用途: 用户 / reviewer 看 memory/ 目录 时需要一个"目录页" — 哪些 memory 存在, 各自是哪个 type, 一句话 描述是什么。 索引自动生成, 不手写, 否则容易"加 memory 忘了同步 索引"。

真实场景: 用户想看 "我项目里有哪些 feedback 类型 的 memory", 直接读 MEMORY.md 看索引 — 按 type 排好 序, 每行 - [feedback] name: description。 这是 Reference 章节 "模式 11 · Cache-friendly 缓存友好" 的延伸 — 索引是 cache miss 时 fallback 的"快速查表"。

设计思想: 派生数据 — 索引是 scan 结果的派生, 每次 create() / delete() 后自动 rebuild, 用户不能手写 (被排除在 scan 之外)。 看 src/memory.ts 第 234 行的 buildIndexContent: 

function buildIndexContent(entries: MemoryEntry[]): string {
  const header = "# Memory Index\n";
  const lines = entries.map(
    (e) => `- [${e.meta.type}] ${e.meta.name}: ${e.meta.description}`,
  );
  return header + "\n" + lines.join("\n") + "\n";
}

实现细节: 索引按 type → name 稳定排序 (看 src/memory.ts 第 247 行的 getStableSortKey), 保证每次 rebuild 出来的 MEMORY.md 内容一致, git diff 只显示 "新增/删除/更新", 不显示"顺序变了"。

相似度去重: Jaccard 教学版

用途: 用户创建 memory 前, 检查是否已存在相似条目 (Jaccard 相似度 ≥ 0.5 就算相似), 给 LLM 返回"已有相似条目, 请确认是要新增还是要更新"。 防止"反复加重复 memory"。

真实场景: LLM 想加 "中文回答" (description), 但 磁盘上已有 "中文交流" (description + body 提到"中文"), 相似度 0.6, 返回"已有相似条目, 是否要更新?", LLM 决定更新而不是新增。

设计思想: 教学版去重 — 不引入 embedding / 向量数据库, 用 Jaccard token 重叠系数, 简单到能 在 30 行内讲清楚。 教学项目优先可读性, 不追求工业级精度。 看 src/memory.ts 第 288 行的 calculateTokenOverlap: 

function calculateTokenOverlap(a: Set<string>, b: Set<string>): number {
  if (a.size === 0 || b.size === 0) return 0;
  let intersection = 0;
  for (const token of a) {
    if (b.has(token)) intersection++;
  }
  const union = a.size + b.size - intersection;
  return union === 0 ? 0 : intersection / union;
}

实现细节: 文本先经过 normalizeForSimilarity (看 src/memory.ts 第 261 行) — 小写 + 标点变空格 + 连续空白压缩, 再 tokenizeForSimilarity 提取长度 ≥ 2 的 token。 教学版的局限: 不识别同义词 ("中文" vs "汉语"), 不识别语序变化 ("用 pnpm" vs "pnpm 是这个项目的包 管理器")。 工业级要做 embedding, 教学版讲 Jaccard 就够了。

buildPromptSection: 启动注入到 system prompt

用途: SessionStart handler 调 memoryManager.buildPromptSection(), 把所有 memory 按 type 排序后拼成一段短文本, 注入 system prompt 的最前面 (稳定前缀)。

真实场景: SessionStart handler 写: const section = memoryManager.buildPromptSection(); if (section) systemPrompt = section + "\n\n" + systemPrompt;。 之后 100 轮对话, system prompt 里的 memory 段不变, cache 命中。

设计思想: 冷启动一次, 热对话复用 — memory 段在 SessionStart 一次写入 system prompt, 之后所有 history 都引用同一份前缀, OpenAI / Anthropic 的 prompt cache 按前缀命中 (第 10 章详细展开)。 这是 Reference 章节 "模式 11 · Cache-friendly 缓存友好" + "模式 12 · Reminder 注入提醒" 的 联合应用。

实现细节: 注入的 section 应该短小精悍 — 每条 memory 只取 description, 不取 body。 body 留给 LLM 需要时 主动调 memory_read(name) 查全量, 避免 system prompt 撑爆。 这就是"摘要 vs 全文" 的设计权衡。

主循环集成: SessionStart handler 读 memory

场景: memory 文件写好了, 怎么进 system prompt

上一节我们看到, 3 条 memory 文件被拼成一段文字, 应该出现在 system prompt 最前面。 但 memory.ts 是纯模块 (管文件 读写), agent.ts 是主循环 (管 LLM 调用和 tool 执行), 这两个模块怎么串起来?

朴素想法 1: "memory.ts 直接 import agent.ts, 在 MemoryManager 里调 agent.addSystemMessage?" 错。 这是 循环依赖 — agent.ts 也需要 MemoryManager, 两个模块互相 import, 一启动就栈溢出。 更糟的是, memory 模块不该知道 "agent 长什么样"。

朴素想法 2: "agent.ts 直接读 memory, 启动时调 memoryManager.buildPromptSection() 拼 system prompt?" 表面 行, 但违反关注点分离 — agent 主循环不该关心 "memory 是什么" / "怎么拼", 它只该调 LLM。 哪天加 Skill 注入 规范, 是不是又改 agent.ts? 加一个 Schedule 提醒是不是又改? agent.ts 会变成"巨型 switch", 谁都不敢动。

正确做法: 用第 08 章的 Hook 系统

SessionStart 是 Hook 的 3 个事件之一 — "harness 启动时发一次"。 写一个 handler, 让它在 SessionStart 触发时:

  1. 调 memoryManager.buildPromptSection() 拿到 memory 段
  2. 返回 {exitCode: 2, message: section} (用 2 = inject, 不是 0)

主循环收到 handler 返回的 inject message, 按延迟注入协议, 追加到 system prompt。 memory.ts 完全不知道 Hook 的存在, agent.ts 完全不知道 Memory 的存在, 只有 handler 这个小桥接文件 知道两边。 这就是 Reference 章节 "模式 3 · 窄接口 + Adapter" 的 应用 — 模块之间不直接耦合, 通过 Composition Root 拼装。

具体代码 (在 index.ts 拼装) 

// src/hooks/memory-inject.ts — handler 实现, 是 memory 和 hook 的桥
import type { HookHandler } from "../hooks.js";
import type { MemoryManager } from "../memory.js";

export function createMemoryInjectHandler(
  memoryManager: MemoryManager,
): HookHandler<{ name: "SessionStart"; payload: { query: string } }> {
  // 返回 handler 函数, 闭包持有 memoryManager
  return (event) => {
    const section = memoryManager.buildPromptSection();
    if (!section) {
      return { exitCode: 0 };   // 没 memory 文件, 啥也不做
    }
    return {
      exitCode: 2,              // 2 = inject, 后续追加到 system prompt
      message: section,         // memory 段文本
    };
  };
}

// src/index.ts — Composition Root, 把 memory + hook 拼装
import { createMemoryManager } from "./memory.js";
import { createHookRunner } from "./hooks.js";
import { createMemoryInjectHandler } from "./hooks/memory-inject.js";

const memoryManager = createMemoryManager({ memoryDir: "~/.memory/", logger });
const hookRunner = createHookRunner({
  SessionStart: [createMemoryInjectHandler(memoryManager)],  // ← 拼装
}, logger);

// agent.ts 完全不知道 memory 的存在, 只调 hookRunner.run(event)
// memory.ts 完全不知道 hook 的存在, 只管文件读写
const agent = createAgent({ hookRunner, ... });

hooks.ts (Hook 系统)
职责: 3 个事件 + run() 接口。
不知道 memory 的存在 (handler 是外部注册的)。
agent.ts (主循环)
职责: 调 LLM, 调 tool, 调 hookRunner.run(event)。
不知道 memory 的存在 (只看到 hookRunner 通用接口)。
hooks/memory-inject.ts (桥接文件)
职责: 调用 memoryManager.buildPromptSection(), 包成 {exitCode: 2, message}。
唯一知道两边的桥, 不在 memory.ts / hooks.ts 里。

关键不变量: 0 循环 import

  • memory.ts 不 import hooks.ts
  • hooks.ts 不 import memory.ts
  • agent.ts 不 import memory.ts
  • hooks/memory-inject.ts 同时 import memory.ts 和 hooks.ts, 是唯一桥接点

这是整个项目架构设计的核心原则 — 模式 3 (依赖注入) + 模式 4 (Composition Root) 都是为了实现这条原则。 完整解释 (为什么不能互相 import / 桥接文件怎么命名 / 仓库里 5 个体现) 见 Reference · 模式 3.5 · 桥接模式 / 0 循环依赖。

为什么 handler 返回 2 (inject) 而不是 0 (continue)

3 个返回码的语义:

  • 0 (continue): 啥也不做, 主循环继续。
  • 1 (block): 阻止当前动作。
  • 2 (inject): 把 message 注入到 system prompt。

memory 段需要追加到 system prompt, 走 2 (inject) 才对。 0 是"我听了但没意见", 主循环不会读 message 字段, memory 段 永远进不了 prompt。 走 1 是"阻止", 完全没意义。 只有 2 能让 memory 段真正生效。 (返回码细节见第 08 章"3 个返回码" 一节。)

幂等: SessionStart handler 只跑 1 次

同一 session 内, SessionStart 事件只发 1 次 (在 agent.run() 第一次被调用时), 不会每轮都发。 所以 memory 段只注入 1 次, 后续 100 轮对话引用同一份 system prompt 前缀, cache 命中。 (cache 细节见第 10 章 "Cache" 一节。)

对 prompt cache 的影响: memory 段是 stable prefix 一部分

用途: 注入 system prompt 的 memory 段应该 位置稳定 + 内容稳定。 位置 = 拼在 system prompt 最前面; 内容 = 同一 session 内不变。

真实场景: 团队加了一条 feedback memory, 重启 session 后 system prompt 里的 memory 段多了 1 行, 整个对话的 cache prefix 变长 100 字符, 后续 100 轮对话 cache 全 miss。 正确做法: 接受这次 miss, 后续 100 轮对话 cache 重新累积。 memory 段应该是"冷启动一次, 后续稳定", 不是"每轮 变"。

设计思想: 冷数据, 不热数据 — memory 是"几乎不变" 的数据, 不应该频繁写。 想要"动态变化" 用 Reminder (第 03 章), 不是 memory。 memory 和 Reminder 职责 切分: memory = 冷, Reminder = 热。

实现细节: 这是和第 05 章 (Skill 稳定前缀) + 第 08 章 (Hook SessionStart 注入) 联合的 cache 策略。 三者 都强调"前缀稳定", 拼在 system prompt 最前面, OpenAI / Anthropic 的 prompt cache 按前缀命中 (前缀 1024 字符左右开始 cache)。

fake test: 用 tmp 目录验证 3 条不变量

用途: Memory 测试不需要真实 memory 目录, 用 os.tmpdir() 临时目录即可 — 测的是"文件系统读写 + 解析 + 校验" 的集成行为, 不是单函数逻辑。

真实场景: 用户写测试验证"加 3 条 memory, 调 scan() 拿到 3 条, 索引 MEMORY.md 包含 3 行"。 用 tmp 目录, 测完清理, 不污染 home 目录。

设计思想: 集成测试覆盖核心 流程 — 写文件 + 读 + 解析 + 校验 + 索引, 一个测试走完整个 链路, 比 5 个单测加起来更稳。 看测试示例: 

test("完整流程: create -> scan -> list -> buildPromptSection", () => {
  const dir = fs.mkdtempSync(path.join(os.tmpdir(), "mem-test-"));
  const mgr = createMemoryManager({ memoryDir: dir, logger: noopLogger });

  mgr.create({ name: "chinese", description: "中文回答", type: "user", body: "..." });
  mgr.create({ name: "commit-format", description: "conventional commit", type: "feedback", body: "..." });
  mgr.create({ name: "use-pnpm", description: "用 pnpm", type: "project", body: "..." });

  const entries = mgr.scan();
  expect(entries).toHaveLength(3);

  // 按 type 排序: feedback 在前, project 在后, user 在中间
  expect(entries[0]!.meta.type).toBe("feedback");
  expect(entries[1]!.meta.type).toBe("user");
  expect(entries[2]!.meta.type).toBe("project");

  // 索引文件自动生成
  const indexContent = fs.readFileSync(path.join(dir, "MEMORY.md"), "utf-8");
  expect(indexContent).toContain("- [feedback] commit-format:");
  expect(indexContent).toContain("- [user] chinese:");
});

test("非法 name 拒绝", () => {
  const dir = fs.mkdtempSync(path.join(os.tmpdir(), "mem-test-"));
  const mgr = createMemoryManager({ memoryDir: dir, logger: noopLogger });
  expect(() => mgr.create({ name: "../escape", description: "x", type: "user", body: "" }))
    .toThrow(/Invalid memory name/);
});

test("frontmatter name != 文件名 警告跳过", () => {
  const dir = fs.mkdtempSync(path.join(os.tmpdir(), "mem-test-"));
  fs.writeFileSync(path.join(dir, "foo.md"), "---\nname: bar\n---\n");
  const mgr = createMemoryManager({ memoryDir: dir, logger: noopLogger });
  const entries = mgr.scan();
  expect(entries).toHaveLength(0);  // 校验失败, 跳过
});

实现细节: 3 个测试覆盖 (a) 完整读写流程, (b) name 合法性校验, (c) name == 文件名校验。 不需要 mock fs, 真实 tmp 目录比 in-memory fs 更稳 — 真实测出 "文件权限 / 编码" 这类边界。

常见误解: 4 类不是权限, 是来源

误解 1: "4 类是权限分级?" 错。 4 类 tag 是 来源 (谁写的), 不是权限 (谁能 改)。 想做权限分级, 加 role-based access, 不是 4 类 tag。

误解 2: "memory 越多越好?" 错。 memory 段注入 system prompt, 多了撑爆 cache prefix 命中率。 应该 5-20 条 精炼条目, 不该 100 条碎片。 LLM 加 memory 前应该先 findSimilar 检查是否已存在。

误解 3: "MEMORY.md 手写也可以?" 不行。 MEMORY.md 是自动生成的索引, scan 时跳过。 手写会被忽略, 而且和 create / delete 后自动 rebuild 的内容冲突。

误解 4: "Jaccard 相似度能识别同义词?" 错。 教学版 Jaccard 只看字面 token 重叠, "中文"和"汉语"不重叠。 想要 同义词识别, 引入 embedding, 不在本章范围。

反例梯度: 4 个常见错误

反例 1 · 不校验 name

让 memory_create({name: "../../etc/passwd"}) 走通, LLM 写入 memory/../../etc/passwd.md, 路径穿越。 错。 任何写入都必须 isValidName 校验, 失败 throw, 不让 LLM 重试。 路径穿越是安全漏洞, 不是"格式问题"。

反例 2 · 不校验 name == 文件名

LLM 改 frontmatter 把 name: foo 改成 name: bar, 但文件名还是 foo.md。 表面无害, 实际 read("foo") 找不到 (文件叫 foo 但 frontmatter 写 bar), read("bar") 也找不到 (文件叫 foo)。 错。 scan 时必须校验 frontmatter name == 文件名, 不一致 warn + 跳过。

反例 3 · 解析失败抛错

用户手改了一个 memory 文件, frontmatter 少一个字段, parser throw。 agent 启动失败。 错。 解析失败应该 warn 日志 + 跳过 单个文件, 不 throw, 不阻止 agent 启动。 harness 容忍坏文件, 用户修好之后下次启动自动恢复。

反例 4 · memory 段写在 history 而非 system

把 buildPromptSection() 的结果 append 到 history[] 第一条 user message, 而不是注入 systemPrompt。 表面能跑, 实际破坏 prompt cache (history 增长每次都变, system 不变)。 错。 memory 段必须 注入 system prompt, 让它成为稳定前缀的一部分。

Validation: 4 条不变量检验

  1. 写读对称: create() 写入的条目, read(name) 必须能读回, 字段 (description / type / body) 完全一致。 验证: tmp 目录, create 一条带特殊字符 body (含中文 + 换行 + JSON), read 回来字符串相等。
  2. name 合法性: create({name: "Foo/../bar"}) 必须 throw, 错误信息含 "Invalid memory name"。 验证: 单测覆盖 大写 / 斜杠 / 点 / 空格 / 空字符串 5 类非法 name。
  3. name == 文件名: 写入 foo.md 但 frontmatter name: bar, scan 跳过该文件 + warn 日志。 验证: tmp 目录, 写一个不一致的文件, scan 拿不到。
  4. 索引派生: create / delete 后 MEMORY.md 内容自动 rebuild, 顺序稳定 (按 type → name 排序), 多次 rebuild 内容 字节相等。 验证: 写 3 条 memory, rebuild, 再 rebuild, 字节 相等 (git diff 显示 0 行)。

回望: 哪些原则在本章兑现了

  • 工厂 + 闭包: createMemoryManager 把 状态藏在闭包, 外部只通过 MemoryManager 接口访问。
  • 派生数据: MEMORY.md 索引是 scan 派生, 不手写, 自动 rebuild, git diff 干净。
  • 3 重身份校验: name 合法字符 + type 合法枚举 + name == 文件名, 防路径穿越 / 类型错乱 / 身份错位。
  • 宽容失败: 解析失败 warn + 跳过, 不 throw, agent 启动不因单条坏 memory 失败。
  • 冷数据进稳定前缀: memory 段注入 system prompt 而非 history, 保持 cache 命中。

前瞻张力: 留给后续章节

多 session 一致性
memory 持久化在磁盘, 跨 session 生效。 但 subagent 嵌套时, 子 agent 是否继承父 agent 的 memory? 当前策略: 继承。 这可能让子 agent 看到不该看到的 user 偏好 (如"中文回答" 是 用户的, 子 agent 跑英文项目不该看)。 第 11 章 Recovery 讨论 subagent 隔离时, memory 隔离会作为子话题。
并发写入冲突
多个 subagent 同时 create() 同一 name, 后写 覆盖先写, 丢 createdAt。 当前不处理。 第 13 章 Async Run 讨论并发时, memory 写入是冲突点之一。
MEMORY.md 是单文件
100 条 memory 时 MEMORY.md 仍然是一份索引, 不分裂。 rebuild 100 条 ~1ms, 规模 OK。 但要监控条目数, 超过 50 该提醒用户"该清理了"。

Prompt Card (本章任务)

Prompt Card · 第 09 章

目标: 给 harness 加 Memory 系统, 4 类 tag 持久化 到 memory/ 目录, SessionStart handler 注入 system prompt。

场景: 用户加 3 条 memory — "中文回答" (user) / "conventional commit" (feedback) / "用 pnpm" (project), 启动 session, system prompt 多了 3 行摘要, 后续 100 轮对话 cache 命中。

模块: src/memory.ts (新) 暴露 createMemoryManager({memoryDir, logger}); src/index.ts (改) Composition Root 创建 + 注入 SessionStart handler; src/system-prompt.ts (改) SessionStart 阶段调 buildPromptSection() 注入。

边界 (LLM 必须遵守的 checklist):

  • 4 类 tag 固定枚举 user / feedback / project / reference, 不可新增
  • name 只允许 [a-z0-9_-]+, 写时校验, 失败 throw
  • type 必须属于 4 类枚举, 写时校验, 失败 throw
  • frontmatter name 必须等于文件名 (去 .md), scan 时校验, 失败 warn + 跳过
  • 解析失败 (frontmatter 缺字段 / type 非法 / 字段含换行) warn + 跳过, 不 throw
  • MEMORY.md 索引是派生数据, 自动 rebuild, scan 时跳过, 用户不手写
  • buildPromptSection() 只取 description, 不取 body, 控制注入体积
  • Jaccard 相似度 ≥ 0.5 返回相似条目, 教学版不引入 embedding

验证 (用 tmp 目录 + vitest, 逐条断言):

  • create → scan → list → buildPromptSection 完整流程, 3 条 memory 按 type 排序
  • create({name: "../escape"}) throw, 错误信息含 "Invalid memory name"
  • 手写 foo.md 但 frontmatter name=bar, scan 返回 0 条, warn 日志
  • create / delete 多次后, MEMORY.md 字节稳定, 排序一致

本章练习

  1. 故意不写 name 合法性校验, 跑 create({name: "../escape"}) 测试, 看"路径穿越" 是否抓到 (文件被写到 memory 目录外)。
  2. 故意不写 name == 文件名校验, 手改 foo.md 的 frontmatter name=bar, 跑 scan, 看"身份错位" 是否抓到 (read 找不到任何条目)。
  3. 故意让 parser throw (不 catch), 跑 scan 一个 frontmatter 缺字段 的文件, 看"宽容失败" 是否抓到 (agent 启动失败 vs. warn + 跳过, 启动成功)。
  4. 把 buildPromptSection() 的结果 append 到 history 第一条 user message, 跑 SessionStart 测试, 看"memory 段进 稳定前缀" 是否抓到 (cache prefix 增长, 后续轮 miss)。

本章小结

Memory 是给 system prompt 的冷数据, 4 类 tag 把 来源分清楚。 核心是 5 个设计:

  • 4 类 tag: user / feedback / project / reference, 固定枚举, 不可新增。
  • Markdown 文件: frontmatter 5 字段 + body, 文件名 = ${name}.md, 双重校验。
  • 3 重身份校验: name 合法字符 + type 合法枚举 + name == 文件名, 缺一不可。
  • 派生索引: MEMORY.md 自动 rebuild, 按 type → name 稳定排序, git diff 干净。
  • Jaccard 教学版去重: 不引入 embedding, 30 行讲清楚, 容忍"中文 vs 汉语" 的局限。

    • 下一章 (第 10 章) 展开 prompt cache 的具体策略 — 稳定前缀的边界 在哪, 哪些字段"必须稳定", 哪些"可以变化", cache miss 后的 fallback。
