第 08 章 · 在 Loop 周围挂钩子

给 Harness 留事实基础: Hook

前面七章让 harness 能聊天、能调工具、能跑子任务、能压缩、能拦权限。 但 harness 的关键事件 (permission 决策、工具执行、状态变化) 现在还没有 统一记录。这一章在 loop 周围挂三个钩子: PreToolUse (执行前)、 PostToolUse (执行后)、SessionStart (会话开始), 让 harness 的关键事件 有 transcript 兜底, 也为 eval 重放和审计留事实基础。

在第 07 章基础上改了什么

这一章在 agent 主循环的关键节点插入 Hook 调用点。PreToolUse 在 permission.check 之后、tool.execute 之前触发, PostToolUse 在 tool.execute 之后、history.add(tool message) 之前触发, SessionStart 在第一次 run() 调用、history.add(user message) 之前触发。Hook 返回 exitCode: 0 (pass through) / 1 (block) / 2 (inject message) 三种 状态, harness 根据 exitCode 决定后续行为。 对应到代码, 改动集中在 3 个文件: src/hooks.ts (新)、 src/agent.ts (改关键节点插入 hookRunner.run())、 src/index.ts (改接线, 注入 hookRunner)。

1. src/hooks.ts: 钩子运行器 (新) 2. src/agent.ts: PreToolUse / PostToolUse / SessionStart 接入点 3. src/transcript.ts: Hook 决策记录 (第 15 章展开)

作者怎么想的: 这一章的思考链

想清楚现象
harness 跑了一个月之后, 出了"为什么这个工具没执行" 的问题, 但 history 里只有"Permission denied" 这条 tool message, 没有"为什么 permission 拒绝" 的具体信息。现象是"决策过程不透明"。
想反例
最朴素的反例是"在 permission 内部 console.log"。这有两个问题: 一是日志进了 stdout, 但 transcript 没记录, eval 重放时无法还原; 二是 console.log 是副作用, 测试时无法断言"permission 拒绝时是否 输出了正确原因"。
想接口和不变量
接口: interface HookRunner { run(event): Promise<HookResult> }。 不变量三条: (1) Hook 是异步的, 但不能修改主消息流 (只能注入新消息 或返回 block), (2) exitCode 1 必须对应到具体副作用, 不能"无理由 block", (3) Hook 失败 (抛错) 必须降级为 pass through, 不能让 harness 因 Hook 崩溃。
想怎么验证
fake HookRunner 预设 PreToolUse 返回 exitCode 1, 跑完后 history 末尾有 tool message "Blocked by PreToolUse hook: ..."; spy 验证 tool.execute 没被调用。

先观察: 两段故意有气味的实现

观察 1 · Hook 抛错让 harness 崩

// 教学简化版, 仓库真实实现见 GitHub 永久链接
const result = await hookRunner.run(event);
if (result.exitCode === 1) {
  // 错误: 没写 tool message
  return;  // 直接 return, tool 也不执行了
}

问:为什么不写 tool message, 直接 return?

答:三件事同时坏掉 —— 协议: tool_call 必须配对 tool_message, 缺一条 messages 序列就断裂; 真实性: LLM 不知道"为什么没执行", 反复重试; 体验: eval 重放时看不到 "Hook 在此刻 block 了什么"。

观察 2 · Hook 抛错让主 loop 崩

// 教学简化版
try {
  const result = await hookRunner.run(event);
} catch (err) {
  throw err;  // 错误: 让 hook 错误冒泡到 agent
}

问:为什么不让 hook 错误冒泡?

答:Hook 是用户扩展点, 第三方 Hook (例如审计 Hook) 不应该有能力让 harness 崩溃。Hook 抛错必须降级为 pass through, 写一条 error tool message 告诉 LLM "Hook 失败, 已忽略"。

接口形状: 在写实现前钉死

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export type HookEventName =
  | "SessionStart" | "PreToolUse" | "PostToolUse" | "SubagentStart" | "SubagentEnd";

export interface HookEvent {
  name: HookEventName;
  payload: Record<string, unknown>;  // 事件专属字段
}

export type HookExitCode = 0 | 1 | 2;
// 0: pass through (继续执行)
// 1: block (阻止后续动作, 但 tool_call 必须配对 tool_message)
// 2: inject (允许后续动作, 追加一条 user 消息到 history)

export interface HookResult {
  exitCode: HookExitCode;
  message?: string;  // exitCode 1 时为拒绝原因, exitCode 2 时为注入消息
}

export interface HookRunner {
  run(event: HookEvent): Promise<HookResult>;
}

loop 接入: 三个事件三个位置

// 教学简化版, 仓库真实实现见 GitHub 永久链接
// SessionStart: 第一次 run() 时触发
if (!sessionStarted) {
  sessionStarted = true;
  const result = await safeRunHook(hookRunner, { name: "SessionStart", payload: { query } });
  if (result.exitCode === 1) {
    return result.message ?? "Session blocked by hook.";  // 整个会话被拒
  }
  // exitCode 2 注入的 message 在用户消息之后追加
}

// 在工具执行路径上
for (const call of assistant.tool_calls) {
  // 1. permission
  const decision = permissionManager.check({ toolName: call.name, args: call.args });
  if (decision.action === "deny") { /* ... */ continue; }
  if (decision.action === "ask") { /* ... */ }

  // 2. PreToolUse Hook (permission 之后, execute 之前)
  const preResult = await safeRunHook(hookRunner, {
    name: "PreToolUse",
    payload: { toolName: call.name, args: call.args, round: loopRound },
  });
  if (preResult.exitCode === 1) {
    history.add({ role: "tool", tool_call_id: call.id,
      content: `Blocked by PreToolUse hook: ${preResult.message ?? ""}` });
    continue;
  }
  // exitCode 2 注入的消息延迟到所有 tool_result 写完后再追加 (避免破坏协议)

  // 3. execute
  const result = await tool.execute(call.args);

  // 4. PostToolUse Hook (execute 之后, tool message 写入之前)
  const postResult = await safeRunHook(hookRunner, {
    name: "PostToolUse",
    payload: { toolName: call.name, args: call.args, output: result.content, error: result.error },
  });
  // PostToolUse exitCode 1 不能撤销已执行的副作用, 只能注入警告
  if (postResult.exitCode === 1) {
    pendingHookMessages.push(`[Hook: PostToolUse] block: ${postResult.message ?? ""}`);
  }
  if (postResult.exitCode === 2) {
    pendingHookMessages.push(`[Hook: PostToolUse] ${postResult.message ?? ""}`);
  }

  history.add({ role: "tool", tool_call_id: call.id, content: result.content });
}

// 5. 所有 tool_result 写完后, 统一追加 Hook 注入的 user 消息
for (const msg of pendingHookMessages) {
  history.add({ role: "user", content: msg }, timing);
}

错误降级: hook 抛错不能崩 harness

// 教学简化版, 仓库真实实现见 GitHub 永久链接
async function safeRunHook(hookRunner: HookRunner, event: HookEvent): Promise<HookResult> {
  try {
    return await hookRunner.run(event);
  } catch (err) {
    logger.warn("Hook %s failed: %s", event.name, err);
    return { exitCode: 0 };  // 降级为 pass through
  }
}

第三方 Hook 不应该有能力让 harness 崩溃。Hook 抛错时, harness 降级 为 pass through, 写一条 warning 到 logger, 主 loop 继续。

反例梯度

新手错法 · A

常见错误:PreToolUse block 时不写 tool message。

为什么错:破坏第 02 章 tool_call 配对协议, messages 序列断裂。

正确做法:block 必写 role: "tool" 消息 "Blocked by PreToolUse hook: ..."。

中级错法 · B

常见错误:Hook 抛错时让主 loop 崩。

为什么错:第三方 Hook 不应该有能力让 harness 崩溃, 主 loop 健壮性优先。

正确做法:safeRunHook() 包 try/catch, 抛错降级为 exitCode 0。

高级错法 · C

常见错误:Hook 注入的 user 消息插在 tool_result 之间。

为什么错:多工具并行调用时, 插在 tool_result 之间会破坏 LLM 协议的 tool_call/tool_result 配对规则。

正确做法:Hook 注入消息延迟到所有 tool_result 写完之后, 统一追加。

边界错法 · D

常见错误:PostToolUse exitCode 1 试图撤销已执行的副作用。

为什么错:工具已经执行 (例如文件已写、命令已跑), 撤销 API 不存在。PostToolUse block 只能注入警告, 不能"假装撤销"。

正确做法:PostToolUse exitCode 1 注入 "[Hook: PostToolUse] block: ..." 警告, 不假装撤销。

回望第 00–07 章: 哪些原则在本章兑现了

  • tool_call 协议沿用:PreToolUse block 必写 tool message, Hook 注入消息延迟追加, 不破坏第 02 章的协议约束。
  • Composition Root 唯一接线:hookRunner 在 index.ts 创建, 注入 agent, 不在 agent.ts 内 new。
  • 错误降级原则:Hook 抛错不崩主 loop, 这条原则在第 11 章 recovery 会进一步展开 (LLM 抛错也降级)。
  • 事实与视图分离:Hook 决策不进 history, 进 transcript (第 15 章展开), history 仍然保持"对话历史" 的单一职责。

Prompt Card (本章任务)

Prompt Card · 第 08 章

目标:在 agent loop 关键节点插入 Hook 调用点, 让 harness 的关键事件 (permission 决策、工具执行、状态变化) 有 transcript 兜底。

场景:用户输入 "清理临时文件", LLM 调用 run_bash, PreToolUse Hook 记录 decision, tool.execute 之后 PostToolUse Hook 记录 result, transcript 留下完整事件流供 eval 重放。

模块: src/hooks.ts (新) 暴露 createNoopHookRunner() 和 createHookRunner(config); src/agent.ts 在 PreToolUse / PostToolUse / SessionStart 三个位置插入 hookRunner.run(); src/index.ts 接线 hookRunner。

边界 (LLM 必须遵守的 checklist):

  • PreToolUse block (exitCode 1) 必写 role: "tool" 消息 "Blocked by PreToolUse hook: ..."
  • Hook 抛错时降级为 exitCode 0, 不让主 loop 崩
  • Hook 注入消息 (exitCode 2) 延迟到所有 tool_result 写完之后统一追加
  • PostToolUse block 仅追加 user 消息警告, 不撤销已执行的副作用
  • SessionStart block 终止 run(), history 长度 = 0 (用户消息未写入)

验证 (用 fake HookRunner + spy registry, 逐条落到 vitest):

  • PreToolUse exitCode 1, spy.execute 没被调用, history 末尾有 tool message "Blocked by PreToolUse hook"
  • fake HookRunner.run 抛错, 主 loop 继续, tool.execute 被调用
  • 2 个 tool_calls + PreToolUse exitCode 2, history 末尾顺序: [tool, tool, user(Hook 注入)]
  • SessionStart exitCode 1, agent.run() 返回 "Session blocked by hook.", history 长度 = 0
  • PostToolUse exitCode 1, tool.execute 仍被调用 1 次, history 末尾追加 user 警告

  2. 让 fake HookRunner.run 抛错, 但 safeRunHook 不包 try/catch, 跑测试, 看"Hook 抛错降级" 是否抓到。
  3. 把 Hook 注入消息直接写 history (不延迟), 跑测试, 看"Hook 注入消息延迟追加" 是否抓到 (2 个 tool_calls 时中间不能有 user message)。

本章小结

本章给 harness 加了 Hook 机制, 在 loop 关键节点 (SessionStart / PreToolUse / PostToolUse) 插入可扩展点。Hook 通过 exitCode 0/1/2 决定 pass-through / block / inject 三种行为, 错误降级原则保证 第三方 Hook 不会让 harness 崩溃。下一章 (第 09 章) 我们会处理"用户偏好 怎么跨会话保留" 的问题——Memory, 把"项目级" 和"用户级" 长期事实 分开, 让 harness 在新会话里也能想起"用户喜欢简洁解释"。

下一章伏笔

第 08 章让 harness 有关键事件的事实记录, 但用户偏好 (例如"我喜欢 简洁解释") 仍然在每次新会话都丢。下一章 Memory 模块会区分"项目级 短期事实" (例如本项目的命名规范) 和"用户级 长期事实" (例如用户的 解释风格偏好), 把后者持久化, 让 harness 在新会话里也能想起。
