第 08 章 · 给主循环挂扩展点

Hook: 在主循环周围挂自定义逻辑

第 07 章的 Permission 是"工具执行前要不要问人" 的硬规则, 但用户经常想 挂更灵活的东西: 审计日志、敏感词提醒、自动追加项目规范、跑完测试后 通知 Slack。 Permission 装不下, 写死在 tool 里又不通用。 这一章 加 Hook 系统: 3 个事件点 (SessionStart / PreToolUse / PostToolUse) × 3 个返回码 (continue / block / inject), 用工厂函数 + 闭包组合, 让扩展逻辑在 loop 周围插桩。 读完后, 你能讲清"Hook 和 Permission 的边界" (Permission 是安全边界, Hook 是扩展点), 并能 用 fake handler 验证"block 短路" 和 "inject 延迟注入" 的协议。

真实失败故事: 用户想"加一行提醒" 却改了 5 个 tool

写代码之前, 先看一个真实痛点。 用户的 team 想给 harness 加一条团队 规范: "所有 bash 命令前检查是否有 --no-verify, 有就警告 LLM 改用 --verify"。 用 Permission 写, 只能 deny, 不能 "警告后继续"。

  1. 写进 tool 里? run_bash.ts 加 if (cmd.includes("--no-verify")) ...。 5 个用户提了 5 个不同规则, tool 里堆满 if/else, 没人维护得了。
  2. 写进 Permission? Permission 的语义是 allow/ask/deny, 不能"警告后继续"。 强行加, Permission 就变成万能垃圾桶。
  3. 写进 agent loop? 改了 5 个 tool 不够, 还要改 agent 的主循环, 任何循环扩展都污染核心。

朴素想法 1: "直接给 agent.ts 加个 switch 监听事件?" 错。 agent 是 通用循环, 业务规则千变万化, agent 不能认识所有规则。 主循环应该 只发事件, 不处理事件。

朴素想法 2: "用 Node EventEmitter 就行了?" 表面行, 但 LLM 的 protocol 决定了你不能在 tool_call 和 tool_result 之间塞消息 (OpenAI API 拒绝)。 需要 Hook 系统 理解延迟注入: 把"补充消息" 留到所有 tool_result 写完后, 统一追加成 user message。

正确做法: 加一个独立模块 src/hooks.ts, 定义 3 个事件 + 3 个返回码, 主循环在固定时机 emit, handler 在外部 注册, 主循环根据返回码决定继续 / 阻止 / 注入。 Permission 仍然是 安全边界, Hook 是扩展点。 这是 Reference 章节 "模式 9 · Observer 观察者模式" 的具体应用。

三个事件点: SessionStart / PreToolUse / PostToolUse

场景: 团队要挂 3 个不同的扩展

团队 leader 想给 harness 加 3 个扩展, 看着都没问题, 但写起来麻烦:

  1. 跑完测试自动 commit: 用户跑 npm test 之后, 看到测试通过, 自动 git add + git commit。 想监听 run_bash 跑完事件。
  2. 工具调用前打印日志: 调试时想看每次工具调用的 名字 + 参数, 输出到 stderr。 想监听 "工具执行前" 事件。
  3. 启动时注入 CLAUDE.md: 团队统一把 CLAUDE.md 内容加载到 system prompt, 让 LLM 看到项目规范。 想监听 "启动" 事件。

这 3 个扩展都不是安全检查 (Permission 不管), 都不是业务核心 (写进 agent.ts 会污染主循环), 都不是单一工具能完成 (跨多个工具或整个 session)。 朴素想法 1: "每个用户自己写个 wrapper 套在 tool 外?" 错。 5 个团队 5 个 wrapper, 互相不兼容, 维护噩梦。 朴素想法 2: "在 agent.ts 主循环里加 3 个 if?" 错。 业务规则千变万化, agent.ts 会变成"巨型 switch", 谁都不敢动。

正确做法: 主循环在3 个固定时机发事件, handler 在外部注册监听。 事件点必须"少而精" — 多一个事件点就是多一份主循环耦合, 永远不会被添加。 看 实际选择的 3 个:

图 1 · 3 个事件点在主循环中的位置
1 · SessionStart
run() 第一次被调用时, 发一次, 携带 query 文本。 用于"启动时注入项目规范 / 加载 MEMORY"。
2 · PreToolUse
权限检查通过后, 工具执行前, 每次 tool_call 都发。 携带 toolCallId / toolName / args / round。
3 · PostToolUse
工具执行后, 每次都有。 携带 output (经 P1 即时压缩) + error 标记。 用于审计 / 自动 commit / Slack 通知。

设计思想: 经典观察者模式 — 主循环是事件 源, handler 是观察者, "事件点" 是订阅点。 关键边界: 主循环 只发事件, 不处理事件 — 业务规则千变万化, agent 不应该认识所有规则。

为什么是这 3 个, 不是更多

用户曾经想加更多: "我想在每个 tool 调用中间 (MidToolUse) 也挂 handler", "我想在 LLM 调用结束后 (AfterLLM) 也挂" — 全部被拒绝。 理由: 主循环每多一个 emit, 就是多一份未来必须维护的耦合。 3 个事件点能覆盖 90% 真实需求 (审计 / 警告 / 自动 commit / 规范 注入), 剩下 10% 用户自己在业务代码里写。

真实场景:

  • 团队要"跑完测试后自动 commit", 用 PostToolUse 监听 run_bash("npm test") 返回 0, 触发 git commit
  • 用户想"工具调用前打印彩色日志", 用 PreToolUse 把 tool name + args 写到 stderr
  • 团队要"每次启动注入 CLAUDE.md 内容" 用 SessionStart 一次性发出 query

设计思想: 经典观察者模式 + 策略模式 组合 — 主循环是事件源, handler 是观察者, 返回码是策略。 关键是"事件点数量要少, 返回码语义要正交":

图 1 · 3 个事件点在主循环中的位置
1 · SessionStart
run() 第一次被调用时, 发一次, 携带 query 文本。 用于"启动时注入项目规范 / 加载 MEMORY"。
2 · PreToolUse
权限检查通过后, 工具执行前, 每次 tool_call 都发。 携带 toolCallId / toolName / args / round。
3 · PostToolUse
工具执行后, 每次都有。 携带 output (经 P1 即时压缩) + error 标记。 用于审计 / 自动 commit / Slack 通知。

实现细节: 事件用 TypeScript discriminated union 定义, 编译期就能窄化 payload 类型。 看 src/hooks.ts 第 76 行的 HookEvent union: 

export type HookEvent =
  | { name: "SessionStart"; payload: { query: string } }
  | {
      name: "PreToolUse";
      payload: {
        toolCallId: string;
        toolName: string;
        args: Record<string, unknown>;
        round: number;
      };
    }
  | {
      name: "PostToolUse";
      payload: {
        toolCallId: string;
        toolName: string;
        args: Record<string, unknown>;
        round: number;
        output: string;
        error: boolean;
      };
    };

为什么是 discriminated union 而不是 3 个独立 type? handler 注册 时只关心自己监听的事件 (如 PostToolUse), union 让 event.name 在 switch 里自动窄化到具体 payload, 不需要 cast, 编译期防错。

为什么 args 是 Record<string, unknown>? 工具 schema 各异, Hook 系统不认识所有 schema, 留 unknown 让 handler 自己 cast / 验证。

三个返回码: 0 / 1 / 2 的语义正交

场景: handler 要告诉主循环"我干了什么"

handler 跑完之后, 主循环要知道"接下来该怎么办"。 朴素做法: handler 返回一个字符串, 主循环 switch (result.action) 判断。 错。 字符串容易拼错 (typo), 编译期不报错, 主循环写 错了就静默 fail。

朴素做法 2: handler 返回一个对象, 比如 {action: "block", message: "..."}。 看起来更好, 但 action 字段还是字符串, 而且对象能加任意字段, 没有"必填" 约束。 团队 leader 想加第 4 种 action (比如 "retry") 时, 又 要改主循环的 switch。

正确做法: 用数字退出码借鉴 shell 传统 (exit 0/1/2), 语义正交 — 3 个码覆盖 3 种独立动作, 任何 组合都有明确含义, 编译期就能校验。 PreToolUse 监听 run_bash 时, 看到 --no-verify 就返回 1 (block, 阻止执行); 看到 curl 就返回 2 (inject, 追加警告消息 到对话历史); 其他返回 0 (continue, 啥都不做)。

图 2 · 3 个返回码的语义对比
exitCode 0 · continue
啥都不做, 继续当前动作。 用于"我听了, 但没意见"。 大多数 handler 走这条。
exitCode 1 · block
阻止当前动作。 PreToolUse 上阻止 tool 执行; PostToolUse 上仅记录警告, 不重做。 SessionStart 上几乎不用。
exitCode 2 · inject
注入一条补充消息后继续。 消息在所有 tool_result 写完后, 统一追加成 user 消息。 用于"提醒 / 警告 / 补充上下文"。

为什么必须正交

如果 3 个码有依赖 (比如"block 必须有 message", "continue 不能有 message"), 写 handler 时要记一堆约束, 容易写错。 正交设计 下, 每个码的 message 字段都是可选的, 互不影响: 0 可以带 message (主循环忽略), 1 可以带 message (写到 tool result), 2 必须带 message (要注入), 但格式都一样, 处理逻辑统一。

exitCode 1 · block
阻止当前动作。 PreToolUse 上阻止 tool 执行; PostToolUse 上仅记录警告, 不重做。 SessionStart 上几乎不用。
exitCode 2 · inject
注入一条补充消息后继续。 消息在所有 tool_result 写完后, 统一追加成 user 消息。 用于"提醒 / 警告 / 补充上下文"。

实现细节: HookResult 接口只有一个 exitCode + 可选 message, 没有"action 字段" "reason 字段"。 看 src/hooks.ts 第 47 行的 HookResult 类型: 

export type HookExitCode = 0 | 1 | 2;

export interface HookResult {
  /** 退出码: 0=继续, 1=阻止, 2=注入补充消息后继续 */
  exitCode: HookExitCode;
  /** 补充说明文本, 给用户 / LLM / 日志看 */
  message?: string;
}

聚合规则: 多个 handler 注册同一事件时, 串行执行, 优先级 1 > 2 > 0。 遇到 block (1) 立即短路, 不执行 后续 handler; inject (2) 的 message 累积, 最后用空行拼接; continue (0) 啥都不做。

handler 抛异常时只记录 warn 日志, 不中断主流程。 这是 Reference 章节 "模式 21 · No Catch Throw" 的体现 — Hook 是扩展机制 不是安全机制, 一个扩展挂掉不能让整个 agent 死掉。

多 handler 聚合: 串行 + 短路 + 累积

场景: 团队同时挂 3 个 handler, 行为怎么定

一个团队想挂 3 个 handler, 看着都合理, 但聚合规则一含糊就出问题:

  1. (a) 内置审计: 每次工具调用都写 audit.log, 永远 continue (返回 0)。
  2. (b) 团队规范检查: 看到 --no-verify 返回 2 (inject 警告消息), 其他 continue。
  3. (c) 安全扫描器: 看到 rm -rf 返回 1 (block, 阻止执行)。

朴素做法 1: "只让挂一个 handler?" 错。 3 个团队 3 个不同需求, 每个 handler 解决一个具体问题, 互不冲突。 朴素做法 2: "并行 执行 3 个 handler, 看哪个先返回?" 错。 并发行为难调试, 出现 block 时谁负责? race condition 难追。 朴素做法 3: "按注册 顺序决定一切?" 错。 含糊, 团队 A 说"我先注册就该我说了算", 团队 B 说"我后注册覆盖你" — 永远吵不完。

正确做法: 显式定义3 条聚合规则:

  1. 串行而非并行: handler 数组按注册顺序一个一个跑, 简单可预测, 调试时打日志能看每个 handler 跑没跑。
  2. block (1) 短路: 第一个返回 1 的 handler 立即短路, 后续 handler 不跑。 既然已经决定 block, 跑更多 handler 没意义。
  3. inject (2) 累积: 多个 handler 返回 2 时, message 用空行拼起来, 最后一起注入。 "最后一个赢" 会丢信息, 累积更安全。

具体走一遍: 假设 LLM 调 rm -rf /tmp/old:

  • handler (a) 审计 跑: 写 audit.log, 返回 0 (continue)。 累积数组: []。
  • handler (b) 规范检查 跑: 命令不含 --no-verify, 返回 0。 累积数组: []。
  • handler (c) 安全扫描 跑: 看到 rm -rf, 返回 1 (block)。 立即短路, 不再跑后续。

主循环看到 1, 阻止 tool 执行, 写 "Permission denied by hook" tool message, 跳到下一轮 LLM。 这就是短路的好处 — 安全扫描是 最后一道防线, 优先级最高, 看到危险就该立即拦下。

再假设 LLM 调 git commit --no-verify -m "x":

  • (a) 审计 返回 0。
  • (b) 规范检查 看到 --no-verify, 返回 2 (inject 警告)。 累积数组: ["请用 --verify 而不是 --no-verify"]。
  • (c) 安全扫描 命令不是 rm -rf, 返回 0。

累积数组非空, 主循环拿到一个聚合结果: {exitCode: 2, message: "请用 --verify 而不是 --no-verify"}, tool 正常 执行, 警告消息稍后注入到 history (下一节"延迟注入")。

实现: 

async function run(event: HookEvent): Promise<HookResult> {
  const list = handlers[event.name] ?? [];
  const injected: string[] = [];

  for (const handler of list) {
    try {
      const result = await handler(event);
      if (result.exitCode === 1) return result;          // 短路
      if (result.exitCode === 2 && result.message) {
        injected.push(result.message);                   // 累积
      }
    } catch (error) {
      logger.warn("Hook %s failed: %s", event.name, ...);
    }
  }
  if (injected.length > 0) {
    return { exitCode: 2, message: injected.join("\n\n") };
  }
  return { exitCode: 0 };
}

设计思想: 串行 + 短路 + 累积 = 简单 + 可预测 + 易调试。 handler 数组存在闭包里, 状态 (handlers + injected) 都在闭包内, 外部只能通过 run() 接口访问。 这是 Reference 章节 "模式 1 · 工厂 + 闭包" 的应用。

延迟注入: 为什么不能在 tool_call 后立刻写 user message

场景: handler 想在 LLM 调用之间塞一条提醒

团队 handler 看到 LLM 调 rm -rf /tmp/old, 想发一条 警告 "建议用 mv 而不是 rm"。 handler 跑完返回 {exitCode: 2, message: "建议用 mv"}, 期望这条消息 出现在 history 里, 让 LLM 下一轮看到。

朴素做法 1: "handler 内部直接 history.add(message) 写消息?" 错。 handler 不知道当前是哪个 tool_call, 不知道 其他 tool_result 写没写完, 写早了破坏 OpenAI API 消息格式。 朴素做法 2: "HookRunner.run() 返回结果后, 主循环立即 history.add(message)?" 也错。 LLM 一次可能发 3 个 tool_call, 第一个 tool_result 写完后就追加 user message, OpenAI API 直接 400 报错。

问题根源: OpenAI API 的硬约束

OpenAI API 要求 assistant 消息里的所有 tool_call 必须有对应的 tool 消息, 中间不能塞 其他 role (包括 user)。 这是消息序列的协议级约束, 不是 OpenAI 文档里说"建议", 是 SDK 实际会 400 报错的硬规则。

具体例子: 一次 LLM 调用发 3 个 tool_call

图 3 · exitCode 2 的延迟注入时机
❌ 错误时机: tool_result 之后立刻追加
assistant(tool_call A,B,C) → tool_result A → user(警告消息) → tool_result B → tool_result C

问题: assistant 消息有 3 个 tool_call, 但中间夹了 user 消息, tool_call B 和 C 的对应 tool_result 还在后面。 OpenAI API 拒绝, 报 400 错误。
✅ 正确时机: 所有 tool_result 写完后追加
assistant(tool_call A,B,C) → tool_result A → tool_result B → tool_result C → user(警告消息) → 下一轮 assistant

好处: 协议合法, OpenAI API 接受; assistant 消息 的所有 tool_call 都有对应 tool_result, user 消息在最后, 序列完整。

正确做法: Hook 系统只返回, 不写入

HookRunner.run() 永远只返回 {exitCode: 2, message: "..."}, 不直接调 history.add()。 主循环缓存所有 inject 消息, 等本轮所有 tool_result 写完, 再统一追加为 user message。 这是 Reference 章节 "模式 4 · 依赖注入" 的应用 — HookRunner 是被注入的策略, 主循环是调用方, 双方协议清晰, 谁写消息由主循环决定。

实现: 这条约束写在 src/hooks.ts 第 28 行的"延迟注入" 注释 里 — 不写测试也会被后续读者注意到。 agent 主循环在 collect tool result 阶段缓存所有 inject 消息, 写完最后 一个 tool_result 后再批量追加为 user message。

createNoopHookRunner: 让 agent 主循环保持直线阅读

场景: 简单任务用不到 hook, 主循环被 6 个 if 刺满

用户的 team 还在调研阶段, 跑最简单的 harness: "读 1 个文件, 改 1 个文件, 结束"。 根本用不到 hook。 但 agent.ts 主循环 6 步里 每步都有 await hookRunner.run(event), 如果 hookRunner 是 undefined, 就要在每步加 if: 

// 朴素实现: 6 个 if 判空
async function run(query) {
  if (hookRunner) await hookRunner.run({ name: "SessionStart", payload: { query } });
  // ... 加载 history
  for (;;) {
    const response = await llm.chat(messages, tools);
    if (response.toolCalls.length === 0) return response.content;
    if (hookRunner) await hookRunner.run({ name: "PreToolUse", payload: { ... } });
    for (const call of response.toolCalls) {
      const result = await tools.invoke(call.name, call.args);
      if (hookRunner) await hookRunner.run({ name: "PostToolUse", payload: { ... } });
      history.add({ role: "tool", tool_call_id: call.id, content: result.content });
    }
  }
}

6 个 if (hookRunner) 像刺猬一样插在主循环, 阅读时 每次都要"哦这一行是判空"。 错。

朴素做法 2: "用 optional chaining hookRunner?.run()?" 也不行。 ?.run() 不会 await, 主循环立刻往下走, hook 还没跑完 tool 就 invoke 了 — 顺序错乱。 这条不行。

正确做法: 空对象模式 (Null Object)

给一个永远返回 { exitCode: 0 } 的对象, 协议完整 但行为无副作用。 主循环永远调 await hookRunner.run(event), 不用判空, 不用 optional chaining, 代码保持直线流。 看 src/hooks.ts 第 220 行的 createNoopHookRunner(): 

export function createNoopHookRunner(): HookRunner {
  return {
    async run() {
      return { exitCode: 0 };
    },
  };
}

实现细节: 在 index.ts 的 Composition Root 里, 如果用户没传 hookRunner 就 createNoopHookRunner() 兜底, 保证 agent.ts 里 await hookRunner.run(event) 永远 不需要判空。 这是 Reference 章节 "模式 1 · 工厂 + 闭包" 的应用 — 闭包返回一个稳定的对象, 行为可预测, 调用方零负担。

handler 注册: Partial<Record> 表达"不监听就是空数组"

场景: 团队只关心 2 个事件, 不想写空数组

团队在 index.ts 里写: 

createHookRunner({
  PreToolUse:  [auditHandler, noVerifyHandler],
  PostToolUse: [slackHandler],
}, logger);

只注册了 2 个事件, 没注册 SessionStart。 朴素做法 1: "强制写完整 3 个键, 没注册的写空数组?" 错。 每次写 SessionStart: [] 噪音, 团队每次都要重复这个空数组, 哪天加第 4 个事件又得改所有 call site。 朴素做法 2: "用可选 字段 handlers?: Record<..., ...>?" 错。 缺字段 运行时是 undefined, 调 handlers.SessionStart.forEach 直接 NPE, 还得加判空。

正确做法: Partial<Record> + 运行时 ?? [] 兜底

编译期允许缺字段 (因为是 Partial), 缺的是 undefined; 运行期用 handlers[event.name] ?? [] 兜底成空数组。 类型安全 + 调用方零负担。 看 src/hooks.ts 第 163 行的 createHookRunner 工厂: 

export function createHookRunner(
  handlers: Partial<Record<HookEventName, HookHandler[]>>,
  logger: Logger,
): HookRunner {
  // handlers[event.name] ?? [] 兜底, Partial 允许 undefined
}

实现细节: 工厂函数返回的对象只有一个 run 方法, 没有 addHandler / removeHandler — 这是故意的简化: handler 在 Composition Root 一次性 注入, 运行期不能动态加 (避免多 handler 顺序的歧义)。 这是 Reference 章节 "模式 3 · 窄接口" 的应用 — 接口只暴露 run, 不暴露 handlers 数组, 闭包状态不外泄。

handler 本身写在哪: 一般写在一个桥接文件里 (如 src/hooks/memory-inject.ts), 这个文件同时 import hooks 和具体业务模块 (memory), 把"业务模块的能力" 包成 hook handler。 这是整个项目架构的核心原则 — 业务模块 之间不直接 import, 通过第 3 个桥接文件集成。 详细解释见 Reference · 模式 3.5 · 桥接模式 / 0 循环依赖。

主循环集成: 在哪 3 个时机 emit

场景: 主循环怎么知道在哪调 hook

团队 leader 看 agent.ts 主循环 6 步, 想知道每个 hook 事件在第几步 emit。 朴素做法 1: "主循环里加 if-else, 每个事件单独一个 if?" 错。 6 步 × 3 个事件 = 18 个 if, 主循环 像刺猬, 改一次业务逻辑就要小心不要碰 hook。 朴素做法 2: "主循环外层包一层 'hook dispatcher'?" 也错。 抽象泄漏 — 业务逻辑被 dispatcher 隔开, 调试时看 dispatcher 跳来跳去, 反而 更难理解。

正确做法: 集成点必须集中, 不能散在 agent.ts 各个角落。 看 3 个事件的具体 emit 时机:

  1. SessionStart (step 0, while 循环开始前): 把 query 包装成事件 { name: "SessionStart", payload: { query } } 发一次。 handler 返回 2 的 message 注入到 system 消息后, 第一轮 assistant 调用之前。
  2. PreToolUse (step 3, call tools 前): 权限检查 通过后, 调 toolRegistry.invoke(...) 之前。 handler 返回 1 阻止 invoke, 写 "Permission denied by hook" tool message; 返回 2 累积 message, 在所有 tool_result 后注入。
  3. PostToolUse (step 3.5, call tools 后, compress 前): invoke 完成后, 把 ToolResult.output 喂给 handler。 handler 返回 1 (block) 在 PostToolUse 上不重做, 只写警告日志 (因为 tool 已经跑完, block 没意义); 返回 2 累积 message。

为什么 PostToolUse 上 block 没用: 朴素想法 "PostToolUse 返回 1 应该撤销 tool 跑过的结果?" 错。 tool 已经执行, 副作用已经发生 (文件已写 / 命令已跑), 撤销需要 复杂的事务回滚, harness 做不到。 PostToolUse 上的 1 只是 "下次类似的操作我要拒绝", 不撤销已经发生的事。

实现: 集成代码写在 agent.ts 里, 调用方式统一是 

const hookResult = await hookRunner.run(event);
if (hookResult.exitCode === 1) {
  // 阻止 / 写 denied message / 跳过 invoke
}
if (hookResult.message) {
  pendingInjects.push(hookResult.message);   // 缓存, 稍后批量注入
}

缓存的 inject 消息在 collect tool results 阶段之后批量追加, 这就是 第 02 章 "message 配对" 协议的具体应用。

对 prompt cache 的影响: SessionStart 注入是 stable prefix 一部分

场景: SessionStart 注入让 cache 命中率跌到 0

团队用 SessionStart 注入"项目规范", 看着是合理的扩展, 但跑了 一个月后账单翻倍。 排查发现: 每次启动 SessionStart 都重新 注入规范, 规范一改 (1 行代码 10 个字符), 后续 100 轮对话 的 system prompt 全 miss, 整个对话的 cache prefix 重新算, 成本 ×10。

朴素想法 1: "SessionStart 不注入不就行了?" 错。 团队规范必须 注入, LLM 不知道项目规范会乱写。 朴素想法 2: "在 system prompt 里写死规范?" 也错。 规范是配置 (git 仓库的 CLAUDE.md), 改规范要重发 harness 体验差。 朴素想法 3: "让 SessionStart 注入的内容每次都一样?" 思路对, 但实现麻烦 — 规范确实偶尔改, 怎么保证"同一 session 内一致"?

正确做法: 幂等性约束

SessionStart 注入必须保证"同一 session 内, 相同 query + 相同 handler 集合 产出相同message"。 handler 应该读稳定数据源 (配置文件 / MEMORY / git commit), 不读动态数据 (时间戳 / 随机数 / 当前时间)。

具体例子: 什么算"稳定"

  • 稳定: 读 CLAUDE.md 文件内容 (一个 session 内不变)。 读 MEMORY.md (一个 session 内不变)。 读 git 当前 commit hash (一个 session 内不变)。
  • 不稳定 (不能用): 读 new Date().toISOString() (每次 emit 都不同, 注入 → cache miss → 注入 → cache miss 死循环)。 读 Math.random() (随机, 同样问题)。

设计思想: SessionStart 是"启动时一次" 的事件, 不是 "每轮一次"。 注入的内容应该只写一次到 system prompt, 后续 100 轮对话引用同一份 system, cache 命中。 这是和第 05 章 (Skill 稳定前缀) + 第 09 章 (Memory 4 类 tags) 联合的 cache 策略。

实现: SessionStart 的 inject 应该在 handler 内部 做"内容去重" — 同一 handler 在同一 session 只 emit 一次 (只 写一次 system message), 后续轮不再追加。 这部分实现在第 10 章 (Cache) 详细展开, 包括 cache-debug.ts 的稳定前缀 hash 工具。

fake test: 用 stub handler 验证 block 短路和 inject 累积

用途: Hook 系统的测试不需要真 hook, 用 stub 即可 — 测的是"主循环按返回码做了什么", 不是 handler 内部逻辑。

真实场景: 用户写一个"检测 rm -rf" 的 handler, 想 验证: (a) rm -rf / 应该 block; (b) rm file.txt 应该 continue; (c) block 时主循环写 "Permission denied by hook" tool message, 不真的跑 rm。 stub handler 返回固定结果就能测。

设计思想: 测协议不测实现 — Hook 系统 的协议是"返回码 → 主循环行为", 这层用 stub 测最稳。 handler 内部 业务逻辑 (rm 检测算法) 用单元测试单独测, 不混在 HookRunner 测试里。 

test("block exitCode 短路后续 handler", async () => {
  const calls: string[] = [];
  const runner = createHookRunner({
    PreToolUse: [
      () => { calls.push("first"); return { exitCode: 0 }; },
      () => { calls.push("second"); return { exitCode: 1, message: "blocked" }; },
      () => { calls.push("third"); return { exitCode: 2, message: "warn" }; },
    ],
  }, noopLogger);
  const result = await runner.run({ name: "PreToolUse", payload: {...} });
  expect(calls).toEqual(["first", "second"]);   // third 没跑
  expect(result).toEqual({ exitCode: 1, message: "blocked" });
});

test("inject 累积: 多个 handler 的 message 用空行拼接", async () => {
  const runner = createHookRunner({
    PostToolUse: [
      () => ({ exitCode: 2, message: "audit: 工具跑完了" }),
      () => ({ exitCode: 2, message: "warn: 命令有点危险" }),
    ],
  }, noopLogger);
  const result = await runner.run({ name: "PostToolUse", payload: {...} });
  expect(result.exitCode).toBe(2);
  expect(result.message).toBe("audit: 工具跑完了\n\nwarn: 命令有点危险");
});

test("handler 抛异常只 warn, 不中断", async () => {
  const runner = createHookRunner({
    PreToolUse: [
      () => { throw new Error("boom"); },
      () => ({ exitCode: 2, message: "next" }),
    ],
  }, noopLogger);
  const result = await runner.run({ name: "PreToolUse", payload: {...} });
  expect(result.message).toBe("next");   // 异常被吞, 后续继续
});

实现细节: 这 3 个 stub 测试覆盖了 Hook 系统 90% 的 协议 — 短路、累积、异常隔离。 handler 自身的业务逻辑测试应该放在 handler-name.test.ts, 不在 hooks.test.ts。

实战: 自己写一个 hook 挂到主循环

前面讲的"主循环在 3 个时机 emit" 和 "handler 是外部注册的函数" — 这一节把它们串起来, 走一遍"自己写一个 hook" 的完整流程。 例子: 写一个 "检测危险命令" 的 PreToolUse handler, 看到 rm -rf / / sudo / curl | sh 就 block。

目标: 走完后, 你能照葫芦画瓢写任意 hook — 比如 "敏感词提醒" / "自动 commit" / "Slack 通知" / "统计工具调用 次数", 都是同一个模板。

步骤 1 · 写 handler 函数

handler 是一个普通 async 函数, 接 HookEvent, 返回 HookResult。 我们写的 "危险命令检测" handler 只关心 PreToolUse 事件, 所以用泛型 HookHandler<...> 精确声明: 

// src/hooks/dangerous-command.ts
import type { HookHandler } from "../hooks.js";

/**
 * 危险命令检测: 看到 rm -rf / / sudo / curl | sh 等危险模式就 block (返回 1)。
 * 其他情况返回 0 continue。
 */
export const dangerousCommandHook: HookHandler<{
  name: "PreToolUse";
  payload: {
    toolCallId: string;
    toolName: string;
    args: Record<string, unknown>;
    round: number;
  };
}> = (event) => {
  // 只关心 run_bash 工具
  if (event.payload.toolName !== "run_bash") {
    return { exitCode: 0 };
  }

  // 取出命令字符串 (args.command 是 unknown, 需要 narrow)
  const command = typeof event.payload.args.command === "string"
    ? event.payload.args.command
    : "";

  // 危险模式黑名单 (教学版只列 3 个, 实战可加更多)
  const dangerousPatterns = [
    { regex: /rm\s+-rf\s+\//,        reason: "rm -rf / 会清空根目录" },
    { regex: /\bsudo\b/,             reason: "sudo 需要管理员权限" },
    { regex: /curl.*\|\s*(bash|sh)/, reason: "curl | sh 远程执行风险" },
  ];

  for (const { regex, reason } of dangerousPatterns) {
    if (regex.test(command)) {
      return {
        exitCode: 1,                    // block
        message: `Dangerous command blocked: ${reason} (command: ${command})`,
      };
    }
  }

  return { exitCode: 0 };               // continue
};

4 个要点:

  1. handler 是普通函数: 可以是 arrow function / async / sync, 没有特殊 wrapper, 直接 (event) => HookResult。
  2. 用泛型 HookHandler<...> 精确声明: 不关心 SessionStart 的 handler 就用 HookHandler<PreToolUse>, TypeScript 帮你窄化 event.payload 到 PreToolUse 专用字段, 不用 if (event.name === "PreToolUse") 判一次。
  3. 不关心的工具直接 continue: 这个 handler 只检查 run_bash, 看到 run_read / run_write 等直接 return { exitCode: 0 }。 不要假设"我应该处理所有工具"。
  4. block 时 message 必填: 写给 LLM 看 (写到 tool result), LLM 据此决定下一步。 不写 message = LLM 不知道为啥被拒。

步骤 2 · 注册到 HookRunner

写完 handler, 下一步注册到 HookRunner。 这一步在 Composition Root (index.ts) 完成。 3 种注册方式:

方式 A: 直接在 index.ts 里写 (简单, 1 个 handler) 

// src/index.ts
import { createHookRunner } from "./hooks.js";
import { dangerousCommandHook } from "./hooks/dangerous-command.js";

const hookRunner = createHookRunner({
  // PreToolUse 上可以注册多个 handler, 按数组顺序串行
  PreToolUse: [
    dangerousCommandHook,
    // 还可以加其他: auditLogHook / noVerifyHook / ...
  ],
  // PostToolUse: 暂时不挂, 以后想加再写
  // SessionStart: 暂时不挂
}, logger);

// 没注册的事件自动是空数组, 啥也不做 (Partial<Record> 兜底)

方式 B: 抽到独立文件 (handler 多 / 想分模块) 

// src/hooks/index.ts
import type { HookHandler } from "../hooks.js";
import { dangerousCommandHook } from "./dangerous-command.js";
import { auditLogHook } from "./audit-log.js";
import { noVerifyHook } from "./no-verify.js";

// 集中导出所有 handler, index.ts 里一次 import
export const preToolUseHandlers: HookHandler[] = [
  auditLogHook,
  dangerousCommandHook,
  noVerifyHook,
];

export const postToolUseHandlers: HookHandler[] = [
  slackNotificationHook,
  autoCommitHook,
];

// src/index.ts
import { preToolUseHandlers, postToolUseHandlers } from "./hooks/index.js";
const hookRunner = createHookRunner({
  PreToolUse:  preToolUseHandlers,
  PostToolUse: postToolUseHandlers,
}, logger);

方式 C: 从配置文件读 (handler 经常变, 不想改代码) 

// config.json
{
  "hooks": {
    "PreToolUse": [
      "./hooks/dangerous-command.js",
      "./hooks/audit-log.js"
    ]
  }
}

// src/index.ts
import config from "./config.json";
const preToolUseHandlers = await Promise.all(
  config.hooks.PreToolUse.map(async (path) => {
    const mod = await import(path);
    return mod.default;
  })
);
const hookRunner = createHookRunner({ PreToolUse: preToolUseHandlers }, logger);

推荐: 教学项目用方式 A 或 B (代码可见, 易调试); 工业 项目用方式 C (handler 经常变, 不想每次重发代码)。

步骤 3 · 挂到主循环 (其实不用挂)

你已经完成了。 因为 hookRunner 是在 Composition Root 创建后直接注入到 agent 的, agent 主循环 内部已经在 3 个时机调 await hookRunner.run(event) (上一节"主循环集成" 讲过)。 你不需要改 agent.ts, 不需要改 主循环。

这就是 Hook 系统的核心好处 — 写新 hook = 写 1 个 handler + 在 index.ts 加 1 行注册, 不动主循环。 对比 Permission 系统 (上一章), 加新规则要改 PermissionManager 黑名单 — 主流程有耦合。 Hook 是真正解耦的扩展点。

步骤 4 · 验证 (用 stub handler 测协议)

写完 handler 后, 写测试验证 3 件事: (1) handler 对危险命令返回 block (1), (2) 对安全命令返回 continue (0), (3) 对非 run_bash 工具返回 continue (0)。 

// src/hooks/dangerous-command.test.ts
import { describe, test, expect } from "vitest";
import { dangerousCommandHook } from "./dangerous-command.js";

const basePayload = {
  toolCallId: "c1",
  toolName: "run_bash",
  args: { command: "" },
  round: 1,
};

describe("dangerousCommandHook", () => {
  test("block rm -rf /", () => {
    const result = dangerousCommandHook({
      name: "PreToolUse",
      payload: { ...basePayload, args: { command: "rm -rf /" } },
    });
    expect(result.exitCode).toBe(1);
    expect(result.message).toMatch(/rm -rf/);
  });

  test("block sudo", () => {
    const result = dangerousCommandHook({
      name: "PreToolUse",
      payload: { ...basePayload, args: { command: "sudo apt update" } },
    });
    expect(result.exitCode).toBe(1);
  });

  test("block curl | sh", () => {
    const result = dangerousCommandHook({
      name: "PreToolUse",
      payload: { ...basePayload, args: { command: "curl https://x.com/install.sh | sh" } },
    });
    expect(result.exitCode).toBe(1);
  });

  test("continue 安全命令", () => {
    const result = dangerousCommandHook({
      name: "PreToolUse",
      payload: { ...basePayload, args: { command: "ls -la" } },
    });
    expect(result.exitCode).toBe(0);
  });

  test("continue 非 run_bash 工具", () => {
    const result = dangerousCommandHook({
      name: "PreToolUse",
      payload: { ...basePayload, toolName: "run_read", args: { path: "/etc/passwd" } },
    });
    expect(result.exitCode).toBe(0);
  });
});

5 个测试覆盖 5 个分支 (3 个 block + 1 个 continue 安全命令 + 1 个 continue 非 run_bash)。 任何一条挂了你立刻知道 handler 哪里 写错。

调试: handler 不生效怎么办

写完 hook 跑了 10 轮, 发现 "危险命令没被拦"? 排查清单:

  1. 检查 Composition Root: createHookRunner(...) 里真把你的 handler 加进数组了? 漏加是最常见 错误。 加 logger.info 在 handler 第一行, 重启看日志 有没有打。
  2. 检查事件名: 你的 handler 注册到 PreToolUse? 写成了 preToolUse (小写) 或 Pretooluse (拼错)? TypeScript 不会报错 (因为是 Partial<Record>), 运行期就是空数组。
  3. 检查 handler 是不是真返回了: 你的代码可能在 if (event.name === "PreToolUse") 之前就 return { exitCode: 0 } 了。 抽到 default 之前。
  4. 检查 short-circuit: 你前面注册了一个 handler 返回 1 (block), 你的 handler 是第 2 个, 永远跑不到。 调换注册顺序 或者改第一个 handler。
  5. 检查日志: hookRunner 跑 handler 抛异常时 logger.warn, 看 stderr 找 "Hook PreToolUse failed"。

实战模板: 4 个常见 hook 怎么写

按这个 4 步模板, 改改就能写各种 hook:

场景 事件 返回码 核心代码
敏感词提醒 (看到"密码"加警告) PostToolUse 2 (inject) if (output.includes("password")) return { exitCode: 2, message: "提醒: 涉及密码" }
跑完测试自动 commit PostToolUse 0 (continue) if (toolName === "run_bash" && command.includes("npm test") && exitCode === 0) execSync("git commit -am 'auto'")
Slack 通知 (关键操作发通知) PostToolUse 0 (continue) if (toolName === "run_bash" && command.includes("deploy")) await slack.post(...)
统计工具调用次数 PostToolUse 0 (continue) counter[toolName] = (counter[toolName] ?? 0) + 1

关键观察: 4 个 hook 都不改 agent.ts, 都不动主循环, 都在 Composition Root 加 1 行注册。 这就是 Hook 系统的核心价值 — 扩展不影响核心。

常见误解: Hook 不是 Permission 替代品

误解 1: "Hook 能取代 Permission?" 错。 Permission 是 安全边界 — 黑名单命令、workspace 外路径、ask 用户, 这些必须 hard 拒绝, 不让 LLM 重试。 Hook 是扩展点 — 审计、提醒、注入上下文, 不替代安全。 两者职责正交。

误解 2: "exitCode 2 = block 的一种?" 错。 2 是 inject 后继续, 不是 block。 把 2 当 1 用, 工具不会 跑, 但 LLM 也不会收到反馈, 以为是协议错误。

误解 3: "PostToolUse 上 block 能重做工具?" 错。 PostToolUse 触发时工具已经跑完, block 没有意义。 block 在 PostToolUse 上只记 warn 日志, 不重做。 想要"重做" 用上层 retry 机制, 不是 Hook。

误解 4: "SessionStart 能 emit 多次?" 不行。 SessionStart 语义是"每个 agent 实例第一次 run() 时", emit 多次 会污染 cache。 主循环只在 while 之前发一次, 不在 循环内重发。

反例梯度: 4 个常见错误

反例 1 · 过度耦合

把 handler 业务逻辑写进 agent.ts 主循环。 改一次 规则要重读 200 行 agent 代码。 错。 Handler 应该注册到 HookRunner, 主循环不出现业务关键字。 agent.ts 里 只调 hookRunner.run(event), 不 import 任何 handler 实现。

反例 2 · 把 2 当 1

用户写 handler 检测 --no-verify, 想"阻止 + 给 LLM 提示改用 --verify", 返回 { exitCode: 1, message: "请用 --verify" }。 错。 1 是 block, message 不会注入, LLM 收到 "blocked" 啥也不知道。 这种情况要返回 { exitCode: 2, message: "..." } (注入) 或 1 (硬阻止) + 主循环负责把 message 写到 tool result。

反例 3 · 忽略延迟注入

主循环在 tool_result 写完后, 立刻把 inject 消息追加到 history 数组。 表面看"消息进 history 了", 但如果 LLM 一轮发了 3 个 tool_call, 第一个 tool_result 后立刻追加 user message, OpenAI API 拒绝 400。 正确做法: 缓存所有 inject 消息, 在本轮所有 tool_result 写完后统一追加。

反例 4 · handler 抛异常拖死 agent

handler 调外部 API 写审计日志, API 挂掉, throw。 主循环 await hookRunner.run(event) 直接抛, agent 死。 错。 Hook 是扩展, 不是核心。 hookRunner.run() 必须 catch 所有 异常, 只 warn 日志, 然后继续。 一个审计 handler 挂掉不能 让 agent 不能用。

Validation: 4 条不变量检验

  1. 协议对称: handler 返回什么 exitCode, 主循环 行为必须和 exitCode 语义一致 — 0 啥也不做, 1 阻止 (PreToolUse 不调 invoke, PostToolUse 只 warn), 2 累积 message 在本轮 tool 结束后注入。 验证: 用 stub handler 返回 0/1/2, 跑主循环 1 轮, 看 history 末尾的 tool message 和 user message 数量是否符合预期。
  2. 延迟注入: 同一轮多个 tool_call 时, inject 消息 必须出现在所有 tool_result 之后。 验证: 写测试 让 LLM 调 3 个 tool (run_bash / run_read / run_write), 第一个 PostToolUse 就返回 2, 看 history 数组里 inject 消息的 index 是 不是在所有 tool_result 之后。
  3. 闭包隔离: 多个 agent 实例 (父子 subagent) 用 各自的 HookRunner, 不共享 handler 数组。 验证: 父 agent handler 调一次, 子 agent handler 不应该被调用; 子 agent 的 handler 调一次, 父 agent 不应该被调用。
  4. 空对象兼容: createNoopHookRunner() 和 真实 createHookRunner(...) 接口完全一致, agent.ts 同一份代码, 不写 if (hookRunner)。 验证: TypeScript 编译通过, 跑 100 轮对话, 行为和挂空 hook 时一致 (只是少了 inject message)。

回望: 哪些原则在本章兑现了

  • 关注点分离: Hook 独立于 Permission, 独立于 ToolRegistry。 主循环只发事件, 不处理业务规则。
  • 工厂 + 闭包: createHookRunner 把 handlers 数组 + 累积 injected 数组藏在闭包, 外部只能调 run。
  • 空对象模式: createNoopHookRunner 让 agent 主循环保持直线流, 不需要判空。
  • 协议对称: 3 个事件 × 3 个返回码, 每个组合的 主循环行为都有明确定义, 单元测试覆盖。
  • 延迟注入: inject 消息不能在 tool_call / tool_result 中间插入, 必须缓存到本轮结束后追加, 这是 LLM 协议层的硬约束。

本章的 handler 例子都是内联 stub。 实战中 handler 要读 配置文件 / 调外部 API, 这部分属于"配置层" 而不是 HookRunner 本身。 第 09 章 Memory 展开"持久化" 机制, SessionStart handler 可以从 MEMORY.md 加载团队规范。

Prompt Card (本章任务)

Prompt Card · 第 08 章

目标: 给 harness 加 Hook 系统, 3 个事件 × 3 个 返回码, 主循环按固定时序 emit, handler 在外部注册。

场景: 团队挂 3 个 handler — 审计 logger 写 audit.log; --no-verify 检查返回 1 阻止; 敏感命令 警告返回 2 注入 user message。 验证 3 个 handler 串联时, block 短路后续 handler, inject 累积 message 用空行拼接。

模块: src/hooks.ts (新) 暴露 createHookRunner(handlers, logger) + createNoopHookRunner(); src/agent.ts (改) 在 3 个固定时机 emit event; src/index.ts (改) Composition Root 注入 HookRunner。

边界 (LLM 必须遵守的 checklist):

  • 3 个事件 SessionStart / PreToolUse / PostToolUse, 不可新增第 4 个事件点
  • 3 个返回码 0 / 1 / 2, 语义正交, 不可合并 / 重命名
  • block 短路后续 handler, inject 累积 message 用 \n\n 拼接
  • handler 异常只 warn 日志, 不中断 agent 主流程
  • exitCode 2 的 message 延迟到本轮所有 tool_result 写完后追加
  • createNoopHookRunner 提供默认空实现, agent.ts 不写 if (hookRunner)
  • HookRunner 接口只有 run(event), 不暴露 handler 数组

验证 (用 stub handler + vitest, 逐条断言):

  • block 短路: 3 个 handler 串联, 第 2 个返回 1, 第 3 个不执行
  • inject 累积: 2 个 handler 都返回 2, message 用空行拼接
  • 异常隔离: handler 抛 Error, 后续 handler 继续执行
  • createNoopHookRunner 和 createHookRunner 接口完全一致, 行为不同
  • 延迟注入: 一轮 3 个 tool_call, 第一个 PostToolUse 返回 2, inject 消息在所有 tool_result 之后

本章练习

  1. 故意把 handler 业务逻辑写进 agent.ts (如 if (event.name === "PreToolUse" && event.payload.args.command?.includes("--no-verify")) ...), 跑测试, 看"主循环只发事件不处理业务" 是否抓到 (agent.ts 行数膨胀 + 改规则必须改 agent)。
  2. 让 handler 返回 { exitCode: 1, message: "请改用 --verify" } 想 "block + 给 LLM 提示", 跑测试, 看"1 是 block, message 不会注入" 是否抓到 (LLM 收不到 message, 一直重复 --no-verify)。
  3. 故意在 tool_result 之后立刻追加 user message (不缓存 inject 消息), 跑多 tool_call 测试, 看"延迟注入" 是否抓到 (OpenAI API 400 error, 或协议断言失败)。
  4. 故意让 handler throw new Error("boom"), 跑测试, 看"异常隔离" 是否抓到 (agent 主流程崩 vs. 后续 handler 继续)。

本章小结

Hook 是给主循环的扩展点, 不是 Permission 的替代品。 核心是 4 个设计:

  • 3 个事件点: SessionStart / PreToolUse / PostToolUse, 少而精, 事件点多就是多一份主循环耦合。
  • 3 个返回码: 0 / 1 / 2, 语义正交 — 0 啥也不做, 1 阻止 (PreToolUse 阻止工具, PostToolUse 只 warn), 2 注入 user message。
  • 多 handler 聚合: 串行执行, block 短路, inject 累积 (用 \n\n 拼接), 异常隔离 (catch + warn, 不中断)。
  • 延迟注入: exitCode 2 的 message 必须在所有 tool_result 写完后追加, 不然破坏 OpenAI API 协议。
