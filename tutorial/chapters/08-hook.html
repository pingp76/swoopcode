第 08 章 · 给主循环挂扩展点

Hook: 在主循环周围挂自定义逻辑

第 07 章的 Permission 是"工具执行前要不要问人" 的硬规则, 但用户经常想 挂更灵活的东西: 审计日志、敏感词提醒、自动追加项目规范、跑完测试后 通知 Slack。 Permission 装不下, 写死在 tool 里又不通用。 这一章 加 Hook 系统: 3 个事件点 (SessionStart / PreToolUse / PostToolUse) × 3 个返回码 (continue / block / inject), 用工厂函数 + 闭包组合, 让扩展逻辑在 loop 周围插桩。 读完后, 你能讲清"Hook 和 Permission 的边界" (Permission 是安全边界, Hook 是扩展点), 并能 用 fake handler 验证"block 短路" 和 "inject 延迟注入" 的协议。

真实失败故事: 用户想"加一行提醒" 却改了 5 个 tool

写代码之前, 先看一个真实痛点。 用户的 team 想给 harness 加一条团队 规范: "所有 bash 命令前检查是否有 --no-verify, 有就警告 LLM 改用 --verify"。 用 Permission 写, 只能 deny, 不能 "警告后继续"。

  1. 写进 tool 里? run_bash.ts 加 if (cmd.includes("--no-verify")) ...。 5 个用户提了 5 个不同规则, tool 里堆满 if/else, 没人维护得了。
  2. 写进 Permission? Permission 的语义是 allow/ask/deny, 不能"警告后继续"。 强行加, Permission 就变成万能垃圾桶。
  3. 写进 agent loop? 改了 5 个 tool 不够, 还要改 agent 的主循环, 任何循环扩展都污染核心。

朴素想法 1: "直接给 agent.ts 加个 switch 监听事件?" 错。 agent 是 通用循环, 业务规则千变万化, agent 不能认识所有规则。 主循环应该 只发事件, 不处理事件。

朴素想法 2: "用 Node EventEmitter 就行了?" 表面行, 但 LLM 的 protocol 决定了你不能在 tool_call 和 tool_result 之间塞消息 (OpenAI API 拒绝)。 需要 Hook 系统 理解延迟注入: 把"补充消息" 留到所有 tool_result 写完后, 统一追加成 user message。

正确做法: 加一个独立模块 src/hooks.ts, 定义 3 个事件 + 3 个返回码, 主循环在固定时机 emit, handler 在外部 注册, 主循环根据返回码决定继续 / 阻止 / 注入。 Permission 仍然是 安全边界, Hook 是扩展点。 这是 Reference 章节 "模式 9 · Observer 观察者模式" 的具体应用。

三个事件点: SessionStart / PreToolUse / PostToolUse

用途: 主循环在 3 个固定时机发事件, handler 在外部注册监听。 事件点必须少而精, 多一个事件点 就是多一份主循环耦合。

真实场景: 团队要"跑完测试后自动 commit", 用 PostToolUse 监听 run_bash("npm test") 返回 0, 触发 git commit; 用户想"工具调用前打印彩色日志", 用 PreToolUse 把 tool name + args 写到 stderr; 团队要"每次启动 注入 CLAUDE.md 内容" 用 SessionStart 一次性发出 query。

设计思想: 经典观察者模式 + 策略模式 组合 — 主循环是事件源, handler 是观察者, 返回码是策略。 关键是"事件点数量要少, 返回码语义要正交":

图 1 · 3 个事件点在主循环中的位置
1 · SessionStart
run() 第一次被调用时, 发一次, 携带 query 文本。 用于"启动时注入项目规范 / 加载 MEMORY"。
2 · PreToolUse
权限检查通过后, 工具执行前, 每次 tool_call 都发。 携带 toolCallId / toolName / args / round。
3 · PostToolUse
工具执行后, 每次都有。 携带 output (经 P1 即时压缩) + error 标记。 用于审计 / 自动 commit / Slack 通知。

实现细节: 事件用 TypeScript discriminated union 定义, 编译期就能窄化 payload 类型。 看 src/hooks.ts 第 76 行的 HookEvent union: 

export type HookEvent =
  | { name: "SessionStart"; payload: { query: string } }
  | {
      name: "PreToolUse";
      payload: {
        toolCallId: string;
        toolName: string;
        args: Record<string, unknown>;
        round: number;
      };
    }
  | {
      name: "PostToolUse";
      payload: {
        toolCallId: string;
        toolName: string;
        args: Record<string, unknown>;
        round: number;
        output: string;
        error: boolean;
      };
    };

为什么是 discriminated union 而不是 3 个独立 type? handler 注册 时只关心自己监听的事件 (如 PostToolUse), union 让 event.name 在 switch 里自动窄化到具体 payload, 不需要 cast, 编译期防错。

为什么 args 是 Record<string, unknown>? 工具 schema 各异, Hook 系统不认识所有 schema, 留 unknown 让 handler 自己 cast / 验证。

三个返回码: 0 / 1 / 2 的语义正交

用途: handler 返回一个退出码, 主循环根据退出码决定 后续动作。 返回码必须正交 (互不依赖), 任何 组合都有明确语义。

真实场景: PreToolUse 监听 run_bash, 看到 --no-verify 就返回 1 (block, 阻止执行); 看到 curl 就返回 2 (inject, 追加警告消息到对话历史); 其他返回 0 (continue, 啥都不做)。

设计思想: 数字退出码借鉴 shell 传统 (exit 0/1/2), 0 = 成功, 1 = 失败, 2 = 特殊用法。 这里的语义是:

图 2 · 3 个返回码的语义对比
exitCode 0 · continue
啥都不做, 继续当前动作。 用于"我听了, 但没意见"。 大多数 handler 走这条。
exitCode 1 · block
阻止当前动作。 PreToolUse 上阻止 tool 执行; PostToolUse 上仅记录警告, 不重做。 SessionStart 上几乎不用。
exitCode 2 · inject
注入一条补充消息后继续。 消息在所有 tool_result 写完后, 统一追加成 user 消息。 用于"提醒 / 警告 / 补充上下文"。

实现细节: HookResult 接口只有一个 exitCode + 可选 message, 没有"action 字段" "reason 字段"。 看 src/hooks.ts 第 47 行的 HookResult 类型: 

export type HookExitCode = 0 | 1 | 2;

export interface HookResult {
  /** 退出码: 0=继续, 1=阻止, 2=注入补充消息后继续 */
  exitCode: HookExitCode;
  /** 补充说明文本, 给用户 / LLM / 日志看 */
  message?: string;
}

聚合规则: 多个 handler 注册同一事件时, 串行执行, 优先级 1 > 2 > 0。 遇到 block (1) 立即短路, 不执行 后续 handler; inject (2) 的 message 累积, 最后用空行拼接; continue (0) 啥都不做。

handler 抛异常时只记录 warn 日志, 不中断主流程。 这是 Reference 章节 "模式 21 · No Catch Throw" 的体现 — Hook 是扩展机制 不是安全机制, 一个扩展挂掉不能让整个 agent 死掉。

多 handler 聚合: 串行 + 短路 + 累积

用途: 多个 handler 可以注册同一事件, 团队 / 用户 / 内置审计可以同时挂。 聚合规则必须明确, 不能 含糊 (不能说"按注册顺序决定一切")。

真实场景: 一个团队想挂 3 个 handler — (a) 内置审计 记录到 audit.log; (b) 团队规范检查 --no-verify; (c) 安全扫描器检测 rm -rf。 handler (c) 返回 1 应该 短路, 跳过 (b); handler (b) 返回 2 累积 message, 不影响 (a) 的 审计写入。

设计思想: 串行而非并行 — 简单、可预测、易调试; block 短路而非 "block 排队" — 第一个 block 命中就够了, 后续没必要 执行; inject 累积而非 "最后一个赢" — 多条提醒都有意义。 看 src/hooks.ts 第 172 行的 run() 实现: 

async function run(event: HookEvent): Promise<HookResult> {
  const list = handlers[event.name] ?? [];
  const injected: string[] = [];

  for (const handler of list) {
    try {
      const result = await handler(event);
      if (result.exitCode === 1) return result;          // 短路
      if (result.exitCode === 2 && result.message) {
        injected.push(result.message);                   // 累积
      }
    } catch (error) {
      logger.warn("Hook %s failed: %s", event.name, ...);
    }
  }
  if (injected.length > 0) {
    return { exitCode: 2, message: injected.join("\n\n") };
  }
  return { exitCode: 0 };
}

实现细节: handler 数组存在闭包里, 用 Partial<Record<HookEventName, HookHandler[]>> 类型保证"只注册需要的事件, 不需要的事件不必写空数组"。 这也是 Reference 章节 "模式 1 · 工厂 + 闭包" 的应用 — 状态 (handlers + injected) 都在闭包内, 外部只能通过 run() 接口访问。

延迟注入: 为什么不能在 tool_call 后立刻写 user message

用途: exitCode 2 的 message 必须延迟到 所有 tool_result 写完后才能追加, 不然会破坏 OpenAI API 消息格式。

真实场景: 假设 LLM 一次发了 3 个 tool_call (run_bash + run_read + run_write), PostToolUse 在每个执行完都触发, handler 返回 2 要注入警告。 如果在第一个 tool_result 后立刻追加 user message, OpenAI API 会拒绝: 它要求 assistant message 的所有 tool_call 必须有对应的 tool_result, 中间不能有 user message。

设计思想: Hook 系统不负责写入, 它只返回"应该注入什么"; 主循环负责"什么时候写"。 这就是 Reference 章节 "模式 4 · 依赖注入" — HookRunner 是被注入的策略, 主循环是 调用方, 双方协议清晰。

图 3 · exitCode 2 的延迟注入时机
❌ 错误时机: tool_result 之后立刻追加
assistant(tool_call A) → tool_result A → user(警告消息) → tool_result B → ... OpenAI 拒绝, 400 error。
✅ 正确时机: 所有 tool_result 写完后追加
assistant(tool_call A,B,C) → tool_result A → tool_result B → tool_result C → user(警告消息) → 下一轮 assistant。 协议合法。

实现细节: 这条约束写在 src/hooks.ts 第 28 行的"延迟注入" 注释 里, 是从 "本阶段限制" 段之后的"关键约束" 部分 — 不写测试也会被 后续读者注意到。 agent 主循环在 collect tool result 阶段缓存 所有 inject 消息, 写完最后一个 tool_result 后再批量追加为 user message。

createNoopHookRunner: 让 agent 主循环保持直线阅读

用途: 默认没有 hook 时, agent 主循环里每个 if (hookRunner) 都会打断阅读。 用 createNoopHookRunner() 做一个"啥都不做" 的默认实现, 让主循环保持直线流。

真实场景: 用户没注册任何 hook, 跑最简单的 harness: "读 1 个文件, 改 1 个文件, 结束"。 主循环 6 步里每步都有 hookRunner.run(...), 如果 hookRunner 是 undefined 就 要写 6 个 if, 阅读起来像刺猬。

设计思想: 空对象模式 (Null Object) — 不是简单 hookRunner?.run(), 而是给一个永远返回 { exitCode: 0 } 的对象, 协议完整但行为无副作用。 看 src/hooks.ts 第 220 行的 createNoopHookRunner(): 

export function createNoopHookRunner(): HookRunner {
  return {
    async run() {
      return { exitCode: 0 };
    },
  };
}

实现细节: 在 index.ts 的 Composition Root 里, 如果用户没传 hookRunner 就 createNoopHookRunner() 兜底, 保证 agent.ts 里 await hookRunner.run(event) 永远 不需要判空。 这是 Reference 章节 "模式 1 · 工厂 + 闭包" 的应用 — 闭包返回一个稳定的对象, 行为可预测。

handler 注册: Partial<Record> 表达"不监听就是空数组"

用途: 创建 HookRunner 时传入 handlers 配置, 同一 事件可注册多个 handler。 没注册的事件 = 空数组 = 啥也不做。

真实场景: index.ts 里 createHookRunner({ PreToolUse: [auditHandler, noVerifyHandler], PostToolUse: [slackHandler] }, logger) — 只注册了 2 个事件, SessionStart 默认空数组。

设计思想: 用 Partial<Record<K, V>> 类型让"只填需要的事件" 成为类型安全的事实 — 漏填的事件是 undefined, 编译期允许, 运行期用 ?? [] 兜底。 看 src/hooks.ts 第 163 行的 createHookRunner 工厂: 

export function createHookRunner(
  handlers: Partial<Record<HookEventName, HookHandler[]>>,
  logger: Logger,
): HookRunner {
  // handlers[event.name] ?? [] 兜底, Partial 允许 undefined
}

实现细节: 工厂函数返回的对象只有一个 run 方法, 没有 addHandler / removeHandler — 这是故意的简化: handler 在 Composition Root 一次性 注入, 运行期不能动态加 (避免多 handler 顺序的歧义)。 这是 Reference 章节 "模式 3 · 窄接口" 的应用 — 接口只暴露 run, 不暴露 handlers 数组, 闭包状态不外泄。

主循环集成: 在哪 3 个时机 emit

用途: Hook 是扩展点, 主循环必须 按固定时序 emit, 时序错了协议就错。 3 个 emit 时机在 agent.ts 里 是:

  1. SessionStart: while 循环开始前, 把 query 包装成事件 { name: "SessionStart", payload: { query } } 发一次。 handler 返回 2 的 message 注入到 system 消息后, 第一轮 assistant 调用之前。
  2. PreToolUse: 权限检查通过后, 调 toolRegistry.invoke(...) 之前。 handler 返回 1 阻止 invoke, 写 "Permission denied by hook" tool message; 返回 2 累积 message, 在所有 tool_result 后注入。
  3. PostToolUse: invoke 完成后, 把 ToolResult.output 喂给 handler。 handler 返回 1 (block) 在 PostToolUse 上不重做, 只写警告日志 (因为 tool 已经跑完, block 没意义); 返回 2 累积 message。

设计思想: 集成点必须集中, 不能散 在 agent.ts 各个角落。 一般 6 步 loop 里 SessionStart 在 step 0 (load history 后), PreToolUse 在 step 3 (call tools 前), PostToolUse 在 step 3.5 (call tools 后, before step 4 compress)。

实现细节: 集成代码写在 agent.ts 里, 调用方式统一是 const hookResult = await hookRunner.run(event); if (hookResult.exitCode === 1) ...; if (hookResult.message) pendingInjects.push(hookResult.message);。 缓存的 inject 消息在 collect tool results 阶段之后批量追加, 这就是 第 02 章 "message 配对" 协议的具体应用。

对 prompt cache 的影响: SessionStart 注入是 stable prefix 一部分

用途: SessionStart 的 inject 消息会进入 system 消息, 而 system 消息是 stable prefix 的核心。 每次启动注入 不同内容, 整个对话的 cache 命中会崩。

真实场景: 团队用 SessionStart 注入"项目规范", 但 规范每行一个 prompt 改了 10 个字符, 后面 100 轮对话 cache 全部 miss, 成本 ×10。 应该把 SessionStart 的注入做成幂等: 启动时一次写, 后续 100 轮都引用, 不再追加。

设计思想: 幂等性约束 — SessionStart 注入必须保证"同一 session 内, 相同 query + 相同 handler 集合" 产出相同 message。 handler 应该读配置文件 / MEMORY, 不读外部随机 数据 (时间戳 / 随机数)。

实现细节: 这是和第 05 章 (Skill 稳定前缀) + 第 09 章 (Memory 4 类 tags) 联合的 cache 策略。 SessionStart 的 inject 应该在 handler 内部做"内容去重" — 同一 handler 在同一 session 只 emit 一次。 这部分实现在第 10 章 (Cache) 详细展开。

fake test: 用 stub handler 验证 block 短路和 inject 累积

用途: Hook 系统的测试不需要真 hook, 用 stub 即可 — 测的是"主循环按返回码做了什么", 不是 handler 内部逻辑。

真实场景: 用户写一个"检测 rm -rf" 的 handler, 想 验证: (a) rm -rf / 应该 block; (b) rm file.txt 应该 continue; (c) block 时主循环写 "Permission denied by hook" tool message, 不真的跑 rm。 stub handler 返回固定结果就能测。

设计思想: 测协议不测实现 — Hook 系统 的协议是"返回码 → 主循环行为", 这层用 stub 测最稳。 handler 内部 业务逻辑 (rm 检测算法) 用单元测试单独测, 不混在 HookRunner 测试里。 

test("block exitCode 短路后续 handler", async () => {
  const calls: string[] = [];
  const runner = createHookRunner({
    PreToolUse: [
      () => { calls.push("first"); return { exitCode: 0 }; },
      () => { calls.push("second"); return { exitCode: 1, message: "blocked" }; },
      () => { calls.push("third"); return { exitCode: 2, message: "warn" }; },
    ],
  }, noopLogger);
  const result = await runner.run({ name: "PreToolUse", payload: {...} });
  expect(calls).toEqual(["first", "second"]);   // third 没跑
  expect(result).toEqual({ exitCode: 1, message: "blocked" });
});

test("inject 累积: 多个 handler 的 message 用空行拼接", async () => {
  const runner = createHookRunner({
    PostToolUse: [
      () => ({ exitCode: 2, message: "audit: 工具跑完了" }),
      () => ({ exitCode: 2, message: "warn: 命令有点危险" }),
    ],
  }, noopLogger);
  const result = await runner.run({ name: "PostToolUse", payload: {...} });
  expect(result.exitCode).toBe(2);
  expect(result.message).toBe("audit: 工具跑完了\n\nwarn: 命令有点危险");
});

test("handler 抛异常只 warn, 不中断", async () => {
  const runner = createHookRunner({
    PreToolUse: [
      () => { throw new Error("boom"); },
      () => ({ exitCode: 2, message: "next" }),
    ],
  }, noopLogger);
  const result = await runner.run({ name: "PreToolUse", payload: {...} });
  expect(result.message).toBe("next");   // 异常被吞, 后续继续
});

实现细节: 这 3 个 stub 测试覆盖了 Hook 系统 90% 的 协议 — 短路、累积、异常隔离。 handler 自身的业务逻辑测试应该放在 handler-name.test.ts, 不在 hooks.test.ts。

常见误解: Hook 不是 Permission 替代品

误解 1: "Hook 能取代 Permission?" 错。 Permission 是 安全边界 — 黑名单命令、workspace 外路径、ask 用户, 这些必须 hard 拒绝, 不让 LLM 重试。 Hook 是扩展点 — 审计、提醒、注入上下文, 不替代安全。 两者职责正交。

误解 2: "exitCode 2 = block 的一种?" 错。 2 是 inject 后继续, 不是 block。 把 2 当 1 用, 工具不会 跑, 但 LLM 也不会收到反馈, 以为是协议错误。

误解 3: "PostToolUse 上 block 能重做工具?" 错。 PostToolUse 触发时工具已经跑完, block 没有意义。 block 在 PostToolUse 上只记 warn 日志, 不重做。 想要"重做" 用上层 retry 机制, 不是 Hook。

误解 4: "SessionStart 能 emit 多次?" 不行。 SessionStart 语义是"每个 agent 实例第一次 run() 时", emit 多次 会污染 cache。 主循环只在 while 之前发一次, 不在 循环内重发。

反例梯度: 4 个常见错误

反例 1 · 过度耦合

把 handler 业务逻辑写进 agent.ts 主循环。 改一次 规则要重读 200 行 agent 代码。 错。 Handler 应该注册到 HookRunner, 主循环不出现业务关键字。 agent.ts 里 只调 hookRunner.run(event), 不 import 任何 handler 实现。

反例 2 · 把 2 当 1

用户写 handler 检测 --no-verify, 想"阻止 + 给 LLM 提示改用 --verify", 返回 { exitCode: 1, message: "请用 --verify" }。 错。 1 是 block, message 不会注入, LLM 收到 "blocked" 啥也不知道。 这种情况要返回 { exitCode: 2, message: "..." } (注入) 或 1 (硬阻止) + 主循环负责把 message 写到 tool result。

反例 3 · 忽略延迟注入

主循环在 tool_result 写完后, 立刻把 inject 消息追加到 history 数组。 表面看"消息进 history 了", 但如果 LLM 一轮发了 3 个 tool_call, 第一个 tool_result 后立刻追加 user message, OpenAI API 拒绝 400。 正确做法: 缓存所有 inject 消息, 在本轮所有 tool_result 写完后统一追加。

反例 4 · handler 抛异常拖死 agent

handler 调外部 API 写审计日志, API 挂掉, throw。 主循环 await hookRunner.run(event) 直接抛, agent 死。 错。 Hook 是扩展, 不是核心。 hookRunner.run() 必须 catch 所有 异常, 只 warn 日志, 然后继续。 一个审计 handler 挂掉不能 让 agent 不能用。

Validation: 4 条不变量检验

  1. 协议对称: handler 返回什么 exitCode, 主循环 行为必须和 exitCode 语义一致 — 0 啥也不做, 1 阻止 (PreToolUse 不调 invoke, PostToolUse 只 warn), 2 累积 message 在本轮 tool 结束后注入。 验证: 用 stub handler 返回 0/1/2, 跑主循环 1 轮, 看 history 末尾的 tool message 和 user message 数量是否符合预期。
  2. 延迟注入: 同一轮多个 tool_call 时, inject 消息 必须出现在所有 tool_result 之后。 验证: 写测试 让 LLM 调 3 个 tool (run_bash / run_read / run_write), 第一个 PostToolUse 就返回 2, 看 history 数组里 inject 消息的 index 是 不是在所有 tool_result 之后。
  3. 闭包隔离: 多个 agent 实例 (父子 subagent) 用 各自的 HookRunner, 不共享 handler 数组。 验证: 父 agent handler 调一次, 子 agent handler 不应该被调用; 子 agent 的 handler 调一次, 父 agent 不应该被调用。
  4. 空对象兼容: createNoopHookRunner() 和 真实 createHookRunner(...) 接口完全一致, agent.ts 同一份代码, 不写 if (hookRunner)。 验证: TypeScript 编译通过, 跑 100 轮对话, 行为和挂空 hook 时一致 (只是少了 inject message)。

回望: 哪些原则在本章兑现了

  • 关注点分离: Hook 独立于 Permission, 独立于 ToolRegistry。 主循环只发事件, 不处理业务规则。
  • 工厂 + 闭包: createHookRunner 把 handlers 数组 + 累积 injected 数组藏在闭包, 外部只能调 run。
  • 空对象模式: createNoopHookRunner 让 agent 主循环保持直线流, 不需要判空。
  • 协议对称: 3 个事件 × 3 个返回码, 每个组合的 主循环行为都有明确定义, 单元测试覆盖。
  • 延迟注入: inject 消息不能在 tool_call / tool_result 中间插入, 必须缓存到本轮结束后追加, 这是 LLM 协议层的硬约束。

本章的 handler 例子都是内联 stub。 实战中 handler 要读 配置文件 / 调外部 API, 这部分属于"配置层" 而不是 HookRunner 本身。 第 09 章 Memory 展开"持久化" 机制, SessionStart handler 可以从 MEMORY.md 加载团队规范。

Prompt Card (本章任务)

Prompt Card · 第 08 章

目标: 给 harness 加 Hook 系统, 3 个事件 × 3 个 返回码, 主循环按固定时序 emit, handler 在外部注册。

场景: 团队挂 3 个 handler — 审计 logger 写 audit.log; --no-verify 检查返回 1 阻止; 敏感命令 警告返回 2 注入 user message。 验证 3 个 handler 串联时, block 短路后续 handler, inject 累积 message 用空行拼接。

模块: src/hooks.ts (新) 暴露 createHookRunner(handlers, logger) + createNoopHookRunner(); src/agent.ts (改) 在 3 个固定时机 emit event; src/index.ts (改) Composition Root 注入 HookRunner。

边界 (LLM 必须遵守的 checklist):

  • 3 个事件 SessionStart / PreToolUse / PostToolUse, 不可新增第 4 个事件点
  • 3 个返回码 0 / 1 / 2, 语义正交, 不可合并 / 重命名
  • block 短路后续 handler, inject 累积 message 用 \n\n 拼接
  • handler 异常只 warn 日志, 不中断 agent 主流程
  • exitCode 2 的 message 延迟到本轮所有 tool_result 写完后追加
  • createNoopHookRunner 提供默认空实现, agent.ts 不写 if (hookRunner)
  • HookRunner 接口只有 run(event), 不暴露 handler 数组

验证 (用 stub handler + vitest, 逐条断言):

  • block 短路: 3 个 handler 串联, 第 2 个返回 1, 第 3 个不执行
  • inject 累积: 2 个 handler 都返回 2, message 用空行拼接
  • 异常隔离: handler 抛 Error, 后续 handler 继续执行
  • createNoopHookRunner 和 createHookRunner 接口完全一致, 行为不同
  • 延迟注入: 一轮 3 个 tool_call, 第一个 PostToolUse 返回 2, inject 消息在所有 tool_result 之后

本章练习

  1. 故意把 handler 业务逻辑写进 agent.ts (如 if (event.name === "PreToolUse" && event.payload.args.command?.includes("--no-verify")) ...), 跑测试, 看"主循环只发事件不处理业务" 是否抓到 (agent.ts 行数膨胀 + 改规则必须改 agent)。
  2. 让 handler 返回 { exitCode: 1, message: "请改用 --verify" } 想 "block + 给 LLM 提示", 跑测试, 看"1 是 block, message 不会注入" 是否抓到 (LLM 收不到 message, 一直重复 --no-verify)。
  3. 故意在 tool_result 之后立刻追加 user message (不缓存 inject 消息), 跑多 tool_call 测试, 看"延迟注入" 是否抓到 (OpenAI API 400 error, 或协议断言失败)。
  4. 故意让 handler throw new Error("boom"), 跑测试, 看"异常隔离" 是否抓到 (agent 主流程崩 vs. 后续 handler 继续)。

本章小结

Hook 是给主循环的扩展点, 不是 Permission 的替代品。 核心是 4 个设计:

  • 3 个事件点: SessionStart / PreToolUse / PostToolUse, 少而精, 事件点多就是多一份主循环耦合。
  • 3 个返回码: 0 / 1 / 2, 语义正交 — 0 啥也不做, 1 阻止 (PreToolUse 阻止工具, PostToolUse 只 warn), 2 注入 user message。
  • 多 handler 聚合: 串行执行, block 短路, inject 累积 (用 \n\n 拼接), 异常隔离 (catch + warn, 不中断)。
  • 延迟注入: exitCode 2 的 message 必须在所有 tool_result 写完后追加, 不然破坏 OpenAI API 协议。
