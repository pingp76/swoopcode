第 08 章 · 在主循环旁边扩展

在 Loop 周围挂钩子：Hook

Hook 是 agent loop 的观察和扩展点。它允许我们在 session start、tool use 前后插入逻辑，但必须避免破坏 tool_call/tool_result 的消息配对。

你已经知道什么，还不知道什么

你已经有权限层。现在想在工具调用前后做一些额外事情：记录日志、注入提醒、阻止某类行为、触发统计。直接改 agent.ts 会让主 loop 越来越臃肿。

Hook 的目标是“可扩展但不失控”。它应该观察事件、返回轻量消息或决策，不应该变成第二套 agent loop。

本章场景

用户运行测试失败后，你想通过 PostToolUse hook 注入一句提醒：“测试失败时请先读失败日志，不要盲改。”这对模型有帮助，但如果插入位置不对，会破坏工具消息顺序。

Hook 系统要提供明确事件：SessionStart、PreToolUse、PostToolUse，并让 hook 输出延迟到安全位置注入。

先试一个朴素方案

朴素方案是在每个工具 handler 里手写 before/after 逻辑，或者让 hook 直接往 History 任意位置插消息。

这会让工具实现和扩展逻辑耦合，也可能把系统提醒插到 assistant tool_call 和 tool_result 中间，导致 LLM 协议非法。 

history.add(assistantToolCall);
history.add({ role: "system", content: "hook reminder" });
history.add(toolResult); // 这里已经破坏了 tool_call/tool_result 邻近关系

朴素方案为什么不够

消息配对被破坏
PostToolUse 的提醒不能插在工具调用组内部。
扩展点污染业务
每个工具都手写 hook 调用，会让工具模块变得不纯。
失败传播不清
hook 失败不应该默认让工具执行失败，除非设计明确允许阻断。

回到 Agent Loop

Hook 分布在 loop 的边缘：SessionStart 在会话启动后；PreToolUse 在权限允许后、工具执行前；PostToolUse 在工具执行后、结果写回安全位置时。

关键原则是：Hook 可以产生 reminder，但 reminder 要进入 session event buffer 或安全注入点，而不是插入工具消息组中间。

架构图解

从原理落到代码

到这里已经有了场景、失败原因和架构图。下面才进入实现层：只抓住接口形状、状态边界和源码入口。

读源码时先找入口函数，再找它读写的状态，最后看测试如何证明这些状态变化是真的发生了。

一次真实运行 walkthrough

  1. 启动 REPL 时触发 SessionStart hook，记录会话信息。
  2. LLM 返回 tool call，权限层先做 allow/ask/deny。
  3. 允许执行后，HookRunner 触发 PreToolUse，观察工具名和参数。
  4. 工具执行并返回 ToolResult。
  5. Agent 写回 tool_result，保持工具配对完整。
  6. HookRunner 触发 PostToolUse，产生的提醒进入 session event buffer。
  7. 下一轮构建 messages 时，reminder 在安全位置注入给 LLM。

关键接口和伪码

Hook 输入是事件上下文，输出是轻量结果。不要让 hook 直接持有 History 可变引用。 

type HookEvent =
  | { kind: "SessionStart"; sessionId: string }
  | { kind: "PreToolUse"; toolName: string; args: unknown }
  | { kind: "PostToolUse"; toolName: string; result: ToolResult };

interface HookRunner {
  run(event: HookEvent): Promise<HookResult[]>;
}

设计直觉

Hook 的设计重点是“不侵入”。主 loop 保持清楚，扩展逻辑通过事件观察。

把 hook 输出延迟成 reminder，是为了同时满足两件事：模型能知道外部提醒，工具消息协议不被破坏。

架构拆分

HookRunner
按稳定顺序执行 hook，收集结果和错误。
SessionEventBuffer
缓存 out-of-band 事件，下一轮安全注入。
Agent
只在固定点触发 hook，不把扩展逻辑写进主流程。

状态与边界

hook context
可观察工具名、参数、结果摘要，但不应暴露敏感内部状态。
reminder
hook 输出可以作为动态消息进入下一轮 LLM。
History
不允许 hook 任意插入工具调用组中间。

Prompt Card

Prompt Card

目标：实现 HookRunner 和 SessionEventBuffer，让 agent 支持 SessionStart/PreToolUse/PostToolUse 扩展点。

场景：工具执行前后需要观察和提醒，但不能把扩展逻辑写死在每个工具里。

模块：实现 hooks.ts、session-events.ts，并在 agent.ts 的固定点触发。

接线：composition root 创建 HookRunner，Agent 注入后触发事件，hook 输出进入 reminder buffer。

边界：hook 不得直接破坏 tool_call/tool_result 配对；hook 失败默认不让主 loop 崩溃。

例外和 corner case：多个 hook 顺序、hook 抛错、PostToolUse 产生提醒、权限拒绝时是否触发 PreToolUse。

验证：测试事件顺序、延迟注入、hook 错误隔离和 agent 集成路径。

源码入口：src/hooks.ts、src/session-events.ts、src/agent.ts。

文档同步：说明 Hook 是扩展点，不是第二套业务流程。

容易踩坑

Design Trap

常见错误：PostToolUse 直接往 History 当前工具组中插入 system message。

为什么错：这会让 LLM 协议认为 tool_result 顺序非法。

正确做法：PostToolUse 输出进入 session event buffer，下一轮在安全位置注入。

怎么验证：构造工具调用后 hook reminder，检查 messages 中 tool_call/tool_result 仍相邻或合法配对。

  • Hook 顺序不稳定会导致调试困难。
  • Hook 吞掉工具错误会掩盖真实失败。
  • PreToolUse 放在权限前会让 hook 观察到本不该执行的敏感参数。

如何验证

Validation

手工 query：启用一个 PostToolUse hook，让它在工具执行后提醒模型如何处理失败输出。

确定性测试：测试 HookRunner 对多个 hook 的顺序和错误隔离。

集成观察：Agent 执行工具后，下一轮 messages 包含 hook reminder。

失败信号：hook reminder 出现在 tool_call 和 tool_result 之间。

  • src/hooks.test.ts 验证 hook 生命周期。
  • src/session-events.test.ts 验证提醒缓冲。
  • src/agent.test.ts 验证 hook 与工具执行集成。

如果实现失败，先查哪里

  • 检查 hook 触发点在权限之后还是之前。
  • 检查 session event buffer 是否被 drain。
  • 检查 hook 错误是否被记录但不吞掉工具结果。
  • 检查当前 messages 是否仍符合 LLM 工具协议。
