第 04 章 · 让 Agent 学会分身

SubAgent: 长任务委托, 父 agent 不被撑爆

第 03 章的 TODO 让 LLM 自己追踪步骤, 但跑 50 轮对话后父 agent 的 history 会被 50 轮工具结果撑爆 — 上下文超出窗口, LLM 反应变慢。 这一章给 agent 加一个"分身"能力: run_subagent 工具, 把 长期子任务委托给子智能体, 父 agent 只看最终结果。 读完后, 你能 讲清"父子隔离" 的 3 个不变量, 并能用 fake LLM 验证子 agent 跑完后 父 history 不被污染。

真实失败故事: 50 轮后父 agent 撑爆

写代码之前, 看一段真实长任务的失败。 跑一个真实场景: 用户让 agent "全项目 code review, 列出 3 个最严重的 bug, 给出修复方案"。

  1. 第 1-10 轮: 父 agent 调 run_read 看 5 个文件, 每个文件 1000 行, history 累积 5000 token。 还撑得住。
  2. 第 25 轮: 看到第 12 个文件时, history 累积 12000 token。 LLM 反应开始慢, 偶尔重复调之前看过的文件。
  3. 第 50 轮: 看到第 25 个文件时, history 累积 25000 token。 LLM 严重失忆, 总结时漏掉 5 个文件, 重复 3 个文件的判断, 修复方案质量大幅下降。
  4. 第 100 轮: history 累积 50000 token, 单次 prompt 超过窗口, LLM SDK 报 400 错误, agent loop 整个崩。

问题本质: 父 agent 扛了"完整探索历史" + "用户 query", 上下文窗口 被撑爆。 即使 LLM 窗口是 200K, 50 轮工具结果也能把它填满。

朴素想法 1: "把 read 改成只读前 100 行?" 治标不治本 — 父 agent 还是扛着工具结果, 还是会撑爆。

朴素想法 2: "压缩, 把前 10 个文件的 read 结果总结成一段?" 能缓解, 但总结会丢失细节, LLM 基于总结给出的修复方案质量下降。

正确做法: 父 agent 调 run_subagent(task="review these 5 files"), 子智能体独立跑, 看 5 个文件, 产出"哪些有 bug" 的总结, 父 agent 只看到总结, 不看到子智能体的全部 history。 上下文从 50000 token 降到 500 token。 成本从 1.5 美元降到 0.05 美元。

朴素反例: 直接调 agent.run 复用 history

// ❌ 反例: 子 agent 复用父 history
function runSubagent(task: string) {
  // 错误: 共享 history
  return createAgent({ llm, history: parentHistory }).run(task);
}

4 件事立刻坏掉:

  1. 父子污染: 子 agent 写进 history 的消息, 父 agent 下 一轮也能看到。 子 agent 调了 50 个工具, 父 agent 全部继承。 上下文没被压缩, 反而共享污染。
  2. 状态错乱: 子 agent 的 TODO 列表和父 agent 混在一起, 进度条闪烁 — 父 agent 看到 [3/5], 子 agent 看到 [1/3], 哪个是当前的? 用户搞不清。
  3. 权限泄露: 子 agent 拿到了父 agent 的全部工具权限 (包括 run_write), 父 agent 不想让子 agent 写文件也防不住。 父 agent "分析 5 个文件", 子 agent 偷偷 run_write 改了 2 个。
  4. 不可调试: 父 history 里混杂父子消息, 出问题不知道 哪一层搞的。 调试时只能二分"父还是子", 但 history 看不到 来源, 只能看 message role / content, 没线索。

3 条不变量

把反例的 4 个坏处倒过来, 就是 SubAgent 的 3 条不变量。 这 3 条 不变量是"写 SubAgent 代码时心里时刻记着的 3 件事", 缺一不可。

不变量 1 · 父子 history 隔离
子 agent 有自己的 history, 父 agent 看不到子 agent 的内部消息。 子 agent 跑完, 父 agent 只看到一条"任务已完成, 总结: ..." 的结果消息 (role: "tool", tool_call_id 是 run_subagent 的 id)。 这条消息的 content 是子 agent 的最终回复, 不是子 agent 调过的 50 个工具结果。
不变量 2 · 父子 TODO 隔离
子 agent 独立 TodoManager, 父 agent 的 TODO 列表不受影响。 父子进度条独立显示 — 父 REPL 顶部是父 TODO, 子 agent 内部 跑自己的 TODO (用户看不到子 TODO, 怕屏幕太乱)。
不变量 3 · 子 agent 工具权限 ≤ 父 agent
子 agent 拿到的是过滤后的 ToolRegistry, 父 agent 有的工具子 agent 不一定有 (例如父可以写, 子只能读)。 这是"最小权限" 原则的体现 — 给子 agent 越少工具, 越不可能 出问题。

为什么共享 LLM, 不共享 history

一个微妙的设计: 父子 agent 共享同一个 LLMClient 实例, 但 不共享 history。 为什么?

共享 LLM 的好处:

  1. 避免双倍费用: 父 agent 调 LLM, 子 agent 也调 LLM, 共享一个 client 不会重复计费 (但 token 仍按子 agent 用量算)。
  2. 复用 LLM 客户端配置: provider / apiKey / 模型名 都是一处配置, 子 agent 自动继承。
  3. 支持 streaming / 缓存 / retry 等中间件: LLM 客户端 通常会包一层 retry / cache, 父子共享时这些中间件也共享。

不共享 history 的原因: 父子上下文是隔离的, LLM 看到的"对话" 是各自 agent 的对话, 不是父子合并的。 共享 history 等于父子 共享上下文, 失去"父只看总结" 的价值。

❌ / ✅ 写法: 

// ❌ 父子都用新的 LLMClient
const childLLM = createOpenAILLMClient({ apiKey });
// 浪费: 子 agent 的 retry / cache / logging 都从零开始

// ✅ 父子共享 LLMClient
const childLLM = parentLLM;   // 同一个实例
// 子 agent 自动获得 retry / cache / logging

工厂模式: 子 agent 怎么创建

用途: 不变量 1 + 2 决定了子 agent 必须是独立实例, 通过 createAgent() 工厂创建, 不复用父的闭包。

真实场景: 父 agent 调 run_subagent(task="review src/agent.ts", tools=["run_read"]), harness 内部用工厂函数创建子 agent 实例, 注入过滤后的工具集, 跑子任务, 拿结果, 返回父。

设计思想: 经典工厂模式 + 依赖注入 (Reference 章节的模式 1 + 模式 3)。 工厂创建独立闭包, 注入 5 件套 (llm / history / todoManager / tools / hooks)。 5 个关键点决定"父子隔 离是不是真成立"。

实现细节: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export function createSubagentTool(deps: {
  parentAgent: Agent;
  factory: AgentFactory;       // createAgent 工厂
  permissionManager: PermissionManager;
}): Tool {
  return {
    name: "run_subagent",
    description: "把一个子任务委托给子 agent, 子 agent 独立 history, " +
                 "完成后只返回最终结果。子 agent 看到的工具是你给的子集。",
    parameters: {
      type: "object",
      properties: {
        task: { type: "string", description: "子 agent 要完成的任务描述" },
        tools: { type: "array", description: "子 agent 可用的工具子集, 留空 = 父的只读子集" },
        maxRounds: { type: "number", description: "子 agent 最大循环轮数, 默认 10" },
      },
      required: ["task"],
    },
    execute: async (args) => {
      // 不变量 1: 独立 history
      const childHistory = createHistory();
      // 不变量 2: 独立 TodoManager
      const childTodoManager = createTodoManager();
      // 不变量 3: 工具权限由父控制
      const childTools = filterToolsForChild(deps.parentAgent.tools, args.tools);
      // 共享 LLM, 不共享 history
      const childAgent = deps.factory({
        llm: deps.parentAgent.llm,
        history: childHistory,
        todoManager: childTodoManager,
        tools: childTools,
        maxRounds: args.maxRounds ?? 10,
      });
      const result = await childAgent.run(args.task);
      // 子 agent 跑完, 父 agent 只看到这条结果 (作为 tool message)
      return { content: result };
    },
  };
}

子 agent 工具怎么过滤

用途: 父 agent 不想让子 agent 拥有的工具, 子 agent 拿不到 schema, 也调不到。 工具权限由父显式声明, 子 agent 自动继承 过滤后的子集。

真实场景: 父 agent 调 run_subagent(task="分析 src/agent.ts", tools=["run_read"]), 子 agent 工具 schema 不含 run_write, 即使子 agent 想偷偷写, 调 run_write 返回 "tool not found" 错误。

父给的子集 子 agent 能调的 典型场景
["run_read", "run_bash"] 读文件 + 跑只读命令 "分析这 5 个文件, 不要改"
["run_read", "run_write", "run_edit"] 读 + 写 + 编辑 "实现这个功能, 改文件"
[] (默认) 父的只读子集 (run_read / run_bash readonly) "查资料, 给我答案"

❌ / ✅: 不要让子 agent 拿到 run_subagent 自身。 

// ❌ 子 agent 可以再 spawn 子子 agent
filterToolsForChild(parentTools, args.tools).includes("run_subagent")
// 递归分身, 上下文被无限拉低, 但费用指数上升

// ✅ 过滤掉 run_subagent, 强制叶子节点
const childTools = filterToolsForChild(parentTools, args.tools);
childTools.remove("run_subagent");  // 永远禁止子 agent 递归

这是硬规矩, 不是配置项。 叶子节点策略, 避免"子生子" 拖垮系统。 递归分身看起来"灵活", 实际是个灾难 — 父给子 1 个任务, 子又 spawn 3 个子任务, 每个子任务又 spawn 3 个, 第 4 层就有 81 个 agent, 费用爆炸, 调试也爆炸。

子 agent 出错怎么办

用途: 子 agent 跑挂了 (maxRounds 用完 / 工具抛 throw), 父 agent 不能假装成功。 3 种处理方式, 跟第 02 章工具失败模式一致。

真实场景: 子 agent 跑 10 轮没完成, 收敛为 maxRounds 触底, 父 agent 收到 "子任务超时" 工具结果, 决定: 重试 / 换工具 / 放弃。 父 agent 不会因为子失败而崩。

情况 父 agent 看到 处理
子 agent 正常完成 "任务完成, 总结: ..." 父 agent 继续下一步
子 agent 超 maxRounds "子任务超时, 已完成: ... / 未完成: ..." 父 agent 决定: 重试 / 换工具 / 放弃
子 agent 抛 throw "子任务失败: error message" 父 agent 决定: 重试 / 换工具 / 放弃

❌ / ✅: 不要让子 agent 失败时父 agent 也崩。 

// ❌ 子 agent 失败, 父 agent 整个崩
async function runSubagent(args) {
  const result = await childAgent.run(args.task);
  return { content: result };     // child 抛了, 这里也抛
}

// ✅ 失败转成 ToolResult, 父 agent 知道
async function runSubagent(args) {
  try {
    const result = await childAgent.run(args.task);
    return { content: result };
  } catch (e) {
    return { content: `子任务失败: ${e.message}\n请决定重试 / 换方案 / 放弃。`, error: true };
  }
}

业务错误 (子 agent 失败) 用 error: true 写回, 父 agent 看到后能继续推理 — "子任务失败了, 我换个方案再试"。 跟第 02 章 的工具失败模式一致。

注意: try/catch 包在 childAgent.run() 外面, 捕获的是"子 agent 抛出的 throw" (比如 maxRounds 用完), 不是"子 agent 返回的业务错误" (比如子 agent 调 run_read 失败)。 后者子 agent 自己用 error: true 处理, 不会抛 throw。

子 agent 跑的时候父怎么显示

用途: 子 agent 跑 10 轮工具, 父 agent 屏幕静止 30 秒, 用户会以为卡了。 REPL 怎么展示子 agent 进度。

真实场景: 父 REPL 渲染时, 在当前 prompt 上方加一行 "↳ [子 agent] 进度", 用户看到子 agent 在跑, 不会以为父 agent 卡死。 子 agent 自己的工具调用细节不显示 (避免父 REPL 被淹没)。

设计思想: 经典反向通知模式 — 父等子, 子 反向告诉父自己的状态 (通过 callback / event)。 不阻塞父, 父 显示当前状态, 用户知道在等什么。

实现细节: 

// 教学简化版, 真实实现见 GitHub 永久链接
function renderSubagentIndicator(childProgress: ChildProgress): string {
  if (!childProgress) return "";
  const { task, currentStep, totalSteps } = childProgress;
  return `\n  ↳ [子 agent] ${task} (${currentStep}/${totalSteps})\n`;
}

用途: SubAgent 隔离是最难写测试的, 因为涉及"父看到的" 和"实际发生的" 的差异。 写一个测试, 验证 3 件事。

设计思想: 用 fake LLM 模拟"父子各自调 LLM" 的序列, 在父 agent.run 结束后, 断言"父 history 看不到子 agent 内部消息"。

实现细节: 

test("subagent 跑完后, 父 history 不被污染", async () => {
  let isChild = false;
  const fakeLLM = createFakeLLM([
    // 父第一次: 调 run_subagent
    {
      content: null,
      toolCalls: [{
        id: "sub1", function: {
          name: "run_subagent",
          arguments: '{"task":"分析 package.json"}',
        },
      }],
      finishReason: "tool_calls",
    },
    // 父第二次: 收到子 agent 结果, 回答
    { content: "Based on subagent: test = npm test", toolCalls: [], finishReason: "stop" },
    // 子 agent 调的工具 (按调用顺序)
    // 第一次: 子 agent 调 run_read
    {
      content: null,
      toolCalls: [{ id: "child_r1", function: { name: "run_read", arguments: '{"path":"package.json"}' } }],
      finishReason: "tool_calls",
    },
    // 第二次: 子 agent 看到结果, 总结
    { content: "package.json says test = npm test", toolCalls: [], finishReason: "stop" },
  ]);

  // 关键: 区分"父 chat" 和"子 chat"
  const callMessages: any[][] = [];
  let parentCallCount = 0;
  let childCallCount = 0;
  const trackingLLM = {
    async chat({ messages }) {
      const isParentCall = !isChild;
      callMessages.push(messages);
      if (isParentCall) parentCallCount++;
      else childCallCount++;
      // ... 返回 fake response (根据 callCount)
    },
  };

  const agent = createAgent({ llm: trackingLLM, history: createHistory(), tools });
  const reply = await agent.run("What's the test command?");

  // 验证 1: 父 history 末尾是 assistant 总结
  const lastParentCall = callMessages[parentCallCount - 1];
  const lastAssistant = lastParentCall.filter(m => m.role === "assistant").pop();
  expect(lastAssistant.content).toContain("Based on subagent");

  // 验证 2: 父 history 看不到子 agent 的 tool_call
  expect(lastParentCall).not.toContainEqual(expect.objectContaining({
    role: "tool", tool_call_id: "child_r1",
  }));

  // 验证 3: 父 history 只看到子 agent 的总结作为 tool result
  expect(lastParentCall).toContainEqual(expect.objectContaining({
    role: "tool", tool_call_id: "sub1",
    content: expect.stringContaining("package.json says test = npm test"),
  }));
});

这条测试如果挂, 你立刻知道父子隔离出了问题。 3 个断言联合 验证: 父看到总结, 看不到子内部消息, run_subagent 配对正确。

3 个常见误解

误解 1 · "子 agent 应该继承父 history"
错。 子 agent 独立 history 是不变量 1, 继承父 history 等于 把子 agent 50 轮工具结果灌给父, 上下文反而没被压缩。 父 想"让子知道上下文" 应该显式传 "context summary" 给子 agent, 不是共享 history。
误解 2 · "子 agent 可以再 spawn 子子 agent"
错。 递归 spawn 看似灵活, 实际是费用爆炸 + 调试噩梦。 强制 叶子节点策略, 子 agent 不拿 run_subagent 工具。
误解 3 · "子 agent 失败应该重试"
错。 子 agent 失败是父 agent 的决策点, 不该 harness 自动重试。 重试掩盖了真实问题, 应该让父 agent 看到 错误, 自己决定: 重试 / 换方案 / 放弃。

反例梯度

新手错法 · A

常见错误: 子 agent 复用父的 history 对象。

为什么错: 子 agent 写进 history 的消息全部进父, 上下文没被压缩, 反而共享污染。

正确做法: 子 agent 通过 createHistory() 独立创建, 父子闭包隔离。

中级错法 · B

常见错误: 父 agent 把所有工具都传给子 agent。

为什么错: 父想"分析文件不要改", 结果子 agent 拿到 run_write 直接改了, 父 agent 防不住。

正确做法: 父显式声明子 agent 的工具子集, 最小权限。

高级错法 · C

常见错误: 子 agent 也能调 run_subagent, 递归 spawn。

为什么错: 上下文被无限拉低, 费用指数上升, 调试噩梦。

正确做法: 过滤工具时强制移除 run_subagent, 叶子节点策略。

边界错法 · D

常见错误: 子 agent 失败时父 agent 整个崩。

为什么错: 子 agent 是异步任务, 它的失败是业务错误 不是 harness 错误, 不该传上去。

正确做法: 子 agent 抛错时, run_subagent 工具返回 { content: "子任务失败: ...", error: true }`, 父 agent 收到后自己决定。

Validation · 第 04 章

父子 history 隔离: 父调 run_subagent, fake LLM 模拟子 agent 调 5 个工具, 验证父 history 看不到这 5 个 tool_call, 只看到子 agent 总结作为 tool result。

子 agent 工具子集: 父声明 tools=["run_read"], 子 agent 工具 schema 不含 run_write, 调 run_write 返回 "tool not found" 错误。

递归禁止: 验证 run_subagent 不在子 agent 工具列表。

子失败不崩: 子 agent 抛错时, run_subagent 工具返回 error: true 的 tool result, 父 history 继续累积, 不抛。

回望: 哪些原则在本章兑现了

  • 工厂模式: 子 agent 走 createAgent() 工厂, 父子闭包隔离天然成立。
  • 依赖注入: 父子共享 LLM, 独立 history / todoManager / tools。
  • 最小权限: 父声明子 agent 工具子集, 强制移除 run_subagent。
  • 错误传播: 子失败不传 throw, 业务错误用 error: true。

目标: 给 agent 加 run_subagent 工具, 长任务可以 委托子 agent, 父子 history / todo / tools 严格隔离。

场景: 用户说"全项目 code review", 父 agent 调 run_subagent(task="review src/agent.ts", tools=["run_read"]), 子 agent 独立 history 跑 5 轮, 父 agent 只看总结, 上下文不被撑爆。

模块: src/tools/run-subagent.ts (新) 实现 run_subagent 工具; src/subagent.ts (新) 工厂函数 createSubagentFromParent(); src/agent.ts (改) 注册 run_subagent 工具; src/index.ts (改) 注入子 agent 工厂。

边界 (LLM 必须遵守的 checklist):

  • 子 agent 通过 createAgent() 工厂创建, 不复用父的 history / todoManager
  • 父子共享 LLM, 独立 history / todoManager / 过滤后 tools
  • 子 agent 工具子集由父显式声明, 强制移除 run_subagent 防止递归
  • 子 agent 失败用 error: true 写回父, 不传 throw
  • 子 agent 跑完, 父 history 只看到一条"子任务完成, 总结: ..." 的 tool result

验证 (用 fake LLM + vitest, 逐条断言):

  • 父 history 看不到子 agent 内部的 tool_call, 只看到 run_subagent 配对的 tool result
  • 父 / 子 TODO 列表相互不可见, 进度条独立
  • 子 agent 拿到 ["run_read"] 子集, 调 run_write 返回 "tool not found"
  • run_subagent 不在子 agent 工具列表, 递归禁止
  • 子 agent 抛错时, run_subagent 返回 error: true, 父 agent 不崩

本章练习

  1. 故意让子 agent 复用父 history, 跑测试, 看"父子 history 隔离" 是否抓到 (父 history 含子 agent 内部消息)。
  2. 让父 agent 不传 tools 子集 (默认全给), 跑测试, 看"最小权限" 是否抓到 (子 agent 能写文件)。
  3. 子 agent 调 run_subagent 递归, 跑测试, 看"递归禁止" 是否抓到。

本章小结

SubAgent 是给长任务的"上下文泄洪" 机制。 核心是 3 个不变量:

  • 父子 history 隔离: 子 agent 独立 history, 父只看到总结。
  • 父子 TODO 隔离: 各自管自己的进度。
  • 子工具 ≤ 父: 最小权限, 禁止递归。
