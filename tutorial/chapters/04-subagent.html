第 04 章 · 委托与隔离

让 Agent 学会分身：SubAgent

SubAgent 让主 agent 可以把局部问题交给一个隔离上下文的小 agent。它看起来像多智能体，其实第一版只需要把“委托”做成一个受控工具。

你已经知道什么，还不知道什么

你已经有工具、TODO 和主 loop。现在要解决的问题不是“怎么让 agent 更聪明”，而是“怎么让它在不污染主上下文的情况下做局部探索”。

SubAgent 的关键直觉是：它本质上是一个工具。主 agent 调用 run_subagent，传入任务；harness 创建 child agent，给它独立 History 和受限工具集，最后把结果作为 tool result 返回父 agent。

本章怎么学

先从本章场景和朴素方案进入问题，不急着记函数名。

再用 loop 位置和架构图建立心智模型：新增模块到底改变了哪条数据流或状态流。

最后再看接口、源码地图和验证方法，把概念压缩成能交给 coding agent 的 Prompt Card。

本章场景

用户要求：“帮我 review 这次重构的风险。”主 agent 可能需要读取多处文件、比较设计文档、整理发现。如果这些探索细节全部进入主 History，主上下文会迅速膨胀。

更好的做法是让主 agent 委托一个 review 子智能体：child 独立搜集证据，最后只把结论和引用返回给父 agent。

先试一个朴素方案

朴素做法是在主 loop 里递归调用 createAgent()，并把父 agent 的所有依赖原样传进去。甚至允许 child 再调用 run_subagent。

这会很快失控：上下文递归、权限扩大、工具循环、父子 History 混在一起。SubAgent 必须从第一天就被看作“受控工具”，而不是随便复制一个完整 agent。 

async function runSubagent(task) {
  const child = createAgent(parentDependencies);
  return child.run(task);
}

朴素方案为什么不够

递归调用失控
如果 child 也能调用 run_subagent，很容易形成树状爆炸，成本、时间和上下文都不可控。
权限被无意扩大
父 agent 在 default 模式下可写文件，不代表 child 做分析时也应该拥有写权限。
父子上下文污染
child 的探索过程如果直接写入父 History，会让主对话变长，也会混淆谁观察了什么。

回到 Agent Loop

SubAgent 插在工具执行层：对父 agent 来说，run_subagent 只是一个普通工具调用；对 harness 来说，它会启动一段新的 agent loop。

这个 child loop 可以复用稳定 system prompt snapshot，但必须拥有独立 History、独立 maxRounds 和经过过滤的 registry。最终返回父 agent 的只是一段总结性 tool result。

从原理落到代码

到这里已经有了场景、失败原因和架构图。下面才进入实现层：只抓住接口形状、状态边界和源码入口。

读源码时先找入口函数，再找它读写的状态，最后看测试如何证明这些状态变化是真的发生了。

一次真实运行 walkthrough

  1. 父 agent 判断当前任务适合委托，返回 run_subagent tool call。
  2. ToolRegistry 找到 subagent 工具，权限层确认是否允许启动 child。
  3. SubAgent 工具创建 child History，不复用父 History 的可变内容。
  4. 组装 child registry，过滤掉 run_subagent 等可能导致递归的工具。
  5. child agent 在自己的 maxRounds 内完成读取、分析或总结。
  6. child 的最终回答被包装成父 agent 的 tool result。
  7. 父 agent 看到这个结果后，继续生成给用户的最终回复或下一步动作。

关键接口和伪码

SubAgent 的接口要非常克制。父 agent 只传任务文本和少量配置；child 不能拿到父内部所有状态的可变引用。 

interface SubagentRequest {
  task: string;
  maxRounds?: number;
  readonly?: boolean;
}

async function runSubagent(request: SubagentRequest): Promise<ToolResult> {
  const childHistory = createHistory();
  const childRegistry = createReadonlyRegistry();
  const child = createAgent({
    history: childHistory,
    tools: childRegistry,
    stablePrompt: parentStablePromptSnapshot,
    maxRounds: request.maxRounds ?? 6,
  });

  const answer = await child.run(request.task);
  return { output: answer };
}

设计直觉

SubAgent 的目标不是炫技，而是控制上下文污染。主 agent 需要结果，不一定需要 child 的每一步思考和观察。

把 SubAgent 做成工具，可以复用工具调用、权限、History 配对和测试框架，同时把递归与权限收窄放在同一个边界上处理。

架构拆分

Parent Agent
决定是否委托，并把 child 结果纳入自己的下一轮推理。
SubAgent Tool
负责创建 child agent、过滤工具、运行并包装结果。
Child Agent
拥有独立 History 和 maxRounds，只执行被委托的局部任务。

状态与边界

共享
稳定 prompt snapshot、只读项目上下文、必要的 logger 或 session 元信息。
隔离
History、TODO、临时工具观察、child 轮次预算。
收窄
工具 registry、权限 profile、可写能力和递归调用能力。

Prompt Card

Prompt Card

目标：实现 run_subagent 工具，让父 agent 可以委托一个隔离上下文的 child agent 完成局部分析。

场景：主 agent 遇到 review、搜索、解释等局部任务时，调用 child agent，最终只把总结结果写回父 History。

模块：实现 tools/subagent.ts，扩展 registry 过滤能力，补充 session parent/child 元信息和测试。

接线：在 composition root 注入 createReadonlyRegistryFn 和稳定 prompt snapshot，不在工具内部重新创建全套共享依赖。

边界：child 不得默认继承父 TODO，不得递归调用 subagent，不得扩大权限。

例外和 corner case：child 超过 maxRounds、child 工具失败、父工具被过滤、空任务输入。

验证：测试 child 能返回结果、过滤工具不可用、maxRounds 生效、父 History 只看到 tool_result 摘要。

源码入口：src/tools/subagent.ts、src/tools/registry.ts、src/session.ts、src/agent.ts。

文档同步：说明 SubAgent 是工具，不是生产级多智能体调度系统。

容易踩坑

Design Trap

常见错误：让 child agent 完整继承父 agent 的工具和权限。

为什么错：委托分析不应该自动获得写权限和递归能力，否则会放大风险。

正确做法：通过 registry filter 和权限 profile 显式收窄 child 能力。

怎么验证：child 尝试调用被过滤工具时，应得到可恢复错误而不是执行成功。

  • 把 child 探索消息全部塞进父 History，会让上下文迅速膨胀。
  • child 没有 maxRounds，会导致父工具调用长时间阻塞。
  • 在 subagent 工具内部重新创建 Memory/Skill/OutputStore 等共享依赖，会让状态不一致。

如何验证

Validation

手工 query：让 agent 委托一个子智能体解释文件，再观察父回复是否只包含总结而不是完整 child 日志。

确定性测试：fake child 返回固定结果，断言父 History 收到 tool_result。

集成观察：测试 registry filter 后 child 无法调用 run_subagent。

失败信号：child 能递归启动 child，或 child 写文件权限大于父期望。

  • src/tools/subagent.test.ts 覆盖 child 运行和工具过滤。
  • 检查 parentSessionId 是否能表达父子关系，但不把 child History 混入父 History。
  • 用 maxRounds 很小的 child 任务验证超限时返回明确错误。

如果实现失败，先查哪里

  • 先看 subagent 工具拿到的 registry 是否已经过滤。
  • 检查 child History 是否新建，而不是引用父 History。
  • 检查 stable prompt snapshot 是否复用稳定前缀，而不是运行中重新生成。
  • 检查 child 错误是否包装成 ToolResult 返回父 loop。

本章练习

写一个 prompt，让 coding agent 只实现只读 SubAgent：child 只能 run_read，不能 run_write，也不能再次 run_subagent。

本章小结

本章让 agent 学会委托，但委托仍然依赖当前已有知识。下一章我们要让 agent 按需加载领域说明，避免把所有技巧都塞进 system prompt。
