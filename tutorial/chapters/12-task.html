第 12 章 · 跨会话的长期任务

Task: 带依赖图的持久化任务组

第 03 章的 TODO 是"当前会话的短期执行清单", 但 LLM 跑长任务 (1 小时跑测试 / 半天跑全项目扫描 / 3 天跑大型迁移) 时, session 早就关掉了, TODO 一起丢。 这一章加 Task 系统: TaskGroup 持久化到 .tasks/groups/*.json, 跨 session 恢复, 带依赖图 (task_2 blockedBy task_1) + 状态机 (pending → in_progress → completed / failed / cancelled) + 派生 ready / blocked (不写入磁盘, 读取时算) + audit events (轻量 审计线索)。 读完后, 你能讲清"Task vs TODO" 的边界 (跨会话持久 化 vs 当前 session), 并能用 tmp 目录验证"环依赖拒绝" + "ready 派生正确" + "terminal status 后不可改" 3 条不变量。

真实失败故事: session 关闭, 跑了 2 小时的任务状态全丢

写代码之前, 先看一个真实痛点。 用户的 team 用 harness 跑大型 迁移: "把 500 个 JS 文件转成 TS, 每天跑 1 小时, 5 天完成"。 团队 leader 看着第 1 天的 TODO 列表很开心, 关电脑下班。 第二天上班开 session, TODO 列表是空的, LLM 不知道"昨天转了 哪些文件" 也 "今天该从哪个开始"。

  1. 症状: 关闭 session = TODO 全丢, LLM 重复跑 已完成的文件 (浪费 token) 或跳过依赖未完成的文件 (出现 编译错误)。
  2. 根因: TODO 是 session 内存状态, 不持久化。 session 关闭 = TODO 数组销毁。
  3. 误诊: 团队以为是 LLM "忘了", 写复杂 prompt 让 LLM "记住" — LLM 没法跨 session 持久化, 写了也丢。
  4. 真正的修法: 持久化 Task 系统 — 任务组写到 .tasks/groups/migration-001.json, 跨 session 恢复; 任务带依赖图, LLM 知道"task_5 依赖 task_3, task_3 没完成所以 task_5 不能开始"; 状态机保证"completed 不能 变回 in_progress"。

朴素想法 1: "把 TODO 写到磁盘不就行了?" 不够。 TODO 是 当前 session 的短期清单, 写磁盘变 长期 任务, 但 TODO 没依赖图 (只能单 链表), 没状态机 (状态字段是字符串, 任意改), 没audit event (谁改的不知道)。 Task 是新的 抽象, 不是"持久化的 TODO"。

朴素想法 2: "用数据库?" 太重。 教学项目要 group.json + git 友好的纯文本, 不用 SQLite/Postgres。 规模也小 (一个项目 几个 group, 几十个 task), 内存里读 + 偶尔写磁盘完全够。

正确做法: 加 src/tasks.ts + src/task-store.ts — TaskStore 管文件 IO, TaskManager 管业务规则 (状态机 + 依赖 校验 + activeTaskGroupId)。 这是 Reference 章节 "模式 7 · State Machine 状态机" + "模式 10 · Atomic Write 原子写" 的 联合应用。

Task vs TODO: 跨会话 vs 当前 session

场景: "5 天任务" 用 TODO 还是 Task

上一章讲 TODO 是当前 session 短期清单。 朴素想法 1: "5 天的迁移用 TODO 就行, 跑一天勾一天?" 错。 session 关闭 后 TODO 数组销毁, 第二天上班 TODO 列表是空的, LLM 不知道 "昨天到哪了"。 朴素想法 2: "5 天的迁移用 Task 就行, 不需要 TODO?" 错。 5 天里有 500 个 task, 一次 session 只跑 50 个, 剩下的 task 不该跟"今天先跑 task_1~50" 混在一起 — 跑完今天后, TODO 应该清空, 明天再起新的 TODO。

正解: 两个抽象各管一段

  • Task (跨 session 长期计划): "5 天迁移" 是一个 TaskGroup 含 500 个 task, 持久化到磁盘, 跨 session 保留, 带依赖图 (task_2 依赖 task_1)。
  • TODO (同 session 短期清单): "今天先跑 task_1~50, 然后看进度调整" 是一个 TODO 列表, session 内存, session 关闭就丢。

两者正交, 不能互相替代:

  • "重要任务被 TODO 化" → 丢状态 (跨 session 后 TODO 没了)
  • "TODO 当 Task 用" → 阻塞依赖检查 (TODO 没依赖图, 任务 ready 状态算不出来)

设计思想: 职责正交, 数据隔离 — Task 持久化在 .tasks/groups/, TODO 在 session 内存; Task 有依赖图, TODO 是单链表; Task 有状态机, TODO 是 简单 3 状态 (pending / in_progress / done)。 看 src/tasks.ts 第 122 行的教学导读注释: 

// TaskManager 是持久化 Task 的业务层。
// 它不关心 group.json 写在哪里, 也不关心 LLM tool 参数长什么样;
// 它只负责长期任务组的状态机:
// - 创建 group / task
// - 校验依赖图
// - 控制 task 状态转换
// - 自动派生 group 完成状态
//
// 这和 todo.ts 的 session TODO 不同:
// TODO 是当前会话的短期执行清单; Task 是跨会话、可恢复、带依赖图的长期计划。
图 1 · Task vs TODO 的边界
Task · 跨会话持久化
存在 .tasks/groups/<id>.json, 关闭 session 不丢。 带依赖图 (DAG) + 状态机 (6 态) + audit events。 适合跑 1 小时以上的任务。

存在 history[] 或 session state, 关闭 session 销毁。 单链表 + 3 态 (pending / in_progress / done)。 适合当前 turn 的执行清单。

实现细节: TaskStore 写 JSON 文件, TaskManager 不直接调 fs, 通过 store 接口。 这是 Reference 章节 "模式 1 · 工厂 + 闭包" + "模式 10 · Atomic Write 原子写" 的应用 — store 接口可以换 (内存 / 文件 / git), TaskManager 不变。

数据结构: 3 个持久化 entity + 2 个派生/临时状态

上面看到痛点"session 关闭, 跑了 2 小时的任务状态全丢", 解决方案是持久化。 这一节先看 Task 系统长什么样 (数据结构 + 关系), 后面各小节再深入每个 entity 的细节。

Task 系统有 3 个持久化 entity (TaskGroup / Task / TaskEvent) + 2 个派生/临时状态 (TaskGroupProgress / activeTaskGroupId), 关系如下:

                    │  createdAt, updatedAt, archivedAt       │

5 个 entity / 状态的各自角色:

TaskGroup (持久化的"项目")

是什么: 一个完整的项目 / 任务集合, 比如 "migration-001" (500 个 JS 转 TS) 或 "weekly-review" (每周例行 任务)。 1 个 TaskGroup 持久化到 1 个 .tasks/groups/<id>.json 文件。

关键字段:

  • id: 唯一标识, 团队全局唯一 (e.g. "migration-001")
  • title: 人看的标题 (e.g. "JS → TS 迁移")
  • scope: project (单项目) 或 multi_project (多项目)
  • status: 4 档 group 级别状态 — active (进行中) / completed (全部 task 完成) / cancelled (取消) / archived (完成后归档)
  • tasks: Task 数组 (详见下)
  • events: TaskEvent 数组 (审计流, 详见下)

什么时候创建: LLM 接到"跑 5 天迁移"这种大任务时 (createGroup({title, tasks})), 跨 session 保留。

Task (具体的"做一件事")

是什么: TaskGroup 里的一个具体动作, 比如 "改 src/a.js" 或 "跑 npm test"。 1 个 TaskGroup 含 N 个 Task (从几个到几千个不等)。

关键字段:

  • id: 唯一标识 (e.g. "task_1" / "task_2" / ...)
  • subject: 人/LLM 看的标题 (e.g. "改 src/a.js")
  • status: 6 档 task 状态 — pending / in_progress / completed / failed / cancelled / deleted
  • blockedBy: 依赖的 task id 列表 (e.g. ["task_3"] 表示"必须等 task_3 完成才能开始")
  • owner: 谁负责 (e.g. "main" / "subagent:explore")
  • note: 备注 (e.g. "auto-failed by schedule at 2026-06-12T09:05")
  • startedAt / completedAt: 可选, 实际跑过的开始/结束时间

派生字段 (不存磁盘):

  • ready: status === pending 且 blockedBy 都 completed
  • blocks: 谁依赖我 (反向索引)

什么时候创建: LLM 创建 TaskGroup 时一起创建 (e.g. 500 个 task 一次性生成), 或后续 addTask() 追加。

TaskEvent (审计流)

是什么: TaskGroup 里的追加日志, 每次业务修改 (创建 / 更新 / 删除 / 归档) 自动加 1 条。 1 个 TaskGroup 含 N 条 Event, N 随时间增长。

关键字段:

  • id: 唯一标识 (e.g. "event_1234567890_abc")
  • timestamp: ISO 时间
  • actor: 谁改的 (e.g. "main" / "subagent:explore" / "schedule")
  • type: 7 种 — group_created / task_added / task_updated / task_deleted / group_completed / group_cancelled / group_archived
  • taskId?: 可选, 关联的 task (e.g. "task_3")
  • message: 人类可读描述 (e.g. "Updated task_3: status in_progress → completed")

什么时候生成: 业务修改时自动追加, LLM 不能直接写 events (e.g. updateTask 内部会追加一条 task_updated 事件)。

TaskGroupProgress (派生统计)

是什么: 1 个 TaskGroup 当前进度的派生统计, 不存磁盘 (从 tasks[] 当前 status 算出来)。 LLM 调 formatTaskGroupView() 时会显示进度条, user 看 "migration-001 进度 50/500"。

关键字段:

  • total / completed / inProgress / ready / blocked / failed / cancelled / deleted: 8 个数字, 各种 status 的 task 数量

派生, 不写入: 跟 ready / blocks 一样, 读取 时从 tasks[] 算, 写入会和 source-of-truth 冲突。

activeTaskGroupId (session 指针)

是什么: "当前 session 正在关注哪个 group" 的指针, 1 个 session 最多 1 个 activeGroup, 跨 session 销毁。

不在磁盘: 跟前面 4 个不同, activeTaskGroupId 是 session 内存 状态, REPL 关闭即销毁, 跨 session 不保留。

怎么用: LLM 调 readGroup(id) 时自动 setActiveGroupId(id), reminder 注入 "active: migration-001" 让 LLM 知道"现在跑哪个 group"。 调别的 group 就自动切。

TaskGroup 结构: 1 group = 1 .json 文件

场景: 5 人同时改 "migration-001" group, git 冲突怎么办

team 5 人同时编辑 "migration-001" group (每人分到不同 task), 每人 commit 自己的修改, PR review 时 git diff 必须能看 "谁加了什么 task / 谁改了什么状态"。 朴素想法 1: "1 个 group = 1 个 JSON 文件" 看 起来对, 但 1 个 task 改 状态, 文件里的 events 数组也加 1 条, 即使只动了 1 行 status 也会改 N 行 events, 5 个人同时改 events 数组, 合并冲突爆炸。 朴素想法 2: "1 个 task 1 个文件, 200 个 task = 200 个文件?" 错。 改 1 个 task 看 1 个文件, 但 5 个人同时 commit 时, group.json 的元数据 (title / scope / events) 各自改各自的, 还是要合并。

正解: 1 个 group = 1 个 JSON 文件

1 个 TaskGroup 持久化到 1 个 .tasks/groups/<id>.json 文件, 包含 3 部分:

  • group 元数据 (title / scope / createdAt / updatedAt)
  • task 列表 (task_1 / task_2 / ... 数组)
  • events 列表 (审计流, 谁改了什么, 什么时候)

5 人同时改, 每人 commit 同一文件的不同区域 (一人 改 task 1, 一人改 task 50, 一人改 events 末尾), git diff 显示清晰, 合并冲突容易解决。 改 1 个 task 看整个文件 (100~200 行, 可读)。

反例: 1 个 task 1 个文件 看起来"职责单一", 实际让 group 元数据管理变难 (title 改 1 次要改 200 个文件)。

设计思想: 人类可读 + git 友好 — JSON 格式, 字段顺序稳定, 多行 (task 数组每行 1 个), git diff 显示"task_3 状态从 in_progress 变 completed" 而不是"第 47 行 改了一个字符不知道改哪"。 看 src/tasks.ts 第 204 行的 group 创建: 

const group: TaskGroupFile = {
  version: 1,
  kind: "task_group",
  id: groupId,
  scope: projectRoots.length > 1 ? "multi_project" : "project",
  projectRoots,
  title,
  status: "active",
  createdAt,
  updatedAt: createdAt,
  tasks,
  events: [
    { id: ..., type: "group_created", message: `Created task group: ${title}` },
  ],
};

实现细节: version: 1 字段给未来 schema 迁移留空间; kind: "task_group" 字段防止和别的 JSON 文件 (memory / skill) 混淆; scope 字段标记单项目 vs 多项目 group, 决定依赖图校验规则。

6 状态状态机: pending → in_progress → completed/failed/cancelled

场景: LLM 想"回滚" 一个 completed task, 该不该让

LLM 跑了 task_1 ~ task_5, 都标 completed。 突然发现 task_3 当时改得不对, 想"回到 pending 重做"。 朴素想法 1: "LLM 想改就改呗, status 字段是 string, 随便填?" 错。 LLM 可能 写出"pending → completed → pending → in_progress" 的死 循环, 审计 log 看不懂, 依赖图算错, 团队 leader 看 group 文件不知道"现在到底跑没跑"。

正解: 6 状态固定枚举 + 显式转换规则

  • 3 个非终态: pending / in_progress / (中间过渡态, 可继续转)
  • 3 个终态: completed / failed / cancelled (锁死, 不可转其他状态)

LLM 完成 task_3, 调 updateTask(groupId, "task_3", {status: "completed"})。 TaskManager 校验: 当前 in_progress → 目标 completed 合法, 更新 + 写 event + 检查依赖 (task_5 现在 ready 了)。 LLM 想 "completed → pending" (回滚), TaskManager 拒绝并 throw — 终态不可改。

想"重新做" 怎么办? 朴素做法: 删掉 task_3, 新建一个 task_6 替代。 group.json 历史完整, 不破坏状态机。

设计思想: 状态机 + 显式转换 — Reference 章节 "模式 7 · State Machine 状态机" 的应用。 看 src/tasks.ts 第 92 行的 TERMINAL_TASK_STATUSES: 

const TERMINAL_TASK_STATUSES: ReadonlySet<TaskStatus> = new Set([
  "completed", "cancelled", "deleted",
]);

const STATUS_SYMBOLS: Record<TaskStatus, string> = {
  pending: "[ ]",
  in_progress: "[>]",
  completed: "[x]",
  failed: "[!]",
  cancelled: "[_]",
  deleted: "[-]",
};
图 2 · 6 状态状态机 + 合法转换
pending · 未开始
[ ] 初始状态。 可转 in_progress / cancelled / deleted。 不可直接 completed。
in_progress · 进行中
[>] 正在做。 可转 completed / failed / cancelled / deleted。 不允许重复 in_progress。
completed · 已完成 (终态)
[x] 成功。 不可转其他状态。 Group 派生完成状态时检查所有非 deleted task 都 completed。
failed · 失败 (终态)
[!] 失败。 不可转其他状态。 Group 派生完成状态时若仍有 in_progress 任务, 仍是 active。
cancelled · 取消 (终态)
[_] 用户取消。 不可转其他状态。 走 archive_group 才能清理磁盘。
deleted · 删除 (终态)
[-] 软删除。 不可转其他状态。 仍被其他 task 依赖时拒绝 delete (悬空引用)。

实现细节: 6 状态分两类 — 3 个非终态 (pending / in_progress / ...), 3 个终态 (completed / cancelled / deleted)。 非终态可继续转换, 终态锁定。 状态转换在 applyStatusTransition 函数内显式定义, 不允许 task.status = "..." 直接赋值。

依赖图: blockedBy + 环检测 + 悬空引用

场景: task 之间有先后, 怎么表达

朴素想法 1: "不写依赖, LLM 自己安排先后?" 错。 LLM 跑 task_2 (改 src/b.js), 不知道 task_1 还没改 src/a.js, b.js import a.js 编译失败。 朴素想法 2: "用注释说 'task_2 必须在 task_1 后跑'?" 错。 注释人不读, LLM 更 不读, 谁也保证不了顺序。

正解: blockedBy 字段。 task 之间有依赖 ("task_5 必须等 task_3 完成才能开始"), 用 blockedBy: ["task_3"] 表达。 LLM 读 group.json 看字段, 不靠注释。

3 条校验规则 (group 写入磁盘前必跑):

  1. 引用存在: task_5.blockedBy 含 task_99, 但 group 没 task_99 → 拒绝 (悬空引用)。
  2. 无环: task_1.blockedBy=[task_2] + task_2.blockedBy=[task_1] → 拒绝 (死锁, 两个 永远等不到对方)。
  3. 不可删被依赖: task_3 仍被 task_5.blockedBy 引用, 删 task_3 → 拒绝 (task_5 找不到依赖, 派生状态算不出来)。
(依赖 task_1 因为 b.js import a.js), task_3 改 src/c.js (依赖 task_1)。

LLM 创建 group 后, 系统的反应:

  • group.json 含依赖图
  • LLM 跑 task_1 完成, 系统自动派生 task_2 / task_3 ready
  • LLM 跑 task_2 完成, 派生 task_3 (如果 task_1 也完成) ready

设计思想: DAG 校验 + 派生状态 — Reference 章节 "模式 7 · State Machine 状态机" 的延伸。 看 src/tasks.ts 第 229 行的 ensureDependencyGraphValid: 

// 创建时就检查依赖图, 避免把有环或引用不存在 task 的 group 写入持久化层。
ensureDependencyGraphValid(group);
图 3 · 依赖图校验的 3 条规则
规则 1 · 引用存在
task_5.blockedBy 含 "task_99", 但 group 没 task_99。 拒绝创建, 错误 "blockedBy references missing task: task_99"。
规则 2 · 无环
task_1.blockedBy=[task_2], task_2.blockedBy=[task_1]。 拒绝创建, 错误 "circular dependency detected: task_1 → task_2 → task_1"。
规则 3 · 不可删被依赖
task_3 仍被 task_5 依赖 (task_5.blockedBy 含 task_3), 拒绝 delete task_3, 错误 "Cannot delete task_3; it is required by task_5"。

实现细节: ensureDependencyGraphValid 在 createGroup / addTask / updateTask 之后都跑一次, 保证任何修改后依赖图 都合法。 3 条规则对应 Reference 章节 "模式 15 · Identity Check 身份校验" 的应用。

ready / blocks: 派生, 不写入磁盘

场景: "task 5 现在能开始吗"

LLM 想"task 5 现在能开始吗?" — 朴素想法 1: "在 task_5 对象里加 ready: true/false 字段, LLM 改 status 时一起更新?" 错。 LLM 改 task_5 status 但忘了更新 ready, 磁盘上 ready 字段过期, 显示 "task_5 ready=true" 但实际 task_3 还没完成, 跑 task_5 编译失败。 派生数据 + 持久化 字段, 双写必出不一致 bug。

正解: 派生不写入

  • ready (任务是否可开始): task.status === "pending" && task.blockedBy 中所有 task 都 completed, 读取时算。
  • blocks (反向依赖, 谁依赖我): 从 blockedBy 反向索引, 读取时算。

磁盘只存 source-of-truth (status / blockedBy / note), 派生状态 (ready / blocks / progress) 读时算, 不写入。 LLM 改 status 自动反映在 ready (下次读取时重算), 不会不一致。

+ blocks: ["task_5"]。 LLM 看到 task_3.ready=true 知道可以开始, 看到 task_3.blocks 含 task_5 知道"我完成会解锁 task_5"。 派生每次重算, 不存在"过期" 问题。

设计思想: 派生数据, 不冗余 — 磁盘只存 source-of-truth (status, blockedBy, ...), 派生 (ready, blocks, progress) 读取时算。 看 src/tasks.ts 第 432 行的 buildTaskGroupView: 

function buildTaskGroupView(group: TaskGroupFile): TaskGroupView {
  const completedIds = new Set(
    group.tasks.filter((task) => task.status === "completed").map((t) => t.id),
  );
  const blocksById = new Map<string, string[]>();
  for (const task of group.tasks) {
    for (const dep of task.blockedBy) {
      blocksByid.set(dep, [...(blocksById.get(dep) ?? []), task.id]);
    }
  }
  const tasks: TaskView[] = group.tasks.map((task) => {
    const missing = task.blockedBy.filter((id) => !completedIds.has(id));
    return {
      ...task,
      blockedBy: [...task.blockedBy],
      ready: task.status === "pending" && missing.length === 0,
      blocks: blocksById.get(task.id) ?? [],
    };
  });
  return { group, tasks };
}

实现细节: ready = (status === "pending") && (blockedBy 都 completed), 这条规则让 LLM 跑完 task_1 立刻看到 task_2 / task_3 变 ready。 blocks 反向索引从 blockedBy 反推, LLM 知道"完成我 会解锁谁"。

audit events: 轻量审计线索

场景: "task_3 是谁标 completed 的"

team leader 想知道"task_3 是谁标 completed 的, 什么时候?", 看 group.json 只能看到当前 status, 看不出历史。 朴素想法 1: "用 git log 看 history?" 错。 LLM 改 task 的 events 是 运行时追加, 不进 git commit (因为每次 LLM 操作都 commit 一下不现实)。 git log 看不到。

正解: events 数组。 每次业务修改 (创建 / 更新 / 删除 / 归档) 追加一条 event 到 group.events[], 含 id / timestamp / actor / type / message 5 个字段。 team leader 查 "task_3 谁改的 completed", 直接 grep events 数组, 找到

{actor: "main", type: "task_updated", message: "Updated task_3", timestamp: "2026-06-12T10:00"}。 actor 标记 "main" (主 agent) 或未来 "subagent:explore" / "user" 等, 帮助定位责任。

设计思想: 事件流, 不状态机 — events 是追加日志, 不参与 status 决策; status 仍以 task.status 字段为准。 这是 Reference 章节 "模式 10 · Atomic Write 原子写" 的延伸 — event 跟着 group 一起写, 不 单独 append (单独 append 会破坏原子性)。

实现细节: 看 src/tasks.ts 第 143 行的 createEvent — event id 用 event_${Date.now()}_${random} 生成, 教学版够用; 工业版应该用 ULID 之类保证排序稳定。

activeTaskGroupId: session 级轻状态, 不持久化

场景: "上次 session 我在跑哪个 group"

user 跑 "migration-001" 跑到一半 session 关闭, 下次开 session 不知道"我之前在跑什么"。 朴素想法 1: "持久化 activeTaskGroupId 到磁盘?" 错。 activeId 是 session 内 "现在关注哪个 group" 的指针, 持久化后下次开 session 自动 指向 migration-001, 但 user 可能想跑别的 group, 反而 干扰。 朴素想法 2: "每次手动 setActiveGroupId?" 麻烦, user 跑 5 天任务每个 session 都要记得设。

正解: session 级轻状态 + 自动 set

activeTaskGroupId 是session 级状态, 不写磁盘 (REPL 关闭就丢)。 LLM 调 readGroup(id) 后自动 set, 工具输出和 reminder 告诉模型"现在关注这个 group"。 不需要 user 手动设, 也不会跨 session 干扰。

具体看 4 种时序:

  • 同 session 内: user 跑 "migration-001", LLM 调 readGroup("migration-001") → activeId 自动设, reminder 注入 "active: migration-001"。
  • 同 session 内换 group: LLM 调 readGroup("other-group") → activeId 自动换, reminder 跟着变。
  • session 关闭: activeId 销毁。
  • 下次开 session: 新建空 activeId, user 主动 setActiveGroupId 或 LLM 调 readGroup 触发自动 set。
user 主动 setActiveGroupId("migration-001") 或 工具调 readGroup("migration-001") 自动 set, reminder 告诉 LLM "现在关注 migration-001, 进度 50/500"。

设计思想: 轻状态, 不持久化 — activeTaskGroupId 是 指针, 真实数据在磁盘 group.json; session 关闭指针丢, 数据不丢。 看 src/tasks.ts 第 137 行: 

// activeTaskGroupId 是 session 级状态, 不写入磁盘。
// 它只是帮助工具输出和 reminder 告诉模型"当前正在关注哪个长期任务组"。
let activeTaskGroupId: string | null = null;

实现细节: readGroup() / createGroup() 自动 set, setActiveGroupId(null) 显式清空; 设置 前先 store.read(groupId) 验证 group 存在, 避免 指针指向不存在的持久化对象。

formatTaskGroupView: 状态符号 + 进度条

场景: 怎么把 group 状态给 LLM / 用户看

LLM 调 readGroup(groupId) 拿到一个 TaskGroupView 对象, 但 LLM 不直接 parse 对象, 它 读字符串。 用户在 REPL 也想看进度, 不可能解析 50 个 task 的 JSON。 需要一个人类可读的格式化函数。

朴素想法 1: "JSON.stringify 整个 group?" 错。 JSON 太啰嗦, LLM 要从 50 行 JSON 里挑出 "task_3 当前啥状态", 难读。 朴素想法 2: "用 markdown 表格?" 也错。 markdown 表格对 LLM 不易 parse (要识别 | 边界, 列对齐), 改格式 LLM 容易搞错列。

正解: 结构化文本

每个 task 一行, 用状态符号 + 缩进表达层级:

  • [ ] pending
  • [>] in_progress
  • [x] completed
  • [!] failed
  • [_] cancelled
  • [-] deleted

末尾加进度条 (completed/total, in_progress, ready, blocked)。 整体像 git status 输出, LLM 容易 parse (按行读, 找 [ ] 找 [x] 找 progress:), 用户也 容易看 (状态符号一眼扫完)。

blockedBy task_1 (未完成), 进度 50/500"。 LLM 据此决定下一步 (完成 task_1 或等 task_1)。

设计思想: 结构化文本 — 用 状态符号 + 缩进表达层级, 不用表格 (markdown 表格 LLM 不易 parse)。 看 src/tasks.ts 第 400 行的 formatTaskGroupView: 

export function formatTaskGroupView(view: TaskGroupView): string {
  const { group, tasks } = view;
  const lines = [`[${group.status}] ${group.id}: ${group.title}`];
  for (const task of tasks) {
    const symbol = task.ready && task.status === "pending" ? "[?]" : STATUS_SYMBOLS[task.status];
    lines.push(`${symbol} ${task.id}: ${task.subject}`);
    lines.push(`    owner: ${task.owner}`);
    lines.push(`    blockedBy: ${task.blockedBy.length > 0 ? task.blockedBy.join(", ") : "-"}`);
    if (task.blockedReason) lines.push(`    blocked: ${task.blockedReason}`);
    if (task.note) lines.push(`    note: ${task.note}`);
  }
  const progress = computeTaskGroupProgress(group);
  lines.push(`progress: ${progress.completed}/${progress.total} completed, ...`);
  return lines.join("\n").trimEnd();
}

实现细节: [?] 符号表示"ready 但 pending" (派生状态), [ ] 表示"pending 但 仍 blocked" (派生状态), 让 LLM 一眼区分"现在能跑" vs "现在 跑不了"。

主循环集成: reminder 注入 active group

场景: LLM 怎么知道 "现在跑哪个 group"

5 天迁移, 每天开 session, LLM 跑新一轮 chat() 前, 应该 知道"我之前跑到 task 50 了, 接下来该跑 task 51"。 朴素 想法 1: "在 system prompt 写死 groupId?" 错。 system prompt 是 stable prefix, 改一次破坏 cache 命中 (第 10 章讲过), 而且一次只能跑一个 group, 切 group 要改 system prompt。 朴素想法 2: "在 LLM 调 readGroup 时返回 groupId?" 接近, 但不够 — LLM 跑 task 51 时不调 readGroup, 就不知道 "我属于 migration-001"。

正解: reminder 注入。 TaskManager 在 Composition Root 创建, 注入 agent 主循环; 每轮 LLM 调用前, reminder 注入到 user message 流末尾, LLM 看到就接着跑。

具体例子: 5 天迁移第 2 天的某一轮, LLM 看到什么

LLM 调 chat() 时, agent 主循环自动在 user message 末尾追加 一条 reminder, LLM 看到的 messages 数组长这样 (简化): 

[
  { role: "user",      content: "继续跑 migration" },     // 用户原始输入
  { role: "assistant", tool_calls: [{ id: "c5", function: { name: "task_update", arguments: "..." }}] },  // 上一轮
  { role: "tool",      tool_call_id: "c5", content: "OK" },
  // ↑ 上面 3 条是 history, 下面这一条是本轮新追加的 reminder
  { role: "user",
    content: "<system-reminder source=\"task\">\n\
Active group: migration-001\n\
Progress: 50/500 (10%)\n\
Next ready tasks: task_51 (改 src/api.js), task_52 (改 src/auth.js), task_53 (改 src/utils.js)\n\
</system-reminder>" },
]

reminder 的5 个字段:

  • source="task": reminder 来源 (区分 task / todo / async-run / schedule / memory 等不同来源, LLM 知道这条 reminder 来自哪里)
  • Active group: 当前 session 关注的 group id, LLM 知道"我现在在跑哪个项目"
  • Progress (50/500): 进度数字, LLM 知道"整体到 哪了"
  • Next ready tasks: 列出最多 3 个可立即跑的 task, 包含 task id + subject (e.g. "task_51 (改 src/api.js)"), LLM 知道"接下来该跑哪个"

LLM 看到这条 reminder, 不用主动查就知道: "我现在跑 migration-001, 整体 10%, 接下来该跑 task_51"。 直接调 task_update({task_id: "task_51", status: "in_progress"})。

reminder 在消息流的什么位置: 

messages: [
  user,           ← 用户原始 query (第 1 轮是 migration 启动 query,
                  后面轮是 "继续跑" / "今天先跑到 task 50")
  assistant,      ← 上一轮 LLM 调的工具
  tool,           ← 上一轮 tool result
  user,           ← ← 本轮的 reminder 插在这里
                  ←    (在所有 history 消息之后, 在本轮 LLM 调用之前)
]
                    ↓
             LLM 调 chat(messages, tools) 时看到这条 reminder

reminder 是本轮临时消息, 不进 history 持久化 (跟 Todo reminder 一样的处理 — 详见第 03 章)。 下一轮 reminder 重新生成, 反映最新 progress。

设计思想: 主动告知 vs 被动查询 — reminder 主动注入 (LLM 不需要问), 工具被动查询 (LLM 主动 调 readGroup 拿完整列表)。 两者配合, LLM 不会 丢上下文。

实现细节: reminder 在 agent.ts 第 3 步 (call LLM) 之前生成, 拼到 user message 末尾。 reminder 内容只含"active group + progress + next 3 ready task", 不含完整 task 列表 (避免 history 撑爆 — task 多了 reminder 会很长)。

常见误解: Task 不替代数据库

误解 1: "Task 替代数据库?" 不替代。 Task 是 跨会话任务计划, 不是通用数据存储。 想要 通用数据库, 用 SQLite / Postgres, Task 不该滥用。

误解 2: "ready 字段写入磁盘?" 错。 ready 是 派生状态, 读取时算。 写入会和 source-of-truth (status) 冲突。

误解 3: "completed 状态能回滚到 pending?" 错。 3 个终态 (completed / cancelled / deleted) 不可转。 "我改主意了" 也不行, 要回滚就新建 group。

误解 4: "events 数组能 1000+ 条?" 性能 OK 但 git diff 难读。 50 个 task 的 group 大约 100 个 event, 工业 规模建议定期 archive 旧 group, 不让 events 无限增长。

反例梯度: 4 个常见错误

反例 1 · 环依赖接受

LLM 创建 group, task_1.blockedBy=[task_2], task_2.blockedBy=[task_1], TaskManager 不校验就写磁盘。 错。 环依赖导致 ready 派生 死循环 (task_1 等 task_2, task_2 等 task_1, 永远没 ready)。 正确: ensureDependencyGraphValid 在 create / addTask / updateTask 后都跑一次, 检测到环 throw, 拒绝写磁盘。

反例 2 · 终态可改

TaskManager 不限制状态转换, LLM 想 completed → pending 直接改。 错。 状态机失去意义, audit trail 出现"completed → pending → in_progress → completed" 死循环, 没人能 解释。 正确: applyStatusTransition 检查当前 status 是不是终态, 终态拒绝任何转换, throw。

反例 3 · ready 写入磁盘

把 ready 字段写入 group.json, LLM 改 status 时同步改 ready。 错。 双写导致不一致 (改 status 忘改 ready, 或反过来)。 正确: 磁盘只存 source-of-truth (status, blockedBy), 派生 (ready, blocks, progress) 读取时算, buildTaskGroupView 永远从 status 重算。

反例 4 · activeTaskGroupId 写磁盘

把 activeTaskGroupId 存到 .tasks/active.json, 跨 session 复用。 错。 active 是 session 级指针, 不是数据; 写到磁盘会和"哪个 group 重要" 的逻辑耦合 (用户 关 session 1 个月后回来, active group 可能已 archive)。 正确: activeTaskGroupId 是闭包内状态, session 关闭丢; 用户新 session 主动 setActiveGroupId(...)。

Validation: 4 条不变量检验

  1. 依赖图无环无悬空: 创建 group / addTask / updateTask 后, 跑 ensureDependencyGraphValid, 含环或悬空 引用 throw。 验证: 3 个 case (环 / 引用不存在 / 引用已 deleted), 全部 throw。
  2. ready 派生正确: task_X 状态从 pending 变 completed, 所有 blockedBy 含 task_X 的 task, 派生 ready 变 true (假设其他 blocker 也 completed)。 验证: 创建 3 个 task (1 独立, 2 依赖 1, 3 依赖 2), 完成 1 后 task_2 ready, 完成 2 后 task_3 ready。
  3. 终态锁定: 3 个终态 (completed / cancelled / deleted) 任何 update 都 throw。 验证: 3 个 case, update status / owner / note / blockedBy 全部 throw。
  4. 删除仍被依赖的任务拒绝: task_X 仍被 task_Y.blockedBy 引用, delete task_X throw。 验证: 创建 2 个 task (2 依赖 1), 试图 delete task_1, 错误含 "required by task_2"。

回望: 哪些原则在本章兑现了

  • 状态机 + 显式转换: 6 状态固定枚举, 转换规则 applyStatusTransition 显式定义, 终态锁定。
  • DAG 校验: 引用存在 + 无环 + 不可删被依赖, 3 条 规则在 create / addTask / updateTask 后都跑。
  • 派生不写入: ready / blocks / progress 读取时 算, 磁盘只存 source-of-truth。
  • 事件流: events 是追加日志, 不驱动状态机, 给 "谁改了什么" 留审计线索。
  • 职责分离: TaskStore 管文件 IO, TaskManager 管 业务规则, store 可换 (内存 / 文件 / git) 不影响 manager。

  • events 是追加日志, 不驱动状态机, id 用 event_${ts}_${random} 生成
  • 删除被依赖 task throw, 错误信息含 "required by" + 任务列表

验证 (用 tmp 目录 + FileTaskStore + vitest, 逐条断言):

  • 环依赖 create throw, 错误信息含 "circular dependency"
  • 引用不存在的 task throw, 错误信息含 "references missing task"
  • task_1 pending → in_progress → completed, 派生 task_2.ready 变 true
  • completed 后 updateTask({status: "in_progress"}) throw, 错误信息含 "terminal status"
  • delete task_1 但 task_2 仍依赖, throw, 错误信息含 "required by task_2"

本章练习

  1. 故意不写 ensureDependencyGraphValid, 创建含环 依赖的 group, 跑测试, 看"环依赖拒绝" 是否抓到 (group 写到 磁盘 vs. throw)。
  2. 故意不在 applyStatusTransition 检查终态, 跑 completed → pending 测试, 看"终态锁定" 是否抓到 (改成功 vs. throw)。
  3. 把 ready 字段加到 TaskItem 接口, 写磁盘, 跑 ready 派生测试, 看"派生不写入" 是否抓到 (磁盘 ready 过期 vs. 派生重算一致)。
  4. 把 activeTaskGroupId 存到 .tasks/active.json, 重启 session 看 active 是否恢复, 看"active 不持久化" 是否 抓到 (新 session 应该没有 active, 让用户主动 set)。

本章小结

Task 是给跨会话长期任务的持久化系统, 区别 于 TODO 的 session 内存清单。 核心是 5 个设计:

  • 6 状态状态机: pending / in_progress / 3 个终态, 显式转换, 终态锁定。
  • DAG 依赖图: 引用存在 + 无环 + 不可删被依赖, ensureDependencyGraphValid 3 处必跑。
  • 派生不写入: ready / blocks / progress 读取时算, 磁盘只存 source-of-truth, 双写会冲突。
  • audit events: 追加日志, 不驱动状态机, 给 "谁改 了什么" 留线索。
  • 职责分离: TaskStore 管文件, TaskManager 管业务, store 可换, manager 不变。

    • 下一章 (第 13 章) 展开 Task 的并发执行 — LLM 跑长任务时怎么"同时跑 3 个独立 task, 不阻塞主对话", 用 Async Run Manager + 并发限制 3 + 输出隔离。
