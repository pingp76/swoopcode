第 12 章 · 跨会话的长期任务

Task: 带依赖图的持久化任务组

第 03 章的 TODO 是"当前会话的短期执行清单", 但 LLM 跑长任务 (1 小时跑测试 / 半天跑全项目扫描 / 3 天跑大型迁移) 时, session 早就关掉了, TODO 一起丢。 这一章加 Task 系统: TaskGroup 持久化到 .tasks/groups/*.json, 跨 session 恢复, 带依赖图 (task_2 blockedBy task_1) + 状态机 (pending → in_progress → completed / failed / cancelled) + 派生 ready / blocked (不写入磁盘, 读取时算) + audit events (轻量 审计线索)。 读完后, 你能讲清"Task vs TODO" 的边界 (跨会话持久 化 vs 当前 session), 并能用 tmp 目录验证"环依赖拒绝" + "ready 派生正确" + "terminal status 后不可改" 3 条不变量。

真实失败故事: session 关闭, 跑了 2 小时的任务状态全丢

写代码之前, 先看一个真实痛点。 用户的 team 用 harness 跑大型 迁移: "把 500 个 JS 文件转成 TS, 每天跑 1 小时, 5 天完成"。 团队 leader 看着第 1 天的 TODO 列表很开心, 关电脑下班。 第二天上班开 session, TODO 列表是空的, LLM 不知道"昨天转了 哪些文件" 也 "今天该从哪个开始"。

  1. 症状: 关闭 session = TODO 全丢, LLM 重复跑 已完成的文件 (浪费 token) 或跳过依赖未完成的文件 (出现 编译错误)。
  2. 根因: TODO 是 session 内存状态, 不持久化。 session 关闭 = TODO 数组销毁。
  3. 误诊: 团队以为是 LLM "忘了", 写复杂 prompt 让 LLM "记住" — LLM 没法跨 session 持久化, 写了也丢。
  4. 真正的修法: 持久化 Task 系统 — 任务组写到 .tasks/groups/migration-001.json, 跨 session 恢复; 任务带依赖图, LLM 知道"task_5 依赖 task_3, task_3 没完成所以 task_5 不能开始"; 状态机保证"completed 不能 变回 in_progress"。

朴素想法 1: "把 TODO 写到磁盘不就行了?" 不够。 TODO 是 当前 session 的短期清单, 写磁盘变 长期 任务, 但 TODO 没依赖图 (只能单 链表), 没状态机 (状态字段是字符串, 任意改), 没audit event (谁改的不知道)。 Task 是新的 抽象, 不是"持久化的 TODO"。

朴素想法 2: "用数据库?" 太重。 教学项目要 group.json + git 友好的纯文本, 不用 SQLite/Postgres。 规模也小 (一个项目 几个 group, 几十个 task), 内存里读 + 偶尔写磁盘完全够。

正确做法: 加 src/tasks.ts + src/task-store.ts — TaskStore 管文件 IO, TaskManager 管业务规则 (状态机 + 依赖 校验 + activeTaskGroupId)。 这是 Reference 章节 "模式 7 · State Machine 状态机" + "模式 10 · Atomic Write 原子写" 的 联合应用。

Task vs TODO: 跨会话 vs 当前 session

用途: 两个抽象职责正交, 不能互相替代。 任何 "应该持久化" 的事用 Task, "只是当前 session 短期跟踪" 用 TODO。 混淆会导致"重要任务被 TODO 化" 丢状态, 或"TODO 当 Task 用" 阻塞依赖检查。

真实场景: 用户让 LLM "跑 5 天的迁移", LLM 创建 TaskGroup "migration-001" 含 500 个 task, 每个 task_1 / task_2 / ... 标记依赖关系; 同一 session 内, LLM 同时维护一个 TODO 列表 "今天先跑 task_1 ~ task_50, 然后 看进度调整", TODO 是"今天的执行清单", Task 是"整个 5 天的 计划"。

设计思想: 职责正交, 数据隔离 — Task 持久化在 .tasks/groups/, TODO 在 session 内存; Task 有依赖图, TODO 是单链表; Task 有状态机, TODO 是 简单 3 状态 (pending / in_progress / done)。 看 src/tasks.ts 第 122 行的教学导读注释: 

// TaskManager 是持久化 Task 的业务层。
// 它不关心 group.json 写在哪里, 也不关心 LLM tool 参数长什么样;
// 它只负责长期任务组的状态机:
// - 创建 group / task
// - 校验依赖图
// - 控制 task 状态转换
// - 自动派生 group 完成状态
//
// 这和 todo.ts 的 session TODO 不同:
// TODO 是当前会话的短期执行清单; Task 是跨会话、可恢复、带依赖图的长期计划。
图 1 · Task vs TODO 的边界
Task · 跨会话持久化
存在 .tasks/groups/<id>.json, 关闭 session 不丢。 带依赖图 (DAG) + 状态机 (6 态) + audit events。 适合跑 1 小时以上的任务。
TODO · session 内存
存在 history[] 或 session state, 关闭 session 销毁。 单链表 + 3 态 (pending / in_progress / done)。 适合当前 turn 的执行清单。

实现细节: TaskStore 写 JSON 文件, TaskManager 不直接调 fs, 通过 store 接口。 这是 Reference 章节 "模式 1 · 工厂 + 闭包" + "模式 10 · Atomic Write 原子写" 的应用 — store 接口可以换 (内存 / 文件 / git), TaskManager 不变。

TaskGroup 结构: 1 group = 1 .json 文件

用途: 1 个 TaskGroup 持久化到 1 个 .tasks/groups/<id>.json 文件, 方便 git diff / review / 合并冲突解决。 1 个 group 包含 1 个 task 列表 + 1 个 events 列表 + 1 个 group 元数据。

真实场景: team 5 人同时编辑 "migration-001" group (分配不同 task), 每人 commit 自己的 group.json 修改, PR review 看 git diff 看清楚"谁加了什么 task / 谁改了什么状态"。 如果 1 个 group = 多个文件, 改 1 个 task 要改多个文件, 合并 冲突爆炸。

设计思想: 人类可读 + git 友好 — JSON 格式, 字段顺序稳定, 多行 (task 数组每行 1 个), git diff 显示"task_3 状态从 in_progress 变 completed" 而不是"第 47 行 改了一个字符不知道改哪"。 看 src/tasks.ts 第 204 行的 group 创建: 

const group: TaskGroupFile = {
  version: 1,
  kind: "task_group",
  id: groupId,
  scope: projectRoots.length > 1 ? "multi_project" : "project",
  projectRoots,
  title,
  status: "active",
  createdAt,
  updatedAt: createdAt,
  tasks,
  events: [
    { id: ..., type: "group_created", message: `Created task group: ${title}` },
  ],
};

实现细节: version: 1 字段给未来 schema 迁移留空间; kind: "task_group" 字段防止和别的 JSON 文件 (memory / skill) 混淆; scope 字段标记单项目 vs 多项目 group, 决定依赖图校验规则。

6 状态状态机: pending → in_progress → completed/failed/cancelled

用途: 任务状态变化必须受控, 任意字符串改会破坏状态机 (LLM 可能写出"pending → completed → pending" 的死循环, 没人能解释)。 6 状态是固定枚举, 转换规则显式定义。

真实场景: LLM 完成 task_3, 调 updateTask(groupId, "task_3", {status: "completed"})。 TaskManager 校验当前 status (in_progress) → 目标 status (completed) 合法, 更新 + 写 event + 检查依赖 (task_5 现在 ready 了)。 如果 LLM 想"completed → pending" (回滚), TaskManager 拒绝并 throw。

设计思想: 状态机 + 显式转换 — Reference 章节 "模式 7 · State Machine 状态机" 的应用。 看 src/tasks.ts 第 92 行的 TERMINAL_TASK_STATUSES: 

const TERMINAL_TASK_STATUSES: ReadonlySet<TaskStatus> = new Set([
  "completed", "cancelled", "deleted",
]);

const STATUS_SYMBOLS: Record<TaskStatus, string> = {
  pending: "[ ]",
  in_progress: "[>]",
  completed: "[x]",
  failed: "[!]",
  cancelled: "[_]",
  deleted: "[-]",
};
图 2 · 6 状态状态机 + 合法转换
pending · 未开始
[ ] 初始状态。 可转 in_progress / cancelled / deleted。 不可直接 completed。
in_progress · 进行中
[>] 正在做。 可转 completed / failed / cancelled / deleted。 不允许重复 in_progress。
completed · 已完成 (终态)
[x] 成功。 不可转其他状态。 Group 派生完成状态时检查所有非 deleted task 都 completed。
failed · 失败 (终态)
[!] 失败。 不可转其他状态。 Group 派生完成状态时若仍有 in_progress 任务, 仍是 active。
cancelled · 取消 (终态)
[_] 用户取消。 不可转其他状态。 走 archive_group 才能清理磁盘。
deleted · 删除 (终态)
[-] 软删除。 不可转其他状态。 仍被其他 task 依赖时拒绝 delete (悬空引用)。

实现细节: 6 状态分两类 — 3 个非终态 (pending / in_progress / ...), 3 个终态 (completed / cancelled / deleted)。 非终态可继续转换, 终态锁定。 状态转换在 applyStatusTransition 函数内显式定义, 不允许 task.status = "..." 直接赋值。

依赖图: blockedBy + 环检测 + 悬空引用

用途: task 之间有依赖 ("task_5 必须等 task_3 完成才能开始"), 用 blockedBy: ["task_3"] 表达。 依赖图必须无环 + 无悬空 (引用 存在的 task), 否则 ready / blocked 派生状态无法解释。

真实场景: 用户迁移 500 个 JS 文件, task_1 改 src/a.js, task_2 改 src/b.js (依赖 task_1 因为 b.js import a.js), task_3 改 src/c.js (依赖 task_1)。

LLM 创建 group 后, 系统的反应:

  • group.json 含依赖图
  • LLM 跑 task_1 完成, 系统自动派生 task_2 / task_3 ready
  • LLM 跑 task_2 完成, 派生 task_3 (如果 task_1 也完成) ready

设计思想: DAG 校验 + 派生状态 — Reference 章节 "模式 7 · State Machine 状态机" 的延伸。 看 src/tasks.ts 第 229 行的 ensureDependencyGraphValid: 

// 创建时就检查依赖图, 避免把有环或引用不存在 task 的 group 写入持久化层。
ensureDependencyGraphValid(group);
图 3 · 依赖图校验的 3 条规则
规则 1 · 引用存在
task_5.blockedBy 含 "task_99", 但 group 没 task_99。 拒绝创建, 错误 "blockedBy references missing task: task_99"。
规则 2 · 无环
task_1.blockedBy=[task_2], task_2.blockedBy=[task_1]。 拒绝创建, 错误 "circular dependency detected: task_1 → task_2 → task_1"。
规则 3 · 不可删被依赖
task_3 仍被 task_5 依赖 (task_5.blockedBy 含 task_3), 拒绝 delete task_3, 错误 "Cannot delete task_3; it is required by task_5"。

实现细节: ensureDependencyGraphValid 在 createGroup / addTask / updateTask 之后都跑一次, 保证任何修改后依赖图 都合法。 3 条规则对应 Reference 章节 "模式 15 · Identity Check 身份校验" 的应用。

ready / blocks: 派生, 不写入磁盘

用途: ready (任务是否可开始) 和 blocks (反向依赖, 谁依赖我) 是派生状态, 读取时算, 不写入磁盘。 写入磁盘会和 source-of-truth 冲突 (LLM 改 status 没及时 rebuild 派生)。

真实场景: LLM 调 readGroup(groupId) 拿到 TaskGroupView, 每个 task 有 ready: true/false + blocks: ["task_5"]。 LLM 看到 task_3.ready=true 知道可以开始, 看到 task_3.blocks 含 task_5 知道"我完成会解锁 task_5"。 派生每次重算, 不存在"过期" 问题。

设计思想: 派生数据, 不冗余 — 磁盘只存 source-of-truth (status, blockedBy, ...), 派生 (ready, blocks, progress) 读取时算。 看 src/tasks.ts 第 432 行的 buildTaskGroupView: 

function buildTaskGroupView(group: TaskGroupFile): TaskGroupView {
  const completedIds = new Set(
    group.tasks.filter((task) => task.status === "completed").map((t) => t.id),
  );
  const blocksById = new Map<string, string[]>();
  for (const task of group.tasks) {
    for (const dep of task.blockedBy) {
      blocksByid.set(dep, [...(blocksById.get(dep) ?? []), task.id]);
    }
  }
  const tasks: TaskView[] = group.tasks.map((task) => {
    const missing = task.blockedBy.filter((id) => !completedIds.has(id));
    return {
      ...task,
      blockedBy: [...task.blockedBy],
      ready: task.status === "pending" && missing.length === 0,
      blocks: blocksById.get(task.id) ?? [],
    };
  });
  return { group, tasks };
}

实现细节: ready = (status === "pending") && (blockedBy 都 completed), 这条规则让 LLM 跑完 task_1 立刻看到 task_2 / task_3 变 ready。 blocks 反向索引从 blockedBy 反推, LLM 知道"完成我 会解锁谁"。

audit events: 轻量审计线索

用途: 每次业务修改 (创建 / 更新 / 删除 / 归档) 追加一条 event 到 group.events[], 含 id / timestamp / actor / type / message。 审计线索轻量, 不驱动状态机, 只给"谁改了什么" 留记录。

真实场景: team leader 想知道"task_3 谁改的 completed", 看 group.events 找到 {actor: "main", type: "task_updated", message: "Updated task_3", timestamp: "2026-06-12T10:00"}。 actor 标记 "main" (主 agent) 或未来 "subagent:explore" / "user" 等, 帮助定位责任。

设计思想: 事件流, 不状态机 — events 是追加日志, 不参与 status 决策; status 仍以 task.status 字段为准。 这是 Reference 章节 "模式 10 · Atomic Write 原子写" 的延伸 — event 跟着 group 一起写, 不 单独 append (单独 append 会破坏原子性)。

实现细节: 看 src/tasks.ts 第 143 行的 createEvent — event id 用 event_${Date.now()}_${random} 生成, 教学版够用; 工业版应该用 ULID 之类保证排序稳定。

activeTaskGroupId: session 级轻状态, 不持久化

用途: "当前正在关注哪个 group" 是 session 级 状态, 不写磁盘。 activeTaskGroupId 是 LLM 调 readGroup(id) 后自动 set, 工具输出和 reminder 告诉模型"现在关注这个 group"。

真实场景: user 跑 "migration-001" 跑到一半 session 关闭, 下次开 session 不知道"我之前在跑什么"。 应该 user 主动 setActiveGroupId("migration-001") 或 工具调 readGroup("migration-001") 自动 set, reminder 告诉 LLM "现在关注 migration-001, 进度 50/500"。

设计思想: 轻状态, 不持久化 — activeTaskGroupId 是 指针, 真实数据在磁盘 group.json; session 关闭指针丢, 数据不丢。 看 src/tasks.ts 第 137 行: 

// activeTaskGroupId 是 session 级状态, 不写入磁盘。
// 它只是帮助工具输出和 reminder 告诉模型"当前正在关注哪个长期任务组"。
let activeTaskGroupId: string | null = null;

实现细节: readGroup() / createGroup() 自动 set, setActiveGroupId(null) 显式清空; 设置 前先 store.read(groupId) 验证 group 存在, 避免 指针指向不存在的持久化对象。

formatTaskGroupView: 状态符号 + 进度条

用途: 给 LLM / 用户读的人类可读格式 — 每个 task 一行, 用状态符号 ([ ] / [>] / [x] / [!] / [_] / [-]) + owner + blockedBy + note, 末尾进度条 (completed/total, in_progress, ready, blocked)。

真实场景: LLM 调 formatTaskGroupView(view) 拿到一段字符串塞回 conversation, 让 LLM 看到"task_3 [>] 正在做, blockedBy task_1 (未完成), 进度 50/500"。 LLM 据此决定下一步 (完成 task_1 或等 task_1)。

设计思想: 结构化文本 — 用 状态符号 + 缩进表达层级, 不用表格 (markdown 表格 LLM 不易 parse)。 看 src/tasks.ts 第 400 行的 formatTaskGroupView: 

export function formatTaskGroupView(view: TaskGroupView): string {
  const { group, tasks } = view;
  const lines = [`[${group.status}] ${group.id}: ${group.title}`];
  for (const task of tasks) {
    const symbol = task.ready && task.status === "pending" ? "[?]" : STATUS_SYMBOLS[task.status];
    lines.push(`${symbol} ${task.id}: ${task.subject}`);
    lines.push(`    owner: ${task.owner}`);
    lines.push(`    blockedBy: ${task.blockedBy.length > 0 ? task.blockedBy.join(", ") : "-"}`);
    if (task.blockedReason) lines.push(`    blocked: ${task.blockedReason}`);
    if (task.note) lines.push(`    note: ${task.note}`);
  }
  const progress = computeTaskGroupProgress(group);
  lines.push(`progress: ${progress.completed}/${progress.total} completed, ...`);
  return lines.join("\n").trimEnd();
}

实现细节: [?] 符号表示"ready 但 pending" (派生状态), [ ] 表示"pending 但 仍 blocked" (派生状态), 让 LLM 一眼区分"现在能跑" vs "现在 跑不了"。

主循环集成: reminder 注入 active group

用途: TaskManager 在 Composition Root 创建, 注入 agent 主循环; 每轮 LLM 调用前, reminder 注入 "[active group: migration-001, progress 50/500, next ready: task_51, task_52]"。 让 LLM 知道"我之前在跑什么, 接下来 该跑什么"。

真实场景: 5 天迁移, 每天开 session 看到 reminder "active: migration-001, progress 100/500, next ready: task_101, task_102"。 LLM 直接接着跑, 不用 问 "我之前到哪了"。

设计思想: 主动告知 vs 被动查询 — reminder 主动注入 (LLM 不需要问), 工具被动查询 (LLM 主动 调 readGroup 拿完整列表)。 两者配合, LLM 不会 丢上下文。

实现细节: reminder 在 agent.ts 第 3 步前生成, 类似 TODO reminder (第 03 章), 拼到 user message 末尾。 reminder 内容只含"active group + progress + next 3 ready task", 不含完整 task 列表 (避免 history 撑爆)。

常见误解: Task 不替代数据库

误解 1: "Task 替代数据库?" 不替代。 Task 是 跨会话任务计划, 不是通用数据存储。 想要 通用数据库, 用 SQLite / Postgres, Task 不该滥用。

误解 2: "ready 字段写入磁盘?" 错。 ready 是 派生状态, 读取时算。 写入会和 source-of-truth (status) 冲突。

误解 3: "completed 状态能回滚到 pending?" 错。 3 个终态 (completed / cancelled / deleted) 不可转。 "我改主意了" 也不行, 要回滚就新建 group。

误解 4: "events 数组能 1000+ 条?" 性能 OK 但 git diff 难读。 50 个 task 的 group 大约 100 个 event, 工业 规模建议定期 archive 旧 group, 不让 events 无限增长。

反例梯度: 4 个常见错误

反例 1 · 环依赖接受

LLM 创建 group, task_1.blockedBy=[task_2], task_2.blockedBy=[task_1], TaskManager 不校验就写磁盘。 错。 环依赖导致 ready 派生 死循环 (task_1 等 task_2, task_2 等 task_1, 永远没 ready)。 正确: ensureDependencyGraphValid 在 create / addTask / updateTask 后都跑一次, 检测到环 throw, 拒绝写磁盘。

反例 2 · 终态可改

TaskManager 不限制状态转换, LLM 想 completed → pending 直接改。 错。 状态机失去意义, audit trail 出现"completed → pending → in_progress → completed" 死循环, 没人能 解释。 正确: applyStatusTransition 检查当前 status 是不是终态, 终态拒绝任何转换, throw。

反例 3 · ready 写入磁盘

把 ready 字段写入 group.json, LLM 改 status 时同步改 ready。 错。 双写导致不一致 (改 status 忘改 ready, 或反过来)。 正确: 磁盘只存 source-of-truth (status, blockedBy), 派生 (ready, blocks, progress) 读取时算, buildTaskGroupView 永远从 status 重算。

反例 4 · activeTaskGroupId 写磁盘

把 activeTaskGroupId 存到 .tasks/active.json, 跨 session 复用。 错。 active 是 session 级指针, 不是数据; 写到磁盘会和"哪个 group 重要" 的逻辑耦合 (用户 关 session 1 个月后回来, active group 可能已 archive)。 正确: activeTaskGroupId 是闭包内状态, session 关闭丢; 用户新 session 主动 setActiveGroupId(...)。

Validation: 4 条不变量检验

  1. 依赖图无环无悬空: 创建 group / addTask / updateTask 后, 跑 ensureDependencyGraphValid, 含环或悬空 引用 throw。 验证: 3 个 case (环 / 引用不存在 / 引用已 deleted), 全部 throw。
  2. ready 派生正确: task_X 状态从 pending 变 completed, 所有 blockedBy 含 task_X 的 task, 派生 ready 变 true (假设其他 blocker 也 completed)。 验证: 创建 3 个 task (1 独立, 2 依赖 1, 3 依赖 2), 完成 1 后 task_2 ready, 完成 2 后 task_3 ready。
  3. 终态锁定: 3 个终态 (completed / cancelled / deleted) 任何 update 都 throw。 验证: 3 个 case, update status / owner / note / blockedBy 全部 throw。
  4. 删除仍被依赖的任务拒绝: task_X 仍被 task_Y.blockedBy 引用, delete task_X throw。 验证: 创建 2 个 task (2 依赖 1), 试图 delete task_1, 错误含 "required by task_2"。

回望: 哪些原则在本章兑现了

  • 状态机 + 显式转换: 6 状态固定枚举, 转换规则 applyStatusTransition 显式定义, 终态锁定。
  • DAG 校验: 引用存在 + 无环 + 不可删被依赖, 3 条 规则在 create / addTask / updateTask 后都跑。
  • 派生不写入: ready / blocks / progress 读取时 算, 磁盘只存 source-of-truth。
  • 事件流: events 是追加日志, 不驱动状态机, 给 "谁改了什么" 留审计线索。
  • 职责分离: TaskStore 管文件 IO, TaskManager 管 业务规则, store 可换 (内存 / 文件 / git) 不影响 manager。

Prompt Card · 第 12 章

目标: 给 harness 加持久化 Task 系统, TaskGroup 写到 .tasks/groups/<id>.json, 跨 session 恢复, 带依赖 图 + 状态机 + 派生 ready/blocks + audit events。

场景: 5 天迁移 500 个 JS → TS, LLM 创建 migration-001 group 含 500 个 task, task_5 blockedBy [task_1, task_3], LLM 跑 task_1 完成, 系统派生 task_2 ready, LLM 跑 task_3 完成, task_5 变 ready, 关 session 不丢状态。

边界 (LLM 必须遵守的 checklist):

  • 6 状态固定枚举 pending / in_progress / completed / failed / cancelled / deleted, 不可新增
  • 3 个终态 (completed / cancelled / deleted) 不可转其他状态, 任何 update throw
  • 依赖图无环 + 无悬空, create / addTask / updateTask 后 ensureDependencyGraphValid 必跑
  • ready / blocks / progress 派生状态不写入磁盘, 读取时 buildTaskGroupView 算
  • activeTaskGroupId 是 session 级闭包内状态, 不写磁盘, 不持久化
  • events 是追加日志, 不驱动状态机, id 用 event_${ts}_${random} 生成
  • 删除被依赖 task throw, 错误信息含 "required by" + 任务列表

验证 (用 tmp 目录 + FileTaskStore + vitest, 逐条断言):

  • 环依赖 create throw, 错误信息含 "circular dependency"
  • 引用不存在的 task throw, 错误信息含 "references missing task"
  • task_1 pending → in_progress → completed, 派生 task_2.ready 变 true
  • completed 后 updateTask({status: "in_progress"}) throw, 错误信息含 "terminal status"
  • delete task_1 但 task_2 仍依赖, throw, 错误信息含 "required by task_2"

  2. 故意不在 applyStatusTransition 检查终态, 跑 completed → pending 测试, 看"终态锁定" 是否抓到 (改成功 vs. throw)。

本章小结

Task 是给跨会话长期任务的持久化系统, 区别 于 TODO 的 session 内存清单。 核心是 5 个设计:

  • 6 状态状态机: pending / in_progress / 3 个终态, 显式转换, 终态锁定。
  • DAG 依赖图: 引用存在 + 无环 + 不可删被依赖, ensureDependencyGraphValid 3 处必跑。
  • 派生不写入: ready / blocks / progress 读取时算, 磁盘只存 source-of-truth, 双写会冲突。
  • audit events: 追加日志, 不驱动状态机, 给 "谁改 了什么" 留线索。
  • 职责分离: TaskStore 管文件, TaskManager 管业务, store 可换, manager 不变。

    • 下一章 (第 13 章) 展开 Task 的并发执行 — LLM 跑长任务时怎么"同时跑 3 个独立 task, 不阻塞主对话", 用 Async Run Manager + 并发限制 3 + 输出隔离。
