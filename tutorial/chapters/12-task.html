第 12 章 · 把长期计划放到磁盘上

长期计划落盘：Persistent Task

TODO 是当前 session 的节奏，Task Group 是跨会话、跨天、跨项目仍然存在的长期计划。本章把任务系统从内存推进到持久化存储。

你已经知道什么，还不知道什么

你已经有 session-local TODO 和 Memory。Persistent Task 既不是 TODO，也不是 Memory：它记录项目中的长期工作计划，而不是用户偏好或本轮步骤。

读这一章时请把注意力放在 agent loop 的接入点：工具执行层和持久化层之间，把长期计划保存为可重建、可校验的项目状态。代码片段只服务于理解架构边界，不要求你逐行背实现。

本章怎么学

先把自己放进一个真实 query：没有 Persistent Task 时，当前 agent 会在哪里卡住。

再故意设计一个朴素方案，观察它如何破坏 loop、状态或安全边界。这个失败比“正确答案”更重要，因为它能帮你写出更好的重建 prompt。

最后才看实现：先看图和数据流，再看少量接口、源码入口和验证方法。读完后，你应该能用自己的话描述 Persistent Task 的职责，而不是只记住文件名。

本章场景

用户说：“接下来三天逐步完成权限重构，今天先做审计，明天实现，后天补 eval。”这不是一次 session 的 TODO，而是一个长期 Task Group。

Task 系统需要能创建 group、添加 task、更新状态、表达依赖，并在重启后仍能恢复。

先试一个朴素方案

朴素方案是把 TODO 数组直接 JSON.stringify 到磁盘。这样确实能重启恢复，但会把临时执行节奏和长期计划混在同一种格式里。

这个朴素方案的问题在于：它把 Persistent Task 当成普通实现细节，而不是 agent harness 的一条稳定边界。 

await writeFile("tasks.json", JSON.stringify(currentTodos));

朴素方案为什么不够

身份不稳定
长期 task 需要 group_id/task_id，不能依赖数组下标。
派生状态漂移
ready/blocked 应由依赖计算得出，落盘后可能和真实状态不一致。
索引损坏不可恢复
index.json 应是可重建索引，不应成为唯一事实来源。

回到 Agent Loop

Persistent Task 接入 loop 的位置是：工具执行层和持久化层之间，把长期计划保存为可重建、可校验的项目状态。理解这个位置，比记住具体函数名更重要。

Task 通过工具暴露给模型：模型可以创建和更新长期计划；当前 active task group 可通过 reminder 影响本轮执行，但存储事实来自磁盘。

从原理落到代码

到这里我们已经知道 Persistent Task 为什么存在、插在 loop 的哪里，以及朴素方案会怎样失败。下面才进入实现层：只抓住能固定架构的接口、状态边界和接线路径。

读源码时不要从文件顶部一路啃到底。更有效的顺序是：先找入口函数，再找它读写的状态，最后看测试如何证明这些状态变化是真的发生了。

一次真实运行 walkthrough

  1. 用户要求创建一个长期计划。
  2. LLM 调用 task group create 工具。
  3. TaskManager 校验 group 标题、projectKey 和任务初始状态。
  4. TaskStore 写入 groups/<group_id>/group.json，并更新 index.json。
  5. 用户继续添加依赖任务，TaskManager 检查依赖是否存在且无环。
  6. 下一次启动时，TaskStore 扫描 group.json，可重建 index。
  7. Agent 当前会话只通过 active group reminder 获得需要执行的上下文。

关键接口和伪码

下面的接口和伪码只保留本章最关键的形状。真正写代码时，可以让 coding agent 参考源码地图中的文件补齐细节。 

interface TaskGroup {
  groupId: string;
  projectKey: string;
  title: string;
  tasks: TaskItem[];
}

interface TaskItem {
  id: string;
  title: string;
  status: "pending" | "in_progress" | "completed" | "cancelled";
  dependsOn: string[];
}

// ready/blocked 由 dependsOn 和 status 推导，不直接落盘。

设计直觉

长期任务的关键是身份和可恢复性。文件名、group_id 和内容里的身份必须一致，否则未来清理、索引重建和审计都会出错。

把 ready/blocked 这类派生状态留在计算层，可以避免存储多年后出现“看起来 blocked 但依赖已完成”的漂移。

架构拆分

TaskStore
负责文件布局、JSON 校验、原子写和索引重建。
TaskManager
负责状态机、依赖图和 active group。
Task tools
给模型暴露 create/list/read/add/update/delete 能力。

状态与边界

持久化事实
TaskGroup、TaskItem、依赖关系和显式状态。
派生状态
ready/blocked、列表排序、active 摘要。
session-local
当前 active group 的 reminder 和正在执行的 TODO。

Prompt Card

Prompt Card

目标：实现 Persistent Task Group，让 agent 能跨会话保存长期项目计划。

场景：用户创建多天任务计划时，agent 用 Task 工具落盘 group 和 task，并能重启后读取。

模块：task-store.ts、tasks.ts、tools/tasks.ts、tests。

接线：composition root 创建 TaskStore/TaskManager，注册 Task 工具，active group 进入 reminder。

边界：不要复用 TODO 格式；ready/blocked 不落盘；index 必须可重建。

例外和 corner case：group_id 不匹配、依赖不存在、依赖成环、删除 task 后依赖收敛。

验证：测试创建、更新、依赖、索引重建和坏 JSON 拒绝。

源码入口：src/task-store.ts、src/tasks.ts、src/tools/tasks.ts。

文档同步：说明 Task 是长期计划，TODO 是短期节奏。

容易踩坑

Design Trap

常见错误：把 ready/blocked 直接写进 group.json。

为什么错：派生状态会随依赖变化而漂移，落盘后很难保持一致。

正确做法：只保存事实状态，读取时重新计算派生状态。

怎么验证：修改依赖状态后，ready/blocked 自动变化且 group.json 不记录它们。

  • 用数组下标当 task id。
  • index.json 损坏后无法恢复。
  • Task 和 TODO 互相覆盖状态。

如何验证

Validation

手工 query：创建带依赖的 group，重启后读取，确认依赖和状态仍在。

确定性测试：TaskStore 坏文件、id 不匹配、索引重建测试。

集成观察：Task 工具返回 active group reminder。

失败信号：删除 index.json 后任务列表全部丢失。

  • src/task-store.test.ts 覆盖持久化。
  • src/tasks.test.ts 覆盖业务状态机。
  • src/tools/tasks.test.ts 覆盖工具层。

如果实现失败，先查哪里

  • 检查 group_id 是否同时匹配目录名和内容。
  • 检查依赖图是否有环。
  • 检查 index 是否可从 group.json 重建。
  • 检查 active group 是否只是会话状态。
