第 13 章 · 后台跑长任务不阻塞主对话

Async Run: 3 并发上限 + 终态幂等 + 输出隔离

第 12 章的 Task 是"长期计划", 但 LLM 想"现在跑 3 个独立测试 (build / lint / typecheck), 不阻塞当前对话" 时, Task 不够 — Task 是计划层, 需要执行层。 这一章加 src/async-runs.ts 模块: AsyncRun = 一次非阻塞运行 实例, 支持 command (跑 bash) 和 subagent (派子 agent) 两种 executor, 最多 3 个同时 running, 5 分钟超时, 终态幂等 (race condition 安全) + 输出隔离 (写到独立文件 + LRU OutputStore)。 读完后, 你能讲清"Async Run vs Task" 的 边界 (执行层 vs 计划层) + "finishRun 收敛" 的核心正确性保证 + "并发限制 3" 的理由。

真实失败故事: LLM 跑 npm test 卡死主对话 10 分钟

写代码之前, 先看一个真实痛点。 用户的 team 用 harness 跑大型 monorepo, LLM 跑一个简单 fix 后想"跑 npm test 验证", 但 npm test 跑 10 分钟才完。 这 10 分钟内 LLM 主循环被阻塞, user 不能 ctrl-c, 不能切到别的对话, 不能问别的任务。

  1. 症状: LLM 调 run_bash("npm test"), 主循环 await 卡 10 分钟, 整个 harness 死锁。
  2. 更糟的并发场景: LLM 想"同时跑 3 个测试套件 (frontend / backend / e2e) 节省时间", 但 run_bash 是同步 的, 必须一个一个跑, 总共 30 分钟。 应该并发。
  3. 更更糟的资源滥用: LLM 想"派 20 个子 agent 跑 20 个独立 task", 20 个子 agent 同时跑, 把 LLM API quota 烧光, 全部 fail。
  4. 真问题: 缺少非阻塞执行层 + 并发限制 + 输出隔离。 LLM 需要的是"启动一个后台 run, 立即拿到 run_id, 继续干别的, 完事后用 run_id 读输出"。

朴素想法 1: "直接用 Node child_process.spawn 不 await?" 半对。 spawn 返回子进程 PID 不 await, 但 LLM 不知道什么时候 跑完, 怎么读输出, 怎么让用户看到结果。 需要一个 Async Run Manager 集中管理。

朴素想法 2: "无限制并发?" 错。 LLM 一次想跑 100 个, 把 API quota 烧光, 用户账号被 ban。 需要并发限制 3 (经验值), 超过就 reject。

正确做法: 加 src/async-runs.ts — createAsyncRunManager 工厂, 内部状态 (records + runningCount + finishedRunIds) 全 在闭包。 finishRun 是所有执行 路径 (命令完成 / 超时 / 异常 / shutdown) 汇合的唯一 入口, 保证幂等。 看 src/async-runs.ts 第 13 行的核心正确性保证注释: 

// 核心正确性保证: finishRun() 是所有终态收敛的唯一入口, 确保:
// 1. 只有 running → 终态的转换有效
// 2. 第一个进入终态的路径 wins (通过 Set 保证)
// 3. late result 不能覆盖 timeout
// 4. 不重复递减 runningCount、不重复推送 notification

Async Run vs Task: 执行层 vs 计划层

用途: Task 是计划层 (跨会话 持久化, 带依赖图, LLM 跑 task_1 等 task_2 ready); Async Run 是执行层 (session 内非阻塞运行, 5 分钟超时, 跑完即丢)。 两者职责正交, 不能互相替代。

真实场景: 用户让 LLM "跑 5 天迁移"。 LLM 先 创建 TaskGroup "migration-001" 含 500 个 task (计划层); 之后 每次开 session, LLM 调 async_run_start({executor: "subagent", prompt: "跑 task_51 ~ task_60"}) 启动 Async Run (执行层)。 Async Run 内部派生独立 sub-agent + 只读工具集, 跑 5 分钟, 完成后更新 task_51 ~ task_60 状态 (回到计划层)。

设计思想: 层间清晰 — Async Run 不持久化, 进程重启丢失; Task 持久化, 跨 session 恢复。 Async Run 的 groupId / persistentTaskId 字段是"链接"到 Task 的桥, 不复制 Task 的状态。 看 src/async-runs.ts 第 7-10 行的核心概念注释: 

// 核心概念:
// - Async Run = 一次非阻塞运行实例, 记录 run_id / status / output / notification
// - 与 Task Group 不同: Async Run 是运行时执行层, 不是长期计划层
// - 第一版只支持 command 和 subagent 两种 executor
// - 第一版最多允许 3 个同时 running 的 async runs
// - 第一版只允许只读探索和诊断命令
图 1 · Async Run vs Task 的边界
Async Run · 运行时执行层
session 内非阻塞, 5 分钟超时, 3 并发上限。 不持久化, 进程重启丢。 适合"现在跑一个 bash / 派一个子 agent"。
Task Group · 长期计划层
跨会话持久化到 .tasks/groups/, 带依赖图 + 状态机。 适合"5 天迁移, 500 个 task"。

实现细节: Async Run 暴露 groupId / persistentTaskId 两个可选字段, 让 run 可以"链接"到 Task, 跑完时回调更新 task 状态 (虽然 LLM 自己写代码更新, 不用 Async Run 自动)。 链接是显式 字段, 不隐式行为。

2 种 executor: command 和 subagent

用途: LLM 启动 Async Run 时指定 executor 类型。 command = 跑一个 bash 命令 (read-only); subagent = 派一个独立 的子 agent (带只读工具集 + 父级 system prompt 快照)。 2 种覆盖 90% 后台任务场景。

真实场景: user 让 LLM "扫一下项目找所有 TODO 注释", LLM 调 async_run_start({executor: "command", command: "grep -rn 'TODO' src/"}); user 让 LLM "派一个子 agent 调研 React 18 新特性", LLM 调 async_run_start({executor: "subagent", prompt: "调研 React 18 新特性", resources: {read_paths: ["docs/"]}})。

设计思想: 封闭类型, 未来扩展点 — 2 种 executor 固定枚举, 不允许动态加第 3 种 (避免混乱); 未来 真要加 (如 MCP / 自定义 runner) 走 P2 阶段。 看 src/async-runs.ts 第 48 行的 AsyncRunExecutor union: 

export type AsyncRunExecutor = "command" | "subagent";
图 2 · 2 种 executor 的资源差异
command · bash 执行器
跑 executeBash(command, timeoutMs), 走 readonly 命令策略 (只允许 ls / cat / grep / find / git status 等只读命令)。 输出写到 output.txt。
subagent · 子 agent 执行器
调 createAgentFn(...) 创建独立 child agent, 派生独立 history / compressor / tools (readPaths 过滤) / 父 system prompt 快照。 maxRounds 默认 8, 上限 20。

实现细节: command 走 launchCommandRunner 内部 setTimeout timeout, subagent 走 launchSubagentRunner 复用 createAgentFn 工厂 + AbortController (shutdown 时主动 abort)。 两个函数返回的 Promise 都不被 start() await, 启动后 立即返回 run_id。

3 并发上限: 经验值的理由

用途: LLM 一次启动的 Async Run 不能超过 3 个。 超过 3 个 reject, 错误信息明确告诉 LLM "当前并发已满, 请等待 现有 run 完"。 这是 Reference 章节 "模式 14 · Concurrent Limit 并发限制" 的具体应用。

真实场景: LLM 想"同时跑 20 个测试套件节省时间", 启动第 4 个时报错 "Maximum concurrent async runs (3) reached"。 LLM 知道 "3 是上限", 调整策略: 跑完 3 个再启动下 3 个, 总时间 节省 67% 但不会爆 quota。

设计思想: 经验值 + 教学友好 — 3 是经验值, 不需要 LLM 配置; 想"我要 10 个" 改源码改常量。 看 src/async-runs.ts 第 208 行的 MAX_CONCURRENCY = 3: 

const MAX_CONCURRENCY = 3;

实现细节: 检查点在 start() 第 657 行 (if (runningCount >= MAX_CONCURRENCY) throw), start() 立刻 reject, 不排队, 不 await。 这条策略叫 "fail fast" — 让 LLM 立即知道"现在过载", 不会傻等。

3 的理由:

  • 经验值: 3 = "够用但不过载"。 1 个太慢, 5+ 个 quota 容易爆, 3 是平衡点。
  • Quota 安全: 3 个 subagent 同时跑, 假设每个调 LLM 5 次, 共 15 次 LLM 调用, 短时间不会触发 rate limit。
  • 教学友好: 3 是单数字, 容易记住, 容易测试 (构造 3 个 running, 第 4 个 reject)。

finishRun 幂等: race condition 的核心防御

用途: Async Run 跑完有 4 条可能路径 — (a) 命令 自然完成; (b) 5 分钟超时; (c) 命令抛错; (d) shutdown 主动 abandon。 这 4 条路径可能竞争 (timeout 和 命令完成几乎同时触发), 必须有唯一收敛点保证 只产生一个终态。

真实场景: 跑 npm test 跑了 4 分 50 秒, setTimeout 在 5 分钟触发 timeout, 同时 npm test 跑完 调回调完成回调。 两个回调都调 finishRun(record, "completed"/"timeout", ...)。 没幂等保护, runningCount 减 2 次, notification 推 2 条, output 写 2 次, 状态错乱。 幂等保护让第一个 wins, 第二个 no-op。

设计思想: 单一收敛点 + 幂等 Set — Reference 章节 "模式 19 · Idempotent 幂等" 的标准应用。 看 src/async-runs.ts 第 397 行的 finishRun: 

function finishRun(
  record: AsyncRunRecord,
  nextStatus: Exclude<AsyncRunStatus, "running">,
  output?: string,
  error?: string,
): boolean {
  // 只有 running 状态才允许进入终态
  if (record.status !== "running") return false;
  // 第一个进入终态的路径 wins (Set 保证幂等)
  if (finishedRunIds.has(record.id)) return false;
  finishedRunIds.add(record.id);
  clearRunRuntime(record.id);
  // ... 写状态 / 写输出 / 推送 notification / 递减 runningCount
}

实现细节: 5 条正确性保证 (见注释第 384-389 行):

  1. running → 终态: 已经终态的 record 拒绝再次 收敛。
  2. 第一个 wins: 用 finishedRunIds: Set 标记已收敛, 重复调用 return false。
  3. late result 不覆盖: 即使 completed 后 timeout 才到, timeout 也被 finishRun 拒绝, 不会把 completed 改成 timeout。
  4. 不重复递减: runningCount 只在第一个 wins 时 减 1, 后续不重复减。
  5. 不重复推送: notification 队列只 push 一次, 不 重复通知。

输出隔离: 文件 + OutputStore LRU

用途: 每个 Async Run 的输出独立写到 async-runs/<runId>/output.txt, 同时登记到 OutputStore (LRU 内存索引), LLM 用 run_id 或 output_id 读。 文件 = 长期存档, LRU = 快速查表。

真实场景: LLM 跑 3 个并发 Async Run (build / lint / typecheck), 每个写到独立 output.txt, 完事后 LLM 调 async_run_check(runId) 拿 preview, 调 async_run_output_read(runId, maxBytes) 拿全文。 3 个 run 互不污染, 不会把 A 的输出塞给 B。

设计思想: 独立文件 + LRU 内存索引 — Reference 章节 "模式 10 · Atomic Write 原子写" 的应用 — finishRun 写 output.txt 用 writeFileSync, 不追加; OutputStore 记录 (runId → outputId) 映射, LRU 淘汰旧 run 的全文 (避免内存爆炸)。

实现细节: 看 src/async-runs.ts 第 426 行的 OutputStore 登记 + 第 446 行的旧 outputPath 写入。 两份同时写 (file + store), file 给 git/archive, store 给快速 LLM 读取。

notification 队列: 后台结果不阻塞当前 LLM

用途: Async Run 完成后, 不能直接插入当前 LLM 调用 (会破坏 OpenAI API 消息格式, 第 08 章延迟注入协议), 也 不能阻塞主循环等 run 完。 正确做法: 跑完后 push 一条 notification 到队列, 下一轮 LLM 调用前调 drainNotifications(), 把 notification 渲染到 user message 末尾。

真实场景: LLM 启动 build / lint / typecheck 3 个 run, 立即拿到 3 个 run_id, 继续干别的。 5 分钟后 3 个 run 陆续完成, 3 条 notification 进队列。 LLM 下一次想"现在 build 怎么样", 调 async_run_drain() 拿 3 条 notification, 看到"build ✓, lint ✗ (1 error), typecheck ✓", 据此决定下一步。

设计思想: 异步结果回传 = 排队, 不插入 — Reference 章节 "模式 9 · Observer 观察者模式" 的延伸。 notification 是短摘要 (runId + status + preview), 全文靠 readOutput 读。 短摘要进 history 不撑爆, 全文不 进 history。

实现细节: notification 在 finishRun 内 push 到 notifications: AsyncRunNotification[] 队列, LLM 调 drainNotifications() 拿所有未读 + 队列 清空。 幂等性保证 (第 4 条) 让重复 finishRun 不重复 push。

shutdown: 进程退出时把 running 标 abandoned

用途: harness 进程退出 (ctrl-c / session 关闭 / eval cleanup) 时, 仍有 running 的 Async Run。 应该把 这些 run 标 abandoned (不是 failed, 是"用户主动放弃"), 清内部状态, abort 子 agent。

真实场景: user 跑了 3 个 Async Run, 第 2 个跑 到一半, user ctrl-c 退出。 期望: harness 优雅退出, 3 个 run 全部 abandoned 状态, 子 agent 收到 abort signal 停止 LLM/tool 调用, 内存状态清空, 写日志 "Async run ar_xxx abandoned: process shutdown"。

设计思想: 运行时生命周期, 非 LLM 工具 — shutdown 是 harness 内部用的, 不暴露给 LLM 工具。 看 src/async-runs.ts 第 164 行的 shutdown 注释: 

// 关闭当前进程内仍在 running 的 async runs。
// 这是运行时生命周期方法, 不是 LLM 工具能力:
// - 普通完成/失败/超时仍由 finishRun() 统一收敛
// - 进程关闭或 eval cleanup 时调用 shutdown(), 把剩余 running run 标记为 abandoned
// - command executor 底层 child_process 仍依赖 executeBash 的 timeout 机制退出;
//   shutdown 的职责是清理 manager 状态、取消内部 timeout、abort 子 Agent

实现细节: shutdown 调 finishRun(record, "abandoned", undefined, reason) 对每个 running record 收敛, 然后 clearRunRuntime 清 timeout + abort 子 agent。 这条保证"进程退出 = 没有 zombie 进程 / 没有 内存泄漏"。

foreground conflict: run_bash 不和 async_run 冲突

用途: LLM 想"用 run_bash 跑命令" (前台同步) vs "用 async_run_start 跑命令" (后台异步), 两者可能冲突 (前台命令占住 terminal, 后台命令也在跑)。 应该禁止 同一命令前后台并发。

真实场景: LLM 启动 async_run_start({command: "npm test"}), 之后又调 run_bash("npm test")。 应该报错"和 async run ar_xxx 冲突", 让 LLM 决定"等 async run 完" 或 "cancel async run 后再前台跑"。

设计思想: 前后台冲突检测 — 看 src/async-runs.ts 第 174 行的 checkForegroundToolConflict: manager 暴露这个方法, run_bash 调用前 check, 防止资源 竞争。 不主动 throw, 返回 blocked + reason, 让 agent 决定如何处理。

实现细节: 冲突检测规则: command 字面相同 (标准化后) + 都在 running 状态 → 冲突。 简单规则, 教学 友好; 工业级要做 AST 级别的"语义冲突" (两个命令读同一 文件 + 写同一文件), 太复杂本阶段不做。

主循环集成: start 立即返回, drain 在每轮前

用途: AsyncRunManager 在 Composition Root 创建, 注入 agent + 注册工具 (async_run_start / check / list / read_output / drain)。 主循环集成点 2 个:

  1. async_run_start 工具: LLM 调, manager 启动 run, 立即返回 record (不 await run 完)。 record 含 run_id / status=running / preview。
  2. async_run_drain 工具 + reminder: LLM 调或 reminder 注入, manager 返回所有未读 notification, 渲染 到 user message 末尾。 主循环在每轮 LLM 调用前 drain 一次。

设计思想: 非阻塞 + 延迟回传 — start 立即返回 + drain 主动拉, LLM 主循环不被后台 run 阻塞。 这是 Reference 章节 "模式 14 · Concurrent Limit" 的应用。

实现细节: agent.ts 在第 3 步 (call LLM) 之前 调 asyncRunManager.drainNotifications(), 如果有 notification 就拼到 user message 末尾, 让 LLM 看到"上次后台 run 跑完了, 结果是这样"。

fake test: 模拟 timeout + 命令完成竞态

用途: AsyncRunManager 测试需要真 time (超时靠 setTimeout) + 假 executor (注入假 command runner, 不真跑 bash)。 这两者结合, 可以测"timeout + 命令完成" 的 race condition。

真实场景: 用户写测试"启动一个 1 秒超时的 command, 0.5 秒后手动调 finishRun(..., "completed") 模拟 命令提前完成; 然后 1 秒后 timeout 触发, 调 finishRun(..., "timeout")。 验证: record.status === "completed" (不是 timeout), runningCount === 0, notification 队列只有 1 条 (不是 2 条)"。 

test("race: timeout 后到不覆盖提前完成", () => {
  vi.useFakeTimers();
  const mgr = createAsyncRunManager({...});

  // 启动一个 1 秒超时的 command
  const record = mgr.start({ title: "x", executor: "command", command: "echo a", timeoutMs: 1000 });

  // 0.5 秒后手动调 finishRun (模拟命令提前完成)
  vi.advanceTimersByTime(500);
  // 这里需要拿到内部 record, 测试可能要 hack 接口
  // 实际测试通过 createAsyncRunManager 注入一个 fake runner

  // 1 秒后 timeout 触发 (fake timer)
  vi.advanceTimersByTime(500);
  // 验证: record.status === "completed"
  expect(mgr.check(record.id)!.status).toBe("completed");
  expect(mgr.list({ includeTerminal: true })).toHaveLength(1);

  vi.useRealTimers();
});

test("并发限制 3: 启动第 4 个 reject", () => {
  const mgr = createAsyncRunManager({...});
  mgr.start({ title: "a", executor: "command", command: "echo a", timeoutMs: 60000 });
  mgr.start({ title: "b", executor: "command", command: "echo b", timeoutMs: 60000 });
  mgr.start({ title: "c", executor: "command", command: "echo c", timeoutMs: 60000 });
  expect(() => mgr.start({ title: "d", executor: "command", command: "echo d", timeoutMs: 60000 }))
    .toThrow(/Maximum concurrent async runs/);
});

test("shutdown 把 running 标 abandoned", () => {
  const mgr = createAsyncRunManager({...});
  const r = mgr.start({ title: "a", executor: "command", command: "echo a", timeoutMs: 60000 });
  mgr.shutdown!("test shutdown");
  expect(mgr.check(r.id)!.status).toBe("abandoned");
});

实现细节: race condition 测试需要 vi.useFakeTimers() + 内部 record 暴露 (生产 API 不暴露, 测试要 hack)。 一般做法: 在 createAsyncRunManager deps 注入一个 fake setTimeout (用 fake timer), record 通过 check() 接口读, 不用 hack 闭包。

常见误解: Async Run 不是后台进程池

误解 1: "Async Run = 启动一个 Node 子进程?" 错。 command executor 内部用 child_process, 但 Async Run 是 管理层, 不是进程本身。 LLM 调 start() 拿到 record, 不是拿到 PID。

误解 2: "无限并发, 多多益善?" 错。 3 是上限, 超就 reject。 LLM 想"再快也要 3 个一组, 跑完再起下一组"。

误解 3: "timeout 一定赢?" 错。 谁先到谁赢, 不分 timeout / completed / failed。 命令提前完成 → completed; timeout 先到 → timeout; 都可能, 看时序。

误解 4: "Async Run 持久化?" 错。 session 内 内存, 进程重启丢。 跨会话跟踪用 Task (第 12 章)。

反例梯度: 4 个常见错误

反例 1 · 没有 finishRun 幂等

让 timeout 和命令完成都直接改 record.status, 没有 Set 标记。 race condition 时 runningCount 减 2 次, notification 推 2 条, output 写 2 次。 错。 正确: finishRun 是唯一收敛点, 用 finishedRunIds: Set 保证幂等, 第二个 调用 return false。

反例 2 · 没有并发限制

start() 不检查 runningCount, LLM 一次想跑 100 个就 真跑 100 个, 把 LLM API quota 烧光。 错。 正确: if (runningCount >= MAX_CONCURRENCY) throw fail fast, 让 LLM 立即知道过载。

反例 3 · 输出直接进 history

Async Run 跑完直接把 output 字符串 append 到 history。 错。 破坏 LLM API 消息格式 (tool_call 后立即 user message, 第 08 章延迟注入), 也撑爆 history。 正确: 输出写到 output.txt + OutputStore, LLM 主动调 readOutput(runId, maxBytes) 拉取。

反例 4 · shutdown 不 abort 子 agent

shutdown() 只清 manager 状态, 不 abort 子 agent 内部 的 LLM/tool 调用。 子 agent 继续跑, 写日志, 调 LLM API, 进程退出时这些调用全 mid-write 报错。 错。 正确: 持有 abortControllers: Map<runId, AbortController>, shutdown 时对每个 running run 调 abortController.abort(), 子 agent 内部 LLM client 收到 abort signal 停止新调用。

Validation: 4 条不变量检验

  1. finishRun 幂等: 对同一 record 调 finishRun 2 次, 第二次 return false, record 状态不变化, runningCount 不重复减。 验证: 模拟 race condition (timeout 和 completed 同时触发), 跑测试断言。
  2. 并发限制: 启动 4 个 run, 第 4 个 throw "Maximum concurrent async runs (3) reached"。 验证: 单测 start 3 个 + start 第 4 个 expect throw。
  3. 状态机合法: running → 4 个终态 (completed / failed / timeout / abandoned) 合法, 终态 → 终态不合法。 验证: 单测, 终态后 start 一条 new task 想覆盖 status, expect throw / no-op。
  4. shutdown 清干净: 启动 3 个 run, shutdown, 所有 run 状态 abandoned, abortControllers / timeoutIds 清空, runningCount === 0。 验证: 跑 shutdown 后调 list({includeTerminal: true}) 全是 abandoned 状态。

回望: 哪些原则在本章兑现了

  • 单一收敛点 + 幂等: finishRun 是 4 条执行路径 的唯一汇合, Set 保证幂等, race condition 安全。
  • fail fast: 并发限制 3 超就 reject, 不排队不 await, LLM 立即知道过载。
  • 异步结果回传 = 排队: notification 队列, 不 插入当前 LLM, 延迟到下一轮 drain。
  • 运行时生命周期: shutdown 是 harness 内部用, 不暴露 LLM 工具, abort 子 agent 避免 zombie。
  • 执行层 vs 计划层: Async Run 内存不持久, Task 持久化, groupId 字段桥接两者。

Prompt Card (本章任务)

Prompt Card · 第 13 章

目标: 给 harness 加 Async Run 系统, LLM 启动 非阻塞后台执行, 立即拿 run_id, 完成后通过 notification 队列 回传。

场景: LLM 跑大型 monorepo fix, 想"同时跑 build / lint / typecheck", 启动 3 个 Async Run (命令) + 派 1 个子 agent 调研 (subagent), 立即拿到 4 个 run_id, 继续干别的。 5 分钟后 3 个 build / lint / typecheck 陆续 完成, 通知进队列, LLM 下一轮 drain 看到结果。

模块: src/async-runs.ts (新) 暴露 createAsyncRunManager(deps); 工具 src/tools/async-run-*.ts (新) 包装 LLM 接口; src/agent.ts (改) 第 3 步前 drain notifications; src/index.ts (改) Composition Root 创建 + shutdown 钩子。

边界 (LLM 必须遵守的 checklist):

  • 2 种 executor 固定枚举 command / subagent, 不可新增第 3 种
  • 3 并发上限, 超就 reject throw, fail fast 不排队
  • 5 分钟超时 (MAX_TIMEOUT_MS = 300_000), 超时和命令完成竞争由 finishRun 收敛
  • finishRun 是所有终态收敛的唯一入口, 用 Set 幂等, 重复调用 return false
  • 输出隔离: 写到 output.txt + OutputStore LRU, 不进 history
  • notification 队列短摘要, LLM 主动 drain, 不插入当前 LLM 调用
  • shutdown 调 finishRun(record, "abandoned") + abortController.abort(), 进程退出不留 zombie
  • 前后台冲突检测: command 字面相同 + 都在 running → 拒绝, 错误含 "conflict"

验证 (用 fake timer + fake executor + vitest, 逐条断言):

  • finishRun 幂等: 同一 record 调 2 次, 第二次 return false, 状态不重复写
  • 并发限制: 启动 3 个后第 4 个 throw "Maximum concurrent async runs"
  • race condition: timeout + completed 竞争, 第一个 wins, 第二个 no-op
  • shutdown: 3 个 running → 全部 abandoned, abortControllers 清空
  • notification 队列: 完成 3 个 run, drain 返回 3 条, 重复 drain 返回 0

本章练习

  1. 故意去掉 finishedRunIds Set, 跑 race condition 测试 (timeout + completed 同时触发), 看"幂等" 是否抓到 (runningCount 减 2 次, notification 推 2 条 vs. 1 条)。
  2. 故意去掉 if (runningCount >= MAX_CONCURRENCY) throw 检查, 启动 5 个 run, 看"并发限制" 是否抓到 (5 个全跑 vs. 第 4 个 throw)。
  3. 故意让 Async Run 跑完直接 append output 到 history, 跑测试 断言 OpenAI API 消息格式, 看"输出隔离" 是否抓到 (消息格式 错误 vs. 写到 output.txt + LRU store)。
  4. 故意不 abort 子 agent, 跑 shutdown 测试, 看"shutdown abort" 是否抓到 (子 agent 内部 LLM 调用 mid-write 报错 vs. abort signal 干净退出)。

本章小结

Async Run 是给后台执行的非阻塞运行层, 区别 于 Task 的长期计划层。 核心是 5 个设计:

  • 2 种 executor: command (只读 bash) / subagent (派生独立 child agent), 固定枚举, 不可新增。
  • 3 并发上限: 经验值, 超就 reject, fail fast 不排队。
  • finishRun 幂等: 4 条执行路径 (完成 / 超时 / 失败 / abandoned) 唯一收敛点, Set 标记, race condition 安全。
  • 输出隔离: output.txt + OutputStore LRU, 不 进 history, LLM 主动 read。
  • notification 队列: 短摘要, 不插入当前 LLM, 延迟到下一轮 drain。

    • 下一章 (第 14 章) 展开 Async Run 的定时触发 — 复用 Async Run 作为执行单元, ScheduleManager 负责"每天 9 点 跑一次" / "每周一 10 点跑一次" 的 cron-like 调度, 用 ID 去重 保证不重复跑。
