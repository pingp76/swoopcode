不阻塞主循环：Async Run

有些命令、子任务或分析需要后台执行。Async Run 让 agent 启动非阻塞运行实例，同时继续与用户对话。

你已经知道什么，还不知道什么

你已经有同步工具和 SubAgent。Async Run 不是长期 Task，也不是 Schedule，它是当前运行期里的后台执行实例。

读这一章时请把注意力放在 agent loop 的接入点：工具执行层启动后台实例，结果通过 notification/reminder 回到后续 turn。代码片段只服务于理解架构边界，不要求你逐行背实现。

本章怎么学

先把自己放进一个真实 query：没有 Async Run 时，当前 agent 会在哪里卡住。

再故意设计一个朴素方案，观察它如何破坏 loop、状态或安全边界。这个失败比“正确答案”更重要，因为它能帮你写出更好的重建 prompt。

最后才看实现：先看图和数据流，再看少量接口、源码入口和验证方法。读完后，你应该能用自己的话描述 Async Run 的职责，而不是只记住文件名。

本章场景

用户说：“先跑完整测试，期间帮我继续看权限代码。”完整测试可能跑很久，主 agent 不应该卡住等待。

Async Run 让 agent 启动一个后台命令或子智能体，返回 run_id；之后可以 check/list/readOutput，并在完成时通过 reminder 通知主 loop。

先试一个朴素方案

朴素方案是直接 await 长命令，直到它结束再回复用户。

这个朴素方案的问题在于：它把 Async Run 当成普通实现细节，而不是 agent harness 的一条稳定边界。 

const output = await runBash("npm test");
return output; // 用户在测试期间无法继续交互

朴素方案为什么不够

主循环阻塞
用户无法继续下达任务，agent 也无法并行分析。
终态竞争
timeout 后 late result 可能覆盖已经结束的状态。
前后台冲突
后台正在读取或测试文件时，前台写操作可能让结果无意义。

回到 Agent Loop

Async Run 接入 loop 的位置是：工具执行层启动后台实例，结果通过 notification/reminder 回到后续 turn。理解这个位置，比记住具体函数名更重要。

Async Run 启动后立即返回 run_id；后台完成时进入 notification 队列，下一轮构建 messages 时以 reminder 注入。

从原理落到代码

到这里我们已经知道 Async Run 为什么存在、插在 loop 的哪里，以及朴素方案会怎样失败。下面才进入实现层：只抓住能固定架构的接口、状态边界和接线路径。

一次真实运行 walkthrough

  1. LLM 调用 async_start，要求后台运行测试。
  2. AsyncRunManager 创建 run_id 和 running 状态。
  3. 后台 executor 开始执行 command 或 subagent。
  4. Agent 立即回复用户 run_id，并继续主 loop。
  5. 用户稍后调用 async_check 或 async_list。
  6. 后台完成后，finishRun 只允许 running 进入终态。
  7. completion notification 在下一轮以 reminder 告知模型和用户。

设计直觉

Async Run 让“等待”变成可管理状态。它不是让 agent 无限并发，而是把后台执行的生命周期显式化。

终态只允许从 running 收敛，是为了防止 timeout 后迟到的输出覆盖最终状态。

架构拆分

AsyncRunManager
管理 run_id、状态机、输出和通知。
Executor
实际执行 command 或 subagent，并把结果交回 manager。
Async tools
提供 start/check/list/output_read 给模型调用。

状态与边界

session-local
run 实例、running 状态和通知队列。
output store
长输出可以通过 handle 读取，不直接塞进 messages。
非持久化
当前 Async Run 不承诺跨重启恢复。

Prompt Card

Prompt Card

目标：实现 Async Run，让 agent 能启动、查询、列出和读取后台运行实例。

场景：长测试或长分析在后台运行，主 agent 能继续响应用户。

模块：async-runs.ts、tools/async-runs.ts、execution-policy.ts、session-events.ts。

接线：composition root 创建 AsyncRunManager，注册 async tools，完成通知进入 reminder。

边界：Async Run 不是持久化 Task；后台命令使用非交互 execution policy；终态不可被 late result 覆盖。

例外和 corner case：timeout、失败、late success、读取不存在 run、前台写冲突。

验证：测试 start/check/list/output_read、终态收敛和通知 drain。

源码入口：src/async-runs.ts、src/tools/async-runs.ts。

文档同步：说明 Async Run 是运行实例，不是长期计划。

容易踩坑

Design Trap

常见错误：timeout 后允许 late result 覆盖状态。

为什么错：用户已经看到 timeout，迟到成功会让审计和后续行为矛盾。

正确做法：finishRun 只允许 running 进入终态，终态后忽略 late result。

怎么验证：模拟 timeout 后再 finish success，状态仍为 timeout。

  • 后台命令绕过 readonly policy。
  • 把 run 当作跨重启任务恢复。
  • 后台完成通知丢失，模型不知道结果可读。

如何验证

Validation

手工 query：启动一个 sleep 或测试命令，期间继续对话，再 check/readOutput。

确定性测试：用 fake executor 控制 succeeded/failed/timeout。

集成观察：完成通知应在下一轮 reminder 中出现。

失败信号：主 loop 被长命令阻塞，或 timeout 被 late success 覆盖。

  • src/async-runs.test.ts 覆盖核心。
  • src/tools/async-runs.test.ts 覆盖工具层。
  • 冲突检测测试覆盖前台写与后台 readPaths。

如果实现失败，先查哪里

  • 检查 run_id 是否唯一。
  • 检查状态转移是否只从 running 到终态。
  • 检查 executor 是否使用正确 policy。
  • 检查 notifications 是否 drain 后不重复。
