第 10 章 · Prompt cache 稳定性

Cache: 怎么让 OpenAI / Anthropic 的 prompt cache 一直命中

第 05 章 Skill 强调"稳定前缀"、第 08 章 Hook SessionStart 注入 也说"稳定前缀"、第 09 章 Memory 注入 system prompt 仍然说"稳定 前缀" — 但到底什么叫稳定? 怎么观察 它稳定不稳定? 这一章加 src/cache-debug.ts 模块 — 计算 system prompt / tools / 稳定前缀的 SHA256 hash, 每轮请求前 对比"上次 vs 这次", 输出 [cache] systemPrompt=stable tools=stable prefix=stable 这种单行日志, 让开发者一眼看出"哪变了, 改一次成本 ×10"。 读完后, 你能讲清"稳定前缀 = system prompt + tools" 的边界, 并能 用 fake tracker 验证"key 排序一致" 的稳定序列化协议。

真实失败故事: 改一行 system prompt 账单 ×10

写代码之前, 先看一个真实痛点。 用户的 team 上线 harness 跑 生产, 第一个月账单正常, 第二个月改了 3 行 system prompt 加 "新规则: 写完代码必须 npm test", 第三个月账单突然 ×10。

  1. 症状: prompt token 从 2k 涨到 20k, 输出 token 不变, 总成本 ×10。
  2. 根因: system prompt 变了, OpenAI 的 prompt cache (Anthropic 类似) 失效, 每次请求都从 0 重新计费输入 token。
  3. 误诊: team 以为是 LLM 流量涨了, 加机器, 没用 — 问题在输入侧重复计费。
  4. 真正的修法: 加 cache debug 工具, 每轮请求前 hash system prompt + tools, 输出"变了" 警告。 改 system prompt 时 先看 hash, 确认是"故意改" 还是"无意改"。

朴素想法 1: "OpenAI 自动 cache, 我不用管?" 错。 OpenAI 的 cache 按前缀 hash 命中, 前缀变化 cache miss, 重新计费。 harness 必须自己保证前缀稳定。

朴素想法 2: "把 system prompt 写死, 永远不变?" 也不行。 用户 加 memory / skill / todo reminder 时, system prompt 必须 反映这些"几乎不变" 的状态, 不能写死成"初始版本"。

正确做法: 加 src/cache-debug.ts — 每轮 LLM 调用 前 hash (system prompt + tools), 与上次对比, 输出 "stable / changed" 标记。 这是 Reference 章节 "模式 11 · Cache-friendly 缓存友好" + "模式 18 · Transcript-First 透传优先" 的联合 应用。

稳定前缀边界: system prompt + tools

用途: OpenAI / Anthropic 的 prompt cache 按 前缀命中, 前缀 = system prompt + tools 定义。 任何"应该稳定的" 东西必须落在前缀内, 任何"每轮会变的" 必须 在 history 里。 边界划错, 成本就乱。

真实场景: team 把 todo reminder 写在 system prompt, todo 状态每轮变, system prompt 每轮变, cache 全 miss。 正确做法: todo reminder 写在 user message (历史), system prompt 写"使用 TodoManager 跟踪任务" (规则), 不写"当前 todo 列表" (状态)。

设计思想: 边界 = 稳定 vs 变化 — 写 cache-debug 模块的第一件事就是定义"稳定 前缀 = 什么"。 看 src/cache-debug.ts 第 119 行的 computeStablePrefixHash: 

function computeStablePrefixHash(
  messages: ChatCompletionMessageParam[],
  tools: ChatCompletionTool[],
): string {
  const systemMsg = messages.find((m) => m.role === "system");
  const systemContent =
    typeof systemMsg?.content === "string"
      ? systemMsg.content
      : JSON.stringify(systemMsg?.content ?? "");
  return sha256(systemContent + stableStringify(tools));
}
图 1 · 稳定前缀的边界
稳定前缀 (cache 命中区)
system prompt (含 memory 段 / skill 段 / 工具规则) + tools 定义 (6 个工具的 JSON schema)。 同一 session 内保持不变, cache 命中。
变化区 (每轮重新计费)
history: user / assistant / tool messages。 每轮增长, LLM 必须读全量, 没办法 cache (因为 cache 是前缀)。

实现细节: 教学版定义"稳定前缀 = system prompt content + tools JSON"。 OpenAI 实际的 cache 算法可能更复杂 (按 token 块 cache, 不一定整个前缀), 但教学版讲 system prompt + tools 这一段已经够用, 不假装"我知道 OpenAI 内部 怎么 cache"。

Hash 计算: SHA256 + 稳定序列化

用途: 缓存调试模块不直接读 LLM API 的 cache 命中率 (那需要后端日志), 而是本地 hash 对比 — 计算 "上一轮 system prompt + tools" 和 "这一轮" 的 hash, 一致 就是稳定, 不一致就是变化。 这是教学版的 cache 观测, 不假装 是真实 cache hit rate。

真实场景: 开发者改 system prompt, 重启 harness, 看到日志: [cache] systemPrompt=changed tools=stable prefix=changed。 知道 system prompt 改了, 决定是"故意改" 还是"无意改"。

设计思想: 本地可观测 — cache 调试 跑在 harness 进程内, 不调外部 API, 不读 LLM 响应头, 只 算 hash 对比。 简单, 可靠, 教学友好。 看 src/cache-debug.ts 第 90 行的 sha256: 

function sha256(input: string): string {
  return createHash("sha256").update(input, "utf-8").digest("hex").slice(0, 8);
}

实现细节: SHA256 取前 8 位 hex 字符 (4 字节) — 足够区分"变了" vs "没变", 不需要完整 64 位。 32 位 hash 碰撞 概率 ~4 × 10⁻¹⁰, 调试用 8 位 (~1.6 × 10⁻⁹) 也够。 用 Node 内置 crypto 模块, 不引入外部依赖。

从这里开始, 全是为了本地调试

上一节讲了 hash 怎么算 (SHA256 + 8 位 hex), 这一节开始讲的 5 个工具 (stableStringify / createCacheDebugTracker / formatCacheDebugLog / loop-integration / changed-actions) — 全部是为了本地调试, 跟真实 cache 命中率 无关。 这一节专门解释"为什么需要这些", 避免你读到后面 糊涂"这堆 hash 工具到底要干啥"。

先说清楚: 教学版不假装读真实 cache hit rate

真实 cache 命中率只有 LLM 后端知道, 存在响应头的 cached_tokens 字段里。 教学版不调 LLM 响应头, 也不接 LLM SDK 的内部 cache 统计, 假装能读真实命中率。 我们 能做的只有 "harness 期望 cache 怎么命中" (基于 prefix hash), 不是 "cache 实际 命中多少" (基于 LLM 后端统计)。 这是教学版的故意简化, 不是 bug。

那为什么还要搞一堆 hash 工具

真正的问题是: 开发者不知道自己改了什么。 团队 leader 改 system prompt, 上线 1 个月账单 ×10, 但他不知道 是"改 system prompt" 还是"改 tools" 还是"加新 skill" 引起 的 cache miss。 没有 hash 工具, 排查靠"猜 + 改一行部署 一次看账单", 几周才查出来。

教学版的 hash 工具解决的是这个排查效率问题: 本地观察 prefix hash 变化, 立刻知道 "哪一段变了", 不用 部署到生产看账单。 提前发现问题, 不让账单当报警器。

stableStringify: 顺序无关的对象序列化

场景: tools 数组顺序变了, hash 也变了 (明明内容一样)

跑 100 轮对话观察 cache 行为, 看到日志: [cache] systemPrompt=stable tools=changed prefix=changed。 意思是 "system prompt 没变, 但 tools 部分变了, 所以整个前缀 hash 变了, cache miss"。 但你确信 tools 内容没改啊! 怎么回事?

朴素做法: 写日志看 tools 是什么: 

// 第 1 次跑
const tools = [run_read, run_bash, run_write];
console.log(JSON.stringify(tools));
// 输出: [{"name":"run_read",...},{"name":"run_bash",...},{"name":"run_write",...}]

// 第 2 次跑 (注册顺序变了, 业务上等价)
const tools = [run_bash, run_read, run_write];
console.log(JSON.stringify(tools));
// 输出: [{"name":"run_bash",...},{"name":"run_read",...},{"name":"run_write",...}]

// 两个字符串不一样! 但内容集合完全一样 (都是 3 个工具, 名字一样)
// → hash 不一样 → cache 误报 miss

问题根源: JSON.stringify 是字面序列化, 数组顺序 / 对象 key 顺序变了, 输出就变, hash 就变。 但 你关心的是 "内容集合是否一样", 不是 "字面是否一样"。 这两个 概念在 hash 时混在一起, 就会误报 cache miss。

正确做法: 写一个"内容 hash" 用的稳定序列化

规则:

  • 对象按 key 字母升序排序: {b: 1, a: 2} 和 {a: 2, b: 1} 内容一样, 序列化输出必须一样 ({"a":2,"b":1})
  • 数组保序: [a, b] 和 [b, a] 内容不一样, 序列化输出必须不一样 (因为数组是有序列表, 顺序有意义)
  • undefined 跳过: 跟 JSON.stringify 行为一致, 避免引入"同一对象两种表示"
  • 递归处理嵌套: 对象套对象, 数组套对象, 都要按规则处理

具体例子: 3 个工具数组, 顺序变了, 序列化结果一样 

// 输入 1 (注册顺序 A)
const tools1 = [
  { name: "run_read",  params: {...} },
  { name: "run_bash",  params: {...} },
  { name: "run_write", params: {...} },
];

// 输入 2 (注册顺序 B, 内容等价)
const tools2 = [
  { name: "run_bash",  params: {...} },
  { name: "run_read",  params: {...} },
  { name: "run_write", params: {...} },
];

// JSON.stringify 行为
JSON.stringify(tools1);  // [{"name":"run_read",...},{"name":"run_bash",...},{"name":"run_write",...}]
JSON.stringify(tools2);  // [{"name":"run_bash",...},{"name":"run_read",...},{"name":"run_write",...}]
// ↑ 字面不同, hash 不同, 误报 cache miss

// stableStringify 行为
stableStringify(tools1);  // [{"name":"run_read",...},{"name":"run_bash",...},{"name":"run_write",...}]
stableStringify(tools2);  // [{"name":"run_bash",...},{"name":"run_read",...},{"name":"run_write",...}]
// ↑ 字面也不同! 数组保序!

// 等等, 数组保序的话, 顺序变了 hash 不也变吗?  对!
// 所以这个例子实际上是"误用", stableStringify 主要解决的是
// 对象 key 顺序问题, 不是数组顺序问题。

关键反直觉点: stableStringify 主要解决"对象 key 顺序", 不是"数组顺序"

数组是有序列表, [a, b] 和 [b, a] 真的不一样, hash 必须不一样。 stableStringify不 解决数组顺序 问题。 它解决的是对象 key 顺序问题 — {a: 1, b: 2} 和 {b: 2, a: 1} 是同一个对象, hash 必须一样。

具体例子: 真正解决的是对象 key 顺序 

// 输入 1 (object key 顺序 A)
const config1 = { name: "run_read", description: "读文件", parameters: {...} };

// 输入 2 (object key 顺序 B, 内容等价)
const config2 = { parameters: {...}, description: "读文件", name: "run_read" };

// JSON.stringify 行为
JSON.stringify(config1);  // {"name":"run_read","description":"读文件","parameters":{...}}
JSON.stringify(config2);  // {"parameters":{...},"description":"读文件","name":"run_read"}
// ↑ 字面不同! hash 不同! 但内容是同一个 config, 应该 cache 命中

// stableStringify 行为
stableStringify(config1);  // {"description":"读文件","name":"run_read","parameters":{...}}
stableStringify(config2);  // {"description":"读文件","name":"run_read","parameters":{...}}
// ↑ 字面也相同! hash 相同! cache 命中

代码: src/cache-debug.ts 第 64 行的 stableStringify: 

function stableStringify(value: unknown): string {
  if (value === null) return "null";
  if (typeof value === "string") return JSON.stringify(value);
  if (Array.isArray(value)) {
    return "[" + value.map(stableStringify).join(",") + "]";
  }
  if (typeof value === "object") {
    const obj = value as Record<string, unknown>;
    const keys = Object.keys(obj).sort();   // ← 关键: key 排序
    const pairs = keys
      .filter((k) => obj[k] !== undefined)
      .map((k) => `${JSON.stringify(k)}:${stableStringify(obj[k])}`);
    return "{" + pairs.join(",") + "}";
  }
  return "null";
}

逐行走一遍 (假设输入是上面的 config1):

  1. value 是对象, 不是 null / string / array。
  2. 进入 "object" 分支, keys = Object.keys(obj).sort() → ["description", "name", "parameters"] (按字母升序)。
  3. 每个 key 生成 "key":value 对, 递归处理 value (如果 value 是 嵌套对象, 也按字母排序)。
  4. 用逗号拼起来, 包 {}。 输出: {"description":"读文件","name":"run_read","parameters":{...}}。

同样的 key 顺序, 同样的输出, 同样的 hash, cache 命中。

教学版的局限 (vs 工业级):

  • 不处理 NaN / BigInt / 循环引用: LLM API 返回值不会有这些类型, 教学版不管。
  • 不保证跨语言一致: Python 的 json.dumps(sort_keys=True) 也做了类似事, 但字符顺序可能和 JS 不一样 (比如 unicode 转义)。 教学版只关心"同一进程内稳定", 不跨进程。
实现细节: 4 条规则 — 普通对象按 key 字母升序 排序; 数组保持原序 (tools 顺序是 LLM 看到的顺序, 不能乱); string 用 JSON.stringify (保留转义); undefined 跳过 (与 JSON.stringify 一致)。 教学版的局限: 不处理 NaN / BigInt / 循环引用, 这些 LLM API 也不会返回, 暂 时不管。

createCacheDebugTracker: 闭包持有 lastSnapshot

用途: tracker 是个轻量对象, 每次 LLM 调用前调 inspect({messages, tools}), 返回"当前快照 + 是否 变化" 的二元判断。 内部用闭包持有 lastSnapshot, 不需要外部状态。

真实场景: agent 主循环在第 3 步 (call LLM) 之前 调 tracker.inspect, 输出 logger.info(formatCacheDebugLog(state))。 之后日志里就有稳定的 [cache] systemPrompt=stable ... 标记, 开发者 tail 日志就能看到变化。

设计思想: 工厂 + 闭包 — 这是 Reference 章节 "模式 1 · 工厂 + 闭包" 的标准应用: lastSnapshot 在闭包内, 外部只通过 inspect 接口访问, 不会泄露 内部状态。 看 src/cache-debug.ts 第 144 行的 createCacheDebugTracker: 

export function createCacheDebugTracker(): {
  inspect(input: { messages, tools }): CacheDebugState;
} {
  let lastSnapshot: CacheDebugSnapshot | null = null;

  return {
    inspect(input): CacheDebugState {
      const current: CacheDebugSnapshot = { ... };
      const state: CacheDebugState = {
        current,
        changed: {
          systemPrompt: lastSnapshot ? lastSnapshot.systemPromptHash !== current.systemPromptHash : false,
          tools: lastSnapshot ? lastSnapshot.toolsHash !== current.toolsHash : false,
          stablePrefix: lastSnapshot ? lastSnapshot.stablePrefixHash !== current.stablePrefixHash : false,
        },
      };
      lastSnapshot = current;
      return state;
    },
  };
}

实现细节: 第一轮调用时 lastSnapshot 为 null, 三个 changed 都是 false (因为没有"上次" 可比, 强制 算 stable), 这是故意的: 第一次 LLM 调用 cache 必然 miss, 输出"stable" 是误导, 但教学版简化了"第一次不输出"。 生产版应该在第一轮输出 [cache] first call, prefix=initial 之类的初始标记。

formatCacheDebugLog: 单行可 tail

用途: 日志格式必须单行可 grep — 开发者 tail agent.log 找 cache 变化, 多行格式 不友好。

真实场景: 开发者跑 harness 100 轮对话, 想看 "什么时候 system prompt 变了"。 跑 grep '\[cache\].*changed' agent.log, 单行格式 一行一个时间点, 找到原因。

设计思想: 结构化日志 — 用 key=value 格式, 容易 grep, 容易解析。 看 src/cache-debug.ts 第 194 行的 formatCacheDebugLog: 

export function formatCacheDebugLog(state: CacheDebugState): string {
  const { current, changed } = state;
  const spStatus = changed.systemPrompt ? "changed" : "stable";
  const toolsStatus = changed.tools ? "changed" : "stable";
  const prefixStatus = changed.stablePrefix ? "changed" : "stable";
  const parts = [
    `[cache] systemPrompt=${spStatus}`,
    `tools=${toolsStatus}`,
    `prefix=${prefixStatus}`,
    `systemHash=${current.systemPromptHash}`,
    `toolsHash=${current.toolsHash}`,
    `msgs=${current.messageCount}`,
    `tools=${current.toolCount}`,
  ];
  return parts.join(" ");
}

实现细节: 输出 7 个字段 — 3 个状态 (stable / changed) + 2 个 hash (8 位) + 2 个数量 (消息数 / 工具数)。 第一轮 prefix=stable 看起来假, 但 0 个 changed 字段都为 false, 后续轮才有意义。 教学版不区分"first call" 标记。

主循环集成: 在哪 1 个时机 inspect

用途: 缓存调试的唯一集成点是 agent 主循环第 3 步 (call LLM) 之前。 调一次 tracker.inspect, 输出日志, 之后 LLM 调用本身不动 (LLM API 自己处理 cache)。

真实场景: agent.ts 里: const state = cacheTracker.inspect({messages, tools}); logger.info(formatCacheDebugLog(state)); const response = await llmClient.chat(messages, tools);

设计思想: 观测点 ≤ 1 — 缓存调试 只在 LLM 调用前 inspect 一次, 不在 LLM 调用后 (响应不改变 cache 状态), 不在 tool 调用前 (tool 调用不改变 system prompt), 不在 user 输入后 (user 输入进 history, 不进 system prompt)。 只有 LLM 调用前才有"前后对比" 的意义。

实现细节: 集成代码应该在 Composition Root 创建 tracker, 注入 agent; agent 第 3 步调 inspect + log。 默认可以 不传 tracker, agent 不输出 cache 日志 (用空对象模式, 类似 createNoopHookRunner)。

实战: 拿到日志后, 怎么找 cache miss 原因

上一节讲完怎么输出日志, 这一节讲拿到日志后 怎么用。 假设你已经跑完一个 100 轮的 agent, 拿到 agent.log 文件 (几百行), 接下来怎么定位"为什么 cache miss 这么多"?

3 步排查法 (从粗到细)

步骤 1 · grep changed, 找出所有变化点 

grep '\[cache\].*changed' agent.log

# 典型输出 (假设跑了 100 轮, 改了 5 次):
[cache] systemPrompt=changed tools=stable prefix=changed systemHash=abc12345 ...   (第 1 轮, 启动)
[cache] systemPrompt=stable  tools=stable prefix=stable  systemHash=abc12345 ...   (第 2-4 轮, 全部命中)
[cache] systemPrompt=stable  tools=changed prefix=changed systemHash=abc12345 toolsHash=def67890 ...   (第 30 轮, 改了 tools)
[cache] systemPrompt=changed tools=stable prefix=changed systemHash=fedcba98 ...   (第 80 轮, 改了 system prompt)
[cache] systemPrompt=stable  tools=stable prefix=stable  systemHash=fedcba98 ...   (第 81-100 轮, 稳定)

看到结果后, 立刻知道 3 件事:

  • 第 1 轮首次启动必然 changed (baseline), 忽略。
  • 第 2-4 轮 stable, 说明稳定 3 轮。
  • 第 30 轮 tools=changed, 改了 tools。
  • 第 80 轮 systemPrompt=changed, 改了 system prompt。

步骤 2 · 看每条 changed 前后的 git diff

看到第 30 轮 tools=changed 了, 但"为什么变" 不知道。 看 git log: 

git log --oneline --before="2026-06-15 14:30"

# 典型输出:
a1b2c3d  2026-06-15 14:25  fix: 给 run_bash 加新参数
d4e5f6a  2026-06-15 14:10  docs: 更新 README
...

a1b2c3d 在 14:25 commit, 第 30 轮在 14:30 (假设), 时间对得上 — tools=changed 是这次 commit 引起的。 打开 diff 看具体改了什么: 

git show a1b2c3d -- src/tools/run-bash.ts

# 看到: 给 run_bash 的 description 加了 10 个字符
- description: "跑 bash 命令"
+ description: "跑 bash 命令, 支持后台执行"

找到原因了: 改了一个 tool 的 description, 10 个字符, hash 变, cache miss 1 次。

步骤 3 · 评估这次变化是否值得

知道原因后, 问自己: 这 10 个字符的描述真的需要改吗?

  • 值得: 描述从"跑 bash 命令" 改成"跑 bash 命令, 支持 后台执行", 这是真功能差异, 改得对。 cache miss 1 次 是合理代价。
  • 不值得: 描述只是改了个错别字, 没必要, 立刻 git revert 恢复, 下次重启 cache 又命中。

具体例子: 一次完整的排查

假设用户跑 200 轮, 账单 ×3, 怀疑 cache miss 太多。 排查:

  1. grep changed: 找到 8 次 changed (200 轮里 8 次 cache miss)。
  2. 逐条看 git log: 8 次分别对应 8 个 commit。
  3. 逐条看 diff: 5 个是"故意改" (加新工具 / 改 description), 3 个是"无意改" (refactor 时手抖改了 system prompt)。
  4. 回滚 3 个无意改: git revert 3 个 commit, 下次重启 cache 重新累积。
  5. 留下 5 个故意改: 不可避免, 接受。

账单从 ×3 降到 ×1.5 (少了 3 次无意 miss)。

常见反模式: 改了 system prompt 不知道为什么

跑了 50 轮, 看到第 25 轮 systemPrompt=changed, 但 git log 没看到最近的 system prompt 改动。 这种最坑 — "我什么都没改, 为什么变了?"

排查方法:

  1. 查 memory 段: SessionStart handler 注入的 memory 段如果改了 (用户加了新 memory 文件), system prompt 就变。 看 ~/.memory/ 目录最近的文件时间。
  2. 查 skill 段: 加载了 skill (e.g. 调 load_skill("react")) 会把 skill 工具描述拼到 system prompt。 第一次加载必然变。
  3. 查 schedule / todo reminder: 启动时 todo reminder 注入到 user message (不进 system), 但 schedule 提醒可能进。 看 reminder 来源。

常见反模式: tools=changed 但没改 tool 代码

跑了 30 轮, 看到 tools=changed, git log 没看到 tool 文件改动。 排查:

  1. 查 ToolRegistry 注册顺序: 工具注册顺序变了 (e.g. index.ts 调整了 import 顺序), 数组顺序变了, 但 stableStringify 应该排除这个。 看到变了, 可能是 stableStringify 没用 (查 cache-debug.ts 配置)。
  2. 查 tool description 动态拼接: 某工具的 description 是动态拼的 (e.g. 包含当前时间戳, 包含 git commit hash)。 错 — description 必须稳定, 不能含动态数据 (第 10 章 "cache 影响" 一节讲过)。 改源码去掉动态部分。

真实 cache hit rate 怎么算 (教学版做不到)

上面说的都是"期望 cache 怎么命中" (基于 prefix hash), 不是 "真实 cache 命中多少" (基于 LLM 后端统计)。 真实 hit rate 要 读 LLM 响应头: 

// OpenAI 响应示例 (在 response.usage 里)
{
  "prompt_tokens": 8000,
  "completion_tokens": 200,
  "cached_tokens": 5000,  // ← 这就是真实 cache 命中的 token 数
}

// 真实 hit rate = cached_tokens / prompt_tokens
//              = 5000 / 8000 = 62.5%

// 教学版 cache-debug 看不到这个字段, 因为没解析 LLM 响应

教学版不能告诉你真实 hit rate 是多少。 我们只能 告诉你"harness 期望命中多少" (prefix 稳定 = 期望 100% 命中)。 真实值可能 60% (LLM 自己有 token 块粒度, 教学版简化成整个 prefix), 但 60% vs 100% 是同一个数量级, 排查"为什么会变" 够用了。

教学版 cache debug 的能力边界

  • 能告诉你: "上一轮 vs 这一轮, prefix hash 变了没, 哪段变了 (system prompt / tools / 整体), 何时变的 (轮次), 哪个 commit 引起的 (git log 对应)。"
  • 不能告诉你: "真实 cache 命中了多少 token, LLM 后端实际走了 cache 还是没走, 哪几个 token 块在 cache 里。"

看到日志后, 重点看 "哪段变了", 知道改了就够了。 真实 hit rate 等生产数据, 教学版负责排查链路, 不负责精度。

prefix changed 的 4 个常见原因

用途: 看到 [cache] prefix=changed 日志, 知道 cache miss, 但不知道为什么。 列 4 个 最常见原因, 帮开发者快速定位。

图 2 · prefix changed 的 4 个常见原因
原因 1 · system prompt 改了 (字面)
开发者改了 src/system-prompt.ts 源码, 拼出来的 system prompt 多了 / 少了一段。 git diff 能看到。
原因 2 · system prompt 改了 (动态)
memory 段加了新条目 (第 09 章), skill 切换了 (第 05 章), todo reminder 注入了状态 (第 03 章)。 看 memory/skill/todo 的变更。
原因 3 · tools 数组变了
注册了第 7 个工具 / 删了第 1 个工具 / 改了某个 tool 的 description。 toolsHash 变, prefix 跟着变。
原因 4 · tool description 改了 (微调)
"改了 1 个 tool 的 description 1 个字符", 字面看微小, hash 变, cache miss。 最容易忽视, 需要严格 review。

实现细节: 4 个原因对应 4 个修法 — (1) 改源码要 故意, 不要随手 commit; (2) 动态注入要走稳定路径 (memory 段 内容稳定 / skill 切换一次写死 / todo reminder 写在 user 而非 system); (3) 工具注册要稳定, 不要在 hot reload 时改顺序; (4) tool description 改动要走 PR review, 不随手 commit。

cache miss 的成本: 10x 不是危言耸听

用途: 让开发者直观感受"改一次 成本 ×10" 的量级, 不是空喊"重视 cache"。

真实场景: harness 跑 100 轮对话, system prompt 2k token, 工具定义 1k token, 稳定前缀 3k token。 OpenAI 的 prompt cache 按 token 块计费, 命中部分 ~10% 折扣, miss 部分 全价。 100 轮中, 1 轮 system prompt 改了, 后续 99 轮 cache 全部重新累积, 3k token × 99 = 297k 输入 token 按全价计费。 比正常 cache 命中多花 ~9 倍输入侧费用。

设计思想: 成本驱动设计 — 谈 cache 不谈钱是空谈, 应该让开发者算出改一次 多少钱。 一般经验: 100 轮对话改 1 次 system prompt, 账单 ×2 到 ×5 (取决于 cache 命中率算法); 1000 轮对话改 1 次, ×10 到 ×20 (累积效应)。

实现细节: harness 启动时输出"当前 system prompt hash + 当前 tools hash", 让用户记录 baseline。 之后 git commit 改了 system-prompt.ts, 重启看到 hash 变, 提醒 "知道这次改的代价吗?"。

fake test: 用稳定消息验证 hash 不变

用途: 缓存调试模块的测试不需要真 LLM, 用 fake messages / tools 即可 — 测的是"序列化 + hash + 对比" 的算法 正确性。

真实场景: 用户写测试验证"messages 相同, tools 顺序不同, toolsHash 相同" (稳定序列化的关键性质)。 用两个 不同顺序的 tools 数组, 调 stableStringify, 输出应该字节相等。

设计思想: 测不变量覆盖 3 个 性质 — (a) 稳定序列化对 key 顺序鲁棒, (b) hash 区分大小写 / 空格, (c) tracker 第一轮与后续轮的不同行为。 看测试示例: 

test("stableStringify 对 key 顺序鲁棒", () => {
  const a = { name: "foo", description: "bar", type: "user" };
  const b = { type: "user", description: "bar", name: "foo" };
  expect(stableStringify(a)).toBe(stableStringify(b));
});

test("hash 区分大小写", () => {
  expect(sha256("Hello")).not.toBe(sha256("hello"));
});

test("tracker 第一轮 changed=false, 第二轮相同输入也 false", () => {
  const tracker = createCacheDebugTracker();
  const input = { messages: [{ role: "system", content: "x" }], tools: [] };
  const s1 = tracker.inspect(input);
  expect(s1.changed.systemPrompt).toBe(false);  // 第一轮无 lastSnapshot
  const s2 = tracker.inspect(input);
  expect(s2.changed.systemPrompt).toBe(false);  // 第二轮 hash 相同
});

test("system prompt 改了, 第二轮 changed=true", () => {
  const tracker = createCacheDebugTracker();
  tracker.inspect({ messages: [{ role: "system", content: "v1" }], tools: [] });
  const s2 = tracker.inspect({ messages: [{ role: "system", content: "v2" }], tools: [] });
  expect(s2.changed.systemPrompt).toBe(true);
  expect(s2.changed.stablePrefix).toBe(true);  // 任何一个变, prefix 都变
});

实现细节: 4 个测试覆盖 (a) 稳定序列化核心, (b) hash 大小写敏感, (c) tracker 状态机, (d) 变化传导。 不 需要 mock crypto, 真实 SHA256 比 mock 更稳。

常见误解: stable 不等于 never change

误解 1: "stable 就应该永远不变?" 错。 stable 是"同一 session 内不变", 跨 session 可以变 (重启 hash 自然 变)。 看的是 session 内变化率, 不是绝对不变。

误解 2: "tools 顺序变了 hash 不变, cache 命中?" 不一定。 stableStringify 让 hash 顺序无关, 但 OpenAI 的 真实 cache看的是 tools 数组的实际顺序 (LLM 看到的 tools 顺序影响 prompt token, prompt token 影响 cache 块)。 稳定序列化只让 harness 观测逻辑 变稳, 不保证 OpenAI 也这么看。

误解 3: "日志输出 stable 就一定 cache 命中?" 错。 日志输出的是 harness 期望 cache 命中, 不 等于 OpenAI 实际 cache 命中。 真实命中率需要 LLM 响应头的 cached_tokens 字段, 教学版不假装读得到。

误解 4: "cache miss 就一定浪费钱?" 不一定。 OpenAI 有最小 cache 块 (1024 token 左右), 小于这个长度根本 不进 cache。 system prompt + tools 不到 1k token 的小项目, cache miss 也无所谓。 谈成本先看规模。

反例梯度: 4 个常见错误

反例 1 · 把 todo reminder 写进 system prompt

第 03 章的 todo 状态每轮变, 用户把"当前 todo 列表" 拼到 system prompt。 错。 system prompt 是稳定前缀, todo 状态 应该写在 user message (历史), 不是 system。 正确做法: 写 "使用 TodoManager 跟踪任务" (规则) 在 system, 写 "[todo] step 1 ✓ step 2 ..." (状态) 在 user message。

反例 2 · 每次 LLM 调用都重建 tools 数组

tools 数组应该创建一次, 在 Composition Root 注入 agent。 每次 LLM 调用前 [...tools] 浅拷贝, 字面看"内容相同", 但稳定序列化之后 hash 仍然相同 (这次 算稳), 但如果有人加了一个字段, 立刻 miss。 错。 一次性 创建, 全程复用, 严禁 hot reload 时重建。

反例 3 · 改 system prompt 不知道

开发者改了 src/system-prompt.ts 1 行, 没意识到 cache miss, 跑生产 1 个月账单 ×5, 才发现。 错。 harness 启动时必须输出当前 system prompt hash, 让 改动可见。 教学版要 console.log(formatCacheDebugLog(state)) 在 startup 时打一次。

反例 4 · JSON.stringify 直接 hash

直接 sha256(JSON.stringify(tools)), tools 数组顺序变了就 hash 变, 误报 cache miss。 错。 应该用 stableStringify 先稳定序列化, 再 hash。 稳定序列化对 key 顺序鲁棒, 数组保序 (因为 LLM 看的就是 这个顺序)。

Validation: 4 条不变量检验

  1. 稳定序列化对 key 顺序鲁棒: 同样内容不同 key 顺序的两个对象, stableStringify 输出字节相等。 验证: 单测覆盖 3 个字段乱序的情况。
  2. Hash 大小写敏感: "Hello"和"hello"hash 不同。 验证: 基础单测, 防止有人把 hash 改成 .toLowerCase() 破坏敏感性。
  3. Tracker 状态机正确: 第一轮 changed=false (无 lastSnapshot), 第二轮相同输入 changed=false (hash 一致), 第二 轮不同输入 changed=true (hash 不一致)。 验证: 3 个连续 inspect, 看 changed 字段。
  4. 变化传导: system prompt 变 → systemPromptHash 变 → stablePrefixHash 变 (因为 prefix = system + tools); tools 变 → toolsHash 变 → stablePrefixHash 变。 验证: 单测 改一边, 看 prefix 是否跟着变。

回望: 哪些原则在本章兑现了

  • 本地可观测: cache 调试跑在 harness 进程内, 不 调外部 API, 不假装读真实 cache hit rate。
  • 稳定序列化: key 排序, 数组保序, undefined 跳过, 与 JSON.stringify 兼容 undefined 行为。
  • 工厂 + 闭包: createCacheDebugTracker 把 lastSnapshot 藏在闭包, 外部只通过 inspect 接口访问。
  • 结构化日志: 单行 key=value 格式, 可 grep, 可解析。
  • 成本驱动设计: 谈 cache 必谈钱, 不空喊"重视 cache"。

前瞻张力: 留给后续章节

真实 cache 命中率
教学版只看 harness 期望的稳定前缀, 不读 LLM 响应头的 cached_tokens。 真实命中率需要 LLM 客户端注入 cache 统计 钩子, 这是 LLM 客户端的功能, 不是 cache-debug 模块。 P2 阶段可以让 llm-client 暴露 cache 命中率, cache-debug 接入 做"实际 vs 期望" 对比。
Cache 失效自动恢复
prefix 变化是"知道为什么变" 的问题, 不是"自动恢复" 的 问题。 真要自动恢复, 需要"检测变化 → 判断是否意外 → 提示 开发者" 的完整流程, 这是 dev 工具范畴, 不是 harness 范畴。
Token 块粒度 cache
OpenAI / Anthropic 的 cache 是按 token 块 (1024 token 左右) 粒度, 不是整个前缀。 教学版简化成"整个前缀", 工业级要 算"哪一段 token 块在 cache 里"。 留给后续优化。

Prompt Card (本章任务)

Prompt Card · 第 10 章

目标: 给 harness 加 cache debug 模块, 每轮 LLM 调用前 hash (system prompt + tools), 输出 stable / changed 标记。

场景: 开发者改 src/system-prompt.ts 1 行, 重启 harness, 看到日志 [cache] systemPrompt=changed tools=stable prefix=changed systemHash=abc12345。 知道 system prompt 改了, 决定是"故意改" 还是"无意改"。

模块: src/cache-debug.ts (新) 暴露 createCacheDebugTracker() + formatCacheDebugLog(state); src/agent.ts (改) 第 3 步 call LLM 前 inspect + log; src/index.ts (改) Composition Root 创建 tracker。

边界 (LLM 必须遵守的 checklist):

  • 稳定前缀定义: system prompt content + tools JSON, 不含 history
  • 稳定序列化对 key 顺序鲁棒 (sort), 数组保序, undefined 跳过
  • Hash 用 SHA256 前 8 位 hex, 不用完整 64 位
  • Tracker inspect 只在 LLM 调用前调一次, 不在 tool 调用前 / 后调
  • 第一轮 changed 全部为 false (无 lastSnapshot, 不假装 stable)
  • 日志格式单行 key=value, 可 grep, 不多行
  • 不假装读真实 cache hit rate, 只算"期望稳定"

验证 (用 fake messages + vitest, 逐条断言):

  • stableStringify 对 key 顺序鲁棒: 3 字段乱序, 字节相等
  • hash 大小写敏感: "Hello" vs "hello" hash 不同
  • tracker 状态机: 第 1 轮 changed=false, 第 2 轮相同输入 changed=false, 不同输入 changed=true
  • 变化传导: system 变 → prefix 变, tools 变 → prefix 变

本章练习

  1. 故意把 todo reminder 拼到 system prompt (第 03 章), 跑 100 轮 对话, 看 cache 日志是否全程 prefix=changed。
  2. 故意用 JSON.stringify(tools) 直接 hash, 跑测试 调换 tools 数组顺序, 看 hash 是否变了 (应该稳定序列化让 hash 不变)。
  3. 故意不在 harness 启动时输出 baseline hash, 改 system-prompt.ts 不重启 harness, 看是否能发现 "system prompt 改了" (不能, 缺观测)。
  4. 故意在 agent.ts 多处调 tracker.inspect (tool 调用前 + LLM 调用前 + LLM 调用后), 看日志是否混乱 (3 个调用点应该只在 LLM 调用前 1 个)。

本章小结

Cache debug 是给 prompt cache 的可观测性, 不 假装读真实 cache hit rate。 核心是 5 个设计:

  • 稳定前缀边界: system prompt + tools, 不含 history。 todo reminder / skill 状态进 history, 不进 system。
  • 本地 hash: SHA256 前 8 位, 不调外部 API, 不 读 LLM 响应头。
  • 稳定序列化: key 排序, 数组保序, undefined 跳过, 避免"顺序变了就误报 miss"。
  • 单点 inspect: 只在 LLM 调用前调一次, 其它时机 无意义。
  • 单行结构化日志: [cache] systemPrompt=stable ..., 可 grep, 可解析。

    • 下一章 (第 11 章) 展开 harness 跑生产时的异常恢复 — 7 类错误 (LLM 4xx/5xx / 工具 throw / JSON parse 失败 / 上下文 撑爆 / user 取消 / network / unknown) 各自的恢复动作。
