第 10 章 · Prompt Cache 友好布局

稳定前缀 + 动态状态: Prompt Cache 友好布局

前面九章让 harness 拥有完整能力, 但每次 LLM 调用都从头发送完整 messages, token 成本居高不下。这一章系统讲 prompt cache 友好布局: 哪些进稳定前缀, 哪些走 reminder, 哪些靠 history 自然增长。基于前九章的所有设计选择 (memory 锚点、reminder 标签、tool 描述作为稳定前缀) 总结出完整原则。

在第 09 章基础上改了什么

这一章不新增模块, 而是基于已有的 memory (第 09 章) / system prompt / tools (第 02 章) / reminders (第 03/04/05 章) / history (第 01 章) 重新组织 prompt 拼装顺序, 最大化 LLM provider 的 prompt cache hit rate。 改动集中在 2 个文件: src/system-prompt.ts (新, 之前散落在 history.ts) 和 src/stable-context.ts (新, 负责 stable snapshot 的 invalidate 逻辑)。

1. src/system-prompt.ts: System prompt 组装 (新) 2. src/stable-context.ts: Stable context snapshot (新) 3. src/cache-debug.ts: Cache hit rate 监控 (新)

作者怎么想的: 这一章的思考链

想清楚现象
长任务下, harness 跑 50 轮 LLM 调用, 总 token 成本是单轮的 50 倍。 但实际上 system prompt + tool 描述 + 前 20 轮 messages 几乎不变。 现象是"harness 重复发送相同内容, 浪费 token"。
想反例
最朴素的反例是"system prompt 里塞 TODO 状态 / skill 状态 / memory 当前值"。这有两个问题: 一是每次 LLM 调用 system prompt 字符串不同, 整个 prompt cache 失效, 二是 token 成本按"全量" 计费, 不按 "增量" 计费。
想接口和不变量
接口: interface SystemPromptProvider { build(query), buildTurnReminders(query) }。 不变量四条: (1) system prompt 拼装顺序固定, 不依赖 query 内容, (2) 动态状态 (TODO / skill 激活集合) 走 reminder 消息, 不进 system prompt 字符串, (3) tool 描述 (按 skill 集合) 在一次 run() 内一旦确定就稳定, (4) history 是自然增量, 不试图"重写为压缩态"。
想怎么验证
fake LLM 跑 3 轮, 每次记录 system prompt + tools 字段的 hash, 断言前 2 轮 hash 相同 (稳定前缀), 第 3 轮 (有 reminder 注入) hash 仍可不同但 tools 字段不变。

先观察: 两段故意有气味的实现

观察 1 · 动态状态进 system prompt

// 教学简化版, 仓库真实实现见 GitHub 永久链接
// 错误: 动态状态拼进 system prompt
const systemPrompt = `${basePrompt}\n\nCurrent TODO: ${JSON.stringify(todos)}\n\nActive skills: ${activeSkills.join(", ")}`;

问:为什么不直接拼字符串?

答:每次 LLM 调用的 system prompt 字符串都不同, 整个 prompt cache 失效。LLM provider (Anthropic / OpenAI) 的 cache 是按 前缀字符串匹配, system prompt 一变, 后面所有 token 都要重新计费。 正确做法: 动态状态走 reminder (user 消息), system prompt 保持稳定。

观察 2 · tools 字段每次都重新拼

// 教学简化版
const tools = activeSkills.tools().map(t => ({ name: t.name, description: t.description, schema: t.schema }));
// 即使 skill 集合没变, 每次拼出的对象字面量都不同 (引用不同)

问:对象字面量内容相同, 为什么 cache 仍然不命中?

答:JSON 序列化后内容相同, 多数 provider 是按 JSON 字符串前缀匹配, 不是对象引用。问题是: 如果代码逻辑 (例如 skill 加载顺序变化) 让 JSON 字符串顺序变了, cache 也会失效。 正确做法: tools 数组的拼装顺序稳定, 加载 skill 时不重新排序。

稳定前缀 / 动态状态 / 自然增量 三段布局

这一节是本章的核心。基于前九章的设计选择, 整理出一份"哪些进哪里" 的清单:

稳定前缀 (system prompt + tools)

  • Base system prompt: 静态指令, 在 history.setSystemPrompt() 时一次性写入, 后续不变。
  • Memory 锚点: "# User Memory" / "# Project Memory" 标题, 加上当前值 (变也只变这一段, prefix 锚点稳定)。
  • Tool 描述 (按 skill 集合): 一旦 skill 集合在 run() 内确定, tools 数组不再变化。

动态状态 (reminder 消息)

  • TODO 状态: 每轮 reminder 注入, 标签 <system-reminder source="todo">。
  • Skill 激活集合变化: 加载新 skill 后追加 reminder 描述。
  • 子智能体输出: 父 agent 调 spawn_subagent 后追加 user 消息 (子智能体输出)。
  • Async run 通知: 第 13 章, 后台 agent 完成后注入 reminder。

自然增量 (history)

  • 用户消息: 每次 run() 追加一条 user message。
  • 助手消息: 每次 LLM 响应追加一条 assistant message。
  • 工具结果: 每次工具执行追加一条 role: "tool" 消息。

history 是 LLM 视角的"对话上下文", 不试图"重写为压缩态"。压缩只 发生在 prepareMessages() 阶段 (第 06 章), history 内部保留原始。

stable snapshot: 显式声明"这部分稳定"

当 LLM 修改了 memory (例如 save_memory 工具调用) 时, system prompt 中的 "# User Memory" 段会变。如果每次 save_memory 都让 system prompt 变, cache 仍然失效。stable-context 模块解决这个问题: 在 memory 内容 之外, 把 "# User Memory" / "# Project Memory" 标题与"当前值"作为 一个 snapshot, 只有当 snapshot 内容真的变了 (字符串 hash 不同) 才 更新 system prompt, 否则复用上次的 system prompt 字符串。 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export function createStableContextManager(history: History) {
  let lastSnapshot = "";

  return {
    buildMessages({ currentQuery, recentFiles }): Message[] {
      const userMemory = memoryStore.renderForPrompt("user");
      const projectMemory = memoryStore.renderForPrompt("project");
      const snapshot = `# User Memory\n${userMemory}\n\n# Project Memory\n${projectMemory}`;

      if (snapshot !== lastSnapshot) {
        // 真的变了, 更新 system prompt
        const base = history.getSystemPrompt() ?? "";
        history.setSystemPrompt(`${base}\n\n${snapshot}`);
        lastSnapshot = snapshot;
      }
      return [];  // 稳定, 不注入 user 消息
    },
    notifyFileChanged(path: string): void {
      // 文件改了, 相关 project memory 可能 stale, 下一轮重新渲染
      lastSnapshot = "";  // 强制刷新
    },
  };
}

关键设计: lastSnapshot 是闭包内状态, 比较的是字符串内容 (不是对象引用)。这意味着: 即使 LLM 频繁调 save_memory 但 value 相同, snapshot 不变, cache 仍然命中。

cache hit rate 监控

cache-debug 模块记录每次 LLM 调用的 cache 命中情况。Anthropic / OpenAI 的 API 响应里有 cache_read_input_tokens / cache_creation_input_tokens 字段, harness 解析后写到 日志和 metrics, 用于评估布局效果。 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
function trackCacheUsage(response: LLMResponse) {
  if (response.usage.cacheReadTokens > 0) {
    cacheDebugTracker.recordHit(response.usage.cacheReadTokens);
  } else {
    cacheDebugTracker.recordMiss();
  }
}

反例梯度

新手错法 · A

常见错误:动态状态拼进 system prompt 字符串。

为什么错:每次 LLM 调用 system prompt 都不同, 整个 cache 失效, token 成本涨 3-5 倍。

正确做法:动态状态走 reminder 消息 (user 角色), system prompt 字符串保持稳定。

中级错法 · B

常见错误:save_memory 每次都让 system prompt 变。

为什么错:即使 memory value 没变, 重新渲染也会让 system prompt 字符串变, cache 失效。

正确做法:stable-context 比较 snapshot 字符串 hash, 内容相同不更新 system prompt。

高级错法 · C

常见错误:tools 数组每次调用都重新 sort, 即使 skill 集合没变。

为什么错:JSON 序列化顺序变了, cache 不命中。

正确做法:tools 数组按注册顺序, 加载 skill 时追加到末尾, 不重新排序。

边界错法 · D

常见错误:项目级 memory 在新项目自动失效, 但 harness 不通知 LLM。

为什么错:LLM 看不到 system prompt 变化, 继续引用旧项目的事实。

正确做法:stable-context 在项目切换时 invalidate, 重新渲染并注入 "Project context changed" reminder。

回望第 00–09 章: 哪些原则在本章兑现了

  • 稳定前缀 / 动态状态 / 自然增量 三段布局:这是前九章所有设计选择的总结, 不是新发明, 而是"显式命名"。
  • 事实与视图分离:memory 是"长期事实" (第 09 章), history 是"对话上下文" (第 01 章), reminders 是"动态状态" (第 03 章)。三者职责严格分离。
  • 稳定 prefix 锚点:"# User Memory" / "# Project Memory" / "<system-reminder source='X'>" 都是锚点, 内容可换, 锚点本身稳定。
  • 不污染 system prompt 字符串:所有动态状态走 reminder / history, 永不拼进 system prompt 字符串。

Prompt Card (本章任务)

Prompt Card · 第 10 章

目标:基于前九章的设计选择, 显式落实 prompt cache 友好布局: 稳定前缀 (system prompt + tools) / 动态状态 (reminders) / 自然增量 (history)。

场景:长任务跑 50 轮, 前 20 轮 system prompt + tools 字符串 hash 相同, cache hit rate 应当 > 80%, 单轮成本下降 3-5 倍。

模块: src/system-prompt.ts (新) 集中管理 system prompt 拼装; src/stable-context.ts (新) 暴露 createStableContextManager(); src/cache-debug.ts (新) 暴露 createCacheDebugTracker(); src/agent.ts prepareMessages 接入 stable context。

边界 (LLM 必须遵守的 checklist):

  • 动态状态 (TODO / skill 激活集合 / 子智能体输出) 永不拼进 system prompt 字符串

  • tools 数组按注册顺序, 加载 skill 时追加到末尾, 不重排
  • cache debug 调用全部在副作用边界 (try / catch 之后), 不在主流程

本章小结

本章系统讲 prompt cache 友好布局: 稳定前缀 (system prompt + tools) / 动态状态 (reminders) / 自然增量 (history)。基于前九章的设计选择 显式命名这套布局, 并用 stable snapshot + cache debug 监控来保证 cache hit rate。下一章 (第 11 章) 我们会处理"LLM 调用失败" 的 情况——Recovery, 在网络异常 / 输出截断 / context overflow 时 自动恢复, 不让 harness 因单次失败就崩。

下一章伏笔

第 10 章让 harness 在长任务下保持稳定成本, 但 LLM 调用本身可能 失败 (rate limit、timeout、context overflow、JSON 损坏)。下一章 Recovery 模块会分类失败模式, 在不同失败下采取不同策略: 重试 退避、强制压缩、续写、终止, 让 harness 在面对不确定的 LLM 时 仍然能持续工作。
