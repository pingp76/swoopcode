第 06 章 · context 撑爆怎么办

压缩: Normalize / Block / Compress 三层防线

第 05 章的 Skill 让工具数不再撑爆 system prompt, 但 messages 本身 仍然会无限增长: 用户多轮对话 + 工具结果累积, 上下文很快超出窗口。 这一章引入三层压缩: Normalize (合并相邻同角色消息) → Block (按消息 block 分组) → Compress (P0 衰减 + P1 即时 + P2 全量), 让 harness 在长任务下保持稳定的 context 长度。 读完后, 你能讲清 "为什么消息 block 是压缩的原子单位", 并能用 fake LLM 验证 P0 衰减 截断后的消息 block 不被破坏。

先看消息结构: block 是什么, 压缩针对哪部分

在讲三层压缩之前, 必须先知道 "压缩针对谁"。 一次跑完的对话, 在 harness 内部是一条一条的 message, 我们用 LLM 视角把 message 排成一列, 如下: 

// 一次对话的 messages 数组 (简化展示, 实际 100+ 条)
[
  { role: "user",      content: "帮我看下 src/agent.ts" },
  { role: "assistant", content: "好的, 我先读一下", tool_calls: [{ id: "c1", function: { name: "run_read", arguments: '{"path":"src/agent.ts"}' }}] },
  { role: "tool",      content: "import xxx ...", tool_call_id: "c1" },
  { role: "assistant", content: "读完了, 我帮你改", tool_calls: [{ id: "c2", function: { name: "run_write", arguments: '...' }}] },
  { role: "tool",      content: "OK, 写完", tool_call_id: "c2" },
  { role: "assistant", content: "改完了" },
  { role: "user",      content: "再帮我看下 src/tools.ts" },
  // ... 继续累积
]

这一列 messages 里有 3 种 role: user (用户发的), assistant (LLM 发的, 可能带 tool_calls), tool (工具执行结果, 必带 tool_call_id)。 LLM 看的对话就是这 3 种角色 交替出现的序列。

block = 压缩操作的最小单位。 我们把上面那列 messages 按"语义完整" 切分成 3 类 block:

图 1 · 一次对话被切成 3 类 block
text block (纯对话)
组成: 只有 user + assistant 消息, 无 tool_calls。
例子: 上面例子里的 {role: "user"} 和 {role: "assistant", content: "改完了"} (无 tool_calls) 组成一个 text block。
特点: 多个连续 user 消息可以合并成一条。
tool_use block (工具调用 + 结果)
组成: 1 条 assistant 消息 (含 tool_calls) + 所有对应的 tool 消息。
例子: 上面例子里 assistant 调 run_read + 对应的 tool result = 1 个 tool_use block (3 条消息组成)。
特点: 不可拆分, 拆开破坏 tool_call_id 配对。
summary block (P2 压缩产物)
组成: 1 条 assistant 消息, 内容是 P2 字符串拼接出来的摘要文本, 不调 LLM。
例子: "User: 帮我看下 agent.ts | Assistant: 读完了 | Tool: run_bash(cat agent.ts) → [前 100 token]" 整段文字。
特点: P0 衰减不再处理 summary block (它本身就是压缩过的)。

压缩针对哪些 block, 不针对哪些?

block 类型 P0 衰减 P1 即时 P2 全量
text block 不衰减 不即时 参与 P2 (被总结)
tool_use block ✓ 衰减 (截断旧 tool result) ✓ 即时 (大输出写文件) 参与 P2 (被总结)
summary block 不衰减 不即时 不参与 (已压过)

压缩后, 哪些会变 / 哪些不变?

  • 会变: tool_use block 的 tool result content (P0 截断 / P1 存文件), text + tool_use block 被 P2 整段总结成 summary block。
  • 不变: tool_use block 的 assistant 消息 (含 tool_calls 和 id), summary block 本身, 整个 system prompt + 工具描述 (system prompt 在第 10 章专门讲, 这里不动)。

压缩绝不改变: tool_call_id 配对, role 字段, assistant 消息里的 tool_calls 数组。 只能改 tool result 的 content。

真实失败故事: 200 轮后 LLM 失忆

写代码之前, 先看一段真实长任务的失败。 跑一段对话: 用户让 agent "分析 50 个文件, 总结常见问题"。

  1. 第 1-10 轮: 调 run_bash 5 次, 调 run_read 10 次, history 累积 8000 token。 LLM 还在跟踪。
  2. 第 25 轮: messages 累积到 200K, LLM 反应开始慢, 偶尔 重复调之前调过的工具。
  3. 第 50 轮: history 累积 80000 token。 LLM 严重失忆, 总结时漏掉 5 个文件, 重复 3 个文件的判断, 修复方案质量大幅下降。
  4. 第 100 轮: history 累积 50000 token, 单次 prompt 超过窗口, LLM SDK 报 400 错误, agent loop 整个崩。

朴素想法 1: "让 LLM 写总结, 把之前的工具结果总结成一段?" 能缓解, 但不根治 — 总结本身也要占 token, 总结会丢失细节, LLM 基于总结给出的判断质量下降。

朴素想法 2: "压缩工具结果?" 压缩针对"单条工具结果太长", 不解决"多步任务的状态丢失" 问题。

正确做法: 三层压缩。 P0 衰减 (轻量, 每轮自动跑) + P1 即时 (单 tool result 太大, 立刻压缩) + P2 全量 (超阈值, 整段 总结)。 三层互补, 各管各的边界。

三层压缩的边界

三层不是"3 个选项", 是"3 道防线, 各管各的时机":

层 何时触发 压缩什么 谁负责 是否可逆
P0 衰减 每轮 LLM 调用前自动 超过 decayThreshold 的旧 tool result 截断 compressor 内部 不可逆 (丢掉细节)
P1 即时 单 tool result 超过阈值 大 tool result 写入 OutputStore, 返回 preview + output_id tool executor 内部 可逆 (通过 output_id 重读)
P2 全量 整个 history 超过阈值 早期 K 个 block 整体压缩成 1 段 summary compressor 内部, 纯字符串拼接 (不调 LLM) 不可逆 (摘要会丢细节)

3 个关键差异:

  1. 触发时机: P0 每轮自动, P1 工具返回时, P2 整个 history 超阈值。 各管各的, 不重叠。
  2. 压缩粒度: P0 单 tool result 截断, P1 单 tool result 存文件, P2 整段 history 总结。 粒度递增。
  3. 是否可逆: P0 不可逆, P1 可逆 (通过 output_id 重读), P2 不可逆。 这是有意的设计 — P1 给用户"后悔" 的机会, P0/P2 默认不可逆。

消息 block 详解: 为什么 tool_use block 不能拆

用途: 既然 block 是压缩的最小单位, 必须保证 block 内部"语义完整"。 tool_use block 是最严格的 — 4 条消息必须一起 处理, 拆了 LLM 看到错位消息。

真实场景: 假设 LLM 在第 5 轮调了 3 个工具 (run_bash + run_read + run_write), 这 3 个 tool_call + 3 个 tool_result 组成 1 个 tool_use block: 

// 1 个 tool_use block = 4 条消息, 不可拆
block = {
  kind: "tool_use",
  messages: [
    { role: "assistant", tool_calls: [
      { id: "c1", function: { name: "run_bash",  arguments: "..." } },
      { id: "c2", function: { name: "run_read",  arguments: "..." } },
      { id: "c3", function: { name: "run_write", arguments: "..." } },
    ]},
    { role: "tool", tool_call_id: "c1", content: "bash 输出" },
    { role: "tool", tool_call_id: "c2", content: "read 输出" },
    { role: "tool", tool_call_id: "c3", content: "write 状态" },
  ],
}

为什么不能拆? 假设压到 token 限制, 我们拆成 "前 2 条 + 后 2 条", 那 LLM 看到的就是: "assistant 调了 3 个工具, 但只收 到 2 个 tool result"。 messages 序列错乱, OpenAI API 可能直接 报 400。

压缩 tool_use block 的正确做法: 整体保留 4 条消息 的结构, 只改 tool 消息的 content (P0 衰减时换成 "[Decayed]" 短串, P1 即时换成 preview + output_id)。

P1 即时压缩: 大 tool result 存文件

P1 是最常用的压缩, 在 tool executor 内部完成。 解决什么问题? 单条 tool result 太大 (例如 cat large.log 返回 5MB), 直接放 history 一次就撑 爆 context。 啥时候触发? tool result content 长度超过阈值 (默认 2000 字符)。 

// 教学简化版
async function runBash(args) {
  const result = await exec(args.command);
  if (result.stdout.length > 2000) {
    // 大输出: 写文件, 返回 preview + output_id
    const outputId = await outputStore.write(result.stdout, { source: "bash", toolCallId: args._toolCallId });
    return {
      content: `[Output truncated, ${result.stdout.length} chars]\n[Preview]:\n${result.stdout.slice(0, 500)}\n...[use run_output_read with output_id: ${outputId} to read full output]`,
      // 不标 error, 这是正常的"输出太大" 情况
    };
  }
  return { content: result.stdout };
}

GitHub · src/tools/run-bash.ts P1 即时压缩实现 (L1)

4 个细节:

  1. 写文件后返回 preview + output_id: LLM 看到 preview 知道大概内容, 看到 output_id 知道如何重读。
  2. 不标 error: 这是正常情况, 不是工具失败。 LLM 收到 不会误以为命令出错。
  3. preview 长度固定 500 字符: 给 LLM 足够的"第一印象", 又不会撑爆单条 tool message。 500 字符大约 100-200 token, 合理。
  4. content 含完整说明: "Output truncated, N chars" + "Preview: ..." + "use run_output_read with output_id: X"。 LLM 看到就知道下一步怎么读完整内容。

为什么 P1 是可逆的? LLM 收到 output_id 后可以调 run_output_read(output_id) 读完整内容 (第 13 章展开)。 这是"后悔药" — 用户让 LLM 看完整 5000 行日志, LLM 不会因为压缩 而丢失信息。

P0 衰减: 截断旧 tool result

解决什么问题? 长任务跑 100 轮, 早期轮 (例如第 1-10 轮) 的 tool result 已经"被 LLM 看到并用过了", 不再重要, 但还在 history 里占 token。 P0 把这些"老掉牙" 的 tool result 截断, 节省 context。

啥时候触发? 每次 LLM 调用前自动跑, 不需要 LLM 介入。 触发条件: tool_use block 距离当前轮次超过 decayThreshold (默认 5 轮)。

具体例子: 跑完 10 轮对话, 触发 P0 前后

先讲清"轮次" 和"block" 的关系: 1 轮对话 = LLM 调 1 次 chat() + 跑 N 个 tool + 写 N 个 tool message。 这 1 轮里 LLM 调了 3 个 tool, 就产生 1 个 tool_use block (1 条 assistant + 3 条 tool)。 所以 10 轮对话可能产生 5-10 个 block, 不等于 10 个 block。

P0 的 age 是按"轮次" 算, 不是按"block 序号"。 每个 block 记着自己产生时的轮次 lastLoopIndex, P0 看 currentLoopIndex - block.lastLoopIndex > 5 就截断。 跑得越久, 旧 block 距当前轮次越大, 越容易被截。

// 教学简化版
function decayOldBlocks(blocks: Block[], currentLoopIndex: number, decayThreshold: number): Block[] {
  return blocks.map(block => {
    if (block.kind !== "tool_use") return block;
    if (currentLoopIndex - block.lastLoopIndex < decayThreshold) return block;
    // 超过 decayThreshold 轮的 tool_use block, 截断 tool result content
    return {
      ...block,
      messages: block.messages.map(m =>
        m.role === "tool"
          ? { ...m, content: `[Decayed, ${m.content.length} chars, use output_id if needed]` }
          : m
      ),
    };
  });
}

GitHub · src/compressor.ts P0 衰减实现 (L1)

关键点:

  1. age 按"对话轮次" 算, 不是按"block 序号": 10 轮对话 可能产生 5-10 个 block, age = currentLoopIndex - block.lastLoopIndex = 多少轮对话之前产生的。 一个 tool_use block 作为一个整体被衰减, 内部所有 tool 消息一起截断。
  2. 截断而非删除: content 替换成 "[Decayed, N chars]", 不是从 messages 里删除。 原因: tool_call_id 配对不能丢, 删了 LLM 看到"assistant 调了 3 个工具, 只收到 2 个 tool" 又错乱。
  3. summary block 不衰减: 它本身就是压缩过的, 衰减没意义。 如果 summary 也衰减, 等于二次压缩, 信息丢失加速。

❌ / ✅: 不要按"消息条数" 衰减, 按"block" 衰减。 

// ❌ 按消息条数, 拆 tool_use block
function decayByMessage(messages, currentLoop) {
  for (let i = 0; i < messages.length; i++) {
    if (messages[i].role === "tool" && currentLoop - i > 5) {
      messages[i].content = "[Decayed]";   // 拆了 tool_use block
    }
  }
}

// ✅ 按 block, tool_use block 整体处理
function decayByBlock(blocks, currentLoop) {
  return blocks.map(block => block.kind === "tool_use" && currentLoop - block.lastLoopIndex > 5
    ? truncateToolResults(block)   // 整个 block 截断, 不拆
    : block);
}

一个细节: 截断过的 block 下一轮还会再截吗?

注意到没有: 上面 truncateToolResults 没看 block 之前 截没截过, 每次都重截一次。 这是有意的设计, 但 容易引起"是不是重复操作" 的疑问。 解释:

为什么每次都重截

  1. decay 是"视图级" 压缩: P0 不改 history 数据, 只在 每次 LLM 调用前, 临时构造一份"用于这次 LLM 调用" 的 messages 列表。 截断是 view-time 操作, 下一轮重新构造时, 又从原始 history 重新截断。 history 本身永远是完整的。
  2. 避免持久化状态: 如果加一个"这个 block 已经截过" 的 标记, 那就要在 history 内部维护状态, 跟"压缩不能改 history" 的原则冲突。 每次重截是"无状态" 实现, 简单可靠。
  3. 成本几乎为 0: truncateToTokens(content, 100) 只是 一个字符串切片, 不调 LLM, 不写文件。 跑 100 轮, 重截 100 次, 总耗时 < 1ms。

具体例子: 同一个 block 跑 10 轮, 每次都"重新截"

图 2.5 · 同一个 block 跨 10 轮的"重截" 行为
第 1 轮 (block 刚产生, age=0, 不截)
tool(c1, "[原始 5000 chars 内容]")
第 7 轮 (age=6, 第一次截)
tool(c1, "[截断到 100 chars: "first 100 chars..."]")
↑ history 仍存 5000 chars 原文, 这次发给 LLM 的视图是截断版
第 8 轮 (age=7, 重截一次)
tool(c1, "[截断到 100 chars: "first 100 chars..."]")
↑ 看起来"内容一样", 但实际是 history 原文 + 截断函数 重新算出来
第 16 轮 (age=15, 还在重截)
tool(c1, "[截断到 100 chars: "first 100 chars..."]")
↑ 仍然 history 原文 (5000 chars) + 重新截断 (100 chars)

那 history 真实占内存多少?

永远是 5000 chars 全文, 不是"5000 → 100 → 100 → 100" 越来越小。 P0 不改 history, 只在 view-time 截。 这意味着 long-running 任务的内存里, history 一直累积原文 — 这也是为什么 还需要 P2 (整段总结) 来真正缩小 history。 P0 只解决"LLM 调用 的入参" 不爆, 不解决"内存里 history 爆"。

trade-off 总结

方案 每次重截 (当前实现) 标记"已截过" (假设实现)
history 改动 不改动 加 metadata 字段
实现复杂度 10 行 20+ 行 + 测试
CPU 成本 几乎为 0 (字符串切片) 0 (但要维护状态)
history 内存 原文持续累积 截断版 (但 P2 之前 memory 仍爆)
可恢复性 任何时候 history 都是完整原文 截断后原文丢失 (除非另存文件)

当前实现"每次重截" 赢在简单 + 可恢复, 输在 history 内存无界 (这要靠 P2 整段总结解决, 不靠 P0 修)。 这就是为什么 P0 + P2 一起用, 而不是只靠 P0。

P0 和 P2 的核心区别: 改不改 history

这个问题容易误解, 我们专门讲一下。 P0 和 P2 都是"压缩", 但 对 history 的影响完全不同:

P0 衰减 P2 全量
改 history 数据吗? 不改, history 内存里永远是原文 改, summary 替换原文, history 真的缩小
持久化吗? 不持久化, 下次 LLM 调用又从原文重新截 持久化, summary 写回 history, 后续 50 轮都是这份 summary
history 内存 无界 (永远累积原文) 有界 (summary 替换后变小)
可逆吗? 可逆 (原文还在, 重截能换回) 不可逆 (原文被替换, 没了)

关键反直觉点: P2 改 history 这件事不总是发生, 取决于 P2 在哪条路径被调用:

  1. 正常 prepareMessages 路径 (每次 LLM 调用前): P2 只算 finalBlocks = compacted.blocks, 喂给当前 LLM 调 用。 不写回 history。 这次 P2 是"视图级", 跟 P0 一样不持久化。
  2. Recovery 路径 (LLM 报 context_length 错误, 第 11 章 展开): compactCurrentHistoryForRecovery 调 history.replaceEntries(newEntries), 真写回。 这时候 summary 真的进了 history, 后续 50 轮都是这份 summary。

为什么要分两条路径?

  • 正常路径不写回: 每轮 prepare messages 都调 P2, 如果 每次都写回, summary 会"叠加" — 第 1 轮 summary 后, 第 2 轮又 把第 1 轮的 summary 跟后面的对话一起再 summary, 反复压缩, 几轮 之后 summary 变成"summary 的 summary 的 summary" , 信息完全丢光。
  • Recovery 路径必须写回: LLM 报 context_length 错误, 说明 "再不缩小 history 整个 agent 死"。 这是真正紧急的兜底, 写回 history 防止下次 LLM 调用又报同样错。

那 P2 总结到底什么时候会真改 history?

只有 1 个场景: LLM 报 context_length 错误 (第 11 章 Recovery) 时。 正常 prepareMessages 跑 P2 只是"这次发给 LLM 的视图变短", history 内存里还是原文 — 这跟 P0 一样, 都是"视图级"。

这条边界很关键, 不然你看到 P2 以为它会真改 history, 写测试时 拿 history 内存去断言 summary, 永远 assert 不过。

P2 全量: 整段 history 总结

解决什么问题? P0 衰减治标不治本 — 长任务跑 200 轮, P0 截断了旧 tool result, 但新 tool result 不断累积, 整体还是 超窗口。 P2 是"重型武器", 把早期多个 block 字符串拼接成 1 段 摘要, 整段替换成 1 段文字。

关键反直觉点: P2 不调 LLM

这是 P2 最容易误解的一点, 先讲清楚: 教学版 P2 不调 LLM, 内部是纯字符串拼接. 每个旧 block 被压成一行 Tool: run_bash(args) → [truncated result], 多个旧 block 拼成一段摘要. 没有网络调用, 没有 token 费用, 耗时 微秒级. (跟第 14 章 Schedule 的 summaryPrompt 调 LLM 生成"用户友好摘要" 是不同的事, 别搞混 — Schedule 的 summary 是给 LLM 看的, P2 的 summary 是给 LLM 上下文用的.)

看 compressor.ts 的 compactHistory: 

// 教学简化版 (实际实现见 src/compressor.ts compactHistory)
function compactHistory(blocks: Block[]): CompactResult {
  const keepCount = cfg.compactKeepRecent;
  const recentBlocks = blocks.slice(-keepCount);   // 末尾 K 个保留
  const oldBlocks = blocks.slice(0, blocks.length - keepCount);   // 前面的全压

  if (oldBlocks.length === 0) return { blocks, summary: "" };

  // 字符串拼接, 不调 LLM
  const summaryLines: string[] = [];
  if (lastSummary) {                              // 上一次的 summary 在闭包内
    summaryLines.push(lastSummary);                // 直接拼进来当"前情提要"
    summaryLines.push("---");
  }
  for (const block of oldBlocks) {
    if (block.type === "tool_use") {
      const calls = extractToolCallSummaries(block.assistant);
      const results = block.toolResults.map(tr => `[${truncateToTokens(extractText(tr), 100)}]`);
      summaryLines.push(`Tool: ${calls.join(", ")} → ${results.join(", ")}`);
    } else if (block.type === "text") {
      const userText = block.user ? extractText(block.user) : "";
      if (userText) summaryLines.push(`User: ${userText}`);
    } else if (block.type === "summary") {
      // summary block 跳过: 它的内容已经在 lastSummary 闭包变量里,
      // 这里再 push 会导致同一份内容在 summary 里出现 2 次.
      // 详见 P2 重复压缩 bug 修复一节.
      continue;
    }
  }
  const summaryText = summaryLines.join("\n");

  lastSummary = summaryText;                      // 缓存, 供下次 P2 复用
  return { blocks: [summaryBlock, ...recentBlocks], summary: summaryText };
}

关键细节: lastSummary 闭包内缓存** (第 168 行 let lastSummary: string | undefined). 第二次 P2 时 把第一次的 summary 拼进去当"前情提要", 不会丢信息. 这是 P2 不需要调 LLM 的关键 — 上一次的摘要已经够"压缩", 这次 只需要把新加的旧 block 拼进去.

啥时候触发? 看 history 原文 (normalize 之后, P0 衰减之前) 的 token 估算是否超过 80% 窗口。 看代码 第 283 行: 

// src/agent.ts 第 283 行
const tokenEstimate = estimateMessagesTokens(normalized);   // ← normalized, 不是 decayed
if (tokenEstimate > threshold) { /* 触发 P2 */ }

为什么是 history 原文, 不是 P0 衰减后?

这是 P2 设计里最微妙的一点, 容易引起疑问, 我们专门讲一下。

方案 检测什么 结果
方案 A: 用原文 token (当前实现) estimateMessagesTokens(normalized), P0 之前 history 原文 8000 chars, 超 80% 窗口阈值, 触发 P2
方案 B: 用 P0 衰减后 token estimateMessagesTokens(decayed), P0 之后 P0 已经把旧 tool result 截到 100 chars, 总 token 大概 2000, 不触发 P2

选 A 不选 B 的理由 (5 个):

  1. history 真实在涨, 不能让 P0 掩盖: P0 截断只是 "发给 LLM 的视图" 变短, history 内存里永远存 8000 chars 原文 (上一节"重截" 讲过)。 如果按方案 B 检测, history 内存会无限 涨, 但 P2 永远不触发, 最终 OOM。 方案 A 看到 真实增长, 触发 P2 把 history 真的缩下来。
  2. LLM 单次调用的费用是真实的: 长任务跑 200 轮, 每轮都 P0 截断后, 看似 token 很少 — 但 LLM 单次调用的费用按"输入 token" 算, 截断到 100 chars 的 tool result 仍然算 100 chars 的费用 (P0 不是 P1, P0 没存文件, 截断版本也是要传给 LLM 的)。 所以"history 原文 8000 chars" ≈ "LLM 实际收到的 token", 不是 错觉。
  3. P0 + P2 各管一段: P0 解决"LLM 单次调用入参" 不爆 (用截断视图, 不改 history), P2 解决"history 内存" 不爆 (用 summary 替换原文, 真改 history)。 两者职责正交, 不能让 P0 替 P2 扛 (反之亦然)。 用方案 A 检测, P2 看到真实 history 大小, 该触发就触发。
  4. 实现简单, 语义清晰: estimateMessagesTokens 拿 normalized messages 直接算, 不依赖 P0 跑过没。 测试也好写 — 喂一段 8000 chars 的 messages, 调 P2 一次, 验证 summary 出现。
  5. 避免"P0 失败导致 P2 不触发": 假设 P0 有 bug, 旧 block 截断失败, content 仍是 5000 chars。 方案 B 检测的是 截断后的 (5000 chars) token, 仍然超阈值, 触发 P2 — 这条 反例说明方案 B 也能触发 P2, 但方案 A 更稳: 不管 P0 跑没跑、 跑成功没, 方案 A 看的是最直接的"history 现在多大"。

// 教学简化版 (实际实现见 src/compressor.ts compactHistory)
async function compactHistory(blocks: Block[], recentKeep: number): Promise<Block[]> {
  if (blocks.length <= recentKeep) return blocks;
  const toCompress = blocks.slice(0, blocks.length - recentKeep);
  const toKeep = blocks.slice(blocks.length - recentKeep);

  // 字符串拼接, 不调 LLM
  const summaryLines: string[] = [];
  if (lastSummary) {
    summaryLines.push(lastSummary);    // 上一次的摘要拼进来
    summaryLines.push("---");
  }
  for (const block of toCompress) {
    if (block.kind === "tool_use") {
      summaryLines.push(`Tool: ${extractToolCallSummaries(block.assistant)} → ${block.toolResults.map(r => `[${truncateToTokens(extractText(r), 100)}]`).join(", ")}`);
    } else if (block.kind === "text") {
      const userText = block.user ? extractText(block.user) : "";
      if (userText) summaryLines.push(`User: ${userText}`);
    } else if (block.kind === "summary") {
      // summary block 跳过, 避免重复压缩
      continue;
    }
  }
  const summary = summaryLines.join("\n");
  lastSummary = summary;   // 闭包内缓存, 供下次 P2 复用

  // 替换为 summary block + 最近的 block
  return [
    { kind: "summary", messages: [{ role: "assistant", content: summary }] },
    ...toKeep,
  ];
}

GitHub · src/compressor.ts compactHistory 实现 (L326)

5 个细节:

  1. 保留最近 K 个 block: 不全压, 留最近 6 个 block 保持近期上下文。 摘要本身也占 token, 压太狠反而失去信息。
  2. 不调 LLM, 纯字符串拼接: P2 内部只用 extractText + truncateToTokens + join("\n"), 不调 LLM, 耗时微秒级, 不花钱。
  3. 不写进 history (正常 prepareMessages 路径): 总结后的 summary block 作为"这次 LLM 调用的视图" 的一部分, 不持久化。 下次 prepare messages 看到原文 + 重新 P2。 (Recovery 路径才 写回, 见上一节"核心区别"。)
  4. 触发条件: history 超过 80% 窗口 (或配置阈值)。 太早触发浪费一点 CPU 算字符串, 太晚触发超窗口 (LLM API 报 400)。
  5. summary block 跳过 for 循环, 不重复压缩: oldBlocks 包含上一次的 summary_old block (因为 P2 正常路径不 写回 history, summary 一直在 blocks 数组里), 但 for 循环遇到 block.type === "summary" 时 continue 跳过 — 它的内容已经在 lastSummary 闭包变量里了, 这里再 push 会导致同一份内容出现 2 次. 这是 P2 不会"把总结 再总结" 的关键 — 避免信息丢失加速 + 避免 summary 越长越触发 P2 的恶性循环. 比如第一次 P2 后 history = [summary, b45-b50], 第二次 P2 时 oldBlocks = [summary, b1-b44] (length 45 > 6), 循环遍历时 summary block 被跳过, 只压 b1-b44.

P2 重复触发的成本: 每轮都跑怎么办

上节讲 P2 不调 LLM, 听上去"挺便宜"。 但还有一个问题: 既然每轮 LLM 调用前都跑 prepareMessages, P2 判断 tokenEstimate > threshold 每轮都跑一次 — 只要 history 原文还超阈值, 接下来 N 轮每轮都跑 P2. 这是重复操作, 真的没问题吗?

实际成本: 接近 0, 但有 1 个 bug 已修

  1. P2 本身微秒级: 字符串拼接 + 几个 truncateToTokens 调用 (也是字符串切片). 跑 1000 次 P2 总耗时 < 10ms, 比 LLM 调 用本身 (几百 ms) 慢 5 个数量级.
  2. 没有网络费用: 跟 P0 一样, P2 不调 LLM, 不付钱.
  3. 有 lastSummary 闭包内 cache (上一节讲过): 第二 次 P2 拼字符串时把上一次的 summary 拼进来, 不丢信息, 不重算.
  4. (已修) 不会重复压缩 summary 自身: P2 之前的实现有 1 个 bug — 第二次 P2 时, oldBlocks 包含上一次的 summary_old block, for 循环遇到 summary block 时 push 一次, 同时 if (lastSummary) 又把同一份内容 push 一次, 同一份内容出现 2 次. 修复: for 循环遇到 block.type === "summary" 时 continue 跳过, 靠 lastSummary 闭包变量承载. 详见下面 "P2 重复压缩 bug" 一节.

所以"不花钱, 不慢, 不丢信息, 不重复" — 工程上是 OK 的. 但仍然不优雅: 每轮都跑 P2 判断 + 重新字符串拼接, 看着别抳. 改进 方向 (留 P2 阶段, 教学版不实现):

改进方向 思路 取舍
(a) 增量估算 不重算全部 normalized token, 只算"上次 P2 之后又新加了多少 token". 没新加就不触发. 省了 N 次 P2, 但要维护"lastCompactedAtLoop" 状态. 状态多了容易飘.
(b) 频率限制 两次 P2 之间至少跑 5 轮. 简单粗暴. 省了 P2, 但中间 5 轮 history 仍然超阈值, LLM 收到的是"上次 P2 的 summary + 新 block", 视图可能比 80% 窗口还长一点.
(c) 真改 history P2 正常路径也写回 history. 下次 prepareMessages 看到的 normalized 已经小了, 自然不超阈值. 最干净的解法, 但失去"可逆"性质. 跟现在"P0/P2 视图级" 的设计哲学冲突.

结论: 教学版"每轮跑 P2 判断 + 重复字符串拼接" 是 教学友好的实现 — 简单 + 不花钱 + 不慢 + 闭包 cache 避免信息丢 失. 工业版可以做 (a) 增量估算, 但不是当前教学项目要解决的事.

已修的 bug: P2 重复压缩 summary 自身

历史: 之前教学版 P2 的实现里, 第二次 P2 触发时 oldBlocks 包含上一次的 summary_old block, 但代码同时 有两条路径拼 summary 内容:

  1. 第 1 条: if (lastSummary) { summaryLines.push(lastSummary); } (第 349-352 行) — 从闭包变量拿
  2. 第 2 条: for (const block of oldBlocks) { ... else if (block.type === "summary") { summaryLines.push(extractText(block.user)); } } (第 354 + 388-391 行) — 从 summary_old block 拿

后果: lastSummary (闭包变量, 第 432 行赋值) 跟 summary_old.user.content (上次 P2 产出的 block) 内容 完全相同, 两条路径各 push 一次, 同一份内容在新 summary 里出现 2 次. 多次 P2 后 summary 越长, 越 频繁触发 P2, 形成恶性循环.

修复: for 循环遇到 block.type === "summary" 时 continue 跳过, 靠 lastSummary 闭包变量承载, 不再从 block 里重复提取. 修改第 388-391 行: 

// 修复前
} else if (block.type === "summary") {
  summaryLines.push(extractText(block.user));   // 重复压缩!
}

// 修复后
} else if (block.type === "summary") {
  // summary block 由 lastSummary 闭包变量承载, 此处跳过避免重复
  continue;
}

修复后行为: 第二次 P2 时, for 循环遍历 oldBlocks (含 summary_old), summary_old block 被跳过; 但 if (lastSummary) 仍然 触发, 把闭包里的 summary 拼进来当"前情提要". 同一份内容 只出现 1 次.

为什么修这个很重要:

  1. 避免恶性循环: 重复压缩 → summary 越长 → 越频繁触发 P2 → 重复压缩更多. 修好后 summary 长度按"压缩比例" 增长, 不按"重复次数" 增长.
  2. 避免 token 浪费: 重复内容占 token 但不携带新信息, 多 花 1 倍 LLM 入参费用 (虽然 P2 不调 LLM, 但 LLM 收到更长的 summary, 仍然多花钱).
  3. 语义清晰: 一份"前情提要" 在 summary 里就该出现 1 次, 重复出现让 LLM 困惑"为什么同一个东西说两遍".

修复后, "P2 重复触发的成本" 一节的结论改写为:

教学版"每轮跑 P2 判断 + 重复字符串拼接" — 简单 + 不花钱 + 不慢 + 闭包 cache 避免信息丢失 + 跳过 summary block 避免重复 压缩.

消息处理管道的顺序 (一个具体例子)

prepare messages 阶段, 7 个步骤按固定顺序跑。 我们用 "10 轮对话, 触发 P0 + P2" 这个具体例子, 看清每一步之后 messages 长什么样。

7 步顺序不能换: getEntries 必须在 annotate 之前 (annotate 需要先有 entry), group 必须在 decay 之前 (decay 按 block 处理), decay 必须在 compact 之前 (compact 要拿到衰减后 的 block), compact 必须在 flatten 之前 (flatten 把 block 还原 成 messages)。 任何一步乱序, 要么 metadata 丢了, 要么压缩 破坏了消息 block。

token 估算: 中英文不同权重

什么时候触发 P2? 需要估算 history 的 token 数。 OpenAI 官方 tokenizer 慢, 我们的简单估算: 

function estimateTokens(text: string): number {
  // 中文字符 × 1.5, 英文 × 0.25, 取较大值
  const chinese = (text.match(/[\u4e00-\u9fa5]/g) || []).length;
  const english = text.length - chinese;
  return Math.ceil(chinese * 1.5 + english * 0.25);
}

设计思想: 跑 10 轮对话, 触发 P0 衰减, 验证 (1) 所有 tool_call_id 仍能找到对应 tool message, (2) 早期 tool result 被 截断, (3) 近期 tool result 完整。 3 件事联合验证。

实现细节: 

test("P0 衰减后 tool_call_id 配对仍然完整", async () => {
  const history = createHistory();
  // 模拟 10 轮对话, 调 10 个工具
  for (let i = 0; i < 10; i++) {
    history.add({ role: "user", content: `Step ${i}` });
    history.add({ role: "assistant", tool_calls: [{ id: `c${i}`, function: { name: "run_bash", arguments: "{}" } }] });
    history.add({ role: "tool", tool_call_id: `c${i}`, content: `Result ${i} with 100 chars content` });
  }
  // 触发 P0 衰减 (假设 currentLoopIndex = 15, threshold = 5)
  const messages = prepareMessages({ history, currentLoopIndex: 15, decayThreshold: 5 });
  // 验证: 所有 tool_call_id 仍然配对
  const callsById = new Map();
  const resultsById = new Map();
  for (const m of messages) {
    if (m.role === "assistant" && m.tool_calls) {
      for (const tc of m.tool_calls) callsById.set(tc.id, tc);
    }
    if (m.role === "tool") resultsById.set(m.tool_call_id, m);
  }
  for (const [id, _] of callsById) {
    expect(resultsById.has(id)).toBe(true);
  }
  // 验证: 早期 (c0-c4) 的 tool result content 是截断标记
  for (let i = 0; i < 5; i++) {
    const result = messages.find(m => m.role === "tool" && m.tool_call_id === `c${i}`);
    expect(result.content).toMatch(/Decayed/);
  }
  // 验证: 近期 (c5-c9) 的 tool result content 完整
  for (let i = 5; i < 10; i++) {
    const result = messages.find(m => m.role === "tool" && m.tool_call_id === `c${i}`);
    expect(result.content).toBe(`Result ${i} with 100 chars content`);
  }
});

这条测试如果挂, 你立刻知道 P0 衰减破坏了 tool_call_id 配对, 或者衰减粒度错了 (衰减了不该衰减的 block)。

3 个常见误解

误解 1 · "压缩就是删除"
错。 P0/P2 是截断/总结, 不是删除。 删除会破坏 tool_call_id 配对, 让 LLM 收到错位消息。 截断保留 role 和 id, 只换 content, 安全。
误解 2 · "压缩可以丢 system prompt"
错。 system prompt 是 stable prefix, 不参与压缩。 只压 history messages。 压 system prompt 等于让 LLM 忘掉行为规则。
误解 3 · "越早压越好"
错。 P0 太激进 (每轮都衰减) 会让 LLM 看不到近期上下文, 质量下降。 阈值要合理: decayThreshold = 5 轮, totalThreshold = 80% 窗口。 调参要看真实任务, 不靠感觉。

反例梯度

新手错法 · A

常见错误: 按消息条数衰减, 拆了 tool_use block。

为什么错: 拆 tool_use block 破坏 tool_call_id 配对, LLM 看到 错位消息, OpenAI API 可能报 400。

正确做法: 按消息 block 衰减, tool_use block 整体处理, 只截断 content 不拆 block。

中级错法 · B

常见错误: P0 衰减时删除 tool message, 留 assistant 调工具的记录。

为什么错: 删 tool message 破坏配对, 下一轮 LLM 看到 "assistant 调了 3 个工具, 只收到 2 个 tool"。

正确做法: 截断 content 为 "[Decayed, N chars]", 保留 role: "tool" 和 tool_call_id。

高级错法 · C

常见错误: 用 spread {...m} 复制消息, 把内部 _loopIndex 字段发给 LLM。

为什么错: _loopIndex 拼到 messages, 每次 chat() 消息数组 不一样, dynamic tail 不稳定, cache 命中受影响。

正确做法: flatten 时清除 _loopIndex 等内部字段, 只发 role / content / tool_calls / tool_call_id。

边界错法 · D

常见错误: 每次 P2 压缩都从最开头开始, summary block 被反复压缩。

为什么错: 反复压缩 summary block, 信息丢失加速, 最后 summary 失去所有细节。

正确做法: summary block 标记为 "kind: summary", P2 只压 非 summary block, summary block 保留原样。

Prompt Card · 第 06 章

目标: 给 harness 加三层压缩 (P0 衰减 / P1 即时 / P2 全量), 长任务下保持稳定的 context 长度。

场景: 跑 200 轮对话, history 累积 80000 token, P0 衰减 早期 tool result, P1 即时压缩大输出, P2 总结超阈值 history。

  • P2 总结保留最近 K 个 block, summary block 不参与下次 P2

本章小结

这一章给 harness 加了 3 层压缩防线:

  • P0 衰减: 每轮自动截断旧 tool result, 轻量。
  • P1 即时: 大 tool result 写文件, 返回 preview + output_id, 可逆。
  • P2 全量: 整段 history 超阈值, 字符串拼接成 1 段 summary, 保留最近 K 个 block。

    • 关键不变量: 消息 block 是压缩的原子单位, tool_use block 不可拆; 内部 metadata 字段 (如 _loopIndex) 不进 LLM messages。 下一章 (第 07 章) 解决"工具执行要不要先问用户" 的问题 — Permission。
