第 06 章 · context 撑爆怎么办

压缩: Normalize / Block / Compress 三层防线

第 05 章的 Skill 让工具数不再撑爆 system prompt, 但 messages 本身 仍然会无限增长: 用户多轮对话 + 工具结果累积, 上下文很快超出窗口。 这一章引入三层压缩: Normalize (合并相邻同角色消息) → Block (按消息块分组) → Compress (P0 衰减 + P1 即时 + P2 全量), 让 harness 在长任务下保持稳定的 context 长度。 读完后, 你能讲清 "为什么消息块是压缩的原子单位", 并能用 fake LLM 验证 P0 衰减 截断后的消息块不被破坏。

真实失败故事: 200 轮后 LLM 失忆

写代码之前, 先看一段真实长任务的失败。 跑一段对话: 用户让 agent "分析 50 个文件, 总结常见问题"。

  1. 第 1-10 轮: 调 run_bash 5 次, 调 run_read 10 次, history 累积 8000 token。 LLM 还在跟踪。
  2. 第 25 轮: messages 累积到 200K, LLM 反应开始慢, 偶尔 重复调之前调过的工具。
  3. 第 50 轮: history 累积 80000 token。 LLM 严重失忆, 总结时漏掉 5 个文件, 重复 3 个文件的判断, 修复方案质量大幅下降。
  4. 第 100 轮: history 累积 50000 token, 单次 prompt 超过窗口, LLM SDK 报 400 错误, agent loop 整个崩。

朴素想法 1: "让 LLM 写总结, 把之前的工具结果总结成一段?" 能缓解, 但不根治 — 总结本身也要占 token, 总结会丢失细节, LLM 基于总结给出的判断质量下降。

朴素想法 2: "压缩工具结果?" 压缩针对"单条工具结果太长", 不解决"多步任务的状态丢失" 问题。

正确做法: 三层压缩。 P0 衰减 (轻量, 每轮自动跑) + P1 即时 (单 tool result 太大, 立刻压缩) + P2 全量 (超阈值, 整段 总结)。 三层互补, 各管各的边界。

三层压缩的边界

三层不是"3 个选项", 是"3 道防线, 各管各的时机":

层 何时触发 压缩什么 谁负责 是否可逆
P0 衰减 每轮 LLM 调用前自动 超过 decayThreshold 的旧 tool result 截断 compressor 内部 不可逆 (丢掉细节)
P1 即时 单 tool result 超过阈值 大 tool result 写入 OutputStore, 返回 preview + output_id tool executor 内部 可逆 (通过 output_id 重读)
P2 全量 整个 history 超过阈值 早期 K 块整体压缩成 1 段 summary compressor 内部 + LLM 总结 不可逆 (总结会丢细节)

3 个关键差异:

  1. 触发时机: P0 每轮自动, P1 工具返回时, P2 整个 history 超阈值。 各管各的, 不重叠。
  2. 压缩粒度: P0 单 tool result 截断, P1 单 tool result 存文件, P2 整段 history 总结。 粒度递增。
  3. 是否可逆: P0 不可逆, P1 可逆 (通过 output_id 重读), P2 不可逆。 这是有意的设计 — P1 给用户"后悔" 的机会, P0/P2 默认不可逆。

消息块: 压缩的原子单位

在讲 P0/P1/P2 之前, 必须先讲消息块 — 压缩的原子单位。

朴素想法: "把消息按 token 数切片, 每 4000 token 一段。" 立刻坏: 一段可能切到 tool_use 块中间, 留下 tool_call 没有 tool_result, messages 序列错乱。

正确做法: 消息块是"语义完整" 的最小单位, 压缩时整块处理。 

type Block =
  | { kind: "text"; messages: Message[] }                      // 纯文本对话
  | { kind: "tool_use"; messages: Message[] }                   // 工具调用 + tool_result 配对
  | { kind: "summary"; messages: Message[] };                   // P2 总结块

GitHub · src/message-block.ts groupToBlocks / flattenToMessages (L1)

3 个块类型的边界:

  1. text 块: 只有 user + assistant 无 tool_calls 的对话。 普通聊天场景, 多个连续 user/assistant 消息可以合并。
  2. tool_use 块: assistant 含 tool_calls + 所有对应的 tool 消息。 这是不可分割的最小单位 — 拆开会破坏 tool_call_id 配对。
  3. summary 块: P2 压缩产生的总结, 一段文字。 后续 P0 衰减不再处理 summary 块 (它本身就是压缩过的)。

为什么 tool_use 块不能拆? 假设 assistant 调用 3 个工具:

  1. assistant(tool_call_1, tool_call_2, tool_call_3)
  2. tool(tool_call_id_1)
  3. tool(tool_call_id_2)
  4. tool(tool_call_id_3)

4 条消息组成一个 tool_use 块。 压缩时只能 4 条一起压, 不能 1-2 留 3-4 走, 否则 LLM 看到"assistant 调了 3 个工具, 只收到 2 个 tool result" — messages 序列错乱。

P1 即时压缩: 大 tool result 存文件

P1 是最常用的压缩, 在 tool executor 内部完成。 触发: tool result content 长度超过阈值 (默认 2000 字符)。 

// 教学简化版
async function runBash(args) {
  const result = await exec(args.command);
  if (result.stdout.length > 2000) {
    // 大输出: 写文件, 返回 preview + output_id
    const outputId = await outputStore.write(result.stdout, { source: "bash", toolCallId: args._toolCallId });
    return {
      content: `[Output truncated, ${result.stdout.length} chars]\n[Preview]:\n${result.stdout.slice(0, 500)}\n...[use run_output_read with output_id: ${outputId} to read full output]`,
      // 不标 error, 这是正常的"输出太大" 情况
    };
  }
  return { content: result.stdout };
}

GitHub · src/tools/run-bash.ts P1 即时压缩实现 (L1)

4 个细节:

  1. 写文件后返回 preview + output_id: LLM 看到 preview 知道大概内容, 看到 output_id 知道如何重读。
  2. 不标 error: 这是正常情况, 不是工具失败。 LLM 收到 不会误以为命令出错。
  3. preview 长度固定 500 字符: 给 LLM 足够的"第一印象", 又不会撑爆单条 tool message。 500 字符大约 100-200 token, 合理。
  4. content 含完整说明: "Output truncated, N chars" + "Preview: ..." + "use run_output_read with output_id: X"。 LLM 看到就知道下一步怎么读完整内容。

为什么 P1 是可逆的? LLM 收到 output_id 后可以调 run_output_read(output_id) 读完整内容 (第 13 章展开)。 这是"后悔药" — 用户让 LLM 看完整 5000 行日志, LLM 不会因为压缩 而丢失信息。

P0 衰减: 截断旧 tool result

P0 在每次 LLM 调用前自动跑, 不需要 LLM 介入。 触发: tool_use 块 超过 decayThreshold (默认 5 轮全局 LLM 调用)。 

// 教学简化版
function decayOldBlocks(blocks: Block[], currentLoopIndex: number, decayThreshold: number): Block[] {
  return blocks.map(block => {
    if (block.kind !== "tool_use") return block;
    if (currentLoopIndex - block.lastLoopIndex < decayThreshold) return block;
    // 超过 decayThreshold 轮的 tool_use 块, 截断 tool result content
    return {
      ...block,
      messages: block.messages.map(m =>
        m.role === "tool"
          ? { ...m, content: `[Decayed, ${m.content.length} chars, use output_id if needed]` }
          : m
      ),
    };
  });
}

GitHub · src/compressor.ts P0 衰减实现 (L1)

关键点:

  1. 按全局 loopIndex 衰减: 不是"每条消息独立 age", 是"整个块的年龄"。 一个 tool_use 块作为一个整体被衰减。
  2. 截断而非删除: content 替换成 "[Decayed, N chars]", 不是从 messages 里删除。 原因: tool_call_id 配对不能丢, 删了 LLM 看到"assistant 调了 3 个工具, 只收到 2 个 tool" 又错乱。
  3. summary 块不衰减: 它本身就是压缩过的, 衰减没意义。 如果 summary 也衰减, 等于二次压缩, 信息丢失加速。

❌ / ✅: 不要按"消息条数" 衰减, 按"块" 衰减。 

// ❌ 按消息条数, 拆 tool_use 块
function decayByMessage(messages, currentLoop) {
  for (let i = 0; i < messages.length; i++) {
    if (messages[i].role === "tool" && currentLoop - i > 5) {
      messages[i].content = "[Decayed]";   // 拆了 tool_use 块
    }
  }
}

// ✅ 按块, tool_use 块整体处理
function decayByBlock(blocks, currentLoop) {
  return blocks.map(block => block.kind === "tool_use" && currentLoop - block.lastLoopIndex > 5
    ? truncateToolResults(block)   // 整个块截断, 不拆
    : block);
}

P2 全量: 整段 history 总结

P2 在 history 超过总阈值时触发, 是一次"重型压缩"。 流程:

  1. 把 history 分块
  2. 保留最近 K 个块 (默认 6)
  3. 之前的所有块 + 当前 summary 调 LLM 总结成 1 段
  4. 替换为 [summary 块] + [最近 K 个块]
// 教学简化版
async function compactHistory(blocks: Block[], recentKeep: number, llm: LLMClient): Promise<Block[]> {
  if (blocks.length <= recentKeep) return blocks;
  const toCompress = blocks.slice(0, blocks.length - recentKeep);
  const toKeep = blocks.slice(blocks.length - recentKeep);

  // 拼接要被压缩的块
  const text = toCompress.map(b => serializeBlock(b)).join("\n---\n");
  // 调 LLM 总结
  const response = await llm.chat({
    messages: [{ role: "user", content: `请总结以下对话的关键信息, 保留事实和决策:\n\n${text}` }],
  });
  const summary = response.content ?? "";

  // 替换为 summary 块 + 最近的块
  return [
    { kind: "summary", messages: [{ role: "assistant", content: summary }] },
    ...toKeep,
  ];
}

GitHub · src/compressor.ts P2 全量压缩实现 (L1)

5 个细节:

  1. 保留最近 K 块: 不全压, 留最近 6 个块保持近期上下文。 压缩总结本身也占 token, 压太狠反而失去信息。
  2. summary 调 LLM 总结: P0/P2 是 harness 内部操作, P2 总结需要 LLM。 这次 LLM 调用不进 history, 是一次性工具。
  3. 不写进 history: 总结后的 summary 块作为 history 的一部分 (kind: "summary"), 下次 prepare messages 时正常处理。
  4. 触发条件: history 超过 80% 窗口 (或配置阈值)。 太早触发浪费 LLM 调用, 太晚触发超窗口。
  5. 可重入: 连续 P2 不会出错, 因为 summary 块不参与下次 P2 (只压非 summary 块)。 这是"分层压缩" 的关键。

消息处理管道的顺序

prepare messages 阶段, 3 个步骤按固定顺序:

  1. getEntries: 从 history 拿所有 entry (带 metadata)。
  2. annotate: 标记每个 entry 的 _loopIndex / _loopRound / _messageSequence (供 P0 衰减判断用)。
  3. normalize: 合并相邻同角色消息 (连续 user 消息合并, 连续 assistant 消息合并)。 减少 messages 数量, 不减少 token。
  4. group: 把 messages 分组成 text/tool_use/summary 块。
  5. decay: P0 衰减, 截断超过 decayThreshold 的 tool_use 块。
  6. compact: 超过总阈值时 P2 总结。
  7. flatten: 把块还原成 messages 数组, 清除内部 _loopIndex 等 metadata。

顺序不能换: getEntries 必须在 annotate 之前, group 必须在 decay 之前, decay 必须在 compact 之前。 任何一步乱序, 要么 metadata 丢了, 要么压缩破坏了消息块。

内部字段清理: _xxx 不发 LLM

history 内部用了 _turnIndex / _loopRound / _loopIndex / _messageSequence 等字段, 这些是 harness 内部记账用的, 不 进 LLM messages。 

// 教学简化版
function flattenToMessages(blocks: Block[]): ChatCompletionMessageParam[] {
  return blocks.flatMap(block =>
    block.messages.map(m => {
      const cleaned: ChatCompletionMessageParam = { role: m.role, content: m.content };
      if (m.tool_calls) cleaned.tool_calls = m.tool_calls;
      if (m.tool_call_id) cleaned.tool_call_id = m.tool_call_id;
      // ❌ 不要 spread m, 否则 _loopIndex 等会进 LLM
      return cleaned;
    })
  );
}

token 估算: 中英文不同权重

什么时候触发 P2? 需要估算 history 的 token 数。 OpenAI 官方 tokenizer 慢, 我们的简单估算: 

function estimateTokens(text: string): number {
  // 中文字符 × 1.5, 英文 × 0.25, 取较大值
  const chinese = (text.match(/[\u4e00-\u9fa5]/g) || []).length;
  const english = text.length - chinese;
  return Math.ceil(chinese * 1.5 + english * 0.25);
}

GitHub · src/compressor.ts estimateTokens 中英文权重 (L1)

为什么不精确? 精确 token 化要调 tiktoken (慢, 启动慢), 或 LLM SDK 的 count_tokens 接口 (要网络)。 估算够用 — P2 触发的 阈值是 80% 窗口, 估算 5-10% 误差不影响触发。

fake LLM 测试: P0 衰减不破坏 tool_call_id 配对

用途: P0 衰减最容易写错的不是衰减逻辑, 是破坏 tool_call_id 配对。 写测试验证 3 件事。

设计思想: 跑 10 轮对话, 触发 P0 衰减, 验证 (1) 所有 tool_call_id 仍能找到对应 tool message, (2) 早期 tool result 被 截断, (3) 近期 tool result 完整。 3 件事联合验证。

实现细节: 

test("P0 衰减后 tool_call_id 配对仍然完整", async () => {
  const history = createHistory();
  // 模拟 10 轮对话, 调 10 个工具
  for (let i = 0; i < 10; i++) {
    history.add({ role: "user", content: `Step ${i}` });
    history.add({ role: "assistant", tool_calls: [{ id: `c${i}`, function: { name: "run_bash", arguments: "{}" } }] });
    history.add({ role: "tool", tool_call_id: `c${i}`, content: `Result ${i} with 100 chars content` });
  }
  // 触发 P0 衰减 (假设 currentLoopIndex = 15, threshold = 5)
  const messages = prepareMessages({ history, currentLoopIndex: 15, decayThreshold: 5 });
  // 验证: 所有 tool_call_id 仍然配对
  const callsById = new Map();
  const resultsById = new Map();
  for (const m of messages) {
    if (m.role === "assistant" && m.tool_calls) {
      for (const tc of m.tool_calls) callsById.set(tc.id, tc);
    }
    if (m.role === "tool") resultsById.set(m.tool_call_id, m);
  }
  for (const [id, _] of callsById) {
    expect(resultsById.has(id)).toBe(true);
  }
  // 验证: 早期 (c0-c4) 的 tool result content 是截断标记
  for (let i = 0; i < 5; i++) {
    const result = messages.find(m => m.role === "tool" && m.tool_call_id === `c${i}`);
    expect(result.content).toMatch(/Decayed/);
  }
  // 验证: 近期 (c5-c9) 的 tool result content 完整
  for (let i = 5; i < 10; i++) {
    const result = messages.find(m => m.role === "tool" && m.tool_call_id === `c${i}`);
    expect(result.content).toBe(`Result ${i} with 100 chars content`);
  }
});

3 个常见误解

误解 1 · "压缩就是删除"
错。 P0/P2 是截断/总结, 不是删除。 删除会破坏 tool_call_id 配对, 让 LLM 收到错位消息。 截断保留 role 和 id, 只换 content, 安全。
误解 2 · "压缩可以丢 system prompt"
错。 system prompt 是 stable prefix, 不参与压缩。 只压 history messages。 压 system prompt 等于让 LLM 忘掉行为规则。
误解 3 · "越早压越好"
错。 P0 太激进 (每轮都衰减) 会让 LLM 看不到近期上下文, 质量下降。 阈值要合理: decayThreshold = 5 轮, totalThreshold = 80% 窗口。 调参要看真实任务, 不靠感觉。

反例梯度

新手错法 · A

常见错误: 按消息条数衰减, 拆了 tool_use 块。

为什么错: 拆 tool_use 块破坏 tool_call_id 配对, LLM 看到 错位消息, OpenAI API 可能报 400。

正确做法: 按消息块衰减, tool_use 块整体处理, 只截断 content 不拆块。

中级错法 · B

常见错误: P0 衰减时删除 tool message, 留 assistant 调工具的记录。

为什么错: 删 tool message 破坏配对, 下一轮 LLM 看到 "assistant 调了 3 个工具, 只收到 2 个 tool"。

正确做法: 截断 content 为 "[Decayed, N chars]", 保留 role: "tool" 和 tool_call_id。

高级错法 · C

常见错误: 用 spread {...m} 复制消息, 把内部 _loopIndex 字段发给 LLM。

为什么错: _loopIndex 拼到 messages, 每次 chat() 消息数组 不一样, dynamic tail 不稳定, cache 命中受影响。

正确做法: flatten 时清除 _loopIndex 等内部字段, 只发 role / content / tool_calls / tool_call_id。

边界错法 · D

常见错误: 每次 P2 压缩都从最开头开始, summary 块被反复压缩。

为什么错: 反复压缩 summary 块, 信息丢失加速, 最后 summary 失去所有细节。

正确做法: summary 块标记为 "kind: summary", P2 只压 非 summary 块, summary 块保留原样。

回望: 哪些原则在本章兑现了

  • 消息块是原子单位: 压缩不能拆 tool_use 块, 否则破坏 tool_call_id 配对。
  • 窄接口: compressor 只暴露 compress / decay / compact, 不暴露内部 history 结构。
  • 分层防御: P0 / P1 / P2 各管各的时机, 不互相替代。
  • 内部 metadata 隔离: _loopIndex 等字段在 history 内部 流转, flatten 时清除。

Prompt Card (本章任务)

Prompt Card · 第 06 章

目标: 给 harness 加三层压缩 (P0 衰减 / P1 即时 / P2 全量), 长任务下保持稳定的 context 长度。

场景: 跑 200 轮对话, history 累积 80000 token, P0 衰减 早期 tool result, P1 即时压缩大输出, P2 总结超阈值 history。

模块: src/normalize.ts (新) 合并相邻同角色消息; src/message-block.ts (新) groupToBlocks() 和 flattenToMessages(); src/compressor.ts (新) createContextCompressor() 暴露 P0 / P1 / P2; src/agent.ts (改) prepare messages 阶段调用压缩; src/output-store.ts (改) 支持 P1 写文件。

边界 (LLM 必须遵守的 checklist):

  • 压缩按消息块, 不按消息条数, tool_use 块不能拆
  • P0 截断 content 为 "[Decayed]", 不删除 role: "tool" 消息
  • P1 大输出写 OutputStore, 返回 preview + output_id, LLM 可重读
  • P2 总结保留最近 K 块, summary 块不参与下次 P2
  • flatten messages 时清除 _loopIndex 等内部字段
  • _loopIndex / _loopRound / _messageSequence 等内部字段不进 LLM messages

验证 (用 fake LLM + vitest, 逐条断言):

  • 10 轮对话触发 P0, 验证所有 tool_call_id 仍能找到对应 tool message
  • run_bash 返回 5000 字符触发 P1, tool message 含 output_id
  • history 超阈值触发 P2, 最后 6 个块原样保留, 之前块被 summary 替换
  • 连续 P2 两次, 第二次不压缩第一次的 summary 块
  • prepare messages 后, LLM 收到的 messages 不含 _loopIndex 字段

本章练习

  1. 故意按消息条数衰减, 跑测试, 看"P0 衰减不破坏配对" 是否抓到 (拆了 tool_use 块)。
  2. 让 run_bash 返回 5000 字符但 P1 阈值设 10000, 跑测试, 看"P1 即时压缩" 是否抓到 (没触发, tool message 撑爆)。
  3. 用 spread {...m} 复制消息, 跑测试, 看"内部字段不发 LLM" 是否抓到 (_loopIndex 进 messages)。

本章小结

这一章给 harness 加了 3 层压缩防线:

  • P0 衰减: 每轮自动截断旧 tool result, 轻量。
  • P1 即时: 大 tool result 写文件, 返回 preview + output_id, 可逆。
  • P2 全量: 整段 history 超阈值, 调 LLM 总结, 保留最近 K 块。

    • 关键不变量: 消息块是压缩的原子单位, tool_use 块 不可拆; 内部 metadata 字段 (如 _loopIndex) 不进 LLM messages。 下一章 (第 07 章) 解决"工具执行要不要先问用户" 的问题 — Permission。
