第 01 章 · 真正落 TypeScript

最小 Agent Loop: 从伪码到可测试模块

第 00 章讲了"为什么需要 harness",并给出了一份 6 段 Prompt Card 模板。 这一章按那张卡片把最小 agent loop 真正写成 TypeScript 模块: History、 LLM Client、Agent、REPL、Composition Root 五件套,加上 fake LLM 的 messages 顺序断言。读完后,你能跑通一份"两轮对话保留上下文"的最小 harness, 并能用测试验证它不是偶然跑通。

作者怎么想的: 这一章的思考链

按第 00 章 6 段 Prompt Card 模板倒推,写这一章 Prompt Card 之前,作者脑子里 经历了下面 4 步。第 00 章第 1 节已经讲过这 4 步的抽象,这里给具体例子。

想清楚现象
现象是"两次 run 之后,模型还能引用第一轮的偏好"。这个现象的最小复现 是 REPL 里连续输入两句"我喜欢简洁"和"我喜欢什么风格"。如果实现错了, 第二句会得到"我不知道你喜欢什么"。
想反例
最朴素的反例是第 00 章的 ask(query): 它只把当前 query 发给 LLM,没有 History。这一章要"治"的就是这种"每次都是新会话"的问题。
想接口和不变量
不变量有三: (1) History 是 messages 的唯一来源, (2) getMessages() 返回浅拷贝,(3) Composition Root 内同一份 history 实例被 agent 和 repl 共享。接口要暴露 add / getMessages / clear 三个动作,不要暴露内部数组。
想怎么验证
验证用 fake LLM,断言"第二轮 LLM 收到的 messages 数量 == 3、第一条 role == user、最后一条 role == user"。这条断言比"最终文本对"强, 因为它要求"消息真的被保存并被重新发送"。

先观察: 三段故意有气味的代码

在写正确实现之前,先看三段常见但"有气味"的最小实现,回答"它为什么 不行"。这一节不是练习,只是训练"看出哪里不对"的眼睛。

观察 1 · 全局单例 History

// history.ts
export const history = createHistory();

// agent.ts
import { history } from "./history.js";
export async function run(query: string) {
  history.add({ role: "user", content: query });
  const msgs = history.getMessages();
  return (await llm.chat(msgs)).content;
}

问:看上去很简洁。为什么 Composition Root 团队普遍禁止 module-level 单例?

答:单例失去三件事 —— 测试时无法替换; 子智能体或 多窗口 REPL 无法隔离; Composition Root 失去了"实例从哪里来"的可观测 性。第 00 章的"中级错法 B"已经讲过这一点。

观察 2 · getMessages 返回内部引用

// history.ts (内部实现)
getMessages(): Message[] {
  return this.messages;
}

问:返回引用比浅拷贝快,为什么不写?

答:LLM client 拼下一轮 messages 时常见写法是 [...msgs, thisTurnUser]。如果 msgs 是内部数组,这个 spread 会让外部 push 写回 history,出现"我没调 add() 但 messages 变长"的隐性 bug。这正是第 00 章的"高级错法 C"。

观察 3 · 把 process.env 塞进 agent.ts

// agent.ts
const apiKey = process.env.ANTHROPIC_API_KEY;
const llm = createLlm({ apiKey });

问:看上去"配置就近使用"很自然,为什么要搬到 config.ts?

答:agent.ts 一旦读环境变量,测试就必须 mock 整个 process.env; 不同部署环境 (本地 / CI / 沙箱) 切换配置就要改业务文件; 子智能体想用同一份 agent 但不同 LLM provider 时也无从换起。 config.ts 是 Composition Root 的"配置版本",不是 agent.ts 的私货。

新增文件清单 (差量表)

这一章会新建 7 个文件。每个文件的第一行 import / export 决定了它的 职责边界; 不属于这个文件职责的代码,不要塞进来。

src/config.ts
读取环境变量并校验; 导出 loadConfig(),返回 { apiKey, model, ... }。本文件不出现业务逻辑。
src/terminal.ts
封装 readline 的输入输出; 暴露 ask() / print() / close()。本文件不出现 LLM 调用,也不出现 History。
src/history.ts
createHistory() 工厂; 内部 messages: Message[]。 暴露 add / getMessages / clear。 getMessages 必返回浅拷贝 (第 00 章不变量 2)。
src/llm.ts
createLlm(config) 工厂; 暴露 chat(messages)。 真实实现调 Anthropic SDK, fake LLM 由测试用同名 interface 注入。
src/agent.ts
createAgent({ history, llm }) 工厂; 暴露 run(query)。本文件不读环境变量, 不 new LLMClient, 不直接用 readline (第 00 章边界 checklist)。
src/repl.ts
createRepl({ agent, terminal }) 工厂; 暴露 start()。循环 ask → agent.run → print, 处理 exit / quit / 空输入。
src/index.ts (Composition Root)
loadConfig → createHistory → createLlm → createTerminal → createAgent(deps) → createRepl(deps) → repl.start()。 本文件不出现 if 分支, 不出现 try/catch 业务逻辑。

loop 路径: 与第 00 章图 00-1 一一对应

下面这段实现直接对应第 00 章的伪码。每行右侧标注了它在图 00-1 中 的节点,方便对照。 

// src/agent.ts
export function createAgent(deps: AgentDeps): Agent {
  const { history, llm } = deps;

  return {
    async run(query: string): Promise<string> {
      // 边界 checklist: 空 query 不写入 history
      if (query.trim().length === 0) {
        return "";
      }

      // 图 00-1 节点 3: 写 user message
      history.add({ role: "user", content: query });

      // 图 00-1 节点 4: 取 messages 浅拷贝
      const messages = history.getMessages();

      // 图 00-1 节点 5: 调 LLM
      const assistant = await llm.chat(messages);

      // 图 00-1 节点 6: 写 assistant message
      history.add(assistant);

      return assistant.content;
    },
  };
}

Composition Root: 把 5 件套装到一根线

index.ts 的存在意义是"5 件套的接线图"。它不出现 if 分支, 也不出现 try/catch,因为这两类代码一旦塞进 Composition Root,后续 子智能体和 e2e 都会变难。 

// src/index.ts
import { loadConfig } from "./config.js";
import { createHistory } from "./history.js";
import { createLlm } from "./llm.js";
import { createTerminal } from "./terminal.js";
import { createAgent } from "./agent.js";
import { createRepl } from "./repl.js";

export async function main(): Promise<void> {
  const config = loadConfig();
  const history = createHistory();
  const llm = createLlm(config);
  const terminal = createTerminal();
  const agent = createAgent({ history, llm });
  const repl = createRepl({ agent, terminal });
  await repl.start();
}

main().catch((err) => {
  console.error(err);
  process.exit(1);
});

REPL 的边界: 退出命令和空输入

REPL 这一层看起来是"输入输出",但它要承担三件事: (1) 拦截 exit / quit 关闭 terminal, (2) 拦截空输入避免 agent 写入 History, (3) 拦截 Ctrl-D / EOF 优雅关闭。 这三件事都不应该让 agent 知道。 

// src/repl.ts
export function createRepl(deps: ReplDeps): Repl {
  const { agent, terminal } = deps;

  return {
    async start(): Promise<void> {
      // eslint-disable-next-line no-constant-condition
      while (true) {
        const raw = await terminal.ask();
        if (raw === null) break;            // EOF
        const query = raw.trim();
        if (query.length === 0) continue;   // 空输入不写入 history
        if (query === "exit" || query === "quit") break;
        const answer = await agent.run(query);
        terminal.print(answer);
      }
      terminal.close();
    },
  };
}

Fake LLM Cookbook: 怎么写 messages 顺序断言

这一节给出一份能直接抄的 fake LLM。它解决第 00 章"红灯 1"中提到的 "只断言最终文本"的隐患。 

// test/_helpers/fake-llm.ts
import type { LLMClient, AssistantMessage, Message } from "../../src/llm.js";

export function createFakeLlm(scripted: string[]): LLMClient & {
  received: Message[][];
} {
  const received: Message[][] = [];
  let i = 0;

  return {
    received,
    async chat(messages: Message[]): Promise<AssistantMessage> {
      // 记录每次调用收到的 messages 浅拷贝
      received.push([...messages]);
      const content = scripted[i++] ?? scripted[scripted.length - 1];
      return { role: "assistant", content };
    },
  };
}

测试中只断言 fake.received[N][M].role / .content,就能在不调真实 LLM 的情况下验证"第二次 LLM 收到了什么"。这种断言比"最终文本对"严格得多,它是后续每一章 Validation 卡片的基础。

Validation: 把 Prompt Card 的 5 条断言落到 vitest

下面 5 条断言直接来自第 00 章 Prompt Card 的"验证"一节。每条都能用 fake LLM 在 <100ms 内跑完。 

// test/agent.test.ts
import { describe, it, expect } from "vitest";
import { createAgent } from "../src/agent.js";
import { createHistory } from "../src/history.js";
import { createFakeLlm } from "./_helpers/fake-llm.js";

describe("agent.run (最小 loop)", () => {
  it("单轮: 透传 fake LLM 返回值", async () => {
    const llm = createFakeLlm(["收到"]);
    const agent = createAgent({ history: createHistory(), llm });
    expect(await agent.run("你好")).toBe("收到");
  });

  it("多轮: 第二次 LLM 收到的 messages 包含第一轮 user 和 assistant", async () => {
    const llm = createFakeLlm(["A1", "A2"]);
    const agent = createAgent({ history: createHistory(), llm });

    await agent.run("u1");
    await agent.run("u2");

    expect(llm.received[1]?.length).toBe(3);                  // [u1, a1, u2]
    expect(llm.received[1]?.[0]?.role).toBe("user");
    expect(llm.received[1]?.[0]?.content).toBe("u1");
    expect(llm.received[1]?.[1]?.role).toBe("assistant");
    expect(llm.received[1]?.[1]?.content).toBe("A1");
    expect(llm.received[1]?.[2]?.role).toBe("user");
  });

  it("空 query 不写入 history", async () => {
    const llm = createFakeLlm(["A"]);
    const history = createHistory();
    const agent = createAgent({ history, llm });

    expect(await agent.run("")).toBe("");
    expect(history.getMessages().length).toBe(0);
    expect(llm.received.length).toBe(0);
  });

  it("getMessages 返回浅拷贝, 外部 push 不影响内部", () => {
    const history = createHistory();
    history.add({ role: "user", content: "u" });

    const snap = history.getMessages();
    snap.push({ role: "user", content: "hacked" });

    expect(history.getMessages().length).toBe(1);
    expect(history.getMessages()[0]?.content).toBe("u");
  });
});

这 4 条断言 (5 条里第 5 条是 REPL 的,放到 repl.test.ts) 已经覆盖了第 00 章 Prompt Card 边界 checklist 的全部 5 条。 第 02 章开始,每一章都会沿用"fake LLM + messages 顺序断言"的模式, 只是 messages 数量和 role 序列变长。

反例梯度: 新手 / 中级 / 高级

本章的反例按"看起来能跑 / 看上去更优雅 / 用对了工具但放错位置"分三档。 读完后你应该能说出"为什么 LLM 经常写出 B 或 C"。

新手错法 · A

常见错误:把 agent.run() 实现成"调一次 LLM,把响应拼接返回",不写 history。

为什么错:多轮上下文立刻断, 第二句会得到"我不知道 你喜欢什么"。

正确做法:严格按图 00-1 的 4 步, 写入 → 取 messages → 调 LLM → 写回。

中级错法 · B

常见错误:把 History 做成 module-level 单例, 在 agent.ts 内 import { history } from "./history.js"。

为什么错:测试时无法替换, 子智能体无法隔离, Composition Root 失去可观测性 (第 00 章中级错法 B)。

正确做法:Composition Root 创建实例, 通过 deps 注入。

高级错法 · C

常见错误:getMessages() 返回内部数组引用。

为什么错:LLM client 拼下一轮 messages 时 [...msgs, thisTurnUser] 会反向污染 history (第 00 章 高级错法 C)。

正确做法:返回 [...this.messages] 浅拷贝, 并在 Validation 里写一条 push 反向断言。

边界错法 · D

常见错误:空字符串 query 仍然写 history 并调 LLM。

为什么错:REPL 误触 (例如连续回车) 会让 history 出现 role: "user", content: "", 既污染上下文又浪费 token。

正确做法:在 agent 层和 repl 层各加一次空输入拦截, 并在 Validation 里断言"空输入不增加 history 长度"。

如何验证 (本章 Validation 卡片)

Validation · 第 01 章

单轮透传:fake LLM 返回 "收到", agent.run("x") === "收到"。

多轮 messages 顺序:连续两次 run 后, fake LLM 第二次 收到的 messages 长度 == 3, role 序列 == [user, assistant, user], content 与第一轮输入一致。

空 query 不写 history:run("") 返回 "", history 长度不变, fake LLM 收到 0 次调用。

getMessages 浅拷贝:外部 push 新元素后, history 内部不受影响 (反向断言)。

Composition Root 实例共享:在 index.ts 内 history 只 new 一次, agent 和 repl 拿到的是同一份 引用 (用 === 断言)。

如果实现失败,先查哪里

  1. 多轮断言失败:看 agent.run() 是不是漏写了 history.add(assistant), 或者 getMessages() 返回了浅拷贝但外部没拿到。
  2. 空 query 仍然写 history:看 agent.run() 顶部是否做了 if (query.trim().length === 0) return ""。
  3. 浅拷贝断言失败:看 getMessages() 是不是 返回了 this.messages 而不是 [...this.messages]。
  4. Composition Root 出现 if 分支:把 if 拆到对应的工厂 函数或 REPL 里, index.ts 只做接线。
  5. fake LLM 收到 0 次调用:检查 llm.received 是不是被 fake 工厂外部覆盖 (要直接读 llm.received, 不要在测试里用 fake.received)。

这一节是"知识回环"。每章末尾都要把"自己兑现了哪些第 00 章原则" 列出来,作为对前章的差量验证。

  • 术语锁定:History / LLMClient / Composition Root / fake LLM 全部以英文出现, 中文仅做"外层运行环境""消息列表"等一次性释义。
  • 防自欺:红灯 1 "跑通 ≠ 正确" 体现在 fake LLM 的 messages 顺序断言上; 红灯 3 "LLM 说做完了" 体现在边界 checklist 的每条都可 翻译为 vitest 断言上; 红灯 4 "卡片漂亮 ≠ 实现漂亮" 体现在反例梯度 4 条具体错法上。
  • Prompt Card 6 段:第 00 章 Prompt Card 是 "6 段模板" 的 演示版, 第 01 章 Prompt Card 已经把 6 段全部按模板写, 不再空话。

前瞻张力: 留给后续章节

现在的 loop 能跑, 但还差几件事: assistant 里出现 tool call 时 (第 02 章)、上下文增长到模型放不下时 (第 06 章)、多轮里 LLM 临时 抽风时 (第 11 章)、需要让另一个 agent 接力时 (第 04 章)。这一节 把这些张力显式钉出来, 后续章节会逐个回来解决。

tool call 分支
当 assistant.content 是空但出现 tool_calls 数组时, loop 不能返回 文本, 而要执行工具、写入 tool result、再发一次 LLM。这条分支在 第 02 章接入, 但 prompt 模板里要提前留出 if 分支的形状。
context 增长
长对话会让 messages 超过模型窗口。History 之后会被 normalize / block / compress (第 06 章), 这意味着 history 不能再"原样返回", 它要知道自己"是 prompt working context", 不等同于"原始记录"。
LLM 异常
LLM 返回空、超时、JSON 损坏时, agent 要决定"重试 / 跳过 / 中止"。 第 11 章会引入 recovery 边界, 现在的 agent.run() 还没考虑这些。
子智能体
第 04 章会让 agent.run() 内部"开第二个 loop", 但第二个 loop 的 History 必须和父 agent 隔离。这意味着 History 不能再 module-level, createHistory() 必须是工厂 (这一章已经做到)。

本次如何 vibe code: 第 01 章的三件套

按第 00 章 vibe-coding 方法论的拆卡 / review / 迭代三件套, 第 01 章的具体操作如下。 后续 02–15 章都会按这个格式给出 "本次如何 vibe code", 请你也按这个格式去写自己的 LLM 对话。

拆卡: 4 轮迭代的具体产物

  1. 第 1 轮 · 接口。只贴 "目标 + 场景 + 模块", 让 LLM 给出 interface History / interface LLMClient / interface Agent / interface Terminal 四个 interface 的草案。本轮不写实现, 只钉形状与命名。
  2. 第 2 轮 · 接线。贴 "模块 + 接线", 让 LLM 给出 src/index.ts 接线代码, 其中 createHistory / createLlm / createAgent / createRepl 都是返回空对象的 stub。本轮 review 重点: history 实例是否 在 index.ts 中只 new 一次, agent 和 repl 拿到的是不是 同一份引用 (=== 断言)。
  3. 第 3 轮 · 边界。贴 "边界 (5 条 checklist)", 让 LLM 按 checklist 写每个工厂的真实实现。本轮 review 重点 (V1 review checklist 的"依赖方向"+"副作用"两栏): agent.ts 是否被偷偷塞进 process.env 读取, getMessages 是否返回浅拷贝。
  4. 第 4 轮 · 验证。贴 "验证 (5 条 vitest 断言)", 让 LLM 写 test/agent.test.ts。本轮 review 重点: fake LLM 是否 真正记录 messages (而不是只返回字符串), 多轮断言是否同时检查 role 和 content。

为什么第 1 轮不允许写实现

LLM 拿到 "目标 + 场景 + 模块" 后, 默认会直接写实现。写实现会触发"假装清单" 里的"假装做了边界检查"。本轮显式禁止 "do not write implementation yet", 强迫 LLM 把注意力放在命名、参数顺序、返回类型上, 后续轮次再补实现。

Review: 第 01 章专属 checklist

在第 00 章通用 review checklist 之上, 第 01 章额外要逐条核对的 5 条:

  1. history 是工厂而不是单例。grep -n 'export const history' src/ 应当 0 行。如果出现, 立即回退到第 2 轮让 LLM 改成工厂。
  2. getMessages 返回浅拷贝。看实现是不是 return [...this.messages], 而不是 return this.messages。 必须配合 Validation 卡片里 "外部 push 不影响 history" 这条反向断言。
  3. agent.run() 内不读环境变量。grep -n 'process.env' src/agent.ts 应当 0 行。
  4. 空 query 在 agent.run() 顶部拦截。看是不是有 if (query.trim().length === 0) return "" 这行, 而不是 REPL 拦截后直接 continue。两层都要有。
  5. Composition Root 内 history 只 new 一次。写一个 __test__.ts (不进 git) 把 agent 和 repl 拿到的 history 引用 === 比对, 不通过则立即重写。

调试: 第 01 章典型伪装

LLM 在 01 章最常犯的两种伪装:

  1. 伪装 A · 把 LLM 调用直接放进 REPL。症状: repl.ts 里有 await llm.chat(...)。这意味着 REPL 绕过了 agent, history 不再唯一。验证: grep -n 'llm.chat' src/repl.ts 应当 0 行; grep -n 'llm.chat' src/agent.ts 应当 ≥ 1 行。
  2. 伪装 B · 把 history 写成 module-level 闭包单例。症状: history.ts 顶层有 let messages: Message[] = []; 之类 module-level 状态。验证: grep -n '^let messages\|^const messages' src/history.ts 应当 0 行 (只能在 createHistory() 函数体内出现)。

迭代: 第 01 章 4 个 commit 节点

推荐的 4 个 commit 节点。每个 commit 跑通对应 vitest 集合后才进下一个:

  1. feat(ch01): 钉 4 个 interface 形状, 不写实现 —— 跑 tsc --noEmit 通过, 没有任何实现代码。
  2. feat(ch01): Composition Root 接线, 工厂为 stub —— 跑通 "Composition Root 实例共享" 那个一次性断言 (history 在 agent 和 repl 间是 ===)。
  3. feat(ch01): 5 条边界 checklist 兑现 —— 跑通 Validation 卡片前 4 条 (单轮透传 / 多轮 messages 顺序 / 空 query / 浅拷贝)。
  4. test(ch01): REPL 边界 (exit / quit / EOF) —— 跑通 第 5 条 Validation (REPL 边界)。

每个 commit message 引用对应 Validation ID 是有意为之: 当某次回归测试 失败时, 你能立刻定位"是哪一条 Validation 被破坏了", 而不必从头读 diff。

本章练习

  1. 按本章 4 条 Validation 断言写测试, 故意把 getMessages() 改回 return this.messages;, 验证测试会失败。
  2. 在 agent.ts 里加一个分支: 如果 query 以 /echo 开头, 不调 LLM, 直接把剩余部分作为 assistant 写回 history。验证多轮 messages 序列仍然正确 (echo 也算一轮)。
  3. 把 index.ts 故意塞进 if (process.env.DRY_RUN) 分支, 看看自己在看代码时能不能立刻识别"这里坏了" (第 00 章红灯 4)。

本章小结

本章按 6 段 Prompt Card 模板兑现了第 00 章的承诺: 7 个文件, 5 件套 + Composition Root + fake LLM, 4 条 messages 顺序断言覆盖了"多轮上下文 真的由 History 承担"这件事。读完后, 你能自己写一份能通过全部 Validation 的实现, 也能识别 LLM 给的代码"看上去对但其实偷懒"的几种典型手法。
