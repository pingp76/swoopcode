第 01 章 · 最小可运行核心

最小 Agent Loop

第 00 章已经把普通 LLM 调用和 agent loop 区分开了。这一章开始落地：用 TypeScript 拆出 REPL、History、LLM Client、Agent 和 Composition Root，让最小 loop 真正跑起来，并且能被 fake LLM 测试。

你已经知道什么，还不知道什么

你已经知道最小 agent loop 的故事：用户输入进入 History，History 变成 messages，LLM 返回 assistant message，assistant 再写回 History。现在的问题是：这个故事在 TypeScript 项目里应该拆成哪些模块？

这一章不会讲模型 API 的所有参数，也不会实现工具调用。我们只关心一个基础工程问题：怎样把最小 loop 写成后续可扩展、可测试、不会一加功能就崩的形状。

本章怎么学

先把第 00 章的图景落到模块：谁接收用户输入，谁保存 History，谁调用 LLM，谁负责组装依赖。不要从 index.ts 一路读到底，而是先看这些职责如何被拆开。

然后观察一个朴素 demo 为什么会失败：它可以聊天，但不可测试、不可替换、不可继续生长。最后再看接口和 fake LLM 测试，确认这个 loop 不是偶然跑通，而是可验证。

本章场景

你准备从零实现这个项目。用户在命令行里输入：“你好，之后请用简洁风格回答。”Agent 应该调用模型回复。下一轮用户问：“我刚才要求什么风格？”Agent 应该把上一轮对话一起发给模型，让模型能基于上下文回答。

这件事听起来像聊天机器人，但我们从第一天就要把它写成 coding agent 的骨架。因为第 02 章工具调用会插进 LLM 返回后；第 07 章权限会插进工具执行前；第 22 章 eval 会替换真实 LLM 来测试 loop 行为。

先试一个朴素方案

最诱人的做法是把所有东西塞进 index.ts：读取环境变量、创建 OpenAI client、启动 readline、维护数组、调用模型、打印结果。 

const messages = [];

rl.on("line", async (query) => {
  messages.push({ role: "user", content: query });
  const response = await openai.chat.completions.create({
    model,
    messages,
  });
  messages.push(response.choices[0].message);
  console.log(response.choices[0].message.content);
});

这段代码很快能跑，但它把交互、状态、模型调用和主循环挤在一起。它适合写 demo，不适合成为后续 14 章继续生长的 agent harness。

朴素方案为什么不够

它很难测试
逻辑绑在 readline 和真实 OpenAI client 上。你想验证“第二轮 messages 包含第一轮内容”，就必须启动终端和真实模型，测试会又慢又不稳定。
它很难扩展
第 02 章要处理 tool call，第 07 章要做权限，第 11 章要做错误恢复。如果所有分支都写在 line 回调里，主流程会很快失控。
它会让共享状态失控
后续子智能体、eval driver、CLI 命令都需要复用或替换部分依赖。如果依赖在模块内部随手 new，就很难保证哪些实例应该共享，哪些应该隔离。

回到 Agent Loop

最小 loop 不是“一个文件里写个 while”，而是几个职责稳定的模块协作。REPL 只把输入交给 Agent；Agent 拥有 loop；History 保存 messages；LLM Client 负责模型协议；Composition Root 创建它们并接线。

架构图解

把模块关系画成依赖方向，会更容易看出谁应该知道谁。

一次真实运行 walkthrough

  1. 启动：index.ts 读取配置，创建 logger、terminal、history、llm、agent、repl。
  2. 接线：history 和 llm 被传给 createAgent()；agent 和 terminal 被传给 createRepl()。
  3. 输入：用户在 REPL 输入一句 query，例如“之后请简洁回答”。
  4. 进入 loop：REPL 调用 agent.run(query)，不自己维护 messages。
  5. 保存用户消息：Agent 调用 history.add({ role: "user", content: query })。
  6. 准备请求：Agent 调用 history.getMessages()，拿到包含历史上下文的 messages。
  7. 调用模型：Agent 调用 llm.chat(messages)，得到 assistant message。
  8. 保存回复：Agent 把 assistant message 写回 History。如果没有 tool call，就把文本返回给 REPL。

这 8 步是后续所有能力的地基。工具调用只是把第 8 步从“直接返回文本”改成“先执行工具，把结果写回 History，再继续下一轮 LLM”。

关键接口和伪码

接口先于实现。我们先定义调用方需要什么，再在内部选择具体实现。

interface Agent {
  run(query: string): Promise<string>;
}

interface History {
  add(message: Message, meta?: MessageTiming): HistoryEntry;
  getMessages(): Message[];
  getEntries(): HistoryEntry[];
  clear(): void;
}

interface LLMClient {
  chat(messages: Message[], tools?: ToolDefinition[]): Promise<LLMResponse>;
}

最小 Agent 的伪码可以先写成下面这样。注意最后的 tool call 分支暂时只是占位，它提醒我们 loop 未来会继续扩展。 

async function run(query: string) {
  history.add(userMessage(query));

  for (let round = 1; round <= maxRounds; round += 1) {
    const messages = history.getMessages();
    const response = await llm.chat(messages);
    history.add(response.message);

    if (!response.toolCalls.length) {
      return response.text;
    }

    // 第 02 章：
    // permission -> execute tool -> append tool result -> continue
  }

  return "达到最大轮次，停止本轮执行。";
}

设计直觉

这一章的核心设计不是“能聊天”，而是“把能聊天这件事写成可替换、可测试、可继续生长的 loop”。所以我们宁愿多几个小模块，也不把所有逻辑塞进一个回调。

工厂函数和闭包是这个项目的基本风格。它们比 class 更直接：createHistory() 内部有私有数组，外部只能通过接口读写；createAgent() 内部拿到依赖，但不负责创建依赖。这种写法让学生很容易看见状态在哪，也让测试可以替换依赖。

架构拆分

config.ts
读取环境变量和模型配置。它只负责配置，不参与 loop。
terminal.ts
封装命令行输入输出。后续权限确认也会复用同一个 terminal。
history.ts
保存 user、assistant、tool 等消息。第 01 章先只用 user 和 assistant。
llm.ts
把 messages 发给模型，并把 provider 返回值收敛成 Agent 能理解的结构。
agent.ts
拥有主循环。它不读取环境变量，不创建 terminal，不知道用户界面长什么样。
repl.ts
负责交互循环：读用户输入，调用 agent.run()，打印最终文本。
index.ts
组装根。创建共享实例，按依赖关系接线，然后启动 REPL。

状态与边界

History 是 session 内工作上下文
它会影响下一次 LLM 请求。第 01 章可以先放在内存里；后续持久化 Task、Memory、Transcript 会分别解决长期状态和审计问题。
System Prompt 与普通消息不同
System prompt 是稳定规则，不是用户对话。实现时最好单独存储，在 getMessages() 时放到消息头部，避免被普通消息处理逻辑误伤。
Agent 不应该拥有 terminal
Agent 返回字符串，REPL 决定如何展示。这样同一个 Agent 可以被 REPL、测试、eval driver、子智能体工具复用。
LLM 失败要能被调用方看见
第 01 章可以先返回清晰错误或抛出错误；后续第 11 章会把错误分类、重试和压缩恢复做完整。

Prompt Card

Prompt Card

目标：实现一个最小 TypeScript Agent Loop：REPL 读取用户输入，Agent 保存 History，调用 LLM，返回 assistant 回复。

场景：这一章只支持普通文本对话，不实现工具调用、权限、压缩、记忆或持久化任务。

模块：新增 config.ts、terminal.ts、history.ts、llm.ts、agent.ts、repl.ts、index.ts。每个模块导出 interface 和 createXxx() 工厂函数。

接线：index.ts 创建 config、logger、terminal、history、llm、agent、repl；共享依赖通过参数注入，不在业务模块内部 new。

边界：REPL 只处理输入输出；Agent 只处理 loop；History 只保存消息；LLM Client 只处理模型请求。不要提前实现 tool registry。

例外和 corner case：空输入不写入 History；exit/quit 退出；LLM 返回空文本时给出可读提示；达到最大轮次时停止。

验证：用 fake LLM 测 agent.run()；断言 History 的 user/assistant 顺序；断言第二轮 messages 包含第一轮上下文；断言 REPL 不把空输入传给 Agent。

源码入口：src/agent.ts、src/history.ts、src/repl.ts、src/index.ts、src/agent.test.ts。

容易踩坑

Design Trap · A

常见错误：在 agent.ts 里直接读取环境变量、创建 LLM client、创建 terminal。

为什么错：Agent 会变得不可测试，也无法在子智能体或 eval driver 中复用。

正确做法：让 index.ts 创建共享依赖，agent.ts 只接收接口。

Design Trap · B

常见错误：不保存 assistant message，只保存 user message。

为什么错：下一轮模型看不到自己上一轮承诺过什么，多轮协作会断裂。

正确做法：每轮 user 和 assistant 都进入 History；后续 tool result 也会进入同一条消息链。

Design Trap · C

常见错误：测试只断言最终文本，不观察传给 LLM 的 messages。

为什么错：最终文本可能碰巧正确，但 history 顺序、system prompt 位置、上下文保留都可能是错的。

正确做法：fake LLM 要记录收到的 messages，测试检查消息顺序和轮次。

如何验证

Validation

单元测试：fake LLM 固定返回“收到”，调用 agent.run("你好") 应返回“收到”。这证明 Agent 能完成最小闭环。

History 测试：添加两条消息后，getMessages() 返回副本且顺序不变。这证明外部无法偷偷改内部数组。

多轮测试：连续两次调用 agent.run()，第二次 fake LLM 收到的 messages 包含第一轮 user 和 assistant。这证明上下文保留正确。

REPL 测试：空输入不调用 Agent，exit 关闭 terminal。这证明交互层边界清晰。

结构检查：index.ts 中能一眼看出依赖创建顺序；业务分支不应散落在 composition root。

如果实现失败，先查哪里

  1. 如果第二轮忘记第一轮，先查 history.add(assistant) 有没有漏。
  2. 如果测试很难写，先查 Agent 是否在内部 new 了真实 LLM 或 terminal。
  3. 如果 system prompt 被当成普通对话压进 History，先查 History 是否把 system prompt 单独存储。
  4. 如果 REPL 行为混乱，先查空输入、退出命令和异常显示是否都在 REPL 层处理。
  5. 如果代码越来越难读，先查 index.ts 是否开始承载业务逻辑。

本章练习

试着写一段 prompt，让另一个 coding agent 实现这一章。写完后用下面三条自检：

  1. 有没有明确“不要实现工具调用”。
  2. 有没有要求 fake LLM 记录收到的 messages。

本章小结

本章把第 00 章的运行图落成 TypeScript 模块：REPL 管交互，Agent 管 loop，History 管上下文，LLM Client 管模型请求，Composition Root 管接线。这个形状会一直延续到后续章节。
