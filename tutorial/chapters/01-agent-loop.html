第 01 章 · 真正落 TypeScript

最小 Agent Loop: 从伪码到可测试模块

第 00 章讲了"为什么需要 harness", 并给出了一份 6 段 Prompt Card 模板。 这一章按那张卡片把最小 agent loop 真正写成 TypeScript 模块: History、 LLM Client、 Agent、 REPL、 Composition Root 五件套, 加上 fake LLM 的 messages 顺序断言。 读完后, 你能跑通一份 "两轮对话保留上下文" 的最小 harness, 并能用测试验证它不是偶然跑通。

为什么 agent 实际是个 loop

在写代码之前, 先把"agent loop" 这个抽象放下, 用一个真实问题看 为什么它必须存在。

假设你做一个"能聊天的程序", 用户输入 "Hello", 程序输出 "Hi!" — 30 行能搞定。 但用户接着输入 "Do you remember?" 时, 朴素实现 就崩了: 程序不知道上一轮说过 "Hello", 也知道上一轮回过 "Hi!", 它只看到这一句 query, 无法回答"remember what"。

朴素想法 1: "在 globalChatHistory 里存所有消息。" 能解决 2 轮对话, 但跑 10 轮、100 轮后, 每次调 LLM 都要把全部 历史塞进去, 费用涨 10 倍、延迟涨 10 倍。

朴素想法 2: "只保留最近 5 轮对话。" 费用降了, 但"我想调出 30 轮前的 tool 调用" 答不上来。 信息 永远在丢。

朴素想法 3: "干脆不用 LLM, 写死状态机?" 那不是 agent, 是 chatbot。 失去 LLM 的灵活性, 一切问题都要 工程师手写。

正确做法: 把"对话"建模成 loop — while (LLM 还要调工具) { 调 LLM; 调工具; 把结果喂回去 }。 history 是"持续累积的状态", loop 是"持续重读的放大镜"。 每一轮 LLM 都看到完整 history, 但只决定"下一步做什么", 上下文由 harness 管理。

这个 loop 不是教学简化 — 它是任何 agent (AutoGPT / Claude Code / Cursor Agent) 的最小骨架。 后面 14 章所有功能 (工具 / 权限 / 压缩 / 子 agent) 都是这个 loop 的扩展, 不 替换它。

Agent Loop 是什么 — 不是什么

在讲代码之前, 把"agent loop" 这个词锁紧, 避免你读到后面 还在想"我到底在写什么"。

Agent loop 是:

  1. 把用户输入写入 history (对话历史)
  2. 把 history 转成 messages, 调一次 LLM, 把回复写回 history
  3. 看 LLM 是否要调工具: 要就执行, 不要就返回文本

Agent loop 不是:

  • "循环调 LLM 直到用户满意" — 没终止条件, 会无限循环
  • "AI 自动完成任务" — agent loop 只是骨架, 任务完成靠工具执行
  • "智能对话" — "智能" 是 LLM 的事, loop 只是"持续放大镜"

任何一上来就讲"完整 agent" 的教程, 都在偷跑前 3 章的债。 你不知道 harness 哪部分是核心、哪部分是装饰, 改的时候不知道 改哪里。 这一章写的 loop 是"骨架", 后面 14 章都是"在骨架上 挂器官"。

真实失败故事: 一个 50 行脚本的第 11 天

假设你按"最小能跑" 的思路写了一个 50 行的 chat 函数 (反例在下一节), 前 3 天跑得挺好。 第 11 天, 你想加一个"上下文超长就压缩" 的功能, 发现做不到:

  1. history 是 module-level 单例, 多 agent 共享: 你想加 subagent 让长任务委托出去, 结果父子 history 串了。
  2. 没有 fake LLM 入口: 想加"基于 mock 验证 messages 顺序" 的测试, 但 openai SDK 是直接调的, mock 要侵入单例 全局变量, 写不出来。
  3. tool 调度写死在函数里: 第 02 章要给 loop 加工具, 结果发现需要改 50 行里的 20 行, 因为 if/else 全在主函数里。

这 3 个坑都是"骨架没分清" 造成的。 这一章写的 5 件套, 就是为了 让"加新功能" 只需要挂新模块, 不改骨架。

朴素反例: 一个文件 50 行的 loop

写 agent 最朴素的做法: 一个文件, 一个函数, 全局变量存 history, 直接调 OpenAI SDK。 50 行能跑通, 但跑 10 分钟后就崩。 

// ❌ 反例: 全局 history + 直接调 SDK + 50 行
let history: Message[] = [];

export async function chat(userInput: string): Promise<string> {
  history.push({ role: "user", content: userInput });
  const completion = await openai.chat.completions.create({
    model: "gpt-4",
    messages: history,
  });
  const reply = completion.choices[0].message.content ?? "";
  history.push({ role: "assistant", content: reply });
  return reply;
}

GitHub · src/agent.ts 真实 agent loop 实现 (L1)

3 件事立刻坏掉:

  1. 测试写不出: openai.chat.completions.create 是真实网络调用, 单测要 mock SDK, 断言 messages 顺序还要侵入全局变量。 实际写测试时, 你会发现自己要么 monkey-patch 全局 openai 客户端 (脏), 要么用 jest.mock (侵入性大)。
  2. 换模型写不出: 换 Claude / Gemini 时, chat.completions.create 的字段名不一样, 整个函数重写。 你想支持多模型路由, 只能复制粘贴 50 行 × 3, 维护噩梦。
  3. 多 agent 共享会污染: 子智能体拿到的 history 和父 agent 一样, 跨上下文泄露。 你想做 subagent, 但子 agent 看到的所有工具调用 都在 history 里, 它误以为是自己的。

这 3 个问题不是"50 行写得不够好", 是"骨架设计错了"。 下面 5 件套就是解药。

3 条不变量

把反例的 3 个坏处倒过来, 就是这一章的 3 条不变量。 不变量是"写代码时 心里时刻记着的 3 件事", 不是"代码里写 3 条注释"。

不变量 1 · History 在闭包内, 不在 module-level
让 history 由 createAgent() 创建, 存在闭包里。 每个 agent 实例有独立 history, 父子隔离自然成立。 这是"工厂模式" 在 agent 里的具体应用 — 后面 04 章 SubAgent 会 再用一次。
不变量 2 · LLM 是窄接口, 不直接调 SDK
定义 LLMClient.chat(messages, tools) 接口, 默认实现调 OpenAI SDK, 测试用 fake。 换模型 = 换 LLMClient 实现, 业务代码不动。 这是"Adapter 模式" 的具体应用 — 后面专题 A 讲 Anthropic / Google 时会扩展。
不变量 3 · Composition Root 唯一组装依赖
所有"用哪个 LLM / 哪个 history / 哪个 logger" 都在 index.ts 的 main() 里一次性创建, 通过 依赖注入传给 agent。 模块内部不 import 其他模块的实现。 这是"控制反转 + 依赖注入" — 让业务模块纯净, 0 改动换测试。

为什么用工厂模式 (而不是单例)

"工厂模式" 在面向对象里是老生常谈, 但在 TypeScript 写 agent 时 有特别的意义, 必须单独讲一遍。

朴素想法: "单例不就行了吗, agent 又不创建多个。" 错。 至少 3 个场景会创建多个 agent 实例:

  1. REPL + SubAgent: REPL 里有一个主 agent, 用户说 "再开一个窗口做这个任务", 主 agent 调 run_subagent 创建子 agent。 父子要独立 history。
  2. 测试隔离: 每个测试 it("...") 需要一个 干净的 agent 实例, 不能被前一个测试污染。
  3. 多 session: 未来 harness 可能支持"同时跑 2 个 agent, 一个做 A 一个做 B", 各自独立 history。

单例写法短 1 行, 但加新场景时改 5 处; 工厂写法长 3 行, 但 加新场景时改 0 处。 这就是工厂模式在 agent harness 里的"复利"。

五件套: 用处 → 场景 → 设计 → 实现

最小 agent loop 拆成 5 个模块, 每个模块单一职责。 拆模块的边界 原则: "如果这一块要换实现, 改的代码越少越好"。

模块 职责 依赖 换实现时改哪里
history.ts 管理对话消息序列 无 替换存储后端 (内存 / 文件 / Redis)
llm.ts LLMClient 接口 + 默认 OpenAI 实现 SDK (可选) 换模型 / 换 SDK
agent.ts agent.run() 6 步骨架 history, llm 几乎不改, 业务逻辑入口
repl.ts readline 循环 + 命令分发 agent, terminal 换交互方式 (CLI / GUI / Web)
index.ts Composition Root, 唯一组装点 所有上述 换测试 / 换部署, 改这一个文件

下面逐个讲。 每个模块按"用途 → 真实场景 → 设计思想 → 实现细节" 展开。

History · 对话历史

用途: 持久化"LLM 当前看到的所有消息"。 它是 loop 的 "事实状态", loop 每次迭代都从 history 读 messages 喂给 LLM, 把 LLM 回复和工具结果追加回 history。

真实场景: 用户说 "刚才我让你改了什么?", LLM 答 "抱歉我不记得"。 原因就是 history 丢了。 一个 agent 跑 10 轮后 history 必须完整保留, 才能让 LLM 看到"过去发生了什么"。

设计思想: History 的 API 必须收窄。 任何"能不能加一个 search" 的诱惑都先拒绝 — 后面有 hook / memory 时再扩展。 这一版的 4 个方法, 覆盖了 80% 的场景: add / getMessages / size / replaceEntries。 多一个方法, 就多一个 "会不会被滥用" 的地方。

实现细节: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
interface History {
  add(entry: HistoryEntry): void;
  getMessages(): ChatCompletionMessageParam[];
  size(): number;
  replaceEntries(entries: HistoryEntry[]): void;  // 压缩用
}

GitHub · src/history.ts History 实现 (L1)

4 个方法的边界:

  • add 只追加, 不修改历史: LLM 已经看到过的消息 不能改, 否则会出现"LLM 之前看到 A, 现在看到 A'" 的不一致。
  • getMessages 是唯一出口: 返回 ChatCompletionMessageParam[], 喂给 LLM。 history 内部可以有 metadata (时间戳 / token 数), 但出口必须干净。
  • size 估算 token, 触发压缩: 简单按字符数估算, 中文字符 × 1.5, 英文 × 0.25。 不需要精确, 数量级对就行。
  • replaceEntries 专给压缩用, 普通路径不调: 第 06 章 P2 全量压缩时, 把"最近 6 块 + 之前总结成 1 块" 整体替换。 这个方法权限大, 必须有显式调用点, 不能被普通业务乱调。

LLMClient · LLM 抽象

用途: 唯一跟"真实 LLM" 打交道的入口。 业务模块 (agent / tool) 不调 OpenAI SDK, 调 LLMClient.chat() 一个 方法。 换模型 / 换 SDK / 测试, 都通过换 LLMClient 实现完成。

真实场景: 跑 100 轮对话后, 发现 Anthropic Claude 比 OpenAI gpt-4 在长上下文表现更好, 想切 Claude。 如果业务代码直接 调 openai.chat.completions.create(), 切换要改 20+ 处; 调 LLMClient.chat(), 切换只要换 LLMClient 实现。

设计思想: 接口故意只暴露 chat。 temperature / max_tokens / streaming 都不在第一版接口里, 是为了不让上层业务和 "这个 LLM SDK 有这个参数" 绑死。 后续章节会逐步加, 但加在 llm.ts 内部, 不外泄到 agent.ts。

实现细节: 

interface LLMClient {
  chat(args: {
    messages: ChatCompletionMessageParam[];
    tools?: ChatCompletionTool[];
  }): Promise<LLMResponse>;
}

interface LLMResponse {
  content: string | null;
  toolCalls: ChatCompletionMessageToolCall[];
  finishReason: string | null;
}

GitHub · src/llm.ts LLMClient 接口 + Adapter (L1)

为什么 LLMResponse 字段这么少? 答: harness 关心的就是"模型说了什么, 调了哪些工具, 为啥停"。 这 3 个字段直接对应 history.add() 的 3 个参数。 多一个字段, 就多一个"会不会被滥用" 的地方。

Agent · 主循环

用途: 整个 harness 的业务入口。 用户调 agent.run(query), agent 负责"写到 history → 调 LLM → 调工具 → 返回结果" 的完整流程。 所有其他模块 (history / llm / tool) 都被 agent 协调。

真实场景: REPL 收到用户输入, 调 agent.run("帮我 看看 README"), agent 内部: 写 user 到 history → 调 LLM → LLM 返回 assistant (调 run_read) → 调 run_read 读 README → 写 tool result 到 history → 调 LLM 续问 → LLM 返回 assistant (没 tool call) → 返回最终回复。

设计思想: 6 步骨架, 每步职责单一, 用 for (;;) 循环而不是 for (let i = 0; i < maxRounds; i++)。 早期版本用后者更安全, 但后续会引入 TODO + SubAgent, "最大轮数" 应该由子 agent 自己的 maxRounds 控制, 不是主 loop 强制。 主 loop 只看"用户有没有得到回复", 终止条件是 LLM 返回无 tool_call。

实现细节: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export function createAgent(deps: { llm: LLMClient; history: History }): Agent {
  return {
    async run(query: string): Promise<string> {
      // 第 1 步: 用户输入写入 history
      deps.history.add({ role: "user", content: query });

      for (;;) {
        // 第 2 步: 从 history 拿 messages
        const messages = deps.history.getMessages();
        // 第 3 步: 调 LLM
        const response = await deps.llm.chat({ messages });
        // 第 4 步: 把回复写回 history
        deps.history.add({
          role: "assistant",
          content: response.content,
          tool_calls: response.toolCalls,
        });
        // 第 5 步 (本章占位): 没有 tool call, 返回最终回复
        if (response.toolCalls.length === 0) {
          return response.content ?? "";
        }
        // 第 5 步 (第 02 章展开): 有 tool call, 执行
        // 第 6 步: 回到第 2 步继续 loop
      }
    },
  };
}

REPL · 终端循环

用途: 让用户和 harness 对话。 用户在终端输入, REPL 调 agent.run(), 把回复打回去。 是 harness 的 "I/O 边界" — 唯一接触终端的地方。

真实场景: 用户跑 npm run dev, 看到 > 提示符, 输入 "Hello", 看到 "Hi!", 输入 /exit 退出。 REPL 是这个交互循环的物理实现。

设计思想: REPL 故意薄。 它只做 3 件事: 读一行, 调 agent.run, 打印回复。 复杂逻辑 (TODO 进度条 / SubAgent 状态) 不进 REPL, 那是 agent 自己的事。 REPL 越薄, 越容易换 (CLI → GUI → Web, REPL 都不动, 业务模块都不动)。

实现细节: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export async function startRepl(agent: Agent, terminal: Terminal): Promise<void> {
  terminal.println("Mini Agent REPL. 输入 /exit 退出。");
  for (;;) {
    const line = await terminal.question("> ");
    if (line === "/exit") break;
    const reply = await agent.run(line);
    terminal.println(reply);
  }
}

Terminal 也是注入的, 不是 readline 直接 import。 这样 测试可以注入"自动回答" 的 fake terminal, 不需要真的 readline。

Composition Root · 唯一组装点

用途: 唯一 import 一切、new 一切的地方。 业务模块 (agent / history / llm) 内部不 import 彼此, 只接收注入。 换测试 = 换 main(), 业务代码 0 改动。

真实场景: 你跑测试, 写 testMain() 注入 fake LLM / fake terminal; 你部署到 GUI, 写 guiMain() 注入 GUI 终端; 你部署到 CI, 写 ciMain() 不启动 REPL, 直接调 agent.run()。 三种 main 共享同一份业务模块。

设计思想: 这是经典依赖反转原则 (DIP) 的 极端形式。 业务模块不再"创建依赖", 而是"接收依赖"。 创建在哪 里? 集中在 1 个 main() 函数。 这避免了"业务模块 互相 new 对方" 的混乱图, 让依赖关系"塌缩成一个点"。

实现细节:

loop 在仓库里的位置

fake LLM 测试: 验证不是偶然跑通

朴素做法"跑通" 是最危险的。 你跑了 1 次, 它通了, 你就觉得 "代码没问题", 改天改一行又挂了, 不知道为什么。

设计思想: 关键洞见是 fake LLM 不仅能"假装模型", 还能"捕获模型收到的所有 messages"。 每次 chat() 调用 时, 真实 LLM 收到什么 messages, fake LLM 也能拿到一份。 我们 用这个能力写断言。

为什么不用 snapshot 测试? snapshot 测试对比 messages 的 完整结构 (role / content / tool_calls 全部字段)。 问题: 任何"无害的字段添加" (比如新增 metadata), 都会让 snapshot 挂。 我们用 toContainEqual 断言关键消息 存在, 不在意其他字段。 测试更稳定。

实现细节: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
function createFakeLLM(responses: LLMResponse[]): LLMClient & { lastMessages: () => any[] } {
  let i = 0;
  const calls: { messages: any[]; tools?: any[] }[] = [];
  return {
    async chat({ messages, tools }) {
      calls.push({ messages, tools });
      if (i >= responses.length) throw new Error("no more fake responses");
      return responses[i++];
    },
    lastMessages() { return calls[calls.length - 1].messages; },
    allCalls() { return calls; },
  };
}

test("agent.run 第二轮能看见第一轮的 user + assistant", async () => {
  const llm = createFakeLLM([
    { content: "Hi!", toolCalls: [], finishReason: "stop" },
    { content: "Yes, I remember.", toolCalls: [], finishReason: "stop" },
  ]);
  const agent = createAgent({ llm, history: createHistory() });

  await agent.run("Hello");
  await agent.run("Do you remember?");

  // 断言: 第二次 chat() 收到的 messages 包含第一次的 user + assistant
  const lastCall = llm.allCalls()[1];
  expect(lastCall.messages).toContainEqual({ role: "user", content: "Hello" });
  expect(lastCall.messages).toContainEqual({ role: "assistant", content: "Hi!" });
  expect(lastCall.messages).toContainEqual({ role: "user", content: "Do you remember?" });
});

3 个常见误解

误解 1 · "History 是给用户看的"
错。 History 是给 LLM 看的, 用户看的是 REPL 渲染。 History 里有 tool_call / tool_result / metadata, 这些不进 REPL 显示。 混了之后, "REPL 怎么不显示完整 tool result?" 这种问题答不上来。
误解 2 · "LLMClient 是单例"
错。 LLMClient 也是工厂创建的, 多个 agent 实例可以共享一个 LLMClient (避免双倍费用), 也可以各自独立 (测试场景)。 "共享 vs 独立" 是 Composition Root 决定的, 不是 LLMClient 决定的。
误解 3 · "Loop 是 for (let i; i < N; i++)"
错。 Loop 是 for (;;), 终止条件是 LLM 返回无 tool_call。 "最大轮数" 是 agent 收到的参数, 由主 agent / SubAgent / 评测 case 各自决定, 不在 loop 内部硬编码。

反例梯度

新手错法 · A

常见错误: module-level let history = []。

为什么错: 子智能体共享父 agent 的 history, 跨上下文泄露。

正确做法: history 在 createAgent() 闭包内。

中级错法 · B

常见错误: agent.ts 直接 import OpenAI from "openai"。

为什么错: 测试要 mock SDK, 换模型要重写。

正确做法: agent.ts 只用 LLMClient 接口, SDK 在 llm.ts 内部 import。

高级错法 · C

常见错误: Composition Root 不存在, 业务模块互相 import。

为什么错: 子智能体无法换 LLM, 测试无法换 fake。

正确做法: index.ts 是唯一 import 一切的地方, 业务模块只接收注入。

边界错法 · D

常见错误: 测试断言"最终回复文本"。

为什么错: LLM 升级后措辞变化, 测试假阳性挂。

正确做法: 用 fake LLM 断言 messages 顺序, 不断言最终回复。
