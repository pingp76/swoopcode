第 03 章 · 让多轮执行有节奏

给 Loop 加一个工作清单: TODO Manager

第 02 章接上 tool call 之后, loop 有一个新问题: 模型可以一次调 10 个工具, 也可以一直调下去不结束。这一章给 loop 加一个"工作清单"——TODO Manager。 模型每轮开始前能看到当前 TODO 状态, 每轮结束后可以更新 TODO。 这是 harness 第一次具备"自我规划"能力。

在第 02 章基础上改了什么

这一章在 agent 主循环里加一个 TODO 模块。loop 跑起来时, 每次 LLM 调用前都把当前 TODO 状态作为 system reminder 注入到 messages, LLM 返回后允许它通过一个特殊工具 (update_todo) 更新 TODO。 对应到代码, 改动集中在 3 个新文件 + 1 处改 agent.ts: src/todo.ts (新)、 src/tools/update_todo.ts (新)、src/agent.ts (改)、 src/index.ts (改接线)。

1. src/todo.ts: TODO 数据结构与状态机 (新) 2. src/agent.ts: 每轮 LLM 调用前注入 TODO reminder 3. src/tools/registry.ts: 注册 update_todo 工具

作者怎么想的: 这一章的思考链

想清楚现象
长任务下 (例如"先读 3 个文件, 改 5 行, 跑测试"), 模型在第 5 轮 已经忘了"我刚才说要先读文件"。harness 必须替它保留一份工作清单, 每轮提醒它"现在在清单第几项"。
想反例
最朴素的反例是"把 TODO 写在 system prompt 里, 让 LLM 自己维护"。 这有两个问题: 一是 LLM 不会真去维护, 它会"假装"维护 (写漂亮 文本, 实际不再看); 二是 system prompt 写 TODO 会破坏第 10 章的 cache-friendly 布局。
想接口和不变量
接口: interface TodoManager { get(), update(items), tickRound() }。 不变量三条: (1) TODO 状态独立于 history, 不进 messages 序列, (2) 每轮 LLM 看到的 TODO 是 reminder 形式 (system-reminder user 消息), 不污染 system prompt 稳定前缀, (3) TODO 状态机: pending → in_progress → completed, 不允许跳跃 (例如从 pending 直接 completed)。
想怎么验证
fake LLM 第一轮返回 assistant 携带 update_todo 工具调用, 跑完后断言 TODO 状态变了; 第二轮 LLM 收到的 messages 包含一条 reminder 消息, content 描述新的 TODO 状态。

先观察: 两段故意有气味的实现

观察 1 · 把 TODO 塞进 system prompt

// 教学简化版, 仓库真实实现见 GitHub 永久链接
const systemPrompt = `你是一个编码助手。当前 TODO: ${JSON.stringify(todos)}`;
const messages = [{ role: "system", content: systemPrompt }, ...history.getMessages()];

问:看上去简洁, 为什么仍然不行?

答:两件事同时坏掉 —— cache: system prompt 一旦包含 TODO 状态, 每次调用 LLM 的 system prompt 都不同, 整个 prompt cache 失效, token 成本涨 3-5 倍; 真实性: LLM 不会真去更新 system prompt 里的 TODO 字符串, harness 必须用结构化数据维护 TODO, 不能用自由文本。

观察 2 · TODO 状态机不限制跳跃

// 教学简化版
class TodoManager {
  update(items) { this.items = items; }  // 直接覆盖
}
}

问:为什么不直接覆盖, 要限制状态机?

答:LLM 倾向于"全部 completed 一笔勾销"。如果不限制, 模型会在第一轮就把所有 TODO 标成 completed, 之后即使它根本没做完, harness 也不知道。这种"假装做完了"是红灯 3 的典型发作。

接口形状: 在写实现前钉死

  isAllCompleted(): boolean;
}

状态机: 不允许跳跃

TODO 状态机是本章的关键不变量。它有三条规则:

  1. 新增: 任意状态都可以新增一条 pending。
  2. 激活: pending → in_progress, 任何时候一个清单只允许一条 in_progress。
  3. 完成: in_progress → completed, 不允许 pending → completed (跳过 in_progress)。

这三条规则保证 TODO 状态对 LLM 是可读的: 任意时刻, "in_progress" 那条就是"我现在在做的事"。如果 LLM 想把 pending 直接标 completed, harness 在 update() 里 reject 这条, 写一条 error tool message 告诉 LLM。

loop 注入点: reminder 而不是 system prompt

TODO 状态变化很频繁 (几乎每轮都可能变), 但它不能进 system prompt。 折中方案: 每轮 LLM 调用前, TODO 状态被格式化成 reminder 字符串, 作为 user 消息注入到 messages 末尾。 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
async function run(query: string): Promise<string> {
  history.add({ role: "user", content: query });

  for (;;) {
    // 第 03 章新增: 每轮注入 TODO reminder
    const reminder = todo.tickRound();
    if (reminder) {
      history.add({ role: "user",
        content: `<system-reminder source="todo">\n${reminder}\n</system-reminder>` });
    }

    const messages = history.getMessages();
    const assistant = await llm.chat(messages);
    history.add(assistant);

    if (!assistant.tool_calls || assistant.tool_calls.length === 0) {
      return assistant.content;
    }

    for (const call of assistant.tool_calls) {
      const tool = registry.get(call.name);
      if (!tool) { /* ... */ continue; }
      const result = await tool.execute(call.args);
      history.add({ role: "tool", tool_call_id: call.id, content: result.content });
    }
  }
}

reminder 用 <system-reminder> 标签包起来, LLM 看到这种 标签就知道"这是 harness 注入的, 不是用户写的", 不会把它当成普通 user 消息去回应。

反例梯度

新手错法 · A

常见错误:TODO 状态作为 module-level 单例。

为什么错:第 04 章子智能体需要独立的 TODO 列表, 单例会让子智能体的 TODO 污染主 agent。

正确做法:本章已示范, createTodoManager() 是工厂, Composition Root 在 index.ts 创建并注入。

中级错法 · B

常见错误:不限制状态机, 允许 LLM 把 pending 直接 completed。

为什么错:LLM 倾向于"全部勾选", harness 必须强制它走过 in_progress, 否则 "in_progress 是不是当前正在做" 这条不变量就崩了。

正确做法:在 update() 入口校验, 拒绝 pending→completed 跳跃, 写 error tool message。

高级错法 · C

常见错误:reminder 直接放在 system prompt 末尾。

为什么错:破坏第 10 章的 cache-friendly 布局, system prompt 应当是稳定前缀, TODO 是动态状态。

正确做法:reminder 作为 user 消息, 标签 <system-reminder source="todo">。

边界错法 · D

常见错误:LLM 调 update_todo 时不传 tool_call_id, 或者 role 写成 "user"。

为什么错:回顾第 02 章的 tool call 协议, role 必须是 "tool", id 必须来自 LLM。

正确做法:第 02 章的约束全部沿用, 不要在本章单独发明一套协议。

Validation · 第 03 章

TODO 状态机拒绝跳跃:fake LLM 调用 update_todo 试图把 pending 直接 completed, harness 写一条 error tool message, TODO 状态没变。

reminder 不在 system prompt:断言 messages[0].role !== "system" 仍然成立, reminder 是 user 角色。

in_progress 唯一:fake LLM 试图把两条都标 in_progress, harness 拒绝, 写 error tool message。

tickRound 幂等:连续两次 todo.tickRound() 返回同一份 reminder 字符串, 不会"消耗"状态。

回望第 00–02 章: 哪些原则在本章兑现了

  • 工厂模式:TODO 是工厂, 不是单例, 为第 04 章子智能体隔离 TODO 列表做准备。
  • 不污染 system prompt:reminder 用 user 消息 + 标签, 保留 cache-friendly 布局 (第 10 章的伏笔)。
  • tool call 协议沿用:update_todo 复用第 02 章的 Tool interface, 不发明新协议。
  • 状态机约束:不变量写进 update() 入口, 用 Validation 卡片"状态机拒绝跳跃"和"in_progress 唯一"反向断言。

Prompt Card (本章任务)

Prompt Card · 第 03 章

目标:给 agent loop 加 TODO Manager, 模型可以看到/更新工作清单, 状态机不允许跳跃。

场景:用户输入 "重构 user 组件, 先读再改再测", agent 第一轮调 update_todo 创建 3 条 pending 并把第一条标 in_progress, 之后每轮 reminder 显示当前 in_progress 那条。

模块: src/todo.ts (新) 暴露 createTodoManager(); src/tools/update_todo.ts (新) 实现 update_todo 工具; src/agent.ts 每轮 LLM 调用前注入 reminder; src/index.ts 接线 todo 到 agent 和 registry。

边界 (LLM 必须遵守的 checklist):

  • TODO 状态机: pending → in_progress → completed, 拒绝 pending→completed 跳跃
  • 同一时刻只允许一条 in_progress
  • reminder 用 <system-reminder source="todo"> 标签, role 是 "user", 不是 "system"
  • tickRound 幂等, 不修改 TODO 状态
  • TODO 数据本身不进 history.getMessages()

  • LLM 调 update_todo 把 pending 直接 completed, 写 error tool message, TODO 状态不变
  • LLM 试图激活第二条 in_progress, 写 error tool message
  • 连续两次 tickRound() 返回同一份 reminder 字符串
  • 跑完一轮后 messages[0].role !== "system" 仍然成立
  • 第二轮 LLM 收到的 messages 包含 reminder 标签, content 描述新 TODO 状态

本章小结

本章给 loop 加了一个"工作清单": TODO Manager。loop 现在每轮能看到 当前 in_progress 那条 TODO, 也能通过 update_todo 工具更新清单。状态机 不允许跳跃, 防止 LLM "全部勾选" 假装做完了。下一章 (第 04 章) 我们 会让 harness 学会"分身"——SubAgent, 让一个 agent 在 loop 内开第二个 agent 处理子任务, 但隔离 history 与 TODO。

下一章伏笔

第 03 章让 loop 有节奏, 但仍然只有一个 agent 在工作。第 04 章 SubAgent 会让 agent 在 loop 内"开第二个 agent" 处理子任务, 关键约束是: 子智能体的 history、todo、permission 都必须和父 agent 隔离, 但 tool registry 可以共享。这一章准备的"工厂模式"就是为这一步服务的。
