第 03 章 · 多轮执行有节奏

TODO Manager: 让 LLM 自己追踪执行步骤

第 02 章的 agent 能调工具了, 但跑 5 轮之后 LLM 已经忘了最初的计划 — 它在调"读 README" 时, 可能忘了"先 run_bash 看看 package.json"。 这一章给 harness 加一个 TODO Manager: LLM 可以创建 / 更新 / 完成 TODO, 这些 TODO 既给 LLM 自己看, 也给用户看。 读完后, 你能讲清 TODO 和 持久化 Task 的边界, 并能用 fake LLM 写一个"完成 3 步 TODO 才返回" 的测试。

真实失败故事: 第 7 轮失忆

写代码之前, 看一段真实长任务的失败。 跑一段对话: 用户让 agent "全项目 code review, 列出 3 个最严重的 bug"。

  1. 第 1-3 轮: agent 看了 README, 看了 2 个文件, 思路清晰。
  2. 第 5 轮: agent 调第 4 个文件, 但忘了之前看过什么, 把"我觉得 X 是 bug" 重复了 1 遍。
  3. 第 7 轮: agent 调第 6 个文件, 又开始重新调第 2 个文件 (重复看), 总结时漏掉 1 个文件, 重复 1 个文件的判断。

原因不是 LLM 笨, 是5 轮工具结果把上下文撑满, LLM 注意力 被稀释。 它"看到" 5 个文件, 但"记得住" 的只有最近 1-2 个。

朴素想法 1: "让 LLM 写总结, 把之前的工具结果总结成一段?" 能缓解, 但不根治 — 总结本身也要占 token, 总结会丢失细节。

朴素想法 2: "压缩工具结果?" 第 06 章会讲压缩, 但压缩针对"单条工具结果太长", 不解决"多步 任务的状态丢失" 问题。

正确做法: 给 harness 加"短期步骤追踪" — TODO Manager。 LLM 自己创建 TODO, 每条 TODO 状态走 turn reminder 注入 history, LLM 每轮都能看到"还有哪些没做", 用户也能在 REPL 看到进度。

为什么不能塞进 system prompt

朴素想法: "TODO 状态既然重要, 直接拼到 system prompt 字符串里?" 立刻坏 3 件事:

  1. 破坏 prompt cache: system prompt 是 stable prefix, LLM provider 缓存这段换算 1/10 价格。 TODO 状态每轮都变, 拼到 system prompt 等于让 cache 命中率归零。 成本涨 5-10 倍。
  2. 污染 system prompt 语义: system prompt 是"行为规则", "当前 TODO 状态" 是"运行时状态"。 混在一起, LLM 不知道哪部分是 规则哪部分是状态。
  3. 每轮重复生成: 拼到 system prompt 字符串, 每次 chat() 都要重新生成完整字符串 (history 拼上)。 多 100 token 不算多, 跑 100 轮就 10000 token 浪费。

解决方式: TODO 状态走 turn reminder — 每轮注入 user message, 状态变化才更新内容, 不变就不重复生成。 详细 reminder 模式见 Reference 章节的"模式 11"。

TODO 数据结构

用途: 持久化"当前 session 的步骤列表"。 给 LLM 看的 "我接下来要做什么", 给用户看的 "[2/5] 进度条"。

真实场景: 用户让 agent 跑 5 步任务, agent 创建 5 条 TODO (read / bash / write / write / verify), 跑过程中 LLM 标 in_progress → completed, 用户在 REPL 看到进度变化, 知道 agent 在做什么。

设计思想: TODO 是session 内的临时状态, 不持久化, 进程结束就丢。 这个边界很重要: 跨会话的长期任务 属于第 12 章的 Task 系统, 不要混。

实现细节: 

type TodoStatus = "pending" | "in_progress" | "completed" | "cancelled";

interface Todo {
  id: string;             // 短 hash, 不用 UUID
  content: string;        // "Read package.json"
  status: TodoStatus;
  createdAt: number;
  updatedAt: number;
}

interface TodoList {
  items: Todo[];
  // 由 TodoManager 维护, LLM 不直接调
  activeId: string | null;  // 当前 in_progress 的 id
}

GitHub · src/todo.ts TodoManager 数据结构 (L1)

4 个状态的语义边界:

状态 含义 谁设置
pending 已建, 没开始 LLM 创建 / 取消后又恢复
in_progress 正在做 LLM 标记
completed 做完 LLM 标记
cancelled 放弃 (用户改主意 / 任务已不必要) LLM 标记

为什么不让 LLM "删除" TODO? 删了之后 history 看不到, 调试时无法 追溯"为什么这一项没了"。 取消比删除更可追溯。 同样的逻辑在 git 里 也成立: git reset --hard 比 git revert 危险。

id 字段为什么用短 hash 不用 UUID? 因为 LLM 要在 tool_call 里 引用 id, 短 hash 比 UUID 更易读、易打字, LLM 出错率低。 短 hash 用内容前 8 个字符的 hash, 冲突概率 session 内可忽略。

6 个工具设计

用途: LLM 怎么操作 TODO? 通过 6 个工具。 设计原则是 "覆盖 LLM 90% 操作, 不留歧义, 不写批量工具"。

真实场景: 跑一个 5 步任务, LLM 操作 TODO 的路径是: create → list (确认) → update in_progress → (做工具调用) → update completed → create next → ... → 全部 completed。 6 个工具覆盖这个循环。

设计思想: 经典CRUD 分解 — create / list / get / update 是 4 个基础操作, clear / set_active 是 2 个辅助操作。 不写"批量更新" 工具 — 批量工具让 LLM "刷" 状态, 失去逐条追踪的 价值。

实现细节: 

run_todo_create(content: string): { id: string; content: string }
run_todo_update(id: string, status: TodoStatus): { ok: boolean }
run_todo_list(): { items: Todo[] }
run_todo_get(id: string): { todo: Todo | null }
run_todo_clear(): { ok: boolean }                  // 清空所有
run_todo_set_active(id: string): { ok: boolean }   // 标记当前在做

GitHub · src/tools/run-todo.ts 6 个 TODO 工具 (L1)

6 个工具的边界:

  1. create / update / list / get 是基础 CRUD: LLM 90% 的操作在这 4 个里。
  2. clear 留给"任务完成 / 用户重置": 不是常用, 但偶尔需要。
  3. set_active 不是 LLM 必需的: 第一个 in_progress 自动成为 active, 只有在 LLM 想"切换" 时用。 这一条留着 但少用, 别给 LLM 添堵。

❌ / ✅: 不要给 LLM 写"批量更新" 工具。 

// ❌ run_todo_bulk_update 一次改 5 条
// LLM 会用它"刷" 所有 TODO 状态, 失去 TODO 的"逐条追踪" 价值

// ✅ 一次只改一条, 显式逐条
// LLM 必须在 history 里看到"第 1 条变 completed" "第 2 条变 in_progress" 的过程

同样的逻辑: 不要写"删除" 工具。 删除是丢弃, 取消是状态变化。 状态变化留在 history 里, 删除不留。

TodoManager — 闭包 + factory

用途: 维护 TODO 状态, 提供 6 个工具给 LLM, 提供 reminder 渲染给用户。

真实场景: 创建 3 条 TODO, 把其中一条标 in_progress, 把第 2 条标 completed, 把第 3 条取消 — 4 个操作, manager 内部维护 activeId 指针, 渲染 reminder 时按 ID 排序。

设计思想: 经典工厂 + 闭包 模式 (Reference 章节的模式 1)。 状态藏在闭包内, 父子 agent 天然隔离。 不用 class, 避免 this 绑定问题。

实现细节: 

export function createTodoManager(): TodoManager {
  const items: Map<string, Todo> = new Map();
  let activeId: string | null = null;

  return {
    create(content: string): Todo {
      const todo: Todo = {
        id: hashId(content),
        content,
        status: "pending",
        createdAt: Date.now(),
        updatedAt: Date.now(),
      };
      items.set(todo.id, todo);
      return todo;
    },
    update(id: string, status: TodoStatus): boolean {
      const todo = items.get(id);
      if (!todo) return false;
      todo.status = status;
      todo.updatedAt = Date.now();
      if (status === "in_progress") activeId = id;
      else if (status === "completed" || status === "cancelled") {
        if (activeId === id) activeId = null;
      }
      return true;
    },
    list(): Todo[] { return [...items.values()].sort((a, b) => a.createdAt - b.createdAt); },
    // ... 其他方法
  };
}

GitHub · src/todo.ts createTodoManager 完整实现 (L1)

activeId 不用 LLM 主动管理, TodoManager 自己根据 update 维护。 这是"harness 的内部状态" 和"LLM 看到的" 的边界 — LLM 调工具, harness 自己维护 active 指针, LLM 不需要知道这个概念。

为什么 TodoManager 是工厂函数, 不是 class?

  1. 闭包天然隔离: 工厂返回的对象捕获 items 和 activeId, 每个 agent 实例独立闭包, 父子隔离自然。
  2. 无 this 绑定问题: TypeScript class 容易被不小心 this.items 在 callback 里丢上下文。 闭包没这个问题。
  3. 易测试: 测试时直接 createTodoManager(), 不用 mock 任何东西。

怎么进 history: turn reminder 模式

用途: TODO 状态需要被 LLM 看到, 但不能破坏 prompt cache。 走 turn reminder 模式 (user message 末尾)。

真实场景: agent 跑 5 步任务, TODO 状态变化, 下一轮 chat() 前 reminder 注入到 messages 末尾, LLM 看到 "进度 [2/5]", 知道还剩 3 步。

设计思想: 经典稳定前缀分离 — TODO 状态 走 messages 末尾 (动态 tail), 不拼到 system prompt (稳定前缀), 保持 prompt cache 命中。 这是第 10 章的核心思想, 本章是它的 第一次具体应用。

实现细节: 

function buildTodoReminder(todos: Todo[]): string {
  if (todos.length === 0) return "";
  const lines = todos.map(t => {
    const icon = t.status === "completed" ? "✓" : t.status === "in_progress" ? "→" : t.status === "cancelled" ? "✗" : "·";
    return `${icon} [${t.id}] ${t.content}`;
  });
  return `<system-reminder source="todo">
当前 TODO 状态:
${lines.join("\n")}
</system-reminder>`;
}

GitHub · src/agent.ts buildTodoReminder 实现 (L1)

注意 4 件事:

  1. 走 user message, 不走 system prompt: reminder 是动态状态, 不进 stable prefix。
  2. 每轮注入, 但只在 TODO 变化时更新内容: 不变就不重复生成。
  3. 用简洁符号: ✓ → ✗ · 比文字描述省 token, LLM 看得更清楚。 一个完成符号 + 一个 id + 描述, 总共 50 字符内, 比"Task t1 is in completed state" 省 70% token。
  4. 空列表返回空字符串: 不污染 history, 不浪费 token。

什么时候注入 reminder? 在 agent.run() 的第 2 步 (拿 messages) 之前, 调 buildTodoReminder(manager.list()), 把字符串作为最后一条 user message 加进 messages。 LLM 看到的 messages 序列:

  1. user: 原 query
  2. assistant: 调 run_todo_create
  3. tool: 创建成功
  4. user: <system-reminder> 当前 TODO 状态: ...</system-reminder>
  5. assistant: 调 run_todo_update
  6. tool: 更新成功
  7. ...

reminder 在每轮的开头出现, LLM 一眼能扫到"还有哪些没做"。

TODO vs Task: 边界

用途: TODO 和 Task 看起来都是"任务列表", 但它们 活在不同的生命周期里。 搞混了, 你要么把"今天的小事" 全建 Task Group (过载), 要么把"3 周后还要跟的计划" 塞进 TODO (丢了)。

真实场景: 用户跑 50 个 session, 用 TODO 管每个 session 内的临时步骤, 用 Task 管跨 session 的长期计划。 如果用错, session 退出后 TODO 列表没了, 3 周前的 Task Group 也不见了。

设计思想: 决策规则 (给 LLM 和教程读者同时用):

  1. 当前 session 的"3 步走完就忘" 任务 → TODO。
  2. "3 周后还要跟" 的计划, 跨项目, 多人协作 → Task。
维度 TODO (本章) Task (第 12 章)
生命周期 session 内, 进程退出就丢 跨会话, 持久化到 agentHome
用户 当前 session 的临时步骤 跨项目的长期计划, 多 owner
数据位置 agent 闭包内 ~/.swoopcode/tasks/
依赖图 无, 自由顺序 有, task 之间可以 blocks / blockedBy
是否阻塞 不阻塞, 仅追踪 可以阻塞另一个 task

这条规则要写进 system prompt, 避免 LLM 混淆。 验证方法: 跑 50 个真实 session, 看 LLM 建 Task Group 的频率。 如果每个 session 都建 3 个 Task Group, 说明 prompt 没强调边界, 改。

用户视角: 怎么看到 TODO 进度

用途: LLM 内部用 TODO 是为了"自己不忘", 用户看到 TODO 状态是副产品。 REPL 怎么展示。

真实场景: 用户跑一个 5 步任务, REPL 顶部展示 [2/5] 改 config.ts ..., 用户随时知道 agent 在做哪一步。 这不是装饰, 是给用户一个"卡住还是正常" 的判断依据。

设计思想: 渲染走独立UI 路径, 跟 LLM 看到的 reminder 走两条路。 数据同源 (manager.list()), UI 渲染不同 — LLM 看到完整 reminder, 用户看到简洁进度条。

实现细节: 

function renderTodoBar(todos: Todo[]): string {
  const total = todos.length;
  const done = todos.filter(t => t.status === "completed").length;
  const current = todos.find(t => t.status === "in_progress");
  if (total === 0) return "";
  const bar = `[${done}/${total}]`;
  return current
    ? `${bar} ${current.content}...`
    : done === total ? `${bar} 全部完成` : `${bar} 等待下一步`;
}

3 个常见误解

误解 1 · "TODO 应该进 system prompt 才有保证"
错。 走 reminder (user message) 和走 system prompt 一样能让 LLM 看到。 区别是 cache 命中率: 走 system prompt 破坏 cache, 走 reminder 不破坏。 功能上等价, 性能上不同。
误解 2 · "TODO 是给用户看的"
一半对一半错。 TODO 主要是给 LLM 看的 (避免失忆), 用户看的是 REPL 顶部那行 [2/5] 进度条。 两个 UI 独立渲染, 数据同源。
误解 3 · "持久化的 Task 也能用 run_todo_*"
错。 跨会话的 Task 走 run_task_*, 不走 run_todo_*。 边界要分清, 不要因为"今天的小事" 用 Task, 也不要因为"3 周的计划" 用 TODO。

反例梯度

新手错法 · A

常见错误: TODO 列表放 module-level 单例。

为什么错: 子智能体共享父 agent 的 TODO, 跨上下文污染。

正确做法: TodoManager 在 createAgent() 闭包内, 父子隔离天然成立。

中级错法 · B

常见错误: TODO 状态拼到 system prompt 字符串。

为什么错: 破坏 prompt cache, 每轮都重传完整 TODO 状态。

正确做法: 走 turn reminder (user message), 状态变化才更新。

高级错法 · C

常见错误: 允许 LLM 删除 TODO。

为什么错: 删了之后 history 看不到, 调试时无法追溯"为什么这一项没了"。

正确做法: 用 cancelled 状态而不是删除, 保留追溯能力。

边界错法 · D

常见错误: TODO 工具和 Task 工具都给 LLM, 让它自己选。

为什么错: LLM 不知道边界, 容易把"今天的小事" 都建 Task Group。

正确做法: system prompt + tool description 都明确"3 步以内走 TODO, 跨会话走 Task"。

Prompt Card (本章任务)

Prompt Card · 第 03 章

目标: 给 harness 加 TODO Manager, LLM 可以创建 / 更新 / 完成 / 取消 TODO, 状态走 turn reminder 注入, 父子 agent 隔离。

场景: 用户说"3 步: 读 package.json, 跑 npm test, 修 失败的 case", LLM 建 3 条 TODO, 逐条标 in_progress → completed, 用户在 REPL 看到 [0/3] → [1/3] → [2/3] → [3/3] 的进度。

边界 (LLM 必须遵守的 checklist):

  • TodoManager 在 createAgent() 闭包内, 不写 module-level 单例
  • TODO 状态走 turn reminder (user message), 不进 system prompt
  • 6 个工具, 不写"批量更新" 工具, 每次只改一条
  • 不允许删除, 用 cancelled 状态代替
  • TODO 和 Task 边界写进 system prompt: "3 步以内走 TODO, 跨会话走 Task"

验证 (用 fake LLM + vitest, 逐条断言):

  • 4 个核心工具 (create / update / list / get) 跑通, 状态正确变化
  • reminder 在 history 最后一条 user message, 不在 system prompt
  • 父 agent 建 TODO 后 spawn 子 agent, 子 agent 看不到父的 TODO
  • TODO 列表为空时, reminder 是空字符串, 不污染 history

本章小结

TODO Manager 是 harness 给 LLM 加的"短期记忆"。 核心是 4 个设计:

  • session 闭包, 不持久化: 区分于第 12 章的 Task 系统, 边界写进 prompt。
  • 6 个工具 + 4 状态: 拒绝"批量更新" 和"删除", 保留追溯能力。
  • 走 reminder, 不走 system prompt: 保持 stable prefix, prompt cache 友好。
  • 工厂模式: TodoManager 闭包, 父子隔离天然成立。
