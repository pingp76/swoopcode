第 14 章 · 定时触发复用 Async Run

Schedule: 5 秒 tick + 6 种 recurrence + missed 不补跑

第 13 章的 Async Run 是"LLM 手动启动后台执行", 但生产里用户 经常想"每天 9 点跑一次全项目测试" / "每周一 10 点扫一下 TODO" / "每 5 分钟查一下 CI 状态"。 这一章加 src/schedules.ts 模块: Schedule 是时间触发器 (不是执行器), 复用 Async Run 作为执行单元, 5 秒 tick 一次扫所有 active schedule, 6 种 recurrence (every_seconds / hourly / daily / weekly / monthly / yearly), occurrence 含 stable id 防重复触发, 启动时检测 missed occurrence 但不补跑, overlap policy 控制"上次还没跑完这次是否跳过"。 读完后, 你能讲清 "Schedule 不是 cron 替代品" 的边界 (in-process tick, 不持久 唤醒), 并能用 fake clock 验证"missed 不补" + "overlap skip" 两条 关键不变量。

真实失败故事: cron 跑了 3 次, 团队 leader 才发现

写代码之前, 先看一个真实痛点。 用户的 team 想"每天 9 点跑一次 全项目 e2e 测试", 写了 bash cron + harness 脚本, 跑了一周。

  1. 症状 1: 重复触发: 网络抖了一下, cron 以为没跑, 第二天发现"同一天跑了 2 次", 资源浪费。
  2. 症状 2: 错过补跑: 周末 9 点团队不上班, harness 进程被关, 错过 1 次。 周一上班发现"周六没跑", 手动补跑, 但周六的 audit log 缺了。
  3. 症状 3: overlap 失控: 9 点的测试跑 1 小时, 10 点 又触发一次, 2 个测试并发, 互相干扰, 结果不可信。
  4. 真问题: 缺少in-process scheduler: 复用 Async Run 作为执行单元, 用 stable occurrence id 防重复, overlap policy 决定"上次没跑完是否跳过", missed 不补跑只 记录审计。 不依赖 OS cron, 不依赖 systemd, 不依赖 k8s cronjob。

朴素想法 1: "直接用 OS cron 调 bash 启动 harness?" 错。 OS cron 不知道"上次跑完没", 不会 dedupe, 不会 overlap skip, 不会写 occurrence 审计。 需要 harness 内部 scheduler 集中管理。

朴素想法 2: "schedule 跑完直接调 run_bash 同步执行?" 错。 schedule 应该复用 Async Run, 让 schedule 本身不执行, 只触发; 执行交给 Async Run 的并发限制 / 终态幂等 / 输出隔离 (第 13 章 全套)。

正确做法: 加 src/schedules.ts — ScheduleManager 是"时间触发器", 内部 5 秒 tick 一次, 扫所有 active schedule, 到点就调 asyncRunManager.start(...) 启动 Async Run。 Schedule 自己不执行, 真实执行交给 Async Run。 看 src/schedules.ts 第 7-12 行的核心设计注释: 

// 核心设计:
// - ScheduleManager.tick(now) 是可测试的核心调度逻辑
// - Schedule 到点后创建 Async Run, 不直接执行命令或运行 LLM
// - occurrence 的 stable id 防止重复触发
// - 启动时检测 missed occurrence, 只记录最近一次, 不补跑

Schedule 是触发器, 不是执行器

用途: Schedule 自己不跑命令 / 不调 LLM, 到点就委托给 Async Run。 两条好处: (a) Schedule 不需要重复实现并发限制 / 终态幂等 / 输出隔离, 复用 Async Run; (b) Schedule 出 bug 不影响 Async Run, 反之 亦然, 关注点分离。

真实场景: 用户设 "每天 9 点跑全项目测试" 的 schedule, 9:00 触发, ScheduleManager 创建 occurrence + 调 asyncRunManager.start({executor: "command", command: "npm test"}), 拿到 run_id, 写 occurrence 记录。 Async Run 跑 5 分钟, 完事后 回调 ScheduleManager 更新 occurrence 终态 (completed / failed / timeout)。 ScheduleManager 再写 notification 让 LLM 看到。

设计思想: 层间清晰 — Reference 章节 "模式 7 · State Machine 状态机" 的延伸。 Schedule 管 active/cancelled/completed 状态, occurrence 管 triggered/skipped/missed/completed/failed/timeout/cancelled 状态, Async Run 管 running/completed/failed/timeout/abandoned 状态。 三层状态机各管各的, 不互相污染。 看 src/schedules.ts 第 613-624 行的教学导读: 

// 教学导读:
// ScheduleManager 是"时间触发器", 不是"执行器"。
// 它只负责判断某个 schedule 是否到点、创建 occurrence 审计记录、
// 然后把真正执行交给 AsyncRunManager。
//
// 这样设计可以避免两套执行生命周期:
// - Async Run 管 running/completed/failed/timeout
// - Schedule 管 active/cancelled/completed 和 occurrence 审计
//
// 学生读这段代码时, 可以把它想成一个小型调度循环:
// scan/reload -> 计算 nextRunAt -> tick 到点 -> 创建 occurrence -> start async run -> finish callback 写回终态
图 1 · Schedule × Async Run 协作时序
1. tick 到点
ScheduleManager.tick(now) 发现 schedule #5 到 9:00 了, 调 asyncRunManager.start({trigger: {kind: "schedule", scheduleId, occurrenceId}})。
2. Async Run 启动
AsyncRunManager 创建 record (status=running), 启动 executor (command 跑 npm test), 拿 run_id 返回给 ScheduleManager。
3. Schedule 写 occurrence
ScheduleManager 创建 occurrence (status=triggered, asyncRunId=<run_id>), 写 store.saveOccurrence(occ), notification 推 "triggered"。
4. Async Run 完成回调
5 分钟后 npm test 跑完, AsyncRunManager.finishRun(record, "completed", output), 触发 onFinish 回调。
5. Schedule 收敛 occurrence
onAsyncRunFinish(record) 找到 occurrence, status=completed, 写 finishedAt + outputRef, notification 推 "completed"。

实现细节: 调度循环用 setInterval(tick, TICK_INTERVAL_MS = 5000), 5 秒一次, 不需要秒级精度 (schedule 大多是分钟 / 小时 / 天级别)。 想秒级精度, 改 TICK_INTERVAL_MS 常量, 不需要改核心逻辑。

6 种 recurrence: 5 分钟到 1 年

用途: LLM / 用户创建 schedule 时指定 recurrence 类型, ScheduleManager 据此算 nextRunAt。 6 种枚举覆盖 90% 场景: every_seconds (5 分钟级) / hourly / daily / weekly / monthly / yearly。 多了维护成本上升, 少了不够用。

真实场景: "每 5 分钟查 CI 状态" 用 every_seconds (5 * 60 = 300 秒); "每天 9 点跑测试" 用 daily (9:00); "每周一 10 点扫 TODO" 用 weekly (mon, 10:00); "每月 1 号清理日志" 用 monthly (1, 00:00); "每年 1 月 1 号跑年度报告" 用 yearly (1/1, 00:00)。 6 种全覆盖。

设计思想: 固定枚举 + 字段类型 — 6 种是固定枚举, 不允许动态加第 7 种 (避免代码膨胀); 每种带 各自字段 (weekly 必须含 weekday, monthly 必须含 day), 编译 期可见, 漏字段直接 TS 报错。 看 src/schedules.ts 第 35-42 行的 recurrence import: 

import type {
  ScheduleTiming, RecurrenceRule, Weekday,
  EverySecondsRule, HourlyRule, DailyRule,
  WeeklyRule, MonthlyRule, YearlyRule,
} from "./schedule-store.js";
图 2 · 6 种 recurrence 的字段差异
every_seconds (5 分钟级)
字段: {kind: "every_seconds", seconds: 300}。 用于"每 5 分钟查 CI" "每 30 秒检查 build 状态"。
hourly (整点触发)
字段: {kind: "hourly", minute: 0}。 用于"每小时整点扫日志"。
daily (每天某时某分)
字段: {kind: "daily", hour: 9, minute: 0}。 用于"每天 9 点跑测试"。
weekly (每周某天某时)
字段: {kind: "weekly", weekday: "mon", hour: 10, minute: 0}。 用于"每周一 10 点扫 TODO"。
monthly (每月某日某时)
字段: {kind: "monthly", day: 1, hour: 0, minute: 0}。 用于"每月 1 号清理日志"。
yearly (每年某月某日)
字段: {kind: "yearly", month: 1, day: 1, hour: 0, minute: 0}。 用于"每年 1 月 1 号年度报告"。

实现细节: nextRunAt 计算纯函数, 输入 (rule + now), 输出 nextRunAt。 测试可以用任意 Date 算, 不需要等真实时间。 教学版不处理"闰秒" / "夏令时" (教学项目优先简单), 工业版要 考虑。

occurrence stable id: 防重复触发的核心

用途: 每个 schedule 到点会触发一次, 那次触发 是一次 occurrence, 必须有 stable id (基于 schedule_id + scheduledAt), 不能每次重启重新生成。 这样: (a) 启动时检测"上次该触发但没触发" 的 occurrence, 不重复创建; (b) Async Run 回调能通过 occurrenceId 找到对应 occurrence 写 回终态。

真实场景: schedule #5 配置 daily 9:00, 周一 9:00 触发 (occurrence #5_2026-06-09T09:00, id stable)。 harness 周二 重启, 启动时检测 "周一 9:00 该触发但没记录", 写一条 missed occurrence (id 仍然是 #5_2026-06-09T09:00, 不会和新触发的混淆)。 周二 9:00 触发时, occurrence id 是 #5_2026-06-10T09:00, 区分清楚。

设计思想: 幂等 id — Reference 章节 "模式 19 · Idempotent 幂等" + "模式 16 · Stable Identity 稳定身份" 的应用。 看 src/schedules.ts 第 155 行的 generateOccurrenceId: 

function generateOccurrenceId(scheduleId: string, scheduledAt: Date): string {
  const y = String(scheduledAt.getFullYear());
  const m = String(scheduledAt.getMonth() + 1).padStart(2, "0");
  const d = String(scheduledAt.getDate()).padStart(2, "0");
  const H = String(scheduledAt.getHours()).padStart(2, "0");
  const M = String(scheduledAt.getMinutes()).padStart(2, "0");
  // ... 拼成 sch___ 形式
}

实现细节: occurrence id = occ_<scheduleId>_<ISO 时间>, 时间精确到分钟 (daily 精确到 hour:minute, every_seconds 精确到 second)。 这样不同次触发生成不同 id, 同一 schedule + 同一时间 永远同一 id, 启动时检测"已存在则跳过" / "已存在则标记 missed"。

missed occurrence: 记录不补跑

用途: harness 进程在 schedule 触发时间点不在线 (周末 / 维护 / 崩溃), 错过触发, 启动时检测到 "上次该跑没跑", 写一条 missed occurrence, 但不补跑 (只记录审计, 不追溯执行)。 这是教学版的故意简化, 工业版可以配置 "missed 时是否补跑"。

真实场景: 周末 harness 进程不在线, 错过周六 9:00 的 daily schedule。 周一上班, 启动 harness, tick 检测到 "周六 9:00 没记录", 写一条 missed occurrence (audit: "missed at 2026-06-08T09:00, recovered at 2026-06-10T09:00")。 周一 9:00 该跑的还是按周一 9:00 跑, 不会"补周六 9:00 一次 + 周一 9:00 一次"。

设计思想: 故意简化 — Reference 章节 "模式 10 · Atomic Write 原子写" 的延伸, 注释明确说 "当前教学实现不回补 missed run, 只记录审计并推进到未来下一次"。 看 src/schedules.ts 第 774 行的注释: 

// 当前教学实现不回补 missed run, 只记录审计并推进到未来下一次。
// 这样可以避免"周末补跑 7 个 daily 跑 7 个命令" 的资源滥用。
// 工业级 Schedule 可能会让用户配置 missed_policy = "skip" | "backfill" | "skip_silently"。
图 3 · 启动时 missed 检测流程
1. 启动 reload
harness 启动, ScheduleManager.reloadActiveSchedules() 扫磁盘读所有 active schedule.json, 计算每个的 lastScheduledAt。
2. 检测 missed
如果 lastScheduledAt < nextRunAt < now (有该跑没跑的时间点), 写 missed occurrence (status=missed, missedAt=now)。
3. 不补跑
只记录审计, 不启动 Async Run 补执行。 nextRunAt 推到未来下一次 (今天 9:00 / 明天 9:00)。
4. 未来照常
之后的 tick 按 nextRunAt 正常触发, 不受 missed 影响。 用户的"周一 9:00 测试" 周一还是按周一 9:00 跑。

实现细节: missed 检测在 tick(now) 第一次调用时跑, 之后只算 "nextRunAt <= now" 的触发, 不再检测 missed。 简化逻辑, 避免每个 tick 都重扫历史。

overlap policy: allow vs skip

用途: schedule 到点时, 上一次 occurrence 还没 跑完 (Async Run 仍在 running), 怎么办? 两种策略: overlapPolicy: "allow" (允许并发, 启动新的 Async Run) 或 overlapPolicy: "skip" (跳过, 写一条 skipped_overlap occurrence)。

真实场景: 用户设 "每 5 分钟跑 CI 检查" 的 schedule, CI 跑了 7 分钟还没完, 下一个 5 分钟到点: allow 策略会启动新的 CI 检查 (总共 2 个 CI 并发, 互相干扰); skip 策略会跳过这次触发, 写 skipped_overlap, 5 分钟后再尝试。 大多数 schedule 用 skip 更安全。

设计思想: 用户可配 + 默认安全 — 字段在 schedule.execution 里, 用户创建 schedule 时指定; 默认 skip (教学项目优先安全, 并发容易出问题)。 看 src/schedules.ts 第 645 行的 runningOccurrences: 

// 跟踪正在 running 的 occurrence, 用于 overlap 检测
const runningOccurrences = new Map<string, Set<string>>(); // scheduleId -> Set<occurrenceId>

实现细节: overlap 检测在 tick 到点时跑: (a) 查 runningOccurrences.get(scheduleId) 是否非空; (b) 非空 + policy=skip → 写 skipped_overlap occurrence, 不启动 Async Run; (c) 非空 + policy=allow → 直接启动新 Async Run (Async Run 自己有 3 并发限制, 可能 reject)。 runningOccurrences 在 Async Run finish 回调里 remove (onAsyncRunFinish 内)。

linked task: 触发后更新 task 状态

用途: schedule 可以链接到一个 task (groupId + taskId), occurrence 触发后根据结果更新 task 状态: linkedTaskUpdate: "never" (不更新) / "append_note" (追加 note) / "mark_failed_on_failure" (failed 时把 task 标 failed)。

真实场景: 用户设 "每天 9 点跑 nightly test" schedule, 链接到 migration-001 group 的 task_nightly_test。 test 通过 → 不改 task 状态; test 失败 → task 标 failed + note 写 "auto-failed by schedule at 2026-06-12T09:05:23, reason: 5 tests failed"。 LLM 下次开 session 看到 task failed, 知道 "夜跑挂了, 去看下"。

设计思想: 桥接 schedule ↔ task — Reference 章节 "模式 6 · Pipeline 管道" 的应用。 schedule 不 知道 task 内部结构, 只通过 linkedTask 字段 + 3 种 update policy 触发 task 状态变化。 task 状态变化由 TaskManager (第 12 章) 处理, 保持职责分离。

实现细节: linked task 写在 schedule.linkedTask = {groupId, taskId}, 触发后 onAsyncRunFinish 回调里根据 result 调 taskManager.updateTask(groupId, taskId, patch)。 3 种 policy 对应不同的 patch 构造。

output policy: save / notify / summary

用途: occurrence 完成后, 输出的处理策略: saveRawOutput: true (写到 OutputStore LRU) / notifyLlm: true (推 notification) / summaryPrompt?: string (用 prompt 让 LLM 生成摘要 塞进 notification)。

真实场景: nightly test schedule 配置 saveRawOutput: true, notifyLlm: true, summaryPrompt: "总结测试结果, 失败的列前 3"。 test 跑完后, 原始输出 5MB 写 OutputStore, LLM 调 summaryPrompt 生成 "5/100 失败, Top 3: [list]", notification 推给 LLM。 LLM 看到摘要, 想看全文调 readOutput(runId)。

设计思想: 短摘要 + 长全文分离 — Reference 章节 "模式 13 · Error→Action 错误转动作" 的延伸。 notification 短, output 长, LLM 按需读, history 不撑爆。

实现细节: summaryPrompt 是可选, 有就调 LLM 生成摘要, 没有就用 Async Run 的 preview 字段。 LLM 调生成 摘要本身是一个小 Async Run (嵌套), 教学版可以简化为 "用 preview 字段当摘要"。

permission profile: readonly / ci / workspace_write

用途: schedule 触发的 Async Run 可能有写操作 (如"每天 9 点跑测试 + 修复") 或纯只读 ("每天 9 点扫日志")。 3 种 permission profile: readonly (只读命令) / ci (允许 build / test / install) / workspace_write (允许 workspace 内写)。

真实场景: nightly test schedule 用 ci profile (允许 npm install + npm test); "每天清理 7 天前的日志" schedule 用 workspace_write (允许 rm 工作区文件); "每天扫 TODO" 用 readonly (只 grep 不改)。 3 种 profile 表达"这个 schedule 的写 权限边界"。

设计思想: 3 种 profile 表达权限边界 — Reference 章节 "模式 3 · 窄接口 + Adapter" 的应用。 permission profile 透传给 Async Run 的 executionPolicy, 让 Async Run 自己 校验。 Schedule 不实现权限, 只声明。

实现细节: profile 字段在 schedule.execution.permissionProfile, 透传到 asyncRunManager.start({..., permissionProfile: "ci"})。 Async Run 内部用 executionPolicy 校验每个 command 是否在 profile 允许范围内, 不在就 reject。

start / stop: 进程内 timer 生命周期

用途: ScheduleManager 自己持有 setInterval timer, start() 启动 timer, stop() 清除 timer + 把 running occurrence 标 abandoned。 进程退出时 stop() 自动调用 (或者外部 hook), 不留 zombie。

真实场景: harness 启动, scheduleManager.start() 启动 5 秒 tick 循环, 开始调度。 user ctrl-c 退出, harness 收到 SIGINT, 调 scheduleManager.stop(), timer 清除 + running occurrence 收敛 abandoned, 干净退出。

设计思想: 进程内 timer, 不持久化 — Reference 章节 "模式 19 · Idempotent 幂等" 的延伸。 timer 是进程级资源, 进程退出 timer 自然消失; schedule 数据 (ScheduleFile) 持久化, 进程重启 reload 即可恢复。 教学版 简单, 工业版可以用 systemd timer / k8s cronjob 替代。

实现细节: timer 在 src/schedules.ts 第 643 行 持有, start() 用 setInterval 5 秒一次, stop() clearInterval + 清理 runningOccurrences。 start / stop 幂等, 多次调用安全。

主循环集成: start 启动 timer, 进程退出 stop

用途: ScheduleManager 在 Composition Root 创建, 注入 agent + 注册工具 (schedule_create / read / list / cancel / list_occurrences)。 主循环集成点 2 个:

  1. index.ts 启动时: scheduleManager.start() 启动 timer, 之后 schedule 开始被调度。
  2. 进程退出 hook: SIGINT / SIGTERM handler 调 scheduleManager.stop() + asyncRunManager.shutdown(), 干净退出。

设计思想: 显式生命周期 — ScheduleManager 不自动 start, 必须显式调, 避免 Composition Root 创建时立即触发未预期的 tick。

实现细节: scheduleManager.start() 是幂等的, 重复调用不创建多个 timer。 stop() 也幂等, 重复调用不抛错。 这是 Reference 章节 "模式 1 · 工厂 + 闭包" 的应用 — 状态在闭包内, 外部不能误改 timer handle。

fake test: 用 fake clock 验证 missed + overlap

实现细节: fake clock 通过 deps.now 注入, ScheduleManager 完全用 deps.now() 拿时间, 不调 new Date()。 fake asyncRunManager 通过 deps.asyncRunManager 注入, 只暴露 start / setOnFinish 两个方法 (用 Pick<AsyncRunManager, "start" | "setOnFinish"> 类型限制)。

常见误解: Schedule 不是 cron 替代品

误解 1: "Schedule = harness 内部的 cron?" 半对。 Schedule 确实像 cron (time-based trigger), 但不 替代 OS cron: (a) Schedule 进程退出时不持久化, 重启依赖 missed detection; (b) Schedule 不支持秒级精度 (5 秒 tick); (c) Schedule 不支持 system reboot 触发。 想要这些特性, 用 OS cron / k8s cronjob 调 harness CLI。

误解 2: "missed occurrence 自动补跑?" 错。 教学版不补跑, 只记录审计。 工业级 Schedule 可能有 missedPolicy: "skip" | "backfill" 配置, 教学版 固定 skip。

误解 3: "overlap=allow 永远安全?" 不一定。 allow 启动新的 Async Run, 但 Async Run 自己有 3 并发限制 (第 13 章), 可能 reject "Maximum concurrent async runs (3) reached"。 overlap=allow 适合"独立任务" (日志清理 + 报告生成), 不适合 "同一资源" (CI 检查 + build)。

误解 4: "schedule 跑完直接改 task 状态?" 错。 通过 linkedTaskUpdate: "append_note" | "mark_failed_on_failure" 3 种 policy 间接改, 不是任意改。 "任意改 task 状态" 仍 走 TaskManager (第 12 章), 保持职责分离。

反例梯度: 4 个常见错误

反例 1 · 不用 Async Run, 自己跑

Schedule 到点直接 await run_bash(command), 不走 Async Run。 错。 Schedule 自己跑没有并发限制, 没有 终态幂等, 没有输出隔离。 复跑 / 重叠触发 / 进程退出 mid-write 全部出问题。 正确: Schedule 只负责触发, 真实执行委托给 Async Run, 复用其全部基础设施 (3 并发 + finishRun 幂等 + OutputStore LRU)。

反例 2 · missed 自动补跑

启动时检测到 3 个 missed occurrence, 立即 3 个并发启动 Async Run 补跑。 错。 周末 7 天 × 1 daily = 7 个 missed, 启动 7 个并发, 立刻触发 Async Run 3 并发限制, 资源滥用。 正确: 只记录 missed audit, 不补跑, 推进到未来下一次。

反例 3 · occurrence id 随机生成

occurrence id 用 Math.random() 生成, 每次 tick 都重新计算。 错。 启动时检测 missed 无法判断"这是不是 已存在 occurrence", 重复创建 + 重复触发。 正确: id 用 scheduleId + scheduledAt 拼, stable + idempotent, 同一时间点永远同一 id。

反例 4 · overlap=allow 默认值

默认 overlap=allow, LLM / user 创建 schedule 不指定就 allow。 错。 大多数 schedule (CI / 测试 / 扫描) 应该 skip, allow 容易出并发问题。 正确: 默认 skip, 用户明确想并发 才显式 allow, 符合"安全默认值" 原则。

Validation: 4 条不变量检验

  1. occurrence id 稳定: 同一 schedule + 同一时间点 调 generateOccurrenceId 2 次, 返回值字节相等。 验证: 单测 覆盖 daily / weekly / monthly, 同一时间点 2 次调用相等。
  2. missed 不补跑: fake clock 跳到"该跑没跑" 的 时间点之后, tick 跑一次, 验证: missed occurrence 写一条, fake asyncRunManager.start 调 0 次 (不补跑)。
  3. overlap=skip 行为: 上次 occurrence 还在 running, 触发点到, 验证: 写 skipped_overlap occurrence, fake asyncRunManager.start 调 0 次。
  4. overlap=allow 行为: 上次 occurrence 还在 running, 触发点到, overlap=allow, 验证: fake asyncRunManager.start 调 1 次 (启动新 run, 假设备用 fakeAsyncRun 不 reject)。

回望: 哪些原则在本章兑现了

  • 触发器, 不是执行器: Schedule 自己不跑, 委托 Async Run, 复用并发限制 / 终态幂等 / 输出隔离。
  • 层间状态机: Schedule 管 active/cancelled, occurrence 管 triggered/skipped/missed, Async Run 管 running/completed, 三层各管各的。
  • 稳定 id: occurrence id = scheduleId + scheduledAt, 幂等, 启动时检测 missed 不重复创建。
  • missed 不补跑: 教学版故意简化, 只记录审计, 避免 资源滥用。
  • overlap 默认 skip: 安全默认, 用户明确 allow 才 启动新 run。

Prompt Card · 第 14 章

目标: 给 harness 加 Schedule 定时调度, 5 秒 tick 一次, 6 种 recurrence, 复用 Async Run 作为执行单元, missed 不补跑, overlap 默认 skip。

模块: src/schedules.ts (新) 暴露 createScheduleManager({store, asyncRunManager, ...}); src/tools/schedule-*.ts (新) 包装 LLM 接口; src/index.ts (改) Composition Root 创建 + start() + SIGINT hook 调 stop()。

边界 (LLM 必须遵守的 checklist):

  • 6 种 recurrence 固定枚举 every_seconds / hourly / daily / weekly / monthly / yearly, 不可新增
  • occurrence id = scheduleId + scheduledAt 拼, stable + idempotent, 不可随机生成
  • missed occurrence 只记录审计, 不补跑, 推进到未来下一次
  • overlap 默认 skip, allow 需显式指定, 上次 running 时 allow 启动新 Async Run
  • Schedule 自己不跑命令 / 不调 LLM, 必须委托给 Async Run, 不重复实现执行生命周期

  • linkedTaskUpdate 3 种 policy: never / append_note / mark_failed_on_failure, 不可任意改 task 状态
  • permissionProfile 3 种: readonly / ci / workspace_write, 透传给 Async Run 校验

验证 (用 fake clock + fake asyncRunManager + vitest, 逐条断言):

  • occurrence id 稳定: 同一 scheduleId + scheduledAt, 2 次调用字节相等
  • missed 不补跑: fake clock 跳到过去, tick 后写 1 条 missed, fakeAsyncRun.start 调 0 次
  • overlap=skip: 上次 running, 触发点写 skipped_overlap, fakeAsyncRun.start 调 0 次
  • overlap=allow: 上次 running, 触发点 fakeAsyncRun.start 调 1 次

本章小结

Schedule 是给"时间触发" 的调度器, 复用 Async Run 作为执行 单元。 核心是 5 个设计:

  • 触发器, 不是执行器: Schedule 自己不跑, 委托 Async Run, 复用其并发 / 终态 / 输出 / abort 全部基础设施。
  • 6 种 recurrence: every_seconds / hourly / daily / weekly / monthly / yearly, 固定枚举, 不可新增。
  • occurrence 稳定 id: scheduleId + scheduledAt 拼, 幂等, 启动时检测 missed 不重复。
  • missed 不补跑: 教学版故意简化, 只记录审计, 推进到未来下一次。
  • overlap 默认 skip: 安全默认, allow 需显式指定。
