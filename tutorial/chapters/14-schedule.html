第 14 章 · 让时间触发 Agent

让时间触发 Agent：Schedule

Schedule 把长期定时规则和实际执行分开：规则持久化，触发时创建 occurrence，真正运行交给 Async Run。

你已经知道什么，还不知道什么

你已经有 Persistent Task 和 Async Run。Schedule 不是另一个执行器，它只是“什么时候启动一次任务”的长期规则。

读这一章时请把注意力放在 agent loop 的接入点：Agent 主动对话之外，由 tick 根据时间规则创建 Async Run 并注入通知。代码片段只服务于理解架构边界，不要求你逐行背实现。

本章怎么学

先把自己放进一个真实 query：没有 Schedule 时，当前 agent 会在哪里卡住。

再故意设计一个朴素方案，观察它如何破坏 loop、状态或安全边界。这个失败比“正确答案”更重要，因为它能帮你写出更好的重建 prompt。

最后才看实现：先看图和数据流，再看少量接口、源码入口和验证方法。读完后，你应该能用自己的话描述 Schedule 的职责，而不是只记住文件名。

本章场景

用户说：“每天早上 9 点检查测试结果，每周五生成一份状态报告。”这些需求不能靠用户每次手动输入。

Schedule 持久化规则；每次到期产生 occurrence；occurrence 触发 Async Run；完成后通知主 session。

先试一个朴素方案

朴素方案是在 Schedule 里直接运行命令并维护 running/succeeded/failed。

这个朴素方案的问题在于：它把 Schedule 当成普通实现细节，而不是 agent harness 的一条稳定边界。 

if (now >= schedule.nextRunAt) {
  schedule.status = "running";
  schedule.output = await runTask(schedule.prompt);
}

朴素方案为什么不够

第二套执行生命周期
Schedule 自己管理 running/timeout，会和 Async Run 重复且不一致。
重启收敛困难
进程死掉后 running occurrence 必须能被识别为 orphaned 并收敛。
时间规则陷阱
timezone、DST、monthly clamp 都会让“下次执行”不只是加毫秒。

回到 Agent Loop

Schedule 接入 loop 的位置是：Agent 主动对话之外，由 tick 根据时间规则创建 Async Run 并注入通知。理解这个位置，比记住具体函数名更重要。

Schedule 的 tick 不在普通用户 turn 内触发，但触发结果最终通过 Async Run notification/reminder 回到 agent loop。

从原理落到代码

到这里我们已经知道 Schedule 为什么存在、插在 loop 的哪里，以及朴素方案会怎样失败。下面才进入实现层：只抓住能固定架构的接口、状态边界和接线路径。

一次真实运行 walkthrough

  1. 用户创建 daily schedule，指定 prompt、timezone 和 overlapPolicy。
  2. ScheduleStore 持久化 schedule 和 nextRunAt。
  3. 后台 tick 扫描当前项目可触发 schedule。
  4. 到期后创建 occurrence，记录触发时间和 prompt snapshot。
  5. ScheduleManager 调用 AsyncRunManager start。
  6. Async Run 完成后，occurrence 进入 succeeded/failed。
  7. 主 loop 下一轮看到 schedule 完成提醒。

interface Schedule {
  id: string;
  projectKey: string;
  prompt: string;
  rrule: string;
  timezone: string;
  overlapPolicy: "skip" | "allow";
  nextRunAt: string;
}

interface Occurrence {
  id: string;
  scheduleId: string;
  status: "queued" | "running" | "succeeded" | "failed" | "orphaned";
}

设计直觉

Schedule 的核心克制是“不执行”。它只决定某个时间点应该启动什么，执行交给已有 Async Run，这样生命周期和输出读取都复用同一套机制。

occurrence 是审计记录：同一个 schedule 每次触发都要留下独立事实，便于解释、重试和清理。

架构拆分

ScheduleStore
保存规则、occurrence 和索引。
ScheduleManager
计算到期、处理 overlap、启动 Async Run。
AsyncRunManager
负责实际执行和输出。

状态与边界

持久化
schedule 规则、occurrence 审计记录。
运行态
Async Run 实例和通知队列。
项目边界
当前项目 manager 只触发当前 projectKey 的 schedule。

Prompt Card

Prompt Card

目标：实现 Schedule 定时系统，让规则持久化，到期时创建 occurrence 并启动 Async Run。

场景：用户创建每天或每周运行的 agent prompt，系统在 tick 时触发。

模块：schedule-store.ts、schedules.ts、tools/schedules.ts、async-runs.ts。

接线：ScheduleManager 注入 AsyncRunManager；tick 到期时 start async run；完成通知进入 session events。

边界：Schedule 不实现第二套执行生命周期；当前项目只触发自己的 schedule。

例外和 corner case：overlap allow/skip、orphaned running、timezone、DST、monthly clamp。

验证：测试 create/list/read/cancel/delete、tick 触发、重启收敛和 overlap。

源码入口：src/schedule-store.ts、src/schedules.ts、src/tools/schedules.ts。

文档同步：说明 Schedule 是时间规则，Async Run 是执行实例。

容易踩坑

Design Trap

常见错误：让 Schedule 自己执行命令并保存输出。

为什么错：会产生第二套 running/timeout/output 生命周期，和 Async Run 不一致。

正确做法：Schedule 只创建 occurrence，并委托 Async Run 执行。

怎么验证：tick 后应该有 occurrence + async run，而不是 schedule 自己带 output。

  • 重启后旧 running occurrence 不收敛。
  • 跨项目 schedule 被当前项目 tick 误触发。
  • overlapPolicy 语义不清导致重复运行。

如何验证

Validation

手工 query：创建一个短间隔 schedule，等待 tick 后查看 occurrence 和 async run。

确定性测试：用 fake clock 测试 nextRunAt 和 overlap。

集成观察：重启 manager 后 orphaned running occurrence 被收敛。

失败信号：schedule 自己保存执行输出，或 tick 触发了其他项目规则。

  • src/schedules.test.ts 覆盖业务层。
  • src/schedule-store.test.ts 覆盖持久化。
  • src/tools/schedules.test.ts 覆盖工具层。

如果实现失败，先查哪里

  • 检查 projectKey。
  • 检查 nextRunAt 计算。
  • 检查 occurrence 和 run_id 关联。
  • 检查 overlapPolicy 分支。
