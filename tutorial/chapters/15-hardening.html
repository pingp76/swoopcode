第 15 章 · 长期运行的卫生系统

Runtime Hardening: 长期运行不会把系统跑坏

前面十四章让 harness 拥有完整能力, 但 schedule / async run / memory / task / output store 这些持久化数据在长期运行下会无限增长, 日志、 transcript、output handle 也会累积。某天启动变慢、磁盘占满、某个 JSON 因为中断写坏。这一章给 harness 加 Runtime Hardening, 覆盖所有 读写边界: 原子写、日志轮转、output handle 引用计数、清理 dry-run、 时间语义统一, 让 harness 在真实使用中慢慢腐烂时不至于崩溃。

在第 14 章基础上改了什么

这一章不新增业务模块, 而是基于已有的 schedule (第 14 章) / async run (第 13 章) / memory (第 09 章) / task (第 12 章) / output store (第 06 章 P1) / transcript (第 08 章) 重新组织所有持久化 IO, 加上 原子写、日志轮转、output handle 引用计数、清理 dry-run、时间语义 统一五个卫生系统。 对应到代码, 改动集中在 5 个文件: src/atomic-write.ts (新)、 src/log-rotation.ts (新)、src/output-store.ts (改加引用计数)、src/cleanup.ts (新)、src/timeline.ts (改时间语义统一)。

1. src/atomic-write.ts: 原子写工具 (新) 2. src/log-rotation.ts: 日志轮转 (新) 3. src/output-store.ts: 引用计数 + 过期清理 (改) 4. src/cleanup.ts: 干跑清理 (新) 5. src/timeline.ts: 时间语义统一 (改)

作者怎么想的: 这一章的思考链

想清楚现象
harness 跑了一个月, 用户突然发现启动变慢、磁盘占满、schedule 触发时间错位 (以为本地时区实际 UTC)。现象是"卫生系统缺失, 长期运行下慢慢腐烂"。
想反例
最朴素的反例是"先不管, 等文件太多再 rm -rf"。这有两个问题: 一是不知道哪些数据可删 (memory / task 是用户长期事实, 删了 用户损失), 二是粗暴删除破坏原子性, 写到一半的文件会留垃圾。
想接口和不变量
接口: interface AtomicWrite { write(path, content) }, interface Cleanup { dryRun(), run() }。 不变量五条: (1) 所有持久化 IO 走 atomic write, 不直接 fs.writeFile, (2) 日志按大小或时间轮转, 不无限增长, (3) output store 引用计数, 没人引用的 output_id 可清理, (4) 清理先 dry-run 给用户看, 用户 确认后才真删, (5) 时间用 epoch ms (number) 不用 Date 字符串。
想怎么验证
fake atomic write 中断 (模拟断电), 跑完后原文件仍然存在, tmp 文件可能被留 (不破坏), 重新启动 harness 后能正常解析; fake cleanup dry-run 返回 5 个待删文件, run() 之前文件仍在, run() 之后文件不在。

先观察: 两段故意有气味的实现

观察 1 · 直接 fs.writeFile, 断电留半截

// 教学简化版, 仓库真实实现见 GitHub 永久链接
// 错误: 直接写, 断电会留半截
import fs from "node:fs/promises";
async function saveMemory(path: string, content: string) {
  await fs.writeFile(path, content);
}

问:为什么不直接 writeFile?

答:三件事同时坏掉 —— 原子性: 写到一半断电, 文件 是半截 JSON, 下次启动解析失败, 用户的偏好丢失; 不可恢复: 解析失败后没有 backup, 用户只能从头开始; 可观测: 写失败时 fs.writeFile 抛错但不告诉用户"为什么失败", 调试困难。

观察 2 · cleanup 实际跑, 不 dry-run

// 教学简化版
// 错误: 不先 dry-run 给用户看
async function cleanup() {
  const oldFiles = await glob(".claude/cache/*.tmp");
  for (const f of oldFiles) await fs.unlink(f);
}

问:为什么不直接删?

答:三件事同时坏掉 —— 误删: 用户可能有临时调研 文件在 tmp 里, 删了用户丢失; 不可逆: 删了之后用户说"等等, 那个文件我有用的", 没了; 不可信: 用户不敢用清理功能, 担心 删错, harness 的卫生功能形同虚设。

原子写: write tmp + fsync + rename

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export async function atomicWriteFile(path: string, content: string | Buffer): Promise<void> {
  const tmpPath = `${path}.tmp.${process.pid}.${Date.now()}`;
  const fh = await fs.open(tmpPath, "w");
  try {
    await fh.writeFile(content);
    await fh.sync();  // fsync 强制刷盘
  } finally {
    await fh.close();
  }
  await fs.rename(tmpPath, path);  // rename 在同一文件系统下是原子的
}

关键: rename 在同一文件系统下是原子操作。要么 tmp 文件 rename 成功, 替换原文件; 要么 rename 失败, 原文件不变。半截写入 不会暴露给后续读取。

日志轮转: 按大小 + 按时间

harness 日志 (LLM 通信 / hook 决策 / schedule 触发) 持续增长。本章 引入日志轮转, 按 maxFileSize (默认 10MB) 或 maxAge (默认 7 天) 切割。轮转策略:

按大小
日志文件超过 maxFileSize 时, 关闭当前文件, 重命名为 app.log.1、app.log.2 等, 启动新的 app.log。保留最近 N 个文件, 更老的删除。
按时间
每天 0 点自动轮转, 跨天的日志按天归档。保留最近 7 天, 更老的 删除。
组合
同时满足时按更严格的策略。本章默认两个都启用, 任一触发即轮转。

output store 引用计数

第 06 章 P1 压缩把大工具输出存到 ~/.cache/run-outputs/<output_id>, history 里只剩占位符。问题是: 如果 LLM 之后再没引用这个 output_id, 文件就成孤儿了, 永远占磁盘。本章加引用计数: 

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export interface OutputHandle {
  id: string;
  path: string;
  size: number;
  refCount: number;        // 当前被多少 messages 引用
  createdAt: number;
  expiresAt: number;        // 引用计数 0 之后, 24h 后可清理
}

每次写入 output 时 refCount = 1 (对应的 tool message 引用一次); 之后 compress / replay 删除或合并该 tool message 时, refCount -= 1; refCount 归 0 后, 24 小时后可清理。cleanup 走 dry-run 列出待清理 文件, 用户确认后才真删。

清理: dry-run + run

// 教学简化版, 仓库真实实现见 GitHub 永久链接
export interface CleanupReport {
  outputOrphans: { id: string; size: number; path: string }[];
  oldLogs: { path: string; size: number; ageDays: number }[];
  rotatedBackups: { path: string; size: number }[];
  totalSize: number;
}

export interface Cleanup {
  // 列出待清理内容, 不真删
  dryRun(): Promise<CleanupReport>;
  // 用户确认后真删
  run(report: CleanupReport): Promise<{ deleted: number; freedBytes: number }>;
}

dry-run 返回的报告分三类: 孤儿 output (引用计数 0 且超过 24h)、 老日志 (超过 maxAge)、轮转备份 (超过保留份数)。用户看完报告, 决定哪些删、哪些保留。

时间语义: 全部用 epoch ms

不同模块用不同时间格式 (Date 字符串 / 秒级 timestamp / 毫秒 timestamp) 会导致排序错乱、跨时区混乱。本章统一时间语义:

  • 持久化存储 (memory / task / schedule): 全部用 epoch ms (number)。
  • 日志输出: epoch ms 数字 + 格式化为 ISO 8601 字符串 (UTC)。
  • UI 显示: 转用户时区 + 友好格式 ("3 小时前")。

内部不允许 Date 字符串。string <-> number 转换只发生在 IO 边界 (读 JSON 文件时 string -> number, 写 UI 时 number -> 字符串)。

反例梯度

新手错法 · A

常见错误:持久化用 fs.writeFile, 不走 atomic write。

为什么错:断电留半截 JSON, 解析失败, 用户偏好丢失。

正确做法:所有持久化 IO 走 atomicWriteFile(), write tmp + fsync + rename。

中级错法 · B

常见错误:cleanup 直接删, 不先 dry-run。

为什么错:误删用户文件, 用户不敢用清理功能, 卫生系统形同虚设。

正确做法:dryRun() 返回报告, 用户确认后 run() 真删。

高级错法 · C

常见错误:output store 不引用计数, 孤儿文件永久占盘。

为什么错:用户跑一个月, ~/.cache/run-outputs/ 撑爆磁盘。

正确做法:写入时 refCount = 1, compress / replay 删引用时 -1, 归 0 24h 后可清理。

边界错法 · D

常见错误:时间用 Date 字符串 ("2024-01-01T00:00:00Z") 存盘。

为什么错:跨时区比较容易出错, 字符串排序不等于时间排序。

正确做法:内部全部用 epoch ms, IO 边界转 ISO 字符串, 内部不允许 Date 字符串。

回望第 00–14 章: 哪些原则在本章兑现了

  • 原子性原则:所有持久化 IO 走 atomic write, 跨进程安全, 断电安全。
  • 不可逆操作必须 dry-run:cleanup / rotation 等会删除数据的操作, 必须先报告, 用户确认后再做。
  • 内部类型稳定:时间用 epoch ms (number), 字符串只在 IO 边界出现, 内部不允许 Date 字符串。
  • 事实与视图分离:output store 是事实, refCount 是视图, 清理是另一个视图, 三者职责清晰分离。
