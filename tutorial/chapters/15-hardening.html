第 15 章 · 跑生产时的运行时安全

Hardening: 日志轮转 / 多 LLM / 多 session / Normalize 协议差异

前 14 章让 harness 在 happy path 上跑得通, 但生产里还要解决 8 件"杂事": 日志文件 撑爆磁盘 (log-rotation) / 3 个 LLM 后端 (OpenAI / Anthropic / 自部署 vLLM) 协议差异 (normalize) / 配置优先级 (CLI > env > config.ts) / 多 session 并发隔离 / prompt 策略层 (system-prompt 拼装) / context 排序 (context-ranking) / 终端兼容 (terminal.ts TTY 检测)。 这一章不深入单个模块, 而是把这些 "安全网" 列清楚, 讲清"为什么需要" + "靠哪条不变量保证" + "出问题怎么定位"。

真实失败故事: harness 跑 3 个月, 磁盘撑爆 100GB

写代码之前, 先看一个真实痛点。 用户的 team 上线 harness 跑 生产, 3 个月后发现 /var/log/harness.log 撑爆 100GB, 系统 报警 "disk usage 95%"。 同时 team 想接 Anthropic Claude (原本 只支持 OpenAI), 改完 LLM 客户端发现 100 个测试里 30 个 fail (thinking_content 字段位置不一样)。

  1. 症状 1: 日志撑爆: 跑 3 个月没轮转, 单个 .log 文件 100GB, 磁盘满, 系统挂。
  2. 症状 2: 换 LLM fail: OpenAI 用 response.choices[0].message.content, Anthropic 用 response.content[0].text, 代码到处写 message.content 拿到 undefined。
  3. 症状 3: 配置混乱: CLI 传 --mode plan, env 传 HARNESS_MODE=auto, 哪个生效? 团队 不知道, 出现"我配的怎么不生效"。
  4. 症状 4: 多 session 互相干扰: user A 和 user B 同时跑 harness, 共享 history 数组, A 看到 B 的对话。
  5. 真问题: 缺运行时安全网 — 日志轮转 / LLM 协议 normalize / 配置优先级 / session 隔离 / prompt 策略 / context 排序 / 终端兼容, 7 个独立但都必需的 模块, 缺一个就出事。

朴素想法 1: "日志不重要, 不用管?" 错。 日志是唯一 出问题时的排查依据, 撑爆 = 排查依据全丢 = 第二次事故无法 复盘。

朴素想法 2: "只支持 OpenAI, 不接其他?" 商业上不行。 team 想 换 Anthropic / 自部署 vLLM 节省成本, 不接 = 锁死单 provider。 需要协议 normalize 层, 内部用统一格式, 外部 adapter 转 provider 协议。

正确做法: 8 个独立模块各自解决一个问题 — log-rotation.ts / normalize.ts / config.ts / session.ts / system-prompt.ts / context-ranking.ts / terminal.ts / cli-commands.ts / llm-providers.ts。 这一章把它们串起来看, 讲清"为什么需要" + "靠哪条不变量保证"。

先梳理语言: 8 个安全网模块是什么

上面看到痛点和朴素方案, 这一节先给词表。 Hardening 这一章讲 8 个独立模块, 每个解决 一个独立的"运行时问题", 但没有任何共同 接口 (不像 Async Run / Task / Schedule 那样是一组紧密 关联的 entity)。 这一节先梳理一遍每个模块 "管啥 / 靠啥不变量保证", 后面 16 个小节读起来不会"谁是谁"。

8 个模块分 3 层: 基础层 (config) / 中间层 (normalize / log-rotation / session / terminal) / 暴露层 (system-prompt / context-ranking / cli-commands / llm-providers / foundation-models)。

1. 基础层 (1 个, 其他都依赖它)

  • config.ts: 配置优先级解决 "我配的怎么不生效"。 读 3 个来源, 按优先级合并: CLI 参数 (--mode plan) > 环境变量 (HARNESS_MODE=auto) > 默认值。 1 个 Config = 1 个 merged 对象, 进程内只读, 不写回。

2. 中间层 (4 个, 通用基础设施)

  • log-rotation.ts: 日志不撑爆 解决 "跑 3 个月 100GB 撑爆磁盘"。 按 size 轮转, 默认 50MB 切一次, 保留 5 个备份 (harness.log, harness.log.1 ... harness.log.5), 超过 5 个的删最早的。 不变量: 单文件不会无限增长。
  • normalize.ts: 多 LLM 协议归一 解决 "换 Anthropic 30 个测试 fail"。 内部用统一格式 {role, content, tool_calls, reasoning_content}, 外部 adapter 转 OpenAI / Anthropic / 自部署协议。 不变量: 下游代码只用内部格式, 换 provider 改 adapter 即可, 不改业务逻辑。
  • session.ts: 多 session 隔离 解决 "A 看到 B 的对话"。 1 个 session = 1 个独立 history 数组 + 1 个独立 tools 闭包, 创建时分配, 用 session_id 寻址。 不变量: session A 的 history 永远不进入 session B。
  • terminal.ts: 终端兼容解决 "在 TTY 里颜色乱码 / 在 pipe 里乱加 ANSI"。 检测 process.stdout.isTTY, 是 TTY 才加颜色, 不是 TTY (pipe / redirect) 输出纯文本。 不变量: 输出 在任何终端 / pipe / 文件重定向下都正确。

3. 暴露层 (5 个, 直接给 LLM / user 用)

  • system-prompt.ts: prompt 拼装 解决 "system prompt 怎么放规则 / tool 描述 / memory / 状态 summary"。 是 1 个函数, 输入: 配置 + memory + 当前状态, 输出: 1 个完整 system prompt 字符串。 不变量: 所有 prompt 走同一入口, 改拼装逻辑不改业务。
  • context-ranking.ts: history 排序 解决 "history 里 100 条 message 哪些优先, 哪些截断, 哪些 丢弃"。 1 个 score 函数, 输入 1 条 message, 输出 score; history 按 score 排序, 截断到 context budget。 不变量: 超 budget 时低 score message 先丢, 高 score message 永远在窗口内。
  • cli-commands.ts: 9 个 /xxx 命令 解决 "用户在 REPL 里怎么调工具"。 1 个 registry, 注册 9 个 user-visible 命令 (/model, /reset, /memory, /skill, /tasks, /async, /schedule, /compress, /exit), REPL 解析 /xxx 前缀路由到 处理器。 不变量: 每个 /xxx 都有 1 个 handler, 错命令提示 "unknown command" 不崩。
  • llm-providers.ts: 3 个 provider 适配 解决 "换 provider 改业务代码"。 是 normalize 的对称层, 3 个 adapter: openai.ts / anthropic.ts / vllm.ts (自部署)。 不变量: 3 个 adapter 暴露同一个 createProvider(config) 工厂, 换 provider 改配置不改代码。
  • foundation-models.ts: 5 个 model 适配 解决 "5 个 model 的 tool calling 格式不一样 (gpt-4o 用 function_calling, claude-3.5 用 tool_use, o1 用 reasoning_content 单独字段 ...)"。 在 llm-providers 之上, 1 个 model = 1 个 schema 补丁。 不变量: 同 provider 跨 model 时换 model name 即可, 改 schema 不改业务。

具体例子: "team 想加 thinking 思考预算配置" 的 4 文件联动

  1. config 加字段: 改 config.ts 的 DEFAULT_CONFIG, 加 thinking_budget: 10000。 同时改 parseConfigFromEnv 支持 HARNESS_THINKING_BUDGET=20000, 改 parseConfigFromCli 支持 --thinking-budget 20000。
  2. cli-commands 加 /thinking: 改 cli-commands.ts 的 registry, 注册 /thinking handler, 调 config.update({thinking_budget: N})。 user 在 REPL 里敲 /thinking 20000 立刻生效。
  3. system-prompt 拼到 prompt: 改 system-prompt.ts 的拼装函数, 在 "## Constraints" 一节加 "thinking_budget: 20000 tokens, 不要超过"。 LLM 看到 这个 budget 后, 调 LLM 时会控制 thinking 长度。
  4. llm-providers + foundation-models 转给 provider: Anthropic 的 adapter 把 thinking_budget=20000 转成 thinking: {type: "enabled", budget_tokens: 20000}, OpenAI 的 adapter 不支持, 转成 reasoning_effort: "medium" 近似 (O1 系列)。 foundation-models 里 o1 / o3 跟 gpt-4o 的 reasoning 字段 位置不一样, 单独处理。

关键观察: 4 个文件联动改, 漏 1 个就出 bug (比如 config 加了但 prompt 没拼, LLM 不知道 budget; 或者 prompt 拼了但 provider 没转, API 报错)。 这就是为什么 hardening 章节讲"模块独立但有依赖" — 独立是手段, 联调是结果。

看完这 8 个模块和具体例子, 后面 16 个小节 (8 个安全网边界 + 8 个模块各自深入 + 主循环集成 + fake test + 常见误解 + 反例梯度 + Validation) 都在讲"模块怎么实现 + 靠啥不变量 保证"。 看到陌生模块, 翻回这节找定位。

8 个安全网的边界

用途: 8 个模块各自独立, 但有层间依赖: config 在最底层 (其他都依赖它的配置), session 在中间层 (多 进程共享资源), cli-commands 在顶层 (暴露 LLM / user 接口)。 理解依赖关系才知道"改一个会不会影响其他"。

真实场景: team 想加 "harness 支持 /thinking 思考预算" 配置, 要改 (1) config.ts 加 thinking_budget 字段; (2) system-prompt.ts 把预算拼到 prompt; (3) cli-commands.ts 加 /thinking CLI 命令; (4) llm-providers.ts 转给具体 provider (Anthropic 接受 thinking 字段, OpenAI 不接受)。 4 个文件联动 改, 漏一个就出 bug。

设计思想: 模块独立 + 协议对齐 — Reference 章节 "模式 1 · 工厂 + 闭包" + "模式 3 · 窄接口 + Adapter" 的延伸。 8 个模块都用工厂 + 闭包, 接口清晰, 改一个不污染其他。

图 1 · 8 个安全网的依赖层次
顶层 · CLI / Tools
cli-commands.ts 暴露 /mode /memory /schedule /task /model-policy 等 CLI 命令; tools/* 暴露 LLM 可调工具。
策略层 · system-prompt / context-ranking
system-prompt.ts 拼装 system message (含 memory / skill / tool 规则); context-ranking.ts 排序 history 中哪些 message 优先。
执行层 · normalize / llm-providers / foundation-models
normalize.ts 把消息归一成 OpenAI ChatCompletionMessageParam; llm-providers.ts 暴露 3 个 provider (openai / anthropic / vllm); foundation-models.ts 列出 5 个常见 model 适配。
会话层 · session / transcript
session.ts 派生独立 history / 隔离多 user; transcript.ts 持久化对话到文件。
基础层 · config / logger / log-rotation / terminal
config.ts 加载配置 (CLI > env > default); logger.ts 输出日志; log-rotation.ts 按 size 轮转; terminal.ts 检测 TTY / 颜色支持。

实现细节: 8 个模块跨 5 层, 顶层依赖底层, 底层 不依赖顶层。 改 config 加字段, 只影响基础层 + 触发其他层 reactive 读; 改 cli-commands 加命令, 只影响顶层。 层间清晰, 维护成本 线性。

log-rotation: 按 size 轮转 + 保留 N 个

用途: 跑生产 harness, 日志文件必须 按 size 轮转, 避免单文件撑爆磁盘。 log-rotation.ts 提供 rotateLogFileIfNeeded(filePath, options) — 写日志前检查 size, 超过 maxBytes 触发轮转。

真实场景: harness 跑 3 个月, 单个 .log 文件 100GB。 正确做法: maxBytes = 50MB, 写日志前检查 size, 超了 rename harness.log → harness.log.1 (覆盖最旧的), 新 harness.log 继续写。 保留 5 个 backup, 总占用 ≤ 50 × 6 = 300MB, 远小于磁盘容量。

设计思想: size-based + 简单 rename — 不引入 logrotate 系统工具, harness 自己处理, 跨平台一致。 看 src/log-rotation.ts 第 29 行的 rotateLogFileIfNeeded: 

export function rotateLogFileIfNeeded(
  filePath: string,
  options: LogRotationOptions,
): { rotated: boolean; removedFiles: string[] };

实现细节: 关键不变量 — (a) 写日志前 检查 size, 不是写完后; (b) rename 走原子 (rename syscall 是 原子的); (c) 保留 N 个 backup, 多的删; (d) 轮转失败 warn 日志, 不 throw, 不影响 harness 继续写日志。 这是 Reference 章节 "模式 10 · Atomic Write 原子写" 的应用。

normalize: 3 个 provider 协议归一

用途: OpenAI / Anthropic / 自部署 vLLM 的协议 格式不同, agent 主循环不应该直接面对 3 套格式。 应该内部 用 OpenAI ChatCompletionMessageParam 统一格式, 外部 调 provider adapter 转协议。 看 src/normalize.ts 第 26 行的 normalizeMessages: 

export function normalizeMessages(
  messages: unknown[],
): ChatCompletionMessageParam[];

真实场景: user 切换 Anthropic Claude, Anthropic 用 {role: "assistant", content: [{type: "text", text: "..."}]} (content 是数组), OpenAI 用 {role: "assistant", content: "..."} (content 是字符串)。 agent 主循环假设 OpenAI 格式, 直接 message.content 拿字符串, 拿到数组报错。 normalize 层把 Anthropic 的 content 数组归一成 OpenAI 的字符串, agent 主循环无感知。

设计思想: 统一内部格式, adapter 翻译 — Reference 章节 "模式 3 · 窄接口 + Adapter" 的标准应用。 选 OpenAI 格式作为"内部标准" 因为最常见, Anthropic / vLLM 写 adapter 把自家协议转成 OpenAI 格式。 看 src/llm-providers.ts — 3 个 provider, 每个一个 adapter 文件, 内部都返回 OpenAI 格式 响应。

实现细节: 5 个常见差异要 normalize — (a) content 字符串 vs 数组; (b) tool_call 在 message 里 vs 在 content 块 里; (c) reasoning_content 字段 (Anthropic) vs reasoning (OpenAI o1); (d) function_call 旧字段 vs tool_calls 新字段; (e) 错误格式 (OpenAI APIError vs AnthropicError)。 normalize 层统一处理, agent 主循环只认"OpenAI 格式"。

config 优先级: CLI > env > default

用途: harness 配置从 3 个来源加载: (1) loadConfig() 读 default + config.ts; (2) 环境变量 HARNESS_* 覆盖 default; (3) CLI 参数 --mode / --debug 覆盖 env。 优先级明确, 不含糊。 看 src/config.ts 第 89 行的 loadConfig: 

export function loadConfig(): Config;

真实场景: team 跑 CI, 想"覆盖 mode=auto + timeout=60s"。 CI 脚本: HARNESS_MODE=auto harness --timeout 60000。 CLI 优先级最高, mode=auto + timeout=60000 都生效; 不传 timeout 用 env 或 default。

设计思想: 显式优先级 + 不打平 — Reference 章节 "模式 5 · Composition Root 组合根" 的应用。 loadConfig() 内部按 (CLI > env > default) 顺序 merge, 不 一次性 flatten (保留"哪个来源覆盖" 信息, 调试时能看到)。 3 条原则:

  1. CLI 最高: --mode plan 永远覆盖 env 和 default。
  2. env 次之: HARNESS_MODE=auto 覆盖 default, 但被 CLI 覆盖。
  3. default 兜底: 没 CLI 也没 env, 用 config.ts 默认值 (mode=default, timeout=30s)。

实现细节: loadConfig 返回的 Config 含 source: { mode: "cli" | "env" | "default" } 字段, 调试时输出 "mode=plan (source: cli)", 让用户知道 配置从哪来, 不再 "我配的怎么不生效"。

多 session 隔离: 独立 history + 独立工具

用途: 多 user / 多项目同时用 harness, 互不 干扰: 独立 history / 独立 tool registry / 独立 session id。 这是 Reference 章节 "模式 1 · 工厂 + 闭包" 的标准应用 — 每次 createSession() 返回新对象, 闭包内 state 互不污染。

真实场景: 30 个 user 同时跑 harness, user A 写到 history 的对话不会跑到 user B 的 history, user A 调的 subagent 不会和 user B 的 subagent 共享 LLM client。 靠 createSession({userId, projectRoot}) 每次返回 独立实例。

设计思想: 工厂 + 闭包 — SessionManager.createSession() 返回的 Session 含独立 history (createHistory) + 独立 tool registry (createToolRegistry) + 独立 llm client (按需创建)。 关键不变量: 状态在闭包内, 外部不能误改。

实现细节: 隔离的边界 — (a) history 数组 互不共享; (b) tool registry 的状态 (call counts / last args) 互不共享; (c) 持久化文件按 sessionId 分目录 (transcript 写到 .sessions/<sessionId>/transcript.jsonl); (d) LLM API key 按 session 注入 (多 key 轮换避免 rate limit)。 Reference 章节 "模式 17 · Test Doubles" 的应用 — 测试时 fake session 互不污染。

system-prompt: 拼装策略层

用途: system prompt 不是"一段写死的字符串", 是策略拼装 — base 规则 + memory 段 + skill 段 + tool 规则 + 当前 context, 按层拼, 缺哪段就跳过。 教学版 关键是 "system prompt 是 stable prefix, 拼完 1 次, 100 轮 对话复用" (第 10 章)。

真实场景: 启动时拼 system prompt = baseRules + memorySection + skillSection + toolRules + contextSummary。 5 段按顺序, 每段独立可测。 测试用 fake memory / fake skill 注入, 验证拼装结果。

设计思想: 分层拼装, 缺段跳过 — Reference 章节 "模式 6 · Pipeline 管道" + "模式 11 · Cache-friendly 缓存友好" 的应用。 每段单独函数, 单独测试, 拼装函数只负责 concat。 缺 memory 段时 memorySection || "", 不 throw。

实现细节: system prompt 的体积控制 — 每段 限制 maxTokens (memory 段 200 token, skill 段 500 token, tool 规则 1000 token, 总量 ≤ 4k token), 超过截断 + 提示 "memory 段过长, 考虑清理"。 这是防止 system prompt 撑爆 cache prefix 命中率 (第 10 章)。

context-ranking: history 中哪些 message 优先

用途: history 100 轮对话, 4000 token, LLM 看不 全? 不是 — LLM 看到全部, 但截断时哪些 message 优先保留? context-ranking 提供排序策略: 系统 消息 / 用户最近输入 / 工具错误 / 长输出摘要等优先级不同。

真实场景: 100 轮对话 history 4000 token 超 模型窗口 8k, 需要截断到 6k。 context-ranking 排序: (1) system message 永不丢; (2) 最近 5 轮 user/assistant/tool 完整保留; (3) 早期长 tool 输出保留摘要; (4) 早期 user/assistant 折叠 成 "1-line summary"。 截断后 LLM 看到"全部 context, 但老的内容 压缩了"。

设计思想: 分级保留 + 摘要折叠 — Reference 章节 "模式 6 · Pipeline 管道" + "模式 13 · Error→Action 错误转动作" 的应用。 截断不是"丢老的内容", 是 "老的内容保留摘要, 新的内容保留全文"。 摘要靠 LLM 生成 (内部 Async Run), 教学版可以简化为"丢最早的 10% 内容"。

实现细节: 排序权重 (a) system = 100; (b) 工具 错误 = 90 (LLM 需要看到错误调整); (c) 最近 5 轮 = 80; (d) 远期 user 消息 = 40; (e) 远期工具输出 = 20 (优先压缩)。 截断 时按权重从高到低保留, 直到达到 maxTokens。

terminal: TTY / 颜色 / 宽度的环境检测

用途: harness 在 TTY 终端 / IDE 内嵌 / pipe 重 定向 3 种环境下行为不同: TTY 用 ANSI 颜色 + 交互式 prompt, pipe 关闭颜色 + 用纯文本, IDE 看 isatty + 父进程名启发式 判断。 terminal.ts 暴露 Terminal 接口, 检测环境 + 调对应输出策略。

真实场景: user 在 vscode 集成终端跑 harness, harness 输出带 ANSI 颜色, vscode 渲染颜色; user pipe 给 less, harness 检测到 not tty 关闭颜色, less 看到 纯文本 (不会乱码)。 同一份代码, 不同环境自动适配。

设计思想: 环境检测 + 降级输出 — Reference 章节 "模式 13 · Error→Action 错误转动作" 的应用。 颜色开启 / 关闭是副作用, 应该从主流程里抽出, 集中到 terminal 接口。

实现细节: 检测点 4 个 — (a) process.stdout.isTTY; (b) process.env.NO_COLOR; (c) process.env.TERM 含 "dumb" 关闭颜色; (d) process.stdout.columns 拿终端宽度 (布局自适应)。 任一不满足 → 关闭颜色 / 用固定 宽度 80 / 禁用交互 prompt。

cli-commands: 9 个 /xxx 命令注册

用途: 用户在 REPL 输 /mode plan / /memory list / /schedule list / /task create / /model-policy show 等 命令, cli-commands.ts 集中注册 + 解析 + 执行。 看 src/cli-commands.ts 第 65 行的 createCliCommandRegistry: 

export function createCliCommandRegistry(): CliCommandRegistry;

真实场景: user 输 /task create "Fix bug #123", registry 解析 create 子命令 + "Fix bug #123" 参数, 调 taskManager.createGroup(...) 创建 task group, 输出 "Created task group task_xxx: Fix bug #123"。 9 个命令 (mode / memory / skill / schedule / task / model-policy / thinking / stable-context 等) 各管一摊, 互不干扰。

设计思想: 注册表 + 工厂 — Reference 章节 "模式 1 · 工厂 + 闭包" + "模式 5 · Composition Root" 的应用。 每个 /xxx 命令是独立工厂, 注册到全局 registry, REPL 调 registry.execute("/task", "create", args) 分发。

实现细节: 9 个命令 不能互相依赖 — /task 不该 import /memory, /schedule 不该 import /mode。 命令 之间的协作通过 Composition Root 注入 (deps), 不用 import 链。 这保证加新命令不影响旧命令。

llm-providers: 3 个 provider 适配

用途: harness 支持 3 个 LLM provider: OpenAI / Anthropic / 自部署 vLLM (OpenAI-compatible)。 llm-providers.ts 暴露 createOpenAIProvider / createAnthropicProvider / createVLLMProvider 3 个工厂, 都返回统一 LLMClient 接口 (chat + stream)。 agent 主循环不直接调 OpenAI SDK, 调 LLMClient 接口。

真实场景: user 想 "省成本, 用自部署 vLLM 跑 7B 模型", 改 LLM_PROVIDER=vllm + VLLM_BASE_URL=http://..., harness 自动用 vLLM provider, agent 主循环 0 改动 (因为都 走 LLMClient 接口)。

设计思想: Adapter 模式 + 统一接口 — Reference 章节 "模式 3 · 窄接口 + Adapter" 的标准应用。 3 个 provider 各写 adapter, 把自家协议转成 OpenAI ChatCompletion 格式 (内部统一标准), agent 主循环只认"OpenAI 格式"。

实现细节: 3 个 provider 的差异 (a) OpenAI 直接 用 openai SDK; (b) Anthropic 用 @anthropic-ai/sdk + 写 adapter 把 Anthropic 响应转 OpenAI 格式; (c) vLLM 用 OpenAI-compatible HTTP API (和 OpenAI 一样, 只换 baseURL)。 Anthropic adapter 是最复杂的 (content 数组 + thinking 字段 + tool_use 块都要转), 教学版只覆盖 80% case。

foundation-models: 5 个常见 model 适配

用途: 不同 LLM model 有不同的能力差异 (上下文窗口 / 支持 function calling / 支持 vision / 思考预算) — GPT-4o 128k + tool + vision; Claude Sonnet 200k + tool + thinking; Qwen2.5-7B 32k + tool (no vision)。 harness 应该 知道"我用的是哪个 model, 它的能力边界在哪", 避免越界调用。

真实场景: user 设 LLM_MODEL=qwen2.5-7b, harness 查 foundation-models.ts 找到 "Qwen2.5-7B: 32k context, tool support: yes, vision: no, thinking: no", LLM 试图调带 图片的 tool 时, harness 提示 "当前 model 不支持 vision, 请用 text 描述"。 防止 LLM "以为能调实际不能调"。

设计思想: 能力清单 + 边界提示 — Reference 章节 "模式 5 · Composition Root 组合根" 的应用。 每个 model 一个清单, 启动时根据 LLM_MODEL 选清单, agent 调工具前 check 能力。

实现细节: 5 个常见 model + 1 个 fallback "unknown" (任意名字走 unknown, 提示"未知 model, 用默认能力, 可能不准")。 教学版不维护完整 model 列表 (5 个够用), 工业版接 litellm 等开源 registry。

主循环集成: 8 个模块在 Composition Root 拼装

用途: 8 个模块在 src/index.ts 的 Composition Root 拼装, 注入 agent。 拼装顺序很重要 — config → logger → terminal → session → llm-providers → system-prompt → context-ranking → cli-commands。 后者依赖前者的实例。

真实场景: Composition Root main() 流程: (1) loadConfig() 读配置; (2) createLogger() 输出启动日志; (3) detectTerminal() 决定颜色 / 交互; (4) createLLMClient() 按 provider 选 adapter; (5) createSession() 派生 session; (6) createSystemPrompt(config) 拼装 system message; (7) createContextRanking() 排序 history; (8) createCliCommandRegistry() 注册 9 个 /xxx 命令; (9) startREPL() 启动交互。

设计思想: 显式依赖链 — Reference 章节 "模式 5 · Composition Root 组合根" 的标准 应用。 main() 函数一气呵成, 没人藏 import, 改模块就改 main。

实现细节: 8 个模块的实例共享一次, 不重复创建 (logger 重复创建 → 日志写到 2 个文件; terminal 重复创建 → 颜色检测 2 次可能不一致; session 重复创建 → 隔离 失效)。 AGENTS.md "共享实例必须字面共享" 规则。

fake test: 用 fake env 验证 4 个不变量

用途: 8 个模块的测试用fake env + tmp dir + in-memory LLM client — 不真连 LLM, 不写真实日志文件, 不依赖真实 TTY。 测的是"模块 行为", 不是"外部依赖"。

真实场景: 用户写测试覆盖 4 条不变量: (a) log-rotation: 写 100MB 日志, 触发 2 次轮转, 总文件数 = 6 (1 个 current + 5 个 backup); (b) normalize: Anthropic 格式 消息 normalize 后变 OpenAI 格式; (c) config 优先级: CLI + env + default 三层, CLI 最高; (d) session 隔离: 2 个 session history 互不影响。

常见误解: hardening 不是新功能

误解 1: "hardening 是新功能?" 错。 hardening 是已有功能的安全网, 不增加业务能力, 只让已有 能力在生产里不出事。 没有它, harness 在开发 机能跑, 在生产跑挂。

误解 2: "只支持一个 LLM 就行, 不需要 normalize?" 错。 商业上不行 — 用户想换 provider 节省成本, 没 normalize 层就要改 agent 主循环 (几千行代码), 不可能。

误解 3: "config 默认值够用, 不需要 CLI / env?" 错。 CI 跑测试时 mode=auto, 日常 mode=default, 团队规范要 mode=plan 培训新成员。 3 种来源必须都能覆盖。

误解 4: "日志不重要, 撑爆就撑爆?" 错。 出问题 时日志是唯一复盘依据。 没日志 = 第二次事故 仍然无解。 轮转是必须的, 不是 nice-to-have。

反例梯度: 4 个常见错误

反例 1 · 日志不轮转

harness 跑 3 个月, harness.log 撑到 100GB, 磁盘满, 系统挂。 排查事故时发现 /var/log 已写满, 之前的日志被 系统 rotate 覆盖, 关键错误日志丢。 错。 正确: 50MB 轮转 + 5 个 backup, 总占用 ≤ 300MB, 关键日志保留 7 天。

反例 2 · 直接调 OpenAI SDK

agent.ts 直接 import OpenAI from "openai"; const client = new OpenAI(...);, 假设 OpenAI 协议。 换 Anthropic 时改 agent.ts 几千行。 错。 正确: 用 LLMClient 抽象接口 + 3 个 provider adapter, agent.ts 只调 llmClient.chat(messages, tools), 换 provider 只改 config。

反例 3 · config 不分优先级

config 一把抓, default 值 + env + CLI 全混在一起, Object.assign(defaults, envVars, cliArgs)。 用户设 CLI 期望覆盖, 实际 env 覆盖了 CLI (Object.assign 后写的赢)。 错。 正确: 显式优先级 (CLI > env > default), 内部按顺序 merge, 不打平, 保留 source 字段调试。

反例 4 · 多 session 共享 history

把 let history: Message[] = [] 放模块顶层, 多个 session 共享同一个数组。 user A 写的对话 user B 看到, 数据泄露 + 互相干扰。 错。 正确: createSession({id}) 每次返回独立实例, history 在闭包内, 互不污染。

Validation: 4 条不变量检验

  1. 日志轮转不丢内容: 写 50MB 日志, 触发轮转, 验证 总内容字节数 = 轮转前字节数 (rename 是原子的, 不丢)。
  2. normalize 对称: 任意 provider 格式 normalize 成 OpenAI 格式, 字段全部对应 (content 字符串 / tool_call 结构 / role 枚举), agent 主循环无差别处理。
  3. config 优先级确定: CLI + env + default 三层各 设不同值, 验证: 最终值 = CLI 值, source.mode = "cli"。
  4. session 隔离: 2 个 session 写各自的 history, 验证 互不影响 (s1.history.length === 1, s2.history.length === 0)。

回望: 哪些原则在本章兑现了

  • Adapter 模式: 3 个 LLM provider 各写 adapter, 内部统一 OpenAI 格式, agent 主循环无差别。
  • 显式优先级: config CLI > env > default, 保留 source 字段调试。
  • 工厂 + 闭包隔离: session / logger / terminal 每次创建独立实例, 闭包内 state 互不污染。
  • 分层拼装: system-prompt / context-ranking 按 层拼, 缺段跳过, 单独函数单独测试。
  • 安全默认值: log-rotation 50MB + 5 backup, terminal 默认 80 宽, config 默认 mode=default。

本章小结

Hardening 是给 harness 跑生产时的8 个安全网, 不增加业务能力, 只让已有能力在生产里不出事。 核心是 5 个原则:

  • Adapter 模式: 3 个 LLM provider + normalize 层, 内部统一格式, 换 provider 0 改动。
  • 显式优先级: config CLI > env > default, 保留 source 字段调试。
  • 工厂 + 闭包隔离: session / logger / terminal 每次独立, 多用户多项目不互污染。
  • 分层拼装: system-prompt / context-ranking 按 层拼, 缺段跳过, 单独函数单独测试。
  • 安全默认: log-rotation 50MB, terminal 80 宽, config mode=default, overlap=skip, missed=skip。
