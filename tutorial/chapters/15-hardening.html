第 15 章 · 长期运行的卫生系统

长期运行不会把系统跑坏：Runtime Hardening

教学项目也会长时间运行。日志、输出、occurrence、task archive 和 eval trace 如果无限增长，最后会把 agentHome 变成垃圾场。

你已经知道什么，还不知道什么

你已经有 Memory、Task、Async Run、Schedule、OutputStore 和日志。它们都可能落盘，也都可能长期增长。

读这一章时请把注意力放在 agent loop 的接入点：持久化边界和运行时维护层，确保长期运行后数据仍可读、可清理、可解释。代码片段只服务于理解架构边界，不要求你逐行背实现。

本章怎么学

先把自己放进一个真实 query：没有 Runtime Hardening 时，当前 agent 会在哪里卡住。

再故意设计一个朴素方案，观察它如何破坏 loop、状态或安全边界。这个失败比“正确答案”更重要，因为它能帮你写出更好的重建 prompt。

最后才看实现：先看图和数据流，再看少量接口、源码入口和验证方法。读完后，你应该能用自己的话描述 Runtime Hardening 的职责，而不是只记住文件名。

本章场景

用户连续几周使用 agent：每天 schedule 跑报告，后台任务输出测试日志，LLM 通信日志持续追加。某天启动变慢，磁盘空间变小，某个 JSON 还因为中断写坏了。

Runtime Hardening 不是炫技优化，而是避免系统在真实使用中慢慢腐烂：原子写、日志轮转、output handle、清理 dry-run、时间语义统一。

先试一个朴素方案

朴素方案是“先不管，等文件太多再 rm -rf logs”。这会让重要审计记录、用户长期记忆和可删除缓存混在一起。

这个朴素方案的问题在于：它把 Runtime Hardening 当成普通实现细节，而不是 agent harness 的一条稳定边界。 

// 危险：不知道哪些数据可删，哪些是长期事实。
rm -rf ~/.learn-claude-code-ts/*

朴素方案为什么不够

半截 JSON
写入中断会留下损坏文件，下次启动解析失败。
无限增长
日志、outputs、occurrences、eval traces 都会累积。
误删长期事实
清理策略如果不分类，可能删掉用户 Memory 或长期 Task。

回到 Agent Loop

Runtime Hardening 接入 loop 的位置是：持久化边界和运行时维护层，确保长期运行后数据仍可读、可清理、可解释。理解这个位置，比记住具体函数名更重要。

Hardening 不只在 agent loop 内，它覆盖所有读写边界：写文件时原子替换，写日志时轮转，暴露大输出时用 output_id，清理时先 dry-run。

架构图解

从原理落到代码

到这里我们已经知道 Runtime Hardening 为什么存在、插在 loop 的哪里，以及朴素方案会怎样失败。下面才进入实现层：只抓住能固定架构的接口、状态边界和接线路径。

读源码时不要从文件顶部一路啃到底。更有效的顺序是：先找入口函数，再找它读写的状态，最后看测试如何证明这些状态变化是真的发生了。

一次真实运行 walkthrough

  1. TaskStore 写 group.json 时先写同目录临时文件。
  2. 写完后校验 JSON，再 rename 覆盖正式文件。
  3. LLM logger 追加日志前检查大小，超过阈值执行轮转。
  4. 大工具输出先登记到 OutputStore，LLM 只拿到 output_id。
  5. 清理命令先 dry-run，列出将删除的日志、旧 output、旧 occurrence。
  6. 用户确认后按类别执行清理，Memory 和 active Task 默认不删。
  7. 时间字段使用 turnIndex/loopRound/messageSequence 等明确语义，避免 round 混用。

async function atomicWriteJson(path: string, value: unknown) {
  const temp = path + ".tmp";
  await writeFile(temp, JSON.stringify(value, null, 2));
  JSON.parse(await readFile(temp, "utf8"));
  await rename(temp, path);
}

设计直觉

鲁棒性设计的核心不是“多加 try/catch”，而是让每类数据都有生命周期：什么是事实，什么是派生索引，什么是缓存，什么可以清理。

清理机制必须保守，尤其不能默认删除 Memory、active Task 和仍有关联的 outputs。

架构拆分

写入安全
原子写、读写对称校验、id 一致性检查。
增长控制
日志轮转、output TTL、occurrence 保留策略。
可解释清理
dry-run 先告诉用户将删除什么，再执行。

状态与边界

长期事实
Memory、TaskGroup、Schedule 规则，默认不自动删除。
审计与缓存
logs、outputs、old occurrences、eval traces，可按策略清理。
时间语义
turnIndex、loopRound、loopIndex、messageSequence、transcript sequence 各有含义。

Prompt Card

Prompt Card

目标：为 agentHome 持久化数据补 Runtime Hardening：原子写、日志轮转、output handle 和清理 dry-run。

场景：agent 长期运行后，数据增长仍可控，坏文件不会破坏启动，清理不会误删长期事实。

模块：atomic-write.ts、log-rotation.ts、output-store.ts、timeline.ts、相关 store 和 tests。

接线：持久化 store 使用 atomic write；logger 使用 rotation；清理命令按类别扫描。

边界：不要默认删除 Memory/active Task；不要把裸路径暴露给 LLM；清理先 dry-run。

例外和 corner case：写入中断、坏 JSON、日志超限、output_id 越权读取、旧 occurrence 清理。

验证：测试原子写、日志轮转、output_id 读取边界和清理计划。

源码入口：src/atomic-write.ts、src/log-rotation.ts、src/output-store.ts、src/timeline.ts。

文档同步：说明鲁棒性是长期运行卫生系统，不是生产化大改。

容易踩坑

Design Trap

常见错误：清理时一刀切删除 agentHome。

为什么错：agentHome 同时包含长期事实、审计日志和缓存，生命周期完全不同。

正确做法：按类别和 TTL 生成 dry-run 计划，默认保护长期事实。

怎么验证：dry-run 输出应明确分类，且不包含 Memory 和 active Task。

  • 原子写临时文件跨目录，rename 不再原子。
  • output path 直接给 LLM。
  • round 同时表示 turn 和 loop，导致压缩年龄判断漂移。

如何验证

Validation

手工 query：制造大日志和旧 output，运行 dry-run，检查计划是否可解释。

确定性测试：测试坏 JSON 不污染正式文件，日志超限后轮转。

集成观察：OutputStore 只能通过 output_id 读取登记输出。

失败信号：清理计划包含长期 Memory，或坏 JSON 让启动崩溃。

  • src/atomic-write.test.ts 验证原子写。
  • src/llm-logger.test.ts 验证轮转。
  • src/output-store.test.ts 验证 output_id。

如果实现失败，先查哪里

  • 先分类数据目录。
  • 检查 store 读写是否对称。
  • 检查清理计划是否 dry-run。
  • 检查时间字段是否语义明确。
