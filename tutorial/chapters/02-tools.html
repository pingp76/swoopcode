第 02 章 · 让模型提出动作

给 Agent 一双手：工具调用

最小 Agent Loop 只能聊天，不能读文件、跑测试或修改代码。这一章把 tool call 接进 loop：模型只提出结构化动作请求，真正执行动作的是 harness。

你已经知道什么，还不知道什么

你已经有了 agent.run(query)、History 和 LLM Client。现在要面对第一个 coding agent 的分水岭：模型不能直接碰你的文件系统，它只能通过工具协议说“我想读这个文件”或“我想运行这个命令”。

本章不会追求完整工具生态，只实现最小的 bash / read / write / edit / editExact 思路。重点是理解 tool schema、tool registry、tool call 和 tool result 如何进入同一个 loop。

本章怎么学

先从本章场景和朴素方案进入问题，不急着记函数名。

最后再看接口、源码地图和验证方法，把概念压缩成能交给 coding agent 的 Prompt Card。

本章场景

用户输入：“帮我看看 src/agent.ts 里主循环怎么写。”如果 agent 只是把这句话交给 LLM，模型只能猜。真正的 coding agent 必须先读取文件，再基于观察结果回答。

于是 LLM 返回的不是最终答案，而是一个结构化 tool call：调用 run_read，参数是目标路径。Harness 检查权限、执行工具、把 tool result 写回 History，然后再把新 messages 发给 LLM。

先试一个朴素方案

最朴素的想法是让模型输出一段 shell 命令，然后本地直接执行。比如模型说“cat src/agent.ts”，harness 就调用 shell。这个方案很诱人，因为看起来不需要 schema、registry 和参数校验。

但这相当于把你的终端交给了模型。模型生成的文本可能包含危险命令，也可能因为格式不稳定导致解析失败。更糟的是，工具结果没有结构化写回，下一轮模型不知道刚才到底执行了什么。 

const answer = await llm.chat(messages);
const command = extractShell(answer.content);
const output = await exec(command);
messages.push({ role: "user", content: output });

朴素方案为什么不够

安全边界消失
自然语言和 shell 字符串混在一起，harness 很难稳定判断哪些命令可执行、哪些路径越界、哪些操作需要用户确认。
模型看不到稳定能力菜单
没有 tool schema 时，模型只能猜工具名字和参数格式。一次叫 read_file，下一次叫 cat_file，tool registry 无法可靠匹配。
tool_call / tool_result 无法配对
LLM 协议要求工具调用和工具结果按顺序成对出现。把输出伪装成普通 user message，会让后续 normalize、compress、replay 都失去事实边界。

回到 Agent Loop

工具调用插在 “LLM 返回后” 和 “下一次 LLM 调用前”。Agent 先把 assistant message 写入 History；如果其中包含 tool calls，就逐个查 registry、做权限检查、执行工具，再把 tool results 写回 History。

这一步完成后，loop 不结束。Agent 会带着新观察结果再次调用 LLM，让模型基于真实文件内容或命令输出生成下一步动作或最终回答。

架构图解

从原理落到代码

到这里已经有了场景、失败原因和架构图。下面才进入实现层：只抓住接口形状、状态边界和源码入口。

读源码时先找入口函数，再找它读写的状态，最后看测试如何证明这些状态变化是真的发生了。

一次真实运行 walkthrough

  1. 用户要求解释某个源码文件，REPL 把 query 交给 agent.run()。
  2. Agent 写入 user message，并把当前 messages 与工具定义一起发给 LLM。
  3. LLM 返回 assistant message，其中包含 run_read 的 tool call。
  4. Agent 先把 assistant message 追加到 History，保留模型提出动作的事实。
  5. Agent 根据工具名从 ToolRegistry 找到实现，并把参数交给对应工具。
  6. 工具在 projectRoot 边界内读取文件，返回 ToolResult。
  7. Agent 把 tool result 追加到 History，形成 tool_call / tool_result 配对。
  8. 下一轮 LLM 看到文件内容后，才能给出基于事实的解释。

关键接口和伪码

最关键的接口不是某个具体工具，而是“模型可见 schema”和“harness 内部执行函数”的分离。schema 稳定给模型看，handler 稳定给 Agent 调用。 

interface ToolDefinition {
  name: string;
  description: string;
  parameters: JsonSchema;
}

interface ToolHandler {
  (args: unknown): Promise<ToolResult>;
}

interface ToolRegistry {
  definitions(): ToolDefinition[];
  execute(name: string, args: unknown): Promise<ToolResult>;
}

while (true) {
  const assistant = await llm.chat(messages, registry.definitions());
  history.add(assistant);
  if (!assistant.toolCalls?.length) return assistant.content;

  for (const call of assistant.toolCalls) {
    const result = await registry.execute(call.name, call.arguments);
    history.addToolResult(call.id, result);
  }
}

设计直觉

Tool Registry 是 agent 的“手册柜台”：模型只知道菜单，Agent 只通过柜台执行动作。这个边界让工具名、参数 schema、权限检查和执行结果都能被测试。

工具结果不是给用户看的日志，而是下一轮 LLM 的观察材料。它必须进入 History，且必须以 tool result 的身份进入，否则后续压缩、回放和错误恢复都会失去结构。

架构拆分

tools/*
每个工具模块只关心自己的参数校验、业务执行和 ToolResult 输出。
ToolRegistry
集中维护工具名、schema、handler 和过滤规则，保证模型看到的工具顺序稳定。
Agent Loop
负责识别 assistant tool calls、执行工具、把 tool results 写回 History 并继续下一轮。

状态与边界

进入 LLM messages
工具定义、assistant tool_call、tool_result 的可见内容。
不直接暴露给 LLM
本地绝对路径、内部 handler、权限实现细节和未登记的大输出文件。
session-local
本轮工具调用顺序和临时执行结果；后续 Transcript 会记录审计事实。

Prompt Card

Prompt Card

目标：实现最小工具调用闭环，让 agent 能通过 run_bash/run_read/run_write/run_edit/run_edit_exact 观察和修改项目。

场景：用户要求读取文件或运行测试时，模型必须先提出结构化 tool call，再由 harness 执行并写回 tool result。

模块：新增 tools/registry.ts、tools/types.ts、tools/bash.ts、tools/files.ts，并扩展 agent.ts 的 tool call 分支。

接线：在 composition root 创建一次 ToolRegistry，把工具定义传给 LLM，把 registry 注入 Agent。

边界：不要绕过 registry 执行模型生成的自然语言命令；文件工具必须限制在 projectRoot。

例外和 corner case：未知工具、非法参数、路径越界、工具抛错、大输出都要变成可恢复 ToolResult。

验证：用 fake LLM 触发 tool call，断言 tool_call/tool_result 顺序、路径拒绝和工具不存在错误。

源码入口：src/agent.ts、src/tools/registry.ts、src/tools/files.ts、src/tools/bash.ts。

文档同步：更新 summary 和教程章节，说明工具是 harness 动作，不是模型自己执行。

容易踩坑

Design Trap

常见错误：把模型输出的 shell 文本直接执行。

为什么错：自然语言不是稳定协议，无法可靠做权限、参数、审计和回放。

正确做法：模型只能返回结构化 tool call；所有执行都经过 registry 和权限层。

怎么验证：测试未知工具和路径越界时，Agent 应返回 tool_result 错误而不是崩溃或执行危险命令。

  • 工具名随意变化会破坏模型习惯，也会让测试难以稳定。
  • tool_result 不能插入到错误位置，否则后续消息标准化会被迫修复甚至无法修复。
  • 文件工具必须先 normalize 路径再检查边界，不能只做字符串前缀判断。

如何验证

Validation

手工 query：让 agent 读取一个存在文件、读取一个不存在文件、尝试读取 ../AGENTS.md 之外路径。

确定性测试：fake LLM 返回固定 tool call，测试 Agent 是否执行 registry 并继续下一轮。

集成观察：运行 files/bash/registry/agent 相关测试，观察工具错误是否回到 LLM 可见消息。

失败信号：模型直接输出命令文本却被执行，或 tool_result 没有对应 tool_call。

  • src/tools/files.test.ts 覆盖 read/write/edit/editExact 的成功和拒绝路径。
  • src/tools/bash.test.ts 覆盖危险命令和命令输出。
  • src/agent.test.ts 覆盖 tool call 进入 loop 后的消息顺序。

如果实现失败，先查哪里

  • 先看 LLM 返回的是不是结构化 tool call，而不是普通文本。
  • 再看 registry 是否注册了同名工具，名称大小写是否一致。
  • 检查 tool result 是否带着正确 call id 写回 History。
  • 路径问题先看 projectRoot 和 normalize 后的绝对路径。

本章小结

本章让 agent 第一次拥有“手”。但这双手还没有工作节奏：遇到一个多步骤任务时，它可能知道能读写文件，却不知道如何持续维护计划。
