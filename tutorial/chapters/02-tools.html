第 02 章 · 给 Agent 一双手

工具调用: 让 LLM 真的能改世界

第 01 章的 loop 跑通了两轮对话保留上下文, 但 LLM 还是只输出文本 — 它没法真的改文件、跑命令。 这一章给 loop 加上一双"手": 工具调用。 读完后, 你能区分 3 种工具调用协议 (Anthropic / OpenAI / Google), 知道 ToolResult 怎么设计, 并能用 fake tool 写一个"先读 README 再回答" 的最小集成测试。

为什么需要工具调用

写代码之前, 先讲为什么 LLM 必须能调工具, 不用会怎样。

LLM 本质是个"下一个 token 预测器"。 给它一段对话, 它能续写; 但让它"真的去看 README 里写了什么" 或者"真的跑 ls" , 它做不到 — 模型没有文件系统、没有 shell、没有网络。 即使你用 "系统提示" 告诉它"假装你能 ls", 它只能瞎编。

工具调用就是让 LLM 输出一个结构化指令 (tool_call), 由 harness 执行, 再把结果 (tool_result) 喂回 LLM。

这条"输出指令 → 执行 → 喂结果" 的循环, 让 LLM 第一次有了"改世界" 的能力。 也是 coding agent 和普通 chatbot 的分水岭。 没有工具 调用的 agent, 只是 chat 套壳; 有工具调用的 agent, 才开始谈得上 "完成用户任务"。

本章要解决的 3 个真实问题:

  1. 协议对齐: 3 家 LLM (Anthropic / OpenAI / Google) 的 tool call 协议不同, harness 怎么选一个统一内部表示?
  2. 工具结果怎么表达: 工具执行完返回什么给 LLM? 字符串 够不够, 还是要支持图片 / 文件?
  3. 工具多了怎么管: 1 个工具写死, 5 个工具还能写死, 30 个工具不写个注册表就乱。

3 种工具调用协议

朴素想法: "OpenAI / Anthropic / Google 都有 tool call, 应该一样吧?" 实际上 3 家协议有 3 个真实差异, adapter 必须收口。 这不是"语法不同" 那么轻, 整个 messages 序列结构都不同。

维度 Anthropic OpenAI Google
工具调用字段 tool_use 块 tool_calls 数组 functionCall 块
content 形状 数组 (text + tool_use) 字符串 + tool_calls 并列 数组 (text + functionCall)
工具结果回传 user message + tool_result 块 tool message + tool_call_id user message + functionResponse 块

字段名不同只是表面, 真正难的是history 序列不同:

  • Anthropic: assistant 输出 tool_use, 下一条必须是 user 消息里的 tool_result 块
  • OpenAI: assistant 输出 tool_calls, 下一条必须是 role:"tool" 的消息, 用 tool_call_id 配对
  • Google: assistant 输出 functionCall, 下一条必须是 user 消息里的 functionResponse 块

注意区别: Anthropic 和 Google 都把 tool result 放在 user message 里 (用 content block 类型区分), OpenAI 单独有一个 role: "tool"。 选哪个? OpenAI 的最简单 (独立 role, 不用 content block 类型判断), 我们就用 OpenAI 风格的统一 history 形状, 未来接 Anthropic / Google 时在 adapter 层翻译 (专题 A 展开)。

OpenAI 风格的 message 形状

选 OpenAI 风格后, harness 内部有 3 种 message role: 

// 1. 用户输入
{ role: "user", content: "What's the test command?" }

// 2. assistant 调工具 (一条消息, content 可能为 null)
{
  role: "assistant",
  content: null,
  tool_calls: [
    {
      id: "call_abc123",                      // 配对 ID
      type: "function",
      function: {
        name: "run_read",
        arguments: '{"path": "package.json"}',  // 字符串, 不是对象
      },
    },
  ],
}

// 3. 工具结果 (role: "tool", 用 tool_call_id 配对上面的调用)
{
  role: "tool",
  tool_call_id: "call_abc123",
  content: "{ \"scripts\": { \"test\": \"npm test\" } }",
}

GitHub · src/llm-adapter.ts OpenAI 风格 message 形状 (L1)

3 个细节决定后面写代码时少踩坑:

  1. assistant 消息的 content 可以是 null: 调工具时 模型可能"只调不解释", content 就是 null。 LLM SDK 接受 content: null, 不用填空字符串。
  2. tool_calls[].function.arguments 是字符串: 字符串, 不是对象。 调 executor 前要 JSON.parse。 模型有时候会给出非法 JSON (比如忘了引号), 解析失败要写 error tool message, 不能让 loop 崩。
  3. tool message 必须配对 tool_call_id: 多工具并发 时 (run_bash 和 run_read 同时调), 每条 tool message 必须用对应的 id 配对, 不然模型不知道 哪个结果对应哪个调用。

朴素反例: 工具逻辑写死在 agent.ts

写 agent 最朴素的做法: 把所有工具的 if/else 塞进 agent.ts。 5 个工具能撑, 10 个工具开始肿, 30 个工具变成大泥球。 

// ❌ 反例: 5 个工具 5 个 if
async function handleToolCall(call) {
  if (call.name === "run_read") {
    return await fs.readFile(call.args.path, "utf8");
  }
  if (call.name === "run_write") {
    await fs.writeFile(call.args.path, call.args.content);
    return "OK";
  }
  if (call.name === "run_bash") {
    const { stdout } = await exec(call.args.command);
    return stdout;
  }
  if (call.name === "run_edit") {
    // ... 50 行编辑逻辑
  }
  if (call.name === "run_todo_create") {
    // ... 10 行 todo 逻辑
  }
  // 新加 run_web_fetch 又要加 if
}

5 件事立刻坏掉:

  1. agent.ts 变成大泥球: 5 个工具 100+ 行, 10 个工具 200+ 行。
  2. 测试没法隔离: 想测"agent 调 run_read 后能不能回答", 必须真的创建文件, 跑 read。 不能用 fake tool。
  3. 权限难加: 想在 run_write 前问用户, 只能改 agent.ts, 工具作者根本不知道有权限层。
  4. 元信息丢失: 工具返回的 metadata (执行耗时、字节数) 没地方放。
  5. 工具作者互相耦合: 改 run_edit 的实现可能影响 run_bash 的逻辑。

解决方式: ToolRegistry 注册表, 把"调用哪个工具" 和 "工具怎么实现" 隔开。

ToolResult 接口 — 工具返回什么

用途: 工具执行完, harness 要把结果喂回 LLM。 这个结果 怎么表达, 决定了 LLM 能不能正确解读。 我们用统一的 ToolResult 接口。

真实场景: 工具作者写 run_bash("ls"), 返回 "README.md\nsrc\n", 这个字符串 LLM 能直接看; 工具作者写 run_web_fetch("https://..."), 返回 HTML + 图片, 纯字符串放不下, 需要 attachments。 统一接口覆盖两种情况。

设计思想: 4 个字段刚好覆盖所有情况 — content 必有 (LLM 看的文本), attachments 可选 (多模态), error 标记业务错误 (区别 throw), metadata 内部信息 (不进 LLM, 只进 transcript)。 不多不少, 多一个字段就多一个"会不会被滥用" 的地方。

实现细节: 

interface ToolResult {
  // 文本结果, 大多数工具用它
  content: string;
  // 可选: 输出图片 / 文件 / 结构化数据
  attachments?: Array<{
    kind: "image" | "file" | "json";
    path?: string;
    data?: string;       // base64 或 JSON 字符串
    mimeType?: string;
  }>;
  // 错误标记: 工具正常完成但没拿到结果 (vs throw 是 harness 错误)
  error?: boolean;
  // 工具内部补充信息 (不进 LLM messages, 只进 transcript)
  metadata?: Record<string, unknown>;
}

GitHub · src/tools/types.ts ToolResult 接口 (L1)

4 个字段的边界:

  1. content 必有: LLM 看到的"工具说了什么"。 即使工具有 attachments, 也要有一个 content 描述 "我找到了这张图, 在 attachments 里"。
  2. attachments 可选: 工具产物, 比如 run_bash 写出的截图, run_read 读到的图片。 LLM 收到时知道这是附件, 不是普通文本。 当前多数工具链 只用 content, attachments 是为多模态工具 (第 09 章 memory 多模态) 留的扩展点。
  3. error: true ≠ throw: 工具正常执行完, 但返回了"命令不存在" 这种业务错误, 标 error: true。 harness 抛 throw 是 harness 内部错误 (权限拒绝、工具超时), 不是工具错误。 两类错误要分开处理 — 见下一节。
  4. metadata 不进 LLM: 工具的内部信息 (执行耗时、字节数), 只进 transcript 调试, 不进 messages。 避免 LLM 被内部细节污染 — LLM 不需要知道"我跑了 200ms", 只需要"命令输出是 X"。

工具失败的 3 种模式

用途: 工具调用的"失败" 有 3 种, 处理方式完全不同。 这是新同学最容易搞混的地方, 必须单独讲。

真实场景: 跑一个 agent 调 3 个工具: run_read 成功, run_bash 因为命令不存在失败, run_write 因为权限被拒未执行。 3 个失败, 3 种处理, 但 harness 必须一致地 把失败信息喂回 LLM, 不能假装成功。

模式 含义 示例 处理
业务错误 工具正常完成, 但结果表示"做错了" run_bash("cat nonexistent") 返回 exit code 1 result.error = true, 写 tool message, LLM 自己决定
权限拒绝 工具被 permission 拦了 用户拒绝 run_write 写 tool message "Permission denied", 不执行, LLM 知道
异常抛出 harness 内部错误, 工具没正常返回 工具超时 / OOM / panic recovery 决定 (第 11 章), loop 不该崩

❌ / ✅: 把业务错误当 throw。 

// ❌ 把业务错误当 throw
async function runRead(args) {
  const content = await fs.readFile(args.path, "utf8");
  if (!content) throw new Error("empty file");   // LLM 看不到这个, messages 断裂
  return { content };
}

// ✅ 业务错误用 error: true
async function runRead(args) {
  try {
    const content = await fs.readFile(args.path, "utf8");
    return { content };
  } catch (e) {
    return { content: `Error: ${e.message}`, error: true };
  }
}

GitHub · src/tools/run-read.ts 业务错误 vs throw 模板 (L1)

业务错误用 error: true, LLM 收到后能继续推理: "啊, 文件不存在, 我换个路径试试"。 异常抛出时, LLM 看不到任何信息, messages 序列还可能断, 整个 loop 崩。

为什么 throw 不行? 假设 LLM 调了 2 个工具: run_read 和 run_bash。 run_read 抛 throw, run_bash 还没执行。 messages 序列里:

  1. user (原 query)
  2. assistant (调 run_read + run_bash)
  3. 工具: throw 退出, 没有 tool message

下一次 chat() 时, LLM 看到一个 assistant 调了 2 个工具, 但只 收到 0 个 tool message, 模型困惑。 messages 序列不完整, OpenAI API 甚至可能直接报 400 错误。

Tool Registry — 工具怎么注册

用途: 工具数量从 1 涨到 5 之后, 把它们写死在 agent.ts 里就难维护。 引入 ToolRegistry 注册表, 把"注册 / 查询 / 调用" 分离。

真实场景: harness 跑久了, 工具数从 5 涨到 30+。 注册表让 工具作者写 register(name, def, executor) 一行就接入, 不需要改 agent.ts 主循环。

设计思想: 经典注册表模式 — 把"按名字查找 对象" 抽成一个数据结构, 外界通过 register() 和 get() 交互。 翻译 (tool def → LLM ChatCompletionTool) 藏 registry 内部, 业务模块不感知。 详细模式解释见 Reference 章节的"模式 5 · 注册表模式"。

实现细节: 

interface ToolRegistry {
  register(name: string, def: ToolDefinition, executor: ToolExecutor): void;
  get(name: string): ToolDefinition | undefined;
  list(): ToolDefinition[];
  getToolDefinitions(): ChatCompletionTool[];   // 转成 OpenAI 格式
  invoke(name: string, args: unknown): Promise<ToolResult>;
}

type ToolDefinition = {
  name: string;
  description: string;       // 给 LLM 看, 决定何时调
  parameters: JSONSchema;     // JSON Schema 描述参数
};

type ToolExecutor = (args: any, ctx: ToolContext) => Promise<ToolResult>;

GitHub · src/tools/registry.ts ToolRegistry 完整实现 (L1)

Registry 的 4 个职责:

  1. 注册: 工具通过 register(name, def, executor) 接入。
  2. 查询: getToolDefinitions() 转成 OpenAI ChatCompletionTool[], 喂给 LLM。
  3. 执行: invoke(name, args) 找到 executor, 调用, 返回 ToolResult。
  4. 权限检查 (第 07 章展开): invoke 之前 调 permissionManager, 拒绝时返回 "Permission denied" 而不执行。

为什么 registry 不直接暴露给 LLM? 因为 LLM 看到的"工具列表" 必须稳定, 不能按 mode 动态增减 — 会破坏 prompt cache (第 10 章)。 工具的"激活 / 停用" 由 Skill 系统管, 不在 registry。

loop 接入: 工具调用是第 5 步

用途: 把第 01 章的 6 步 loop 补上第 5 步, 让 LLM 调的工具真的 被执行。 这是 loop 从"聊天" 升级为"agent" 的关键一步。

真实场景: LLM 返回 tool_calls: [{ name: "run_read", args: {path: "package.json"} }], harness 找到 run_read 的 executor, 调它, 拿 ToolResult, 写回 history, 回到 loop 顶部让 LLM 看到结果。

设计思想: 5 个关键细节决定"工具调用是不是真稳":

  1. 工具参数是 JSON 字符串: call.function.arguments 是字符串, 调 invoke 前 JSON.parse。 解析失败要 写 error tool message, 不能 throw 让 loop 崩。
  2. 每个 tool_call 都写一条 tool message: 即使工具 抛错, 也要写 role: "tool" 消息告诉 LLM "这个调用失败了" 。 不写会让 messages 序列断裂, LLM 收到错位消息。
  3. catch 包到 invoke 内部: 不让 throw 逃出 loop。 业务错误用 error: true 表达, harness 错误用 recovery (第 11 章) 处理。
  4. load_skill 特殊处理: 第 05 章展开 — load_skill 激活 skill 而不执行, 激活后写 tool message。
  5. 工具是并行还是串行? 第一版用串行, 简单可调试。 第 13 章再讲并行 / 异步。

实现细节: 

export function createAgent(deps: { llm: LLMClient; history: History; tools: ToolRegistry }): Agent {
  return {
    async run(query: string): Promise<string> {
      deps.history.add({ role: "user", content: query });

      for (;;) {
        const messages = deps.history.getMessages();
        const toolDefs = deps.tools.getToolDefinitions();
        const response = await deps.llm.chat({ messages, tools: toolDefs });
        deps.history.add({
          role: "assistant",
          content: response.content,
          tool_calls: response.toolCalls,
        });
        if (response.toolCalls.length === 0) {
          return response.content ?? "";
        }

        // 5. 执行工具调用
        for (const call of response.toolCalls) {
          let result: ToolResult;
          try {
            const args = JSON.parse(call.function.arguments);
            result = await deps.tools.invoke(call.function.name, args);
          } catch (e) {
            // JSON 解析失败也算业务错误, 不让 loop 崩
            result = { content: `Tool error: ${e.message}`, error: true };
          }
          deps.history.add({
            role: "tool",
            tool_call_id: call.id,
            content: result.content,
          });
        }
        // 6. 回到 loop 顶部
      }
    },
  };
}

tool description 怎么写 LLM 才用对

用途: 工具被 LLM 调对调错, 80% 取决于 description 怎么写。 这一节总结几个常见 pattern, 帮工具作者写好 description。

test("agent.run 调 run_read 后能基于结果回答", async () => {
  const fakeTools = createToolRegistry();
  fakeTools.register(
    "run_read",
    { name: "run_read", description: "Read a file", parameters: {} },

带工具的 loop 数据流

反例梯度

新手错法 · A

常见错误: 工具执行失败直接 throw。

为什么错: throw 会让 agent.run 整个崩, messages 序列断裂, LLM 收不到任何信息。

正确做法: 业务错误用 { content: "Error: ...", error: true } 返回, LLM 看到错误后能继续推理。

中级错法 · B

常见错误: 工具调成功不写 role: "tool" 消息。

为什么错: messages 序列断裂, 下一次 LLM 收到的 assistant 消息没有对应 tool_result, 模型困惑。

正确做法: 每次 tool_call 必须配对一条 tool message, 不管 成功失败。

高级错法 · C

常见错误: tool description 写"这是一个工具"。

为什么错: LLM 看了不知道何时调, 永远不调。

正确做法: 写"读取文件内容, 需要时用 这个工具, 不要用 cat 命令代替"。

边界错法 · D

常见错误: 工具参数 schema 用 TypeScript 类型, 不写 JSON Schema。

为什么错: LLM 看不到类型, 模型给出来的参数不符合期望。

正确做法: 用 JSON Schema 描述, harness 在 invoke 前做一次 校验, 不通过返回 error。

如何验证 (本章 Validation 卡片)

Validation · 第 02 章

调一次工具再回答: fake tool 返回固定内容, fake LLM 第一轮 调工具, 第二轮基于工具结果回答, 最终回复匹配 fake tool 的内容。

tool message 必写: 工具调成功 / 业务失败 / 权限拒绝, history 末尾都有 role: "tool" 消息, content 描述结果。

参数 JSON 解析失败不崩: LLM 给出非法 JSON args, tool executor 收到解析错误返回, history 写 error tool message, loop 不抛。

tool description 含使用场景: 每个工具 description 至少含 一句"何时用", 而不是空泛的"工具" 二字。

回望: 哪些原则在本章兑现了

  • 依赖注入: ToolRegistry 注入 agent, 不在内部 new。
  • 工厂模式: 工具通过 register(name, def, executor) 注册, registry 是单例, 但 executor 闭包内捕获上下文。
  • 窄接口: ToolResult 只暴露 4 个字段, 工具元信息不外泄。
  • fake 测试: 不依赖真实文件系统, 用 fake tool 验证 loop。

Prompt Card (本章任务)

Prompt Card · 第 02 章

目标: 给 agent loop 加上工具调用能力, LLM 输出 tool_call, harness 执行, 把 tool_result 写回 history。

场景: 用户问"这个项目用什么 test 命令?", agent 调 run_read("package.json"), 看到结果, 回答 "npm test"。

模块: src/tools/types.ts (新) 暴露 ToolResult 接口; src/tools/registry.ts (新) 暴露 createToolRegistry(); src/tools/run-read.ts (新) 真实 run_read 工具; src/agent.ts (改) 接收 ToolRegistry, 第 5 步执行工具; src/index.ts (改) 注册核心工具。

边界 (LLM 必须遵守的 checklist):

  • ToolResult 只暴露 content / attachments / error / metadata 4 个字段, 工具元信息不外泄
  • 工具参数是 JSON Schema, 不是 TypeScript 类型
  • 业务错误用 error: true 返回, 不 throw
  • 每次 tool_call 必须配对一条 role: "tool" 消息, 不管成功失败
  • tool description 必须含"何时用" 的场景描述, 不能空泛

验证 (用 fake LLM + fake tool, 逐条落到 vitest):

  • fake LLM 第一轮调 run_read, fake tool 返回固定内容, 第二轮 LLM 收到 tool_result, 最终回复基于工具内容
  • history 累积 user + assistant(tool_call) + tool + assistant, 不断裂
  • 业务错误 (fake tool 返回 error: true) 不抛, LLM 继续推理
  • 非法 JSON args 返回 error tool message, loop 不崩

本章练习

  1. 故意让 tool executor throw new Error(), 跑测试, 看 "业务错误不崩" 是否抓到 (loop 崩, history 断裂)。
  2. 让 tool executor 调成功但不写 tool message, 跑测试, 看 "tool message 必写" 是否抓到 (下次 chat() 收到错位消息)。
  3. 写一个 tool description = "这是一个工具", 跑测试, 看 LLM 是否会调它 (不会调, 描述太抽象)。

本章小结

这一章给 loop 加上了工具调用。 核心是 3 个设计选择:

  • OpenAI 风格的统一 history 形状: 内部一律用 tool_calls + role: "tool", 未来接 Anthropic / Google 时在 adapter 层翻译。
  • ToolResult 4 字段: content / attachments / error / metadata, 业务错误用 error: true, 不用 throw。
  • Tool Registry: 工具注册后转成 JSON Schema 给 LLM, LLM 调的工具名 + 参数 + 结果都进 history。

    • 下一章 (第 03 章) 给 agent 加一个"短期记忆" — TODO Manager, 让多轮执行有节奏。
